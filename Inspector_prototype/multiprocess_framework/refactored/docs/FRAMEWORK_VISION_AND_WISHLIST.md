## 1. Как можно представить «самый мощный» каркас

**Цель:** сложность живёт в библиотеке; приложение собирается как декларативный конструктор (процессы, каналы, данные, UI-слоты), а не как копипаста очередей и бесконечных `while True`.

### Вдохновение (не только Python)

- **Erlang/OTP, Akka:** супервизоры, деревья процессов, «let it crash», явные контракты сообщений, перезапуск по политике.
- **ROS2 / идея DDS:** темы, сервисы, QoS (надёжность, дедлайны), discovery — удобно для пайплайнов «сенсор → обработка → визуализация».
- **gRPC / Cap’n Proto:** строгие контракты и эволюция схем без «всё dict и надеемся».
- **Game engines:** мир = системы, которые читают/пишут компоненты; UI — отдельный клиент к тому же состоянию (хорошо стыкуется с регистрами + командами).
- **Electron / Tauri:** главный процесс vs renderer — аналог «оркестратор + изолированные воркеры», UI тонкий.

### Архитектура «мечты» (слоями)

**A. Модель приложения (DSL / декларация)**  
Одна структура: какие **процессы**, какие **транспорты** (очередь, SHM, сокет), какие **схемы сообщений** и **shared state**. Запуск: `app = AppSpec(...); app.run()`. Под капотом — генерация `proc_dict`, пайпов, политик рестарта.

**B. Сообщения и состояние — раздельно**

- **Команды / RPC / события** — короткий жизненный цикл, идемпотентность где нужно, `correlation_id`, таймауты, отмена.
- **Состояние (регистры, снапшоты)** — diff, подписки, ограничение частоты синхронизации, при необходимости разруливание конфликтов.

Идея **«два механизма: register vs command — норма»** хорошо совпадает с такой моделью.

**C. Транспорт как плагин**  
Сегодня `multiprocessing.Queue` + SHM, завтра — ZMQ/NANOMSG, zero-copy контракты или отдельный процесс-мост. Приложение опирается на **capabilities** (задержка, размер сообщения, broadcast), а не на детали реализации.

**D. Наблюдаемость из коробки**  
Трейс одного `command_id` через процессы, метрики очередей, backpressure, единый контекст (`trace_id`, `process`, `worker`). Логирование — событие первого класса.

**E. PyQt как адаптер, не как сердце**  
UI-процесс: подписка на **проекции состояния** + отправка **намерений** (команды). Тяжёлая логика не застревает в слотах. Идеал: привязка регистр ↔ виджет через один тонкий слой (в сторону `FrontendRegistersBridge` + единого sender команд).

**F. Тестирование**  
In-process **симуляция** топологии без лишних форков, детерминированные очереди, property-based тесты сериализации на границе.

**G. Эргономика «конструктора»**  
Минимум кода для нового процесса: явные точки входа, автогенерация или единый каталог команд из типов/реестра. Направление `process(Config)` + схемы уже близко к этому.

## 2. Что уже есть в Multiprocess Framework и прототипе

| Идея «мечты» | Текущая реализация (по докам и `multiprocess_prototype`) |
|--------------|------------------------------------------------------------|
| Спрятать хаос `multiprocessing` | **16 модулей**, единый жизненный цикл, `BaseManager`, graceful shutdown |
| Единый язык IPC | **`Message` + типы**, `MessageAdapter`, Dict at Boundary |
| Типизация внутри, простота на границе | **Pydantic / `SchemaBase`**, `process()` → `proc_dict` для лаунчера |
| Масштабирование процессов | **`SystemLauncher`**, явное `add_process` |
| Состояние между процессами | **Регистры**, `RegistersManager`, схемы в приложении |
| PyQt не в ядре | **`frontend_module`** + `GuiProcess` / `FrontendLauncher` в прототипе |
| Реальный домен (CV) | **SHM для кадров**, очереди для управления |
| Зрелость мышления про UI | **[FRONTEND_COMMAND_LAUNCHER_ROADMAP.md](./FRONTEND_COMMAND_LAUNCHER_ROADMAP.md):** один канонический путь outbound COMMAND, ограничители сложности, ADR |

Итог: направление совпадает с «идеалом»; расхождение в основном в **степени автоматизации, наблюдаемости и сменяемости транспорта**, а не в базовой архитектурной идее.

## 3. Где расхождение сильнее всего

1. **«Конструктор» vs объём знаний**  
   У идеального фреймворка низкий порог после короткого введения. Здесь документация глубокая и правильная, но **вход высокий**: много модулей, два пути (registers / commands), чеклисты. Для «массового» API не хватает **одного вертикального среза** «от нуля до трёх процессов + одна кнопка» на минимуме кода без чтения всего каталога.

2. **Граница dict — сила и ограничение**  
   Dict at Boundary упрощает pickle и эволюцию; «мечта» часто тянет к **версионируемым схемам на wire** и codegen. Компромисс: оставить dict, но усилить **контракты** (версия сообщения, обязательные ключи, миграции на приёме).

3. **Супервизия и деградация**  
   Сильный акцент на порядке shutdown; в «мечте» дополнительно **политики рестарта процесса**, изоляция сбоя воркера, circuit breaker на очередях. Для Python это тяжело, но **hooks** на «процесс умер» / «очередь переполнена» приблизили бы каркас к OTP-подобному мышлению.

4. **Транспорт зафиксирован практикой**  
   Прототип честно заточен под **SHM + очереди**. Универсальная библиотека имела бы **абстракцию канала** с метриками и подменой в тестах без мока половины процесса.

5. **Frontend как продукт**  
   Закрытие roadmap (`RoutedCommandSender`, отсутствие дублей handler/mixin) — маркер зрелости. Когда **ARCHITECTURE.md** прототипа и код полностью совпадают по одному пути GUI→очередь, разрыв с «идеалом» по UI сузится.

## 4. Черновик структуры документа «Пожелания / советы»

Глобальный порядок работ и волны приближения к идеалу: **[FRAMEWORK_IDEAL_GLOBAL_PLAN.md](./FRAMEWORK_IDEAL_GLOBAL_PLAN.md)**. Ниже — исходный список тем (детализация по мере реализации переносится в план и ADR).

1. **Целевой пользователь фреймворка** — внутренняя команда / open-source / оба (от этого зависит агрессивность упрощения публичного API).
2. **Vertical slice «минимальное приложение»** — учебный пример в репозитории (не Inspector): 3 процесса, без камеры.
3. **Контракты на границе** — опциональная `schema_version` в `Message`, политика неизвестных полей.
4. **Наблюдаемость** — единый `correlation_id` / `trace_id` в COMMAND и в логах; одна страница «как отладить путь кнопки».
5. **Политики отказов** — поведение лаунчера при падении дочернего процесса (restart N раз, затем останов всей системы).
6. **Тестовый режим** — `FakeRouter` / in-process topology для pytest без DISPLAY и без лишних форков где возможно.
7. **Публичная поверхность** — что экспортировать из `interfaces.py` vs внутреннее; правило для приложений «не импортировать core напрямую».
8. **Синхронизация roadmap и кода** — чеклист: нет второго пути для GUI command без ADR (см. FRONTEND_COMMAND_LAUNCHER_ROADMAP).

## 5. Краткий вывод

Текущая реализация и документация попадают в **верхний сегмент** того, что обычно делают в Python под multiprocess + Qt: явные слои, Dict at Boundary, регистры, оркестратор, отдельный frontend-модуль и осознанная дорожная карта по GUI-командам.

«Идеальный вселенский» фреймворк отличался бы в первую очередь **эргономикой входа, политиками жизни/смерти процессов, сменяемым транспортом и жёсткой наблюдаемостью**; **базовая архитектурная идея уже совпадает** с описанным здесь видением.

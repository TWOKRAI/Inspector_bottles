## Основная проблема

### Почему Python multiprocessing сложен?

В сыром виде, `multiprocessing` — это боль:

```python
# ❌ Проблемы сырого multiprocessing
def worker(queue_in, queue_out, logger_queue, error_queue, config, stop_flag, pause_flag, ...):
    """30+ параметров, трудно отслеживать"""
    while True:
        try:
            msg = queue_in.get(timeout=1)
            result = process(msg)
            queue_out.put(result)
        except Exception as e:
            error_queue.put(str(e))  # как логировать?
            logger_queue.put(f"ERROR: {e}")  # структура?
        finally:
            # cleanup что? когда?
```

**Пять главных проблем:**

1. **Нет единого протокола** — каждый процесс использует свои очереди и соглашения
2. **Нет структуры логирования** — логи идут в разные места, без контекста
3. **Нет управления жизненным циклом** — как gracefully остановить все?
4. **Нет типизации** — что передавать между процессами? Пиклировать что?
5. **Нет масштабируемости** — добавить процесс = дублировать 200 строк кода

### Решение: Архитектурный фреймворк

**Multiprocess Framework** — это **набор 16 модулей**, которые создают **единый архитектурный каркас** для решения всех пяти проблем:

✅ **Единый протокол сообщений** (Message с 9 типами)  
✅ **Единая система логирования** (через ObservableMixin)  
✅ **Управление жизненным циклом** (graceful shutdown, signal handling)  
✅ **Типизация через Pydantic** (SchemaBase, FieldMeta)  
✅ **Масштабируемость** (наследование от BaseManager)

## Два взаимодополняющих взгляда

Архитектура фреймворка может быть понята двумя способами, которые **дополняют** друг друга:

### 1️⃣ Взгляд через иерархию управления (Тройца создания циклов)

```
ProcessManagerCore (Сверхэго) — Мозг системы
    ↓ создает и управляет
ProcessModule (Эго) — Организм процесса
    ↓ использует
WorkerManager (Ид) — Мышцы потоков
```

**Применение:** Когда нужно понять управление и контроль.

### 2️⃣ Взгляд через биологическую аналогию (Живой организм)

```
Организм (живой объект)
    ├─ Мозг (ProcessManager) — управление
    ├─ Нервная система (RouterManager) — коммуникация
    ├─ Органы (менеджеры) — функции
    ├─ Мышцы (WorkerManager) — выполнение
    ├─ Память (LoggerManager) — логирование
    ├─ Архив (SharedResources) — хранилище
    └─ ДНК (DataSchema) — конфигурация
```

**Применение:** Когда нужно интуитивно понять, как всё работает вместе.

**Оба взгляда описывают ОДНУ архитектуру с разных углов зрения.**

## Тройца создания циклов

### Философская идея

В психоанализе Фрейда выделяются три уровня психики:

- **Сверхэго** (Superego) — высший уровень контроля, нормы, мораль
- **Эго** (Ego) — средний уровень, реальность, выполнение
- **Ид** (Id) — низший уровень, инстинкты, энергия

Эта же триада лежит в основе нашей архитектуры:

### 1️⃣ ProcessManagerCore (Сверхэго) — Высший уровень контроля

**Роль:** Управляет всей системой на уровне ОС.

**Аналогия:**
- В психике: нормы и правила поведения
- В организме: мозг, который принимает стратегические решения
- В системе: точка входа, которая создает и управляет процессами

**Ответственность:**
- Создание и уничтожение процессов ОС
- Мониторинг состояния всех процессов
- Координация между процессами
- Управление жизненным циклом системы
- Обработка сигналов (SIGTERM, SIGINT)

**Характеристика:** Контролирует, но не выполняет сама. Определяет правила, которым подчиняются нижние уровни.

### 2️⃣ ProcessModule (Эго) — Средний уровень реальности

**Роль:** Выполняет реальную работу в одном процессе.

**Аналогия:**
- В психике: рациональное мышление, выполнение действий
- В организме: каждый орган, который выполняет функцию
- В системе: каждый запущенный процесс, который работает

**Ответственность:**
- Жизненный цикл процесса (initialize/shutdown)
- Управление менеджерами процесса (logger, command, router, worker)
- Обработка входящих команд и данных
- Внутренняя координация работы

**Характеристика:** Выполняет работу, следуя правилам сверхэго, используя инстинктивную энергию ида.

**Важный момент:** ProcessModule **повинуется** указаниям ProcessManagerCore, но **независим** в том, как выполняет работу.

### 3️⃣ WorkerManager (Ид) — Низший уровень энергии

**Роль:** Управляет потоками выполнения внутри процесса.

**Аналогия:**
- В психике: инстинкты и первичные импульсы
- В организме: мышцы, которые выполняют движения
- В системе: потоки, которые циклически выполняют работу

**Ответственность:**
- Создание и управление потоками-воркерами
- Приоритеты выполнения (REALTIME, NORMAL, BATCH, BACKGROUND)
- Зависимости между потоками
- Автоматический перезапуск при ошибках

**Характеристика:** Слепая энергия. Просто выполняет приказы, циклически, не задумываясь.

### Взаимодействие трёх уровней

```
ProcessManagerCore (Сверхэго)
    ↓ "Создавай процесс VisionProcess"
    │ (управление, контроль)
    ↓
ProcessModule (Эго)
    │ Инициализирует себя
    │ Создаёт менеджеры
    ↓ "Начни обрабатывать видео"
    │ (выполнение, координация)
    ↓
WorkerManager (Ид)
    │ Создаёт потоки
    ├→ vision_capture_worker (REALTIME)
    ├→ frame_processor_worker (NORMAL)
    └→ result_publisher_worker (NORMAL)
    ↓ Циклическое выполнение
```

**Ключевая идея:** Каждый уровень имеет свою ответственность, и вместе они создают **целостную систему**, где нет противоречий.

## Аналогия с живым организмом

Система функционирует как живой организм:

### 🧠 Мозг — ProcessManagerCore

**Функция:** Принимает стратегические решения, управляет всеми органами.

**Действия:**
- Решает, какие процессы нужны системе
- Запускает их при необходимости
- Останавливает их по сигналам

### 🫀 Сердце процесса — ProcessModule

**Функция:** Каждый процесс — это независимая "сущность", которая живет своей жизнью.

**Действия:**
- Инициализируется при запуске
- Работает независимо
- Отвечает на команды
- Останавливается при сигнале

### 🫀 Нервная система — RouterManager

**Функция:** Связывает все компоненты через сообщения (как нейротрансмиттеры).

**Действия:**
- Маршрутизирует сообщения между процессами
- Доставляет команды
- Собирает результаты

### 🩸 Кровь — Message (Protocol)

**Функция:** Единая форма, в которой передаются данные между компонентами.

**Действия:**
- 9 типов сообщений (command, response, event, data, etc.)
- Dict at Boundary (безопасная передача между процессами)
- correlation_id для отслеживания запросов

### 💪 Мышцы — WorkerManager

**Функция:** Выполняет циклическую работу.

**Действия:**
- Потоки выполняют задачи
- Приоритеты определяют срочность
- Автоматический перезапуск при ошибках

### 🧬 ДНК — DataSchema

**Функция:** Описывает структуру данных каждого компонента.

**Действия:**
- Определяет, какие поля есть у каждого процесса
- Валидирует данные через Pydantic
- Позволяет другим процессам узнать структуру

### 📚 Память — LoggerManager

**Функция:** Запоминает всё, что происходит.

**Действия:**
- Логирует события
- Собирает метрики
- Позволяет отследить, что произошло

### 🗂️ Архив — SharedResources

**Функция:** Хранилище, где хранятся все данные системы.

**Действия:**
- Регистр состояния каждого процесса
- Реестр очередей
- Хранилище конфигурации

## Архитектурные слои

### Слой 1: Фундамент (Foundation)

```
BaseManager + ObservableMixin ← все наследуют эту базу
    ├─ base_manager
    ├─ data_schema_module
    └─ message_module
```

**Роль:** Предоставить общие механизмы для всех остальных модулей.

**Зачем:**
- **BaseManager** — единый жизненный цикл для всех компонентов
- **ObservableMixin** — единая система логирования
- **Message** — единый протокол коммуникации

**Философия:** "Explicit is Better Than Implicit" — все зависимости явные.

### Слой 2: Инфраструктура (Infrastructure)

```
logger_module
error_module
config_module
console_module
shared_resources_module
channel_routing_module ← новая база для трёх менеджеров
registers_module
```

**Роль:** Предоставить утилиты и сервисы.

**Философия:** "Separation of Concerns" — каждый модуль делает одно хорошо.

### Слой 3: Коммуникация (Communication)

```
router_module ← отправка/получение
dispatch_module ← маршрутизация
command_module ← обработка команд
```

**Роль:** Обеспечить коммуникацию между процессами.

**Философия:** "Message-Driven" — всё работает через сообщения.

### Слой 4: Процесс (Process)

```
worker_module ← управление потоками
process_module ← базовый процесс
```

**Роль:** Управление потоками и процессами.

**Философия:** "Graceful Everything" — graceful инициализация, завершение, деградация.

### Слой 5: Оркестрация (Orchestration)

```
process_manager_module ← запуск всей системы
```

**Роль:** Точка входа, которая запускает всю систему.

**Философия:** "Bootstrap" — простая точка входа для сложной системы.

## Фундаментальные принципы

### 1. Explicit is Better Than Implicit

**Принцип:**
- Все зависимости передаются явно через конструктор
- Нет скрытых глобалов
- Нет магических импортов

**Пример:**
```python
# ✅ Хорошо: явные зависимости
manager = RouterManager(
    name="router",
    logger=logger_manager,  # явно!
    config=config_manager,   # явно!
)

# ❌ Плохо: скрытая зависимость
manager = RouterManager(name="router")
# где manager.logger? откуда оно взялось?
```

**Применение:** Когда нейросеть анализирует код, она сразу видит все зависимости.

### 2. Separation of Concerns

**Принцип:**
- Каждый модуль отвечает за одно
- Нет смешивания ответственности
- Легко заменить один модуль на другой

**Пример:**
```python
# ✅ Хорошо: каждый модуль отвечает за своё
- message_module: как передавать данные
- router_module: куда отправлять
- command_module: что делать с данными
- logger_module: как логировать

# ❌ Плохо: всё в одном модуле
- mega_module: Message + Router + Logger + everything
```

**Применение:** Модули можно разрабатывать и тестировать независимо.

### 3. Graceful Everything

**Принцип:**
- Graceful инициализация (try-except-retry)
- Graceful завершение (signal handling, timeout cascades)
- Graceful деградация (fallback strategies)
- Graceful error handling (не падать на первой ошибке)

**Пример:**
```python
# Graceful shutdown: дать каждому процессу время
Ctrl+C / SIGTERM
    ↓
ProcessRegistry.stop_all(timeout=5s)
    ├─ Set stop_event for each process
    ├─ Wait for graceful exit (up to 5s)
    ├─ If hanging: SIGTERM
    ├─ If still hanging: SIGKILL
    └─ Guarantee: all dead in 5s
```

**Применение:** Система не падает на ошибки, а продолжает работу.

### 4. Type Safety

**Принцип:**
- Все данные типизированы через Pydantic
- Все сообщения имеют известный тип
- Все команды валидированы

**Пример:**
```python
# ✅ Типизированное сообщение
msg = MessageFactory.command(
    targets=["detector"],
    command="detect",  # type: str
    args={"frame_id": 1}  # type: dict
)

# ❌ Без типизации
put_in_queue({"some": "data"})  # что это?
```

**Применение:** Нейросеть может автоматически проверить типы.

### 5. Modularity

**Принцип:**
- 16 независимых модулей
- Каждый может быть развернут отдельно
- Каждый имеет свои тесты

**Пример:**
```python
# Можно тестировать отдельно
- test_router_module.py
- test_logger_module.py
- test_process_module.py

# Можно интегрировать выборочно
app = SystemLauncher(processes=[...])
```

**Применение:** Не нужно запускать всю систему для тестирования одного модуля.

### 6. Dict at Boundary

**Принцип:**
- На границах процессов только dict (pickle-safe)
- Внутри процесса можно использовать Pydantic

**Пример:**
```python
# ✅ На границе: dict
msg.to_dict()  # {"type": "command", "targets": [...]}
queue.put(msg.to_dict())

# ❌ На границе: Pydantic
queue.put(msg)  # может не сериализоваться!

# ✅ Внутри процесса: Pydantic
msg = Message.from_dict(raw)  # восстанавливаем объект
```

**Применение:** Гарантирует, что данные передаются корректно между процессами.

## Жизненный цикл системы

### Инициализация (Startup)

```
1. SystemLauncher.__init__()
   ↓ создаёт конфиг
2. SystemLauncher.run()
   ↓ запускает manager
3. ProcessManagerCore.initialize()
   ├─ Инициализирует платформу
   ├─ Инициализирует SharedResources
   ├─ Инициализирует ConfigManager
   └─ Готово к запуску процессов
4. ProcessManagerCore.create_process()
   ├─ Создаёт ProcessData в SharedResources
   ├─ Регистрирует очереди в QueueRegistry
   └─ Создаёт процесс ОС (fork/spawn)
5. ProcessModule.initialize() (в дочернем процессе)
   ├─ reinitialize_in_child() — переинициализирует многопроцессные структуры
   ├─ Инициализирует менеджеры (logger, router, command, worker)
   └─ Запускает системные потоки
```

### Выполнение (Runtime)

```
ProcessModule.run()
    ├─ WorkerManager.start_all_workers()
    │  └─ Потоки начинают циклическую работу
    │
    ├─ Основной цикл процесса
    │  ├─ Получает сообщения через RouterManager
    │  ├─ Обрабатывает команды через CommandManager
    │  └─ Отправляет результаты
    │
    └─ Ожидает stop_event
```

### Завершение (Shutdown)

```
Ctrl+C / SIGTERM
    ↓
ProcessManagerCore.signal_handler()
    ↓
ProcessRegistry.stop_all(timeout=5s)
    ├─ Для каждого процесса:
    │  ├─ Set stop_event
    │  ├─ Wait for graceful exit (up to 5s)
    │  ├─ If hanging: send SIGTERM
    │  └─ If still hanging: send SIGKILL
    │
    └─ All processes dead
    ↓
Flush logs, close connections
    ↓
Exit code 0
```

## Преимущества и философия

### Преимущество 1: Модульность

**Что даёт:**
- Каждый модуль можно развивать независимо
- Каждый модуль можно тестировать отдельно
- Легко добавлять новые модули

**Философия:** "Don't Repeat Yourself" (DRY) — код не дублируется.

### Преимущество 2: Type Safety

**Что даёт:**
- IDE может подсказывать методы
- Нейросеть может проверить типы
- Меньше runtime ошибок

**Философия:** "Fail Fast" — ошибки видны сразу, а не в боевых условиях.

### Преимущество 3: Observable & Traceable

**Что даёт:**
- Всё логируется через единую систему
- Можно отследить путь сообщения через всю систему
- Есть корреляция между логами разных процессов

**Философия:** "Observability" — видеть состояние системы в любой момент.

### Преимущество 4: Graceful & Resilient

**Что даёт:**
- Система не падает на первой ошибке
- Graceful shutdown гарантирует целостность данных
- Автоматический перезапуск потоков при ошибках

**Философия:** "Resilience" — система работает при ошибках.

### Преимущество 5: Масштабируемость

**Что даёт:**
- Легко добавить новый процесс (просто наследовать от ProcessModule)
- Не нужно менять код других процессов
- Система автоматически её найдёт и включит

**Философия:** "Open/Closed Principle" — открыто для расширения, закрыто для изменения.

## Когда использовать этот фреймворк

### ✅ Идеально подходит для

- **Видеообработка** (camera → detector → tracker → UI)
- **IoT-приложения** (sensors → processing → storage)
- **Микросервисы на одной машине** (lightweight service architecture)
- **Мониторинг систем** (multi-source data collection)
- **Приложения реального времени** (low-latency requirements)

### ❌ Не подходит для

- Простые скрипты (используйте raw `multiprocessing`)
- Распределённые системы (используйте Docker + Kubernetes)
- Async-only код (используйте `asyncio`)
- Машинное обучение (используйте `torch.distributed` или Ray)

## Резюме: Философия в одном абзаце

**Multiprocess Framework** построен на философии **"живой организм"**, где каждый компонент имеет чёткую роль и взаимодействует с другими через единый протокол. Система иерархична (Мозг → Организм → Мышцы), явна в зависимостях, типобезопасна, и выглядит как естественная расширение архитектуры самой операционной системы. Каждый процесс — это независимая сущность, которая следует правилам фреймворка, но имеет свободу в том, как выполнять работу.

**Ключевая идея:** Если ты понимаешь, как работает живой организм, ты поймёшь, как работает этот фреймворк. Если ты можешь анализировать и улучшать живой организм, ты можешь анализировать и улучшать эту систему.

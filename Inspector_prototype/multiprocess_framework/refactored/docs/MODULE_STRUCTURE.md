# Структура модулей - Стандарт

## Общая структура модуля

Все модули должны следовать единой структуре для консистентности и понятности:

```
module_name/
├── __init__.py              # Публичный API модуля (экспорт только публичных классов/функций)
├── interfaces.py            # Интерфейсы модуля (если нужны)
├── core/                    # Основные классы модуля
│   ├── __init__.py
│   └── main_class.py        # Основной публичный класс
├── validators/              # Валидаторы (внутренние)
│   ├── __init__.py
│   └── validator.py
├── converters/              # Конвертеры (внутренние)
│   ├── __init__.py
│   └── converter.py
├── factories/               # Фабрики (публичные)
│   ├── __init__.py
│   └── factory.py
├── types/                   # Типы, константы, схемы, исключения
│   ├── __init__.py
│   ├── enums.py             # Перечисления
│   ├── constants.py         # Константы
│   ├── schemas.py           # Схемы данных
│   └── exceptions.py        # Исключения
├── utils.py                 # Внутренние утилиты (если нужны)
├── README.md                # Документация модуля
└── tests/                   # Тесты модуля
    ├── __init__.py
    ├── test_core.py          # Тесты основных классов
    └── conftest.py           # Конфигурация pytest
```

## Принципы организации

### 1. Понятные папки по функциональности

- **core/** - основные классы модуля (публичные)
- **validators/** - валидаторы (внутренние, но в отдельных файлах)
- **converters/** - конвертеры (внутренние)
- **factories/** - фабрики (публичные)
- **types/** - типы, константы, исключения (публичные)
- **utils.py** - утилиты (внутренние, если простые)

### 2. Разделение публичного и внутреннего внутри файлов

Внутри файлов методы разделяются комментариями:

```python
class Message:
    """
    Публичный класс.
    
    Разделение методов:
    - Публичные методы (используются извне) - без префикса _
    - Внутренние методы (используются только внутри класса) - с префиксом _
    """
    
    # ========================================================================
    # ПУБЛИЧНЫЙ API
    # ========================================================================
    
    def create(cls, ...):
        """Публичный метод."""
        pass
    
    def validate(self):
        """Публичный метод."""
        pass
    
    # ========================================================================
    # ВНУТРЕННИЕ МЕТОДЫ (не используются извне)
    # ========================================================================
    
    def _sync_to_dict(self):
        """Внутренний метод."""
        pass
```

### 3. Единственная ответственность

Каждый файл должен иметь одну четкую ответственность:
- `core/main_class.py` - основной класс модуля
- `factories/factory.py` - фабрики для создания объектов
- `validators/validator.py` - валидация
- `converters/converter.py` - конвертация
- `utils.py` - утилиты (если простые)

### 4. Публичный API через __init__.py

Модуль экспортирует только публичные классы и функции:

```python
# module_name/__init__.py
from .core import MainClass
from .factories import Factory, create_object
from .types import TypeEnum, ExceptionClass

__all__ = [
    'MainClass',
    'Factory',
    'create_object',
    'TypeEnum',
    'ExceptionClass',
]
```

### 5. Внутренние классы не экспортируются

Внутренние классы используются только внутри модуля:

```python
# Внутри модуля
from .validators.validator import Validator

# Извне модуля - НЕ ДОСТУПНО
# from module_name.validators.validator import Validator  # НЕ РЕКОМЕНДУЕТСЯ
```

## Пример: Message Module

```
message/
├── __init__.py              # Экспорт: Message, MessageType, create_message
├── interfaces.py            # IMessage, IMessageFactory (если нужны)
├── core/                    # Основные классы
│   ├── __init__.py
│   └── message.py           # Класс Message (публичные и внутренние методы разделены комментариями)
├── validators/              # Валидаторы (внутренние)
│   ├── __init__.py
│   └── message_validator.py # MessageValidator (внутренний класс)
├── converters/             # Конвертеры (внутренние)
│   ├── __init__.py
│   └── message_converter.py # MessageConverter (внутренний класс)
├── factories/               # Фабрики (публичные)
│   ├── __init__.py
│   └── message_factory.py  # MessageFactory, create_message, parse_message
├── types/                   # Типы и константы
│   ├── __init__.py
│   ├── message_types.py    # MessageType, Priority, LogLevel, MessageSchema
│   └── exceptions.py       # MessageValidationError
├── utils.py                 # Утилиты (generate_message_id, apply_type_defaults)
├── README.md
└── tests/
    └── test_message.py      # Тесты Message
```

## Правила именования

### Публичные классы и функции
- PascalCase для классов: `Message`, `MessageFactory`
- snake_case для функций: `create_message`, `parse_message`
- Явные имена, отражающие назначение

### Внутренние классы и функции
- Префикс `_` для приватных методов класса
- Имена должны отражать внутреннее назначение
- Внутренние классы в отдельных папках (validators/, converters/)

### Файлы
- snake_case: `message.py`, `message_types.py`
- Описательные имена: `message_validator.py`, `message_converter.py`

## Документация

### README.md модуля должен содержать:
1. Описание модуля и его роли в архитектуре
2. Публичный API (только то, что экспортируется из `__init__.py`)
3. Примеры использования
4. Интеграция с другими модулями
5. Структура модуля (какие папки за что отвечают)

### Docstrings
- Все публичные классы и методы должны иметь docstrings
- Внутренние классы могут иметь краткие docstrings
- Используйте русский язык для docstrings
- Разделяйте публичные и внутренние методы комментариями в коде

## Тестирование

### Структура тестов
- `test_core.py` - тесты основных классов (обязательно)
- `test_factories.py` - тесты фабрик (если есть)
- `conftest.py` - фикстуры pytest

### Покрытие
- Публичный API должен быть покрыт тестами на 100%
- Внутренние классы - по необходимости (сложная логика)

## Миграция существующих модулей

При рефакторинге существующих модулей:

1. Создать структуру папок (`core/`, `validators/`, `converters/`, `factories/`, `types/`)
2. Разделить код по функциональности
3. Вынести типы и константы в `types/`
4. Разделить публичные и внутренние методы комментариями в файлах
5. Обновить `__init__.py` для экспорта только публичного API
6. Обновить тесты под новую структуру
7. Обновить документацию

## Преимущества структуры

1. **Понятная организация** - папки по функциональности
2. **Легкая навигация** - структура предсказуема
3. **Разделение ответственности** - каждый файл отвечает за одну вещь
4. **Тестируемость** - легко тестировать публичный API
5. **Расширяемость** - легко добавлять новые компоненты
6. **Консистентность** - все модули следуют одному стандарту

# Упрощённая архитектура: предложение и решения

Документ учитывает уточнения по review и задаёт один источник истины, простые схемы, DNA как hot-reload, карту маршрутизации и рецепты через VersionManager.


---

## 1. Схемы — один раз в одном месте

**Проблема:** Дублирование — регистры как Pydantic-модели и отдельно в SchemaRegistry.

**Решение: схемы только в data_schema, регистры их используют.**

- **Единственное место определения схем:** классы моделей регистров (DrawRegisters, CameraRegisters и т.д.) объявляются один раз.
- **SchemaRegistry** хранит только ссылки на эти классы (по имени регистра или имени схемы). Никакого дублирования: регистр = схема, схема = класс Pydantic.
- **registers_module** не дублирует схемы. Он либо:
  - получает классы из конфига/списка при инициализации и сам создаёт экземпляры, либо
  - запрашивает у SchemaRegistry экземпляр по имени схемы, если регистры зарегистрированы там.
- **Валидация рецептов:** рецепт — это dict (снимок регистров). Валидируем через ту же схему: либо составная схема «Recipe» из зарегистрированных классов регистров, либо пошаговая валидация каждого блока по имени регистра через SchemaRegistry.

**Итог:** Один класс = одна схема. SchemaRegistry хранит классы; RegistersManager работает с экземплярами (созданными из этих классов или переданными). Удобно и без дублирования.

## 2. DNA — создание и hot-reload

**Назначение DNA:**

- Описание компонента для **создания** (модуль, класс, ресурсы, параметры).
- **Редактирование в реальном времени:** поменяли конфиг в файле → нажали «Обновить» в приложении → компонент/процесс пересоздаётся без перезапуска приложения.

**Use cases:**

1. Старт: загрузка конфига → создание процессов/компонентов по DNA.
2. Hot-reload: перечитали конфиг → сравнили с текущим DNA → пересоздали изменённые компоненты/процессы.
3. Редактирование свойств: изменение полей DNA (например параметров создания) и применение без перезапуска.

**Где хранить:** ProcessData + StorageManager (как сейчас). ProcessDataContainer для ДНК компонентов. Менеджер конфигураций читает/пишет через data_schema (валидация, конвертеры).

**Итог:** DNA = конфиг для создания и пересоздания; один инструментарий — data_schema; сценарий «меняем файл → обновить → пересоздание» закладывается в архитектуру.

## 3. Регистры и карта маршрутизации

**Идея:**

- У каждого процесса **свой набор регистров** (свои модели и экземпляры).
- **Главный менеджер процессов** перед созданием процессов:
  - импортирует все модули со списками/моделями регистров (из App и из описаний процессов),
  - строит **карту маршрутизации:** register + field → (process, channel) или (router, channel),
  - передаёт эту карту каждому процессу при создании (или после) и держит её в своём RouterManager.
- Сообщение в канал: **команда** (записать, считать и т.д.) + **ключ** (register/field или составной) + **значение**.
- Роутер процесса, получив сообщение, решает, что делать (записать в локальные/глобальные переменные по регистрам, вызвать обработчик и т.д.).

**Один источник истины по маршрутам:** карта строится один раз из всех регистров и передаётся в роутер; процессы не дублируют логику построения карты, только используют её.

**Реализация:**

- В главном процессе: сбор всех классов/модулей регистров → обход полей с `routing` → построение словаря `(register, field) -> (router, channel)` или `(process_id, channel)`.
- Роутер главного процесса по сообщению `{ command, register, field, value }` определяет целевой процесс и канал и отправляет туда.
- В процессе: свой RouterManager, свой RegistersManager (локальные переменные/экземпляры); обработчик сообщений по команде обновляет регистры или отвечает чтением.

## 4. Один источник истины по данным

**Разделение ответственности:**

| Данные | Где хранятся | Кто отвечает |
|--------|--------------|--------------|
| Конфиг процесса, ДНК, конфиги менеджеров | ProcessData (через StorageManager) | data_schema_module (StorageManager, SchemaRegistry, конвертеры) |
| Текущие значения регистров процесса | Память процесса (экземпляры в RegistersManager) или ProcessData.custom["registers"] при необходимости персиста | registers_module (RegistersManager) |
| Рецепты (снимки регистров) | Файл Data/Recipes/value_settings.yaml + при необходимости версии в VersionManager | RecipeManager + data_schema (валидация, VersionManager) |

**Один источник истины:**

- **ProcessData** — единый контейнер состояния процесса (конфиг, менеджеры, ДНК, при желании снимок регистров).
- **StorageManager** — единственная точка записи/чтения для конфигов и менеджеров в ProcessData.
- **RegistersManager** — единственная точка доступа к значениям регистров процесса (память или чтение/запись в ProcessData.custom["registers"], если решите хранить там).
- Рецепты: единственный формат снимка = dict из RegistersManager.model_dump_all(); единственное место хранения файлов — Data/Recipes; версии — только через VersionManager (см. ниже).

Таким образом не два источника истины, а один: ProcessData + файлы рецептов; доступ только через фасады (StorageManager, RegistersManager, RecipeManager).

## 5. Ошибки роутинга и будущий ErrorManager

**Сценарии:**

- Роутер не нашёл маршрут для register/field.
- Процесс-получатель недоступен.
- Сообщение не доставлено (таймаут, очередь переполнена).

**Решение:** все такие случаи обрабатываются в роутере/обработчике и **передаются в менеджер ошибок (ErrorManager)**, который будет реализован позже.

**Контракт для ErrorManager (зафиксировать в архитектуре):**

- Роутер/обработчик при ошибке вызывает что-то вроде: `error_manager.report(type, context)`.
- `type`: например `ROUTING_NOT_FOUND`, `PROCESS_UNREACHABLE`, `MESSAGE_LOST`.
- `context`: dict (register, field, process_id, channel, message_id, exception и т.д.).
- ErrorManager пока может логировать и складывать в очередь/буфер; позже — UI, алерты, повторные попытки.

В документе архитектуры указать: «Ошибки роутинга и доставки сообщений обрабатываются и передаются в ErrorManager (интерфейс и реализация — отдельно)».

## 6. Рецепты через VersionManager

**Идея:** использовать тот же принцип и по возможности тот же формат, что и VersionManager в data_schema.

**Как сейчас устроен VersionManager:** внутри хранит `data` как dict (model_dump()), ключ — (manager_type, manager_name). Версии лежат в ProcessData.custom[VERSIONS_KEY] или можно вынести в файл.

**Предложение для рецептов:**

1. **Сохранение рецепта:**
   - RegistersManager.model_dump_all() → snapshot (dict).
   - Опционально: валидация snapshot по схемам регистров (SchemaRegistry).
   - Запись в YAML (Data/Recipes/value_settings.yaml) в текущем формате приложения.
   - Если нужна история: вызывать **тот же VersionManager** с обобщённым методом, например `save_document(scope="recipe", name=recipe_id, data=snapshot)` (или использовать существующий ключ как manager_type="Recipe", manager_name=recipe_id, если добавить в VersionManager приём dict вместо только BaseManagerModel).

2. **Расширение VersionManager (минимальное):**
   - Добавить метод `save_document(scope, name, data: dict, ...)` / `get_document(scope, name, version)`.
   - Внутри — та же структура версий (VersionInfo с data), тот же лимит версий. Хранение: либо в ProcessData.custom для scope="recipe", либо в отдельном файле/каталоге для рецептов (например Data/Recipes/versions/).
   - Тогда рецепты и конфиги менеджеров используют один механизм версий и один формат хранения версий.

3. **Загрузка рецепта:**
   - Чтение из YAML (или get_document для нужной версии).
   - Валидация по схемам регистров.
   - RegistersManager.model_validate_all(snapshot).

**Итог:** рецепты используют общие инструменты data_schema (схемы для валидации, конвертеры при необходимости) и VersionManager для истории — без дублирования логики версий.

---

## 7. Краткий flow сохранения рецепта

```
UI (Sort_widget)
  → RecipeManager.save_recipe(recipe_id, params?)
  → RegistersManager.model_dump_all()  → snapshot (dict)
  → SchemaRegistry.validate(snapshot) или поэлементная валидация по регистрам
  → запись в Data/Recipes/value_settings.yaml (текущий формат)
  → при необходимости: VersionManager.save_document("recipe", recipe_id, snapshot)
  → при ошибках валидации/записи → ErrorManager.report(...)
```

Загрузка — обратный порядок: чтение YAML или get_document → валидация → model_validate_all.

## 8. Что зафиксировать в основном документе архитектуры

Рекомендуется обновить ARCHITECTURE_REGISTERS_ROUTER.md и ARCHITECTURE_REVIEW.md так:

1. **Схемы:** Схемы регистров живут в одном месте (классы Pydantic); SchemaRegistry хранит ссылки на них; RegistersManager не дублирует схемы, а использует эти классы или реестр. Валидация рецептов — через те же схемы.

2. **DNA:** Назначение — создание и пересоздание компонентов/процессов; сценарий hot-reload (изменение конфига → обновить → пересоздание). Хранение и валидация — data_schema (StorageManager, ProcessDataContainer).

3. **Карта маршрутизации:** Строится в главном менеджере процессов при старте (импорт всех регистров, обход routing) и передаётся в роутер; сообщения = команда + ключ + значение; ошибки роутинга/доставки → ErrorManager (контракт описан, реализация позже).

4. **Один источник истины:** ProcessData — контейнер состояния процесса; StorageManager — для конфигов/ДНК/менеджеров; RegistersManager — для регистров; рецепты — файл + при необходимости VersionManager.

5. **Рецепты и VersionManager:** Сохранение/загрузка рецепта по flow выше; при необходимости истории — расширить VersionManager методом save_document/get_document для произвольного dict и использовать для рецептов тот же принцип и формат.

6. **Ошибки:** Обработка ошибок роутинга и доставки и передача в ErrorManager зафиксированы в архитектуре; детали API ErrorManager — в отдельном месте.

## 9. Оценка после упрощения

- **Схемы:** 9/10 — один раз в одном месте, без дублирования, удобно.
- **DNA:** 9/10 — ясный use case (создание + hot-reload), один инструментарий.
- **Регистры и карта:** 8/10 — понятная модель; остаётся аккуратно реализовать сбор регистров из всех процессов и построение карты.
- **Один источник истины:** 9/10 — чёткое разделение: ProcessData + фасады.
- **Ошибки:** 8/10 — контракт под ErrorManager снимает неопределённость.
- **Рецепты + VersionManager:** 8/10 — один механизм версий; нужно только добавить save_document/get_document (или эквивалент) и решить, где хранить версии рецептов (ProcessData или файлы).

**Общая оценка упрощённого варианта: 8.5/10** — рационально, реализуемо, без лишних сущностей.

## 10. Реализация (refactored + App)

- **Схемы:** Классы регистров (App) регистрируются в SchemaRegistry по имени регистра (`register_app_schemas()` в `App/Registers/schema_bridge.py`). Валидация рецепта — `SchemaRegistry.validate_recipe(snapshot)`.
- **DNA:** ProcessDataContainer и StorageManager (ДНК в ProcessData.custom). Hot-reload — через перечитывание конфига и пересоздание компонентов.
- **Карта маршрутизации:** `build_routing_map(registers)` в `registers_module/routing_map.py`; отправка — `send_register_message(router, routing_map, register, field, value, error_callback=error_manager.report)`.
- **ErrorManager:** Контракт `report(error_type, context)` реализован в `App/Managers/error_manager.py`; типы `ROUTING_NOT_FOUND`, `PROCESS_UNREACHABLE`, `MESSAGE_LOST` в `registers_module.routing_map`.
- **Рецепты:** `RecipeManager.save_recipe()` / `load_recipe()` — model_dump_all → validate_recipe → YAML → опционально `VersionManager.save_document("recipe", id, snapshot)`; загрузка — YAML/get_document → validate → model_validate_all. Ошибки — `ErrorManager.report(VALIDATION_ERROR, RECIPE_SAVE_ERROR, RECIPE_LOAD_ERROR)`.
- **VersionManager:** Добавлены `save_document(scope, name, data)` и `get_document(scope, name, version)` для рецептов и произвольных документов.
- **Один источник истины:** ProcessData + StorageManager (конфиги/ДНК); RegistersManager (значения регистров); рецепты — файл YAML + при необходимости VersionManager.

# Чеклист для валидации модулей

Этот документ содержит чеклист для проверки модулей на соответствие стандартам фреймворка.

## Общая структура модуля

- [ ] Модуль имеет директорию `modules/{module_name}/`
- [ ] Есть файл `__init__.py` в корне модуля
- [ ] Есть файл `README.md` с описанием модуля
- [ ] Есть файл `interfaces.py` с интерфейсами модуля
- [ ] Есть директория `core/` с основной реализацией
- [ ] Есть директория `tests/` с тестами

## Интерфейсы

- [ ] Все интерфейсы определены в `interfaces.py`
- [ ] Интерфейсы наследуются от `ABC` из `abc`
- [ ] Все методы интерфейсов имеют `@abstractmethod`
- [ ] Все методы интерфейсов имеют docstrings
- [ ] Все методы интерфейсов имеют type hints
- [ ] Для каждого интерфейса `IXxx` есть реализация `Xxx` в `core/`

## Импорты

- [ ] Нет зависимостей от старого кода (`multiprocess_framework.modules.*`)
- [ ] Все импорты используют относительные пути (`from ..base_manager`)
- [ ] Нет циклических импортов
- [ ] Импорты упорядочены (стандартная библиотека, сторонние, локальные)

## Документация

- [ ] Все публичные классы имеют docstrings
- [ ] Все публичные методы имеют docstrings
- [ ] Docstrings описывают параметры и возвращаемые значения
- [ ] Есть примеры использования в README.md
- [ ] Документация актуальна (соответствует коду)

## Type Hints

- [ ] Все публичные методы имеют type hints для параметров
- [ ] Все публичные методы имеют type hints для возвращаемых значений
- [ ] Используются типы из `typing` где необходимо
- [ ] Нет использования `Any` без необходимости

## Тесты

- [ ] Есть тесты для всех публичных методов
- [ ] Тесты покрывают основные сценарии использования
- [ ] Тесты покрывают edge cases
- [ ] Тесты используют pytest или unittest
- [ ] Тесты проходят (95%+ успешность)
- [ ] Есть интеграционные тесты для взаимодействия с другими модулями

## Соответствие архитектуре

- [ ] Менеджеры наследуются от `BaseManager`
- [ ] Менеджеры используют `ObservableMixin` для логирования
- [ ] Менеджеры реализуют интерфейс из `interfaces.py`
- [ ] Менеджеры имеют стандартный жизненный цикл (`initialize()`, `shutdown()`)
- [ ] Адаптеры наследуются от `BaseAdapter`

## Качество кода

- [ ] Код следует PEP 8
- [ ] Нет неиспользуемого кода
- [ ] Нет закомментированного кода
- [ ] Обработка ошибок реализована корректно
- [ ] Логирование используется через `ObservableMixin`

## Производительность

- [ ] Нет очевидных проблем с производительностью
- [ ] Используются эффективные структуры данных
- [ ] Нет утечек памяти (для долгоживущих процессов)

## Пример использования

- [ ] В README.md есть пример создания и использования модуля
- [ ] Примеры работают и актуальны
- [ ] Примеры демонстрируют основные возможности модуля

## Интеграция

- [ ] Модуль интегрируется с другими модулями корректно
- [ ] Нет конфликтов с другими модулями
- [ ] Модуль может использоваться независимо (если это предполагается)

## Как использовать чеклист

1. Откройте этот файл
2. Пройдитесь по каждому пункту для вашего модуля
3. Отметьте выполненные пункты
4. Исправьте проблемы для непройденных пунктов
5. Запустите валидатор: `python -m multiprocess_framework.refactored.tools.validate_all_modules {module_name}`

## Автоматическая проверка

Большинство пунктов можно проверить автоматически с помощью валидатора:

```bash
# Проверка конкретного модуля
python -m multiprocess_framework.refactored.tools.validate_all_modules base_manager

# Проверка всех модулей
python -m multiprocess_framework.refactored.tools.validate_all_modules
```

Валидатор проверит:
- Структуру модуля
- Импорты (зависимости от старого кода)
- Интерфейсы
- Документацию
- Type hints
- Тесты

Ручная проверка требуется для:
- Качества кода (PEP 8, неиспользуемый код)
- Производительности
- Примеров использования
- Интеграции

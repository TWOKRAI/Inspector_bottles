# config_module — Архитектурные решения

> Ссылки: [`../../DECISIONS.md`](../../DECISIONS.md)

**Глобальный контекст:** решение о роли модуля как тонкой обёртки зафиксировано в главном [`DECISIONS.md`](../../DECISIONS.md) — раздел **ADR-023: config_module — тонкая обёртка над data_schema_module**.  
Ниже — уточнения, специфичные для `config_module` (модульные **ADR-CFG-001…004**, ранее ADR-143…146; глобальные **ADR-024…027** относятся к router / prototype / SharedMemory / rendered_frame).

## ADR-CFG-001 (was ADR-143): Dict at Boundary для ConfigStore

**Статус:** принято  
**Дата:** 2026-03-15 (уточнено 2026-04-09)  
**Контекст:** ConfigStore должен хранить конфиги между процессами безопасно (pickle-safe).  
**Решение:**

- На границе (ConfigStore): имена → **plain `dict`**, не объекты `Config`.
- Внутри процесса: экземпляры **`Config`** (runtime, подписки, RLock).
- `ConfigManager.sync_config(name)` вызывает `config_store.store(name, config.data)` — **`config.data`** даёт копию-словарь.
- `load_config_from_storage(name)` читает `dict` и делает `Config(initial_data=data)` или `existing.update(data)`.

**Последствия:** ConfigStore не сериализует `Config`; межпроцессная граница остаётся простой и предсказуемой.

## ADR-CFG-002 (was ADR-144): Нет файловой I/O в config_module

**Статус:** принято  
**Дата:** 2026-03-15  
**Контекст:** модуль должен быть ортогонален источнику конфигов (файлы, env, API).  
**Решение:**

- `Config` принимает и отдаёт структуры как **dict**; файлов JSON/YAML/TOML не читает и не пишет.
- Загрузка/сохранение файлов — зона **`DataConverter`** и прикладного кода (`data_schema_module` и выше).

**Последствия:** модуль проще тестировать; нет скрытых зависимостей от ФС.

## ADR-CFG-003 (was ADR-145): Пять компонентов модуля

**Статус:** принято  
**Дата:** 2026-03-15  
**Решение:** публичный контур из пяти частей с одной зоной ответственности каждая:

| Компонент | Назначение |
|-----------|------------|
| `Config` | Контейнер одной конфигурации (dot-notation, подписки, RLock) |
| `ConfigManager` | Коллекция `Config`, lifecycle, синхронизация с ConfigStore |
| `ConfigSection` | View на поддерево конфигурации |
| `ConfigSchemaAdapter` | Мост SchemaBase → параметры для UI/реестра |
| `ConfigManagerConfig` | SchemaBase: настройки самого менеджера (не payload хранилища) |

## ADR-CFG-004 (was ADR-146): Env-fallback как опциональная возможность

**Статус:** принято  
**Дата:** 2026-03-15  
**Контекст:** `Config.get()` может подхватывать значения из окружения, без обязательной привязки к env.  
**Решение:**

- Если в конструктор `Config` передан **`env_prefix`** — включён поиск переменных вида `{ENV_PREFIX}_{KEY}` (точки в ключе → подчёркивания).
- Если префикс не задан — переменные окружения для fallback не используются.

**Последствия:** один и тот же API работает и в тестах без env, и в деплое с префиксом.

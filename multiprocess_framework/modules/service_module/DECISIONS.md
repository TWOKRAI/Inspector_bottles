## ADR-SVC-001: Singleton ServiceRegistry через `__new__` + Lock

**Дата:** 2026-05-25
**Статус:** Принято
**Task context:** Task 3.2 — ServiceRegistry

### Контекст

Нужен единый реестр сервисов, доступный из любой точки приложения без явной передачи через DI: декоратор `@register_service` срабатывает при импорте файла и не имеет доступа к контексту приложения. При конкурентном импорте из нескольких потоков (GUI-поток + backend-поток) нужна thread-safety.

### Альтернативы

**A. Module-level global dict**
- Плюсы: просто, нет бойлерплейта
- Минусы: нет thread-safety; тест не может изолированно сбросить состояние без патчинга модуля; неочевидный контракт для пользователя

**B. Classmethod-registry (все данные в атрибутах класса)**
- Плюсы: те же что singleton, нет нужды в `__new__`
- Минусы: семантически неочевидно («вызов класса как функции без смысла»); тяжелее мокать в тестах; `ServiceRegistry.register(...)` vs `ServiceRegistry().register(...)` — ломает ожидания

**C. Singleton через `__new__` + `threading.Lock` (double-checked locking)**
- Плюсы: thread-safe; стандартный Python-паттерн; `ServiceRegistry()` всегда возвращает тот же объект; `clear()` изолирует тесты через тот же singleton без патчинга
- Минусы: немного больше кода в `__new__`

### Решение

Выбран вариант C. `ServiceRegistry.__new__` проверяет `_instance` дважды (до и после захвата `_init_lock`), что предотвращает двойную инициализацию при гонке потоков.

### Последствия

- Архитектурные изменения: нет (паттерн уже применяется в `PluginManager`)
- Тесты: каждый тест вызывает `ServiceRegistry().clear()` через autouse-фикстуру
- Ограничения: singleton живёт на уровне процесса; при тестировании в subprocess необходимо повторное `discover()`
- Известные риски: при использовании `multiprocessing` (fork) каждый дочерний процесс получает свой экземпляр singleton — это ожидаемое поведение (реестр инициализируется через `discover()` в каждом процессе отдельно)

## ADR-SVC-002: `IService` через Protocol + `@runtime_checkable` (не ABC)

**Дата:** 2026-05-25
**Статус:** Принято
**Task context:** Task 3.1 — interfaces.py

### Контекст

Нужен интерфейс для сервисов с возможностью runtime-проверки совместимости (`isinstance`). Существующие реализации в `Services/` (`WebcamCameraService` и другие) уже содержат методы `start/stop/get_status` и атрибут `name`, но объявлены без явного наследования от какого-либо базового класса. Требование: фреймворк не должен форсировать изменения в application-слое.

### Альтернативы

**A. ABC с `@abstractmethod`**
- Плюсы: явный контракт при определении класса; статический анализ без `isinstance`
- Минусы: каждый существующий сервис нужно переписать (`class WebcamCameraService(IService):`); нарушает принцип «фреймворк не диктует structure application-слою»; делает `IService` зависимостью времени определения класса

**B. `@runtime_checkable class IService(Protocol)` (structural subtyping)**
- Плюсы: `isinstance(obj, IService)` работает без явного наследования; существующие сервисы структурно совместимы без правок; stub-классы в тестах не требуют `import IService`; Protocol удобнее при расширении интерфейса (добавить `health()` через optional method)
- Минусы: `isinstance` проверяет только наличие атрибутов, не их сигнатуры; IDE хуже показывает «что нужно реализовать»

**C. Metaclass-проверка при регистрации**
- Плюсы: полная проверка на этапе регистрации
- Минусы: чрезмерная сложность; нестандартный паттерн; трудно отлаживать

### Решение

Выбран вариант B. Structural subtyping через Protocol соответствует «duck typing» духу Python и позволяет подключить любой класс с нужными методами без изменения его определения.

### Последствия

- Архитектурные изменения: `interfaces.py` содержит только stdlib-зависимости
- Ограничения: `isinstance(obj, IService)` не проверяет сигнатуры методов (например, `start()` без `config: dict` пройдёт проверку); явную проверку при необходимости выполняет application-слой
- Паттерн согласован с `IRouter` из `state_store_module` (ADR-SS-001) — единообразие в фреймворке

## ADR-SVC-003: `ServiceRegistry` хранит классы, не экземпляры

**Дата:** 2026-05-25
**Статус:** Принято
**Task context:** Task 3.2 — ServiceRegistry, Task 3.3 — scanner

### Контекст

При регистрации `@register_service` доступен только класс (`decorator(cls)`). Инстанцирование сервиса — потенциально тяжёлая операция (открытие камеры, подключение к БД) и не должна срабатывать при import-time. Кроме того, разные компоненты application-слоя могут требовать разные параметры конструктора.

### Альтернативы

**A. Хранить экземпляр в `ServiceEntry.instance` (eager instantiation)**
- Плюсы: `ServiceRegistry.get("sql").instance.start()` без дополнительного шага
- Минусы: `@register_service` выполняется при импорте → I/O при импорте; если `Services/sql/service.py` импортируется до доступности БД — ошибка при импорте; нельзя передать параметры конструктора

**B. Ленивая инициализация через `__getattr__` на `ServiceEntry`**
- Плюсы: экземпляр создаётся при первом обращении
- Минусы: неочевидный контракт; трудно управлять lifecycle (когда вызвать `stop()`?); скрывает момент инстанцирования от разработчика

**C. Хранить класс в `ServiceEntry.cls`, инстанцировать в application-слое**
- Плюсы: регистрация при импорте дешёвая (нет I/O); вызывающий контролирует параметры конструктора (`WebcamCameraService(device_index=0)`); разные подписчики могут создавать разные экземпляры или использовать один через DI; тестирование: легко подменить `entry.cls = MockService`
- Минусы: нужен дополнительный шаг инстанцирования в вызывающем коде

### Решение

Выбран вариант C. `ServiceEntry.cls: type` хранит ссылку на класс. Инстанцирование и вызов `start()` — ответственность application-слоя (Task 3.7 — `ServicesPresenter.start_service(name)`).

### Последствия

- Архитектурные изменения: `ServicesPresenter` (Task 3.7) хранит `_instances: dict[str, IService]` — кэш запущенных экземпляров
- Ограничения: `ServiceRegistry` не предоставляет «текущий запущенный экземпляр» — это обязанность presenter-а или adapter-а
- Известные риски: при ошибке в конструкторе сервиса ошибка возникает в application-слое, не при регистрации — что является желаемым поведением

# 04. Сравнение вычислительных платформ

## 1. Варианты архитектуры

| Вариант | Описание | Схема |
|---------|----------|-------|
| **A. Один промПК** | Один мощный iROBO на все 4 линии | Централизованная |
| **B. Несколько Jetson** | По 1 Jetson на линию (4 шт) | Распределённая |
| **C. Гибрид** | промПК (центр) + Jetson (edge) на линию | Гибридная |
| **D. Несколько промПК** | По 1 промПК на 2 линии (2 шт) | Полуцентрализованная |

## 2. Детальное сравнение

### Вариант A: Один промышленный ПК (iROBO-3000)

```
┌─────────────────────────────────────┐
│  iROBO-3000 (i7-12700 + RTX A2000) │
│                                     │
│  Process_Line1  ──► GPU inference   │
│  Process_Line2  ──► GPU inference   │
│  Process_Line3  ──► GPU inference   │
│  Process_Line4  ──► GPU inference   │
│  Process_HMI    ──► PyQt5           │
│  Process_PLC    ──► Modbus/EtherCAT │
│  Process_Archive──► SQLite/NAS      │
│                                     │
│  16 камер → GigE Switch → PCIe NIC  │
└─────────────────────────────────────┘
```

| Параметр | Оценка |
|----------|--------|
| **CPU** | i7-12700: 12C/20T, 4.9 GHz boost. 4 линии × (grab + preprocess + postprocess) = 8-12 ядер. **Достаточно.** |
| **GPU** | RTX A2000: 6 GB VRAM, 3328 CUDA cores. YOLOv8m FP16: ~8 мс/кадр. 4 линии × 4 камеры = 16 кадров за цикл инспекции. При batch=16: ~30 мс. **Достаточно.** При нагрузке 40 шт/мин × 4 линии = 160 рулонов/мин = 2.7/сек × 4 кадра = 10.7 инференсов/сек. Легко. |
| **RAM** | 32 GB DDR4. 4 процесса × ~2 GB + GPU shared + PyQt + OS = ~16-20 GB. **Достаточно с запасом.** |
| **Сеть** | 3 LAN (1x 2.5GbE + 2x 1GbE) + PCIe 4-port NIC. 16 камер через GigE Switch → 2 NIC порта. **Решаемо.** |
| **Хранение** | SSD 512 GB — для ОС+ПО. Нужен NAS/HDD для архива (1 год). |
| **Стоимость** | ~300-400 тыс. руб (ПК) + ~80-120 тыс. (GPU) + ~50 тыс. (доп. NIC) = **~430-570 тыс. руб** |
| **Надёжность** | Промышленный корпус, пассивное охлаждение, IP56 (зависит от корпуса). Единая точка отказа — падение ПК = все 4 линии стоят. |

**Плюсы:**
- Простая архитектура, один комп = одна точка администрирования
- Все данные локально, нет задержек сети между узлами
- Фреймворк multiprocess_framework идеально ложится (multiprocessing внутри одного хоста)
- Один GPU обслуживает все линии эффективнее (GPU batching)
- Меньше точек отказа по железу

**Минусы:**
- Single point of failure: падение ПК → все 4 линии стоят
- 16 камер на один хост — много GigE трафика (но trigger mode решает)
- При обновлении ПО — downtime всех линий

### Вариант B: Несколько Jetson (распределённая)

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│Jetson AGX│  │Jetson AGX│  │Jetson AGX│  │Jetson AGX│
│ Линия 1  │  │ Линия 2  │  │ Линия 3  │  │ Линия 4  │
│ 4 камеры │  │ 4 камеры │  │ 4 камеры │  │ 4 камеры │
│ Inference│  │ Inference│  │ Inference│  │ Inference│
└─────┬────┘  └─────┬────┘  └─────┬────┘  └─────┬────┘
      │             │             │             │
      └──────────┬──┴─────────────┴──┬──────────┘
                 │  Ethernet Switch  │
                 └────────┬──────────┘
                          │
                 ┌────────┴────────┐
                 │  Головной ПК    │
                 │  (HMI, архив,   │
                 │   PLC связь)    │
                 └─────────────────┘
```

**Jetson варианты:**

| Модель | GPU | CPU | RAM | Цена (ориент.) | YOLO FPS (v8m) |
|--------|-----|-----|-----|----------------|----------------|
| Jetson Orin Nano 8GB | 1024 CUDA | 6C A78AE | 8 GB | ~35-50 тыс. | ~30-40 FPS |
| Jetson Orin NX 16GB | 2048 CUDA | 8C A78AE | 16 GB | ~70-100 тыс. | ~60-80 FPS |
| Jetson AGX Orin 32GB | 2048 CUDA | 12C A78AE | 32 GB | ~150-200 тыс. | ~100+ FPS |

**Для нашей задачи:** Jetson Orin NX 16GB достаточно на 1 линию (4 камеры, ~10 инференсов/сек).

| Параметр | Оценка |
|----------|--------|
| **Стоимость** | 4 × Orin NX (~80 тыс.) + головной ПК (~150 тыс.) + сеть = **~520-700 тыс. руб** |
| **Надёжность** | Отказ 1 Jetson → только 1 линия стоит. Изоляция сбоев. |
| **GigE камеры** | Jetson имеет 1-2 Ethernet порта. Для 4 камер нужен USB-Ethernet или GigE switch на каждый Jetson. **Проблема: Jetson не имеет PCIe для доп. NIC.** |
| **SDK** | NVIDIA JetPack: CUDA, TensorRT, DeepStream. **Hikrobot SDK на ARM64 — нужно проверить совместимость.** |
| **ПО** | Нужна распределённая архитектура: Jetson ↔ головной ПК по сети. Фреймворк multiprocess_framework рассчитан на один хост. **Потребуется доработка для межмашинного IPC.** |
| **Обслуживание** | 4+1 устройств вместо 1. Обновление ПО на каждом. |

**Плюсы:**
- Изоляция: падение одного узла не затрагивает остальные
- Масштабируемость: добавить 5-ю линию = добавить Jetson
- Jetson = встраиваемый, компактный, низкое энергопотребление

**Минусы:**
- **Hikrobot GigE SDK на ARM64** — не гарантирована полная поддержка
- Сложнее архитектура ПО (распределённая система)
- Фреймворк нужно адаптировать для multi-host
- Больше точек обслуживания (4+1 устройство)
- Не промышленный корпус (нужен кастомный или IP65 кожух)
- Синхронизация данных между узлами (архив, статистика)

### Вариант C: Гибрид (промПК + Jetson edge)

```
┌───────────────────────────────────┐
│  промПК iROBO-3000                │
│  (HMI, архив, PLC, координация)  │
│  + GPU для тяжёлых задач          │
└──────────────┬────────────────────┘
               │ Ethernet
    ┌──────────┼──────────┐
    │          │          │
┌───┴───┐ ┌───┴───┐ ┌───┴───┐
│Jetson │ │Jetson │ │Jetson │ × 4
│Edge   │ │Edge   │ │Edge   │
│Grab + │ │Grab + │ │Grab + │
│Preproc│ │Preproc│ │Preproc│
└───────┘ └───────┘ └───────┘
```

**Идея:** Jetson рядом с камерами делает grab + preprocessing. Тяжёлый inference на центральном GPU. Результат обратно на Jetson для управления.

**Минусы:** Двойная сетевая нагрузка (кадры на центр, результат обратно). Latency сети добавляется к pipeline. Сложность без реального выигрыша — если центральный GPU справляется сам, зачем edge?

**Вердикт:** Избыточная сложность для данной задачи. Имеет смысл только если 16 камер не помещаются на одном GigE сегменте.

### Вариант D: Два промПК (по 2 линии)

```
┌──────────────────────┐     ┌──────────────────────┐
│  промПК #1           │     │  промПК #2           │
│  Линия 1 + Линия 2  │     │  Линия 3 + Линия 4  │
│  8 камер + GPU       │     │  8 камер + GPU       │
│  HMI (primary)       │     │  HMI (secondary)     │
└──────────┬───────────┘     └──────────┬───────────┘
           │                            │
           └────────────┬───────────────┘
                        │ Ethernet
                ┌───────┴────────┐
                │  Общая сеть    │
                │  PLC, NAS,     │
                │  предприятие   │
                └────────────────┘
```

| Параметр | Оценка |
|----------|--------|
| **Стоимость** | 2 × iROBO (~350 тыс.) + 2 × GPU (~100 тыс.) = **~700-900 тыс. руб** |
| **Надёжность** | Отказ 1 ПК → 2 линии стоят, 2 работают |
| **ПО** | Фреймворк работает как есть на каждом ПК. Нужна синхронизация HMI и архива между двумя. |

**Плюсы:** Баланс надёжности и простоты. **Минусы:** Дороже варианта A, сложнее в администрировании.

## 4. Рекомендация

### Основной вариант: **A — один промышленный ПК iROBO-3000**

**Обоснование:**
1. **Минимальная стоимость** — самый дешёвый вариант
2. **Фреймворк работает as-is** — multiprocess_framework спроектирован для multiprocessing на одном хосте
3. **Полная совместимость с Hikrobot SDK** — x86_64, Linux/Windows
4. **Простота обслуживания** — один компьютер, одна точка администрирования
5. **GPU batching** — один GPU обрабатывает batch из всех линий эффективнее 4 отдельных
6. **Производительности хватает** с запасом (см. расчёт в 07_PERFORMANCE_CALC.md)

### Митигация единственной точки отказа

- **UPS (ИБП)** обязателен для промПК — защита от скачков и отключений
- **Watchdog** — hardware watchdog iROBO перезагрузит систему при зависании
- **Auto-restart** — systemd/supervisor автоматически перезапускает ПО
- **Горячий резерв** — опционально: второй ПК в режиме standby (можно добавить позже)
- **Мониторинг** — Grafana/Prometheus dashboards, алерты при деградации

### Когда переходить на распределённую:

- Если заказчик добавляет 5+ линий
- Если нужна полная отказоустойчивость (линия не должна стоять даже при отказе ПК)
- Если появятся задачи, требующие больше GPU (3D-инспекция, тяжёлые модели)

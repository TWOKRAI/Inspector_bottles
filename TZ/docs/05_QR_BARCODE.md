# 05. Считывание QR-кодов и штрихкодов

## 1. Где нужны QR/штрихкоды в процессе

| Точка | Что читаем | Зачем | Этап ТЗ |
|-------|-----------|-------|---------|
| На рулоне (этикетка) | Штрихкод/QR на плёнке ОПП | Идентификация продукта, прослеживаемость | Этап 2 (инспекция) |
| На коробе | Этикетка от этикетировщика | Верификация: правильная этикетка на правильном коробе | Этап 6 |
| На паллете | Паллетная этикетка | Учёт: сколько коробов, какой продукт, когда | Этап 7 |

> **Из ТЗ (п.60):** "Описание машинного зрения и систем автоматического обеспечения качества (контроль дефектов и контроль количества вложений)". Прямого требования по QR нет, но **наклейка этикетки** (Этап 6) и **счётчики прохождения** (п.64) подразумевают идентификацию.

## 2. Подходы к реализации

### 2.1. Вариант A: Отдельные промышленные сканеры (стандартный)

```
  Конвейер ──► [Промышленный сканер Datalogic/Cognex] ──► PLC ──► ПК
```

| Плюсы | Минусы |
|-------|--------|
| Надёжный, проверенный в промышленности | Дорого: 50-150 тыс. за сканер |
| Работает автономно, не нагружает ПК | Отдельная интеграция с PLC |
| IP65/67 из коробки | Нет гибкости (только штрихкоды) |
| Высокая скорость декодирования | Каждая точка — отдельное устройство |

### 2.2. Вариант B: Программная реализация через камеры ТЗ (наш выбор)

```
  Конвейер ──► [Hikrobot камера] ──► GigE ──► ПК ──► Python (pyzbar/OpenCV)
```

| Плюсы | Минусы |
|-------|--------|
| **Камеры уже стоят** — дополнительное оборудование не нужно | Нагрузка на CPU/GPU (незначительная) |
| Гибкость: QR, DataMatrix, EAN-13, Code128, любой формат | Нужна оптика/освещение, подходящие для чтения кодов |
| Единая система: инспекция + считывание в одном pipeline | Зависимость от качества печати и освещения |
| Стоимость: ~0 руб (программное) | При плохой печати — хуже спец. сканера |
| Полный контроль: лог, верификация, аналитика | |

### 2.3. Вариант C: Гибрид (камеры + 1-2 промышленных сканера)

Камеры для QR на рулонах (уже стоят), промышленный сканер на выходе (верификация этикетки короба).

## 3. Рекомендация: Вариант B (программная реализация)

**Обоснование:**
- Камеры Hikrobot 5MP с Global Shutter дают достаточное разрешение для декодирования
- При trigger mode (1 кадр на рулон) нагрузка на декодирование минимальна
- Экономия: не нужно покупать отдельные сканеры (4-8 шт × 50-150 тыс. = 200-1200 тыс. руб)
- Полная интеграция с системой инспекции — один pipeline, один архив

## 4. Техническая реализация

### 4.1. Библиотеки декодирования

| Библиотека | Форматы | Скорость | Надёжность | Лицензия |
|------------|---------|----------|------------|----------|
| **pyzbar** (libzbar) | 1D (EAN, Code128, Code39) + QR | ~5-15 мс | Хорошая | LGPL |
| **OpenCV** (cv2.QRCodeDetector) | Только QR | ~3-10 мс | Средняя | Apache 2.0 |
| **pyboof** (BoofCV) | 1D + 2D + DataMatrix | ~5-20 мс | Хорошая | Apache 2.0 |
| **zxing-cpp** (python bindings) | Все основные | ~3-10 мс | Отличная | Apache 2.0 |
| **Dynamsoft** | Все + плохое качество | ~5-15 мс | Отличная | Коммерческая |

**Рекомендация:** `pyzbar` для MVP (бесплатный, стабильный, покрывает все нужные форматы). Если потребуется надёжность при плохой печати — `Dynamsoft Barcode Reader`.

### 4.2. Pipeline считывания

```python
# Упрощённый pipeline (в рамках multiprocess_framework)

# 1. Получение кадра (в CameraGrabProcess)
frame = camera.grab()  # trigger mode, синхронно с инспекцией

# 2. ROI для штрихкода (известная область на рулоне/этикетке)
barcode_roi = frame[y1:y2, x1:x2]

# 3. Preprocessing для улучшения читаемости
gray = cv2.cvtColor(barcode_roi, cv2.COLOR_BGR2GRAY)
# Адаптивная бинаризация (хорошо для неравномерного освещения)
binary = cv2.adaptiveThreshold(gray, 255, cv2.ADAPTIVE_THRESH_GAUSSIAN_C,
                                cv2.THRESH_BINARY, 11, 2)

# 4. Декодирование
from pyzbar.pyzbar import decode
results = decode(binary)

# 5. Валидация
for barcode in results:
    data = barcode.data.decode('utf-8')
    barcode_type = barcode.type  # 'QRCODE', 'EAN13', 'CODE128', etc.
    # Проверка формата, контрольной суммы, соответствия номенклатуре
```

### 4.3. Позиции камер для считывания кодов

| Точка считывания | Камера | Где код | Особенности |
|-----------------|--------|---------|-------------|
| QR/штрихкод на рулоне | Cam1 или Cam2 (поверхность) | На плёнке ОПП, боковая поверхность | Код виден при вращении рулона; нужно ловить момент |
| Этикетка на коробе | Доп. камера (Cam5) или Cam "контроль вложений" | На боку или сверху короба | После этикетировщика, перед паллетайзером |

### 4.4. Типы кодов (вероятные)

| Тип | Применение | Ёмкость | Минимальный модуль |
|-----|-----------|---------|-------------------|
| EAN-13 | Товарный штрихкод на рулоне | 13 цифр | ~0.33 мм |
| Code 128 | Внутренний код производства | Произвольный текст | ~0.25 мм |
| QR Code | Расширенная информация | До 4296 символов | ~0.5 мм |
| DataMatrix | Компактный 2D код | До 2335 символов | ~0.3 мм |
| GS1-128 | Логистический код на коробе | GTIN + серия + дата | ~0.25 мм |

> Конкретные типы кодов нужно уточнить у заказчика.

### 4.5. Разрешение камеры для кодов

**Правило: минимум 3-4 пикселя на наименьший модуль (элемент) штрихкода.**

- Code 128, модуль 0.3 мм → нужно ≥1.2 мм FOV/px
- Наша камера 5MP: при FOV 460 мм → 0.18 мм/px → **3.0 мм / 0.18 = 16.7 px на модуль — отлично**
- При FOV 70 мм (торец) → 0.027 мм/px → **ещё лучше**

**Вывод:** разрешения камер Hikrobot 5MP более чем достаточно для декодирования штрихкодов.

## 5. Считывание при вращении рулона

**Проблема:** QR/штрихкод на плёнке рулона виден только с одной стороны. При вращении на 360° нужно поймать момент, когда код обращён к камере.

**Решения:**

1. **Мультикадровый захват:** снять 8-12 кадров за полный оборот, декодировать каждый. Код будет прочитан в 1-3 кадрах.
   - При 360° / 12 кадров = 30° на кадр — код (занимает ~60-90° дуги) попадёт минимум в 2 кадра

2. **Линейное сканирование:** если конвейер с энкодером — "разворачивать" поверхность в панораму, затем искать код на развёртке

3. **Дополнительная камера "code reader":** одна дешёвая камера 2MP специально для чтения кодов, направлена на зону, где код гарантированно виден (если есть фиксированная ориентация подачи)

**Рекомендация:** Вариант 1 (мультикадровый) — самый простой, работает без дополнительного оборудования.

## 6. Верификация этикетки на коробе

### 6.1. Что проверяем

- Этикетка наклеена (визуально: есть/нет)
- Штрихкод/QR читаем
- Содержимое соответствует продукту в коробе
- Дата/партия корректны

### 6.2. Реализация

Дополнительная камера (Cam5) после этикетировщика, перед паллетайзером. Или использовать камеру системы контроля вложений.

```
  ───► Заклейщик ──► Этикетировщик ──► [Cam5: верификация] ──► Паллетайзер
```

## 7. Интеграция с системой учёта

```
  Считанный код ──► Валидация ──► БД (SQLite)
                                     │
                                     ├── Счётчик продукции (ТЗ п.64)
                                     ├── Связка: рулон → короб → паллет
                                     ├── Статистика по номенклатурам
                                     └── Экспорт в Excel/dashboard (ТЗ п.87)
```

Каждый считанный код записывается с timestamp, номером линии, результатом инспекции (годен/брак), номером короба и позицией на паллете. Это обеспечивает полную прослеживаемость.

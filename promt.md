Контекст
У нас есть многопроцессный фреймворк на Python, который разрабатывался долгое время и прошёл множество рефакторингов. В текущем состоянии связи между модулями нарушены, присутствуют ошибки импортов, циклические зависимости, несоответствие документации и кода. Модули должны быть рабочими но где то работать не так как нужно, потому что могут быть ошибки.

Роли
Ты — тимлид и сеньор-архитектор. Ты анализируешь текущее состояние, разрабатываешь целевую архитектуру и составляешь детальный поэтапный план рефакторинга.

Исполнители — разные модели AI (Sonnet, Haiku и др.), которые будут выполнять твой план. Для оптимизации затрат некоторые этапы можно поручать более лёгким моделям (например, Haiku), если задача не требует глубокого анализа или сложного рефакторинга. Сложные этапы, требующие архитектурных решений, должен выполнять Sonnet.

Цель рефакторинга
Привести фреймворк в рабочее состояние, сделав его:

Модульным — каждый модуль автономен, имеет чёткие границы и минимальные внешние зависимости (только через стабильные интерфейсы).

Тестируемым — каждый модуль покрыт модульными тестами, интеграционные тесты проверяют связку.

Документированным — каждый модуль содержит README с описанием, примерами, зависимостями, инструкцией по изолированному тестированию, с содержанием классов модулей и их связей и ссылками, описание интерфейсов.

Соответствующим лучшим практикам Python и паттернам проектирования (SOLID, композиция вместо наследования, dependency injection, интерфейсы/протоколы и т.д.).

Важно: Не нужно заботиться об обратной совместимости. Можно ломать API, переименовывать, переписывать с нуля. Главное — получить чистую, рабочую, хорошо документированную систему.

## Философия и метафоры

Система построена по аналогии с **живым организмом** и **человеческим обществом**. Это помогает понять роли компонентов и их взаимодействие.

- **Человеческая раса** → все объекты фреймворка происходят от единого базового класса `BaseManager`. Это «кирпичики», из которых строится всё.
- **Должности** → каждый модуль выполняет определённую функцию, подобно должностям в организации.
- **Отделы** → `ProcessModule` (начальник отдела) включает в себя других менеджеров, которые работают внутри него.
- **Работяги** → `WorkerManager` управляет потоками (воркерами), которые выполняют реальную циклическую работу.
- **Почта** → `RouterManager` отвечает за маршрутизацию сообщений между компонентами. Его помощник — `DispatchModule` (почтальон).
- **Команды** → `CommandManager` обрабатывает команды, также используя `DispatchModule` как помощника.
- **Язык общения** → все менеджеры общаются через объекты `Message` (модуль `message_module`).
- **Проектировщики** → `data_schema_module` отвечает за описание структур данных (конфигураций, регистров) с помощью Pydantic моделей. Это «чертежи» для всех компонентов.
- **Библиотека ресурсов** → `shared_resources_module` — центральное хранилище общих данных: очереди, события, разделяемая память, состояния процессов.
- **Генеральный директор** →  `process_manager_module` — это особый процесс который также строится на ProcessModule, который запускается всегда и управляет созданием всех остальных процессов. Он собирает конфигурации, настраивает общую библиотеку и распределяет ресурсы и контролирует в реальном времени все.
- **Братья** → `LoggerManager` и `ErrorManager` близки по функциональности, поэтому `ErrorManager` наследуется от `LoggerManager`, добавляя специфику обработки ошибок.
 и тому подобное

## Базовые строительные блоки

### `BaseManager`
Абстрактный базовый класс для всех менеджеров в системе. Определяет единый жизненный цикл:
- `initialize()` — инициализация менеджера.
- `shutdown()` — корректное завершение работы.
- `get_stats()` — сбор метрик и статистики.

Все менеджеры наследуются от `BaseManager`, что обеспечивает единообразие и предсказуемость.

### `BaseAdapter`
Базовый класс для адаптеров, которые предоставляют упрощённый интерфейс к менеджерам для использования в процессах (например, `LoggerAdapter`, `CommandAdapter`, `RouterAdapter`). Адаптеры тоже следуют жизненному циклу `BaseManager`.

### `ObservableMixin`
Примесь, добавляющая возможность подписки на события и уведомления. Используется в менеджерах, которые должны оповещать об изменениях своего состояния (например, `ProcessStateRegistry`).

### `Message`
Универсальный контейнер для обмена данными между компонентами. Содержит тип, приоритет, отправителя, получателя и полезную нагрузку. Все коммуникации внутри фреймворка строятся на сообщениях.

## Основные роли (должности)

### ProcessManagerCore (Генеральный директор)
- **Тип:** специальный процесс, всегда запущен.
- **Задачи:**
  - Чтение глобальной конфигурации.
  - чтение конфигурации других процессов и их воркеров 
  - формирование общего документа со всеми конфигурациями
  
  - Создание `SharedResourcesManager` (библиотеки).
  - создание очередей, событий и разделенной памяти в `SharedResourcesManager` по всем конфигурациям 
  - создание процессов и передача им список или ссылки на общий `SharedResourcesManager`
  - Запуск и остановка дочерних процессов (`ProcessModule`).
  - Мониторинг состояния всех процессов.
- **Реализация:** находится в модуле `process_manager_module`.

### ProcessModule (Начальник отдела)
- **Тип:** базовый класс для любого пользовательского процесса.
- **Задачи:**
  - Управление жизненным циклом своего процесса.
  - Создание и координация внутренних менеджеров:
    - `WorkerManager` (работяги)
    - `RouterManager` (начальник почты) (по дефолту один уже встроен и он опрашивает или отправляет сообщения но можно при необходимости и еще создавать для каждого воркера)
    - `CommandManager` (исполнитель команд) он раздает команды в зависимости от сообщений и зарегестрированных функций. Ему отправляет роутер менеджер сообщение если указано что ему, а там может быть команда. 
    - `LoggerManager` (логгер)
    - `ErrorManager` (обработчик ошибок)
    - "message_module" (язык общения всех менеджеров и логгера и ошибок)
  - Взаимодействие с `SharedResourcesManager` для доступа к общим данным.
- **Реализация:** модуль `process_module`.

### WorkerManager (Бригадир работяг)
- **Тип:** менеджер потоков внутри процесса.
- **Задачи:**
  - Регистрация и запуск воркеров (функций, работающих в цикле).
  - Управление приоритетами и зависимостями между воркерами.
  - Автоматический перезапуск упавших воркеров.
  -взаимодействие с другими менеджерами через  RouterManager
- **Реализация:** модуль `worker_module`.

### RouterManager (Начальник почты)
- **Тип:** менеджер маршрутизации сообщений.
- **Задачи:**
  - Приём сообщений от отправителей.
  - Определение канала доставки (очередь, событие, прямая отправка).
  - Передача сообщения адресату с помощью `DispatchModule` (почтальона).
- **Реализация:** модуль `router_module`, использует `dispatch_module`.

### CommandManager (Диспетчер команд)
- **Тип:** менеджер обработки команд.
- **Задачи:**
  - Регистрация обработчиков команд по их именам.
  - При поступлении сообщения-команды — поиск обработчика и выполнение.
  - Использует `DispatchModule` для вызова обработчиков.
- **Реализация:** модуль `command_module`, использует `dispatch_module`.

### DispatchModule (Почтальон)
- **Тип:** вспомогательный модуль, не является менеджером (но может быть оформлен как менеджер).
- **Задачи:**
  - Синхронный или асинхронный вызов зарегистрированных обработчиков.
  - Поддержка различных стратегий диспетчеризации (циклический перебор, приоритетный и т.д.).
  - Используется в `RouterManager` и `CommandManager` и другие.
- **Реализация:** модуль `dispatch_module`.

### LoggerManager (Летописец)
- **Тип:** менеджер логирования.
- **Задачи:**
  - Приём сообщений с уровнем логирования.
  - Запись логов в различные каналы (файл, консоль, очередь).
  - Батчинг для оптимизации.
- **Реализация:** модуль `logger_module`.

### ErrorManager (Аварийный комиссар)
- **Тип:** специализированный менеджер ошибок, наследник `LoggerManager`.
- **Задачи:**
  - Логирование ошибок с повышенным приоритетом.
  - Уведомление вышестоящих менеджеров (например, `ProcessManagerCore`) о критических сбоях.
  - Возможность выполнения корректирующих действий.
- **Реализация:** модуль `logger_module` (внутри `error_manager.py`).

### SharedResourcesManager (Библиотекарь)
- **Тип:** менеджер общих ресурсов.
- **Задачи:**
  - Хранение состояний всех процессов (`ProcessStateRegistry`).
  - Управление очередями (`QueueRegistry`).
  - Управление событиями (`EventManager`).
  - Управление разделяемой памятью (`MemoryManager`).
- **Реализация:** модуль `shared_resources_module`.

### DataSchemaModule (Проектировщик)
- **Тип:** модуль, не являющийся менеджером (но может содержать менеджеры).
- **Задачи:**
  - Определение Pydantic-схем для конфигураций и данных.
  - Регистрация схем компонентов.
  - Валидация и сериализация/десериализация данных.
  - Предоставление единого способа описания структуры всех конфигураций.
- **Реализация:** модуль `data_schema_module`.

### ConfigManager (Управляющий документацией) (нужно посмотреть связь с DataSchemaModule чтоб не было дублирования)
- **Тип:** менеджер конфигурации.
- **Задачи:**
  - Загрузка конфигураций из файлов (JSON, YAML).
  - Предоставление доступа к настройкам по разделам.
  - Использует `data_schema_module` для валидации и структурирования.
- **Реализация:** модуль `config_module`.

### ConsoleManager (Диспетчер консоли)
- **Тип:** менеджер для работы с консольным выводом.
- **Задачи:**
  - Перенаправление stdout/stderr.
  - Управление цветным выводом.
  - Интеграция с `LoggerManager`.
- **Реализация:** модуль `console_module`.

### GUIProcessModule (Начальник отдела с графическим интерфейсом) (пока не рассматриваем)
- **Тип:** наследник `ProcessModule` для процессов, имеющих GUI (PyQt/PySide).
- **Задачи:**
  - Интеграция с циклом событий Qt.
  - Специфическая обработка окон.
- **Реализация:** модуль `gui_module`.

## Взаимодействие между модулями

### Принципы
1. **Все общаются через сообщения.** Никаких прямых вызовов методов между процессами (кроме локальных вызовов внутри одного процесса).
2. **Центральное хранилище состояний** — `SharedResourcesManager`. Любой процесс может читать/обновлять состояние других процессов через этот менеджер.
3. **Конфигурация строится на схемах.** Каждый компонент описывает свою конфигурацию в виде Pydantic-модели, зарегистрированной в `data_schema_module`. каждый менеджер или объект подвязан к своей дата схеме и по ней создается. 
4. **Зависимости через интерфейсы.** Модули зависят от абстракций (например, `IRouter`, `ICommandHandler`), что позволяет легко подменять реализации при тестировании.

### Пример: отправка сообщения от процесса A к процессу B
1. Процесс A создаёт объект `Message` (тип, целевой процесс, данные).
2. Вызывает `self.router.send(message)` (в `ProcessModule`).
3. `RouterManager` процесса A определяет канал (например, очередь процесса B) и передаёт сообщение в `DispatchModule`.
4. `DispatchModule` помещает сообщение в очередь (через `SharedResourcesManager.get_queue(process_B, 'default')`).
5. Процесс B в своём системном потоке (воркере) слушает очередь, получает сообщение через `RouterManager.receive()` и обрабатывает.

### Пример: обновление состояния процесса
1. Процесс A изменяет своё состояние (статус, метрики) через `self.update_process_state(...)`.
2. `ProcessModule` вызывает `SharedResourcesManager.update_process_state(...)`.
3. `SharedResourcesManager` обновляет запись в `ProcessStateRegistry` и генерирует событие `PROCESS_STATE_CHANGED` через `EventManager`.
4. Подписчики (например, `ProcessManagerCore`) получают уведомление и реагируют.

## Жизненный цикл системы

1. **Запуск**
   - Пользователь создаёт `SystemLauncher` с конфигурацией.
   - `SystemLauncher` запускает `ProcessManagerBootstrap`, который создаёт процесс `ProcessManager`.
   - `ProcessManager` инициализирует `SharedResourcesManager`, `ConfigManager`, `ConsoleManager`.
   - На основе конфигурации `ProcessManager` через `ProcessManagerCore` создаёт дочерние процессы (`ProcessModule`), передавая им ссылки на общие ресурсы.

2. **Инициализация процесса**
   - В каждом дочернем процессе запускается целевая функция, которая создаёт экземпляр `ProcessModule`.
   - `ProcessModule.initialize()` создаёт внутренние менеджеры:
     - `WorkerManager`
     - `RouterManager` (с `DispatchModule`)
     - `CommandManager` (с `DispatchModule`)
     - `LoggerManager`, `ErrorManager`
   - Регистрирует свои системные воркеры (например, для приёма сообщений).
   - Обновляет состояние в `SharedResourcesManager` на `ready`.

3. **Работа**
   - `ProcessModule.run()` запускает пользовательскую логику (обычно цикл).
   - Воркеры, зарегистрированные в `WorkerManager`, выполняются в своих потоках.
   - Коммуникация идёт через сообщения.

4. **Завершение**
   - При получении сигнала остановки (или по команде от `ProcessManager`) вызывается `shutdown()`.
   - `ProcessModule.shutdown()` останавливает воркеры, завершает менеджеры, обновляет состояние.
   - Процесс завершается.

Проектирование целевой архитектуры

На основе метафор (человеческая раса, должности) и реальных модулей сформируй новую, упрощённую архитектуру.

Определи точный набор модулей и их ответственность (учитывая уже существующие наработки, но с возможностью их переработки).

Для каждого модуля опиши:

Публичный API (классы, методы).

Зависимости (от каких других модулей зависит, через какие интерфейсы).

Способ тестирования в изоляции (какие моки/заглушки нужны).

Нарисуй схему зависимостей между модулями (можно в текстовом виде или Mermaid).

## Заключение

Предложенная архитектура обеспечивает:
- **Модульность** — каждый компонент автономен и может заменяться.
- **Единообразие** — все менеджеры наследуют `BaseManager`.
- **Гибкость** — взаимодействие через сообщения и централизованные ресурсы.
- **Понятность** — метафоры помогают быстро вникнуть в роли компонентов.

Документация структурирована так, чтобы разработчик мог начать с любого модуля и углубляться по необходимости, имея чёткие ссылки на смежные компоненты.

Лишнюю документацию убираем которая в папке docs вообще много лишнего и проще новую писать, а старую отправить в archive.

# vfd_comm — архитектурные решения

## ADR-VC-001: Транспорт-агностик через RegisterTransport (мост без импорта робота)

**Дата:** 2026-06-11 · **Статус:** принято

**Контекст.** ПЧ физически висит на RS-485 за роботом: ПК общается с ним через
mailbox-регистры робота (Lua ретранслирует). Прямой линии к ПЧ нет, но может
появиться.

**Решение.** `VfdClient(transport: RegisterTransport, register_map=...)`:
- сервисы НЕ связаны: vfd_comm не импортирует robot_comm; связку
  `transport=runtime.get_client()` делает плагин vfd_control (слой Plugins);
- мост = RobotClient (реализует RegisterTransport), прямое подключение =
  ModbusDevice(rtu) + DIRECT_MAP — без правки клиента;
- слой Lua скрывает slave id ПЧ (P14.00) — клиент про RS-485 не знает (SRP).

**Отвергнуто:** vfd_* методы внутри RobotClient (как в pc_full.py) — смешивает
домены, не масштабируется на прямое подключение и второй ПЧ.

## ADR-VC-002: poll() — пульс VFD_FLAG как опрос (компенсация ограничения Lua)

**Дата:** 2026-06-11 · **Статус:** принято (до Lua-улучшения №1)

В текущем Lua зеркало 0x1210+ (включая heartbeat) обновляется ТОЛЬКО при
обработке VFD_FLAG (`vfd_poll_publish` зовётся из `handle_vfd`). Периодический
read_status без команд читал бы замороженный снимок. Решение: `poll()` пишет
FLAG=1 без смены команды — Lua кэширует last_cmd/last_freq, на RS-485 уходит
только чтение статуса; `ensure_alive()` детектит обрыв моста по динамике
heartbeat (stale_polls_limit подряд без роста → VfdBridgeStaleError).
После Lua-улучшения «публикация в idle» poll() останется работать (пульс
станет избыточным, но безвредным).

## ADR-VC-003: Две карты на одно устройство (BRIDGE + DIRECT)

**Дата:** 2026-06-11 · **Статус:** принято

При прямом RTU меняется не только транспорт, но и карта (0x2000/0x2100/0x3000
вместо mailbox) и состав статуса (heartbeat/comm_errors — телеметрия моста,
у самого GD20 их нет). Поэтому `VFDStatus` с опциональными полями и
параметризация клиента картой. DIRECT_MAP — закладка по мануалу goodrive20,
активируется при появлении физической линии.

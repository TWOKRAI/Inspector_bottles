# modbus — архитектурные решения

## ADR-MB-001: Универсальный транспортный модуль + тонкие сервисы устройств

**Дата:** 2026-06-11 · **Статус:** принято · **Refs:** plans/robot-vfd-services.md

**Контекст.** Появились устройства с разными способами подключения (робот Delta
по TCP, ПЧ GD20 за RS-485-мостом робота, будущие сканеры/датчики). Первый
черновик предлагал каждому сервису свою pymodbus-обёртку (у рабочего кода
робота была собственная — из-за атомарных мульти-записей, которых ModbusDevice
не умел).

**Решение (владелец, 2026-06-11).** Один универсальный `Services/modbus` для
всех устройств; сервис устройства — тонкая надстройка «выбор соединения +
карта регистров + доменные методы». Для этого в modbus добавлены:

1. **`ModbusDevice.transaction(ops)`** — серия записей под ОДНИМ Lock,
   abort на первой ошибке (маркер-флаг не пишется поверх частичных данных).
   Примитив mailbox-протоколов: данные → маркер последним.
2. **`RegisterTransport`** (Protocol, interfaces.py) — минимальный контракт
   «устройство как пространство регистров»: `read_registers` + `transaction` +
   `is_connected`. Реализуют `ModbusDevice` (прямое подключение) и клиенты-
   мосты (RobotClient: ПК→робот по TCP, робот→ПЧ по RS-485).
3. **`RegisterMap`** (core/register_map.py) — декларативная карта регистров
   устройства: `Reg`/`RegDW`/`RegBlock`+`Field` со scale/signed/word_order
   поверх `sdk/datatypes`. Новый регистр устройства = одна строка в карте.
4. **TCP_NODELAY** (опция конфига) — боевой опыт робота: −40 мс лагов.

**Паттерн нового устройства** — README, раздел «Как добавить сервис нового
устройства». Эталоны: `Services/robot_comm` (прямое TCP), `Services/vfd_comm`
(мост через Protocol, без импорта robot_comm).

**Отвергнуто:** pymodbus-обёртка в каждом сервисе устройства — дубль
reconnect/телеметрии/кодеков, расползание Lock-семантик.

## ADR-MB-002: Reconnect — у владельца соединения, не в sdk

**Дата:** 2026-06-11 · **Статус:** принято

Тихий reconnect+retry внутри sdk разорвал бы атомарность `transaction`
(серия частично уйдёт в старое соединение, частично в новое). sdk только
детектирует обрыв (`ModbusConnectionError`); переподключение — ответственность
владельца соединения (плагина) по образцу `modbus_sink._ensure_connected`
(throttled). Транзакция при обрыве — fail-fast целиком. Следствие: держите
`timeout_sec` малым (transaction держит Lock на время I/O) и `retries=1`.

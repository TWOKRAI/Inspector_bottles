# robot_comm — архитектурные решения

## ADR-RC-001: Транспорт — универсальный Services/modbus, не своя pymodbus-обёртка

**Дата:** 2026-06-11 · **Статус:** принято

**Контекст.** Рабочий `pc_full.py` держал собственную обёртку ModbusTcpClient с
`_atomic` (серия записей под одним Lock). План калибровки изначально предлагал
повторить её в robot_comm (долг P1.1 «мигрировать потом»).

**Решение.** Долг закрыт до возникновения: примитивы подняты в `Services/modbus`
(`ModbusDevice.transaction`, `RegisterTransport`, `RegisterMap`), robot_comm —
тонкая надстройка «карта + методы». Новые устройства повторяют паттерн.

**Последствия.** Один транспортный стек на все устройства; reconnect и
телеметрия — в одном месте; vfd_comm не знает про робота (Protocol-мост).

## ADR-RC-002: Abort-семантика транзакций (осознанное отличие от pc_full)

**Дата:** 2026-06-11 · **Статус:** принято

У оригинала `_atomic` латентный баг: `ok = (not rr.isError()) and ok` — серия
продолжалась после ошибки, маркер-флаг мог лечь поверх частично записанных
данных. `ModbusDevice.transaction` обрывается на ПЕРВОЙ ошибке — маркер не
пишется, частичный mailbox без флага инертен для Lua. Инвариант «маркер
последним» обязателен в каждом вызове (send_job/set_config/draw_*/vfd_*).

## ADR-RC-003: Модель владельца соединения (один master на процесс)

**Дата:** 2026-06-11 · **Статус:** superseded by ADR-DH-001 (2026-06-11)
> **Замещено:** runtime.py и co-location модель удалены. Владелец соединения —
> процесс `devices` (always-on), см. `Services/device_hub/DECISIONS.md` ADR-DH-001.

`RobotClient` НЕ singleton и НЕ создаётся каждым плагином. Владелец — плагин
`robot_io`: создаёт/коннектит/публикует в `runtime.py`/закрывает. Потребители —
`runtime.get_client()` (read-only владение). `service.py` — карточка каталога
БЕЗ собственного соединения: второй TCP-master к одному mailbox дал бы гонку
по проводу (Lock каждого клиента локален процессу). Все плагины робота обязаны
жить в одном `process_name` рецепта; мульти-процессный шаринг — IPC-канал
(отложено, см. план robot-calibration Фаза 4+).

## ADR-RC-004: Симулятор — два уровня с одним ядром

**Дата:** 2026-06-11 · **Статус:** принято

pymodbus 3.13 удалил классический мутируемый datastore (сервер принимает только
SimDevice; spike подтвердил — деприкейченные шеллы неработоспособны). Решение:

- `RobotSimCore` — чистая логика (list[int] + tick() как Lua Motion-цикл);
- `FakeRobotTransport` — in-process, чтение тикает ядро (детерминизм, ~90% тестов);
- TCP `sim_robot` — публичный хук `SimDevice(action=...)` отдаёт живой список
  регистров сервера (`attach`), фоновый ticker крутит ядро над ним. Хук обязан
  быть async-ФУНКЦИЕЙ (замыкание) — инстанс с async `__call__` не проходит
  валидацию pymodbus.

Sim сознательно воспроизводит ограничение реального Lua: зеркало ПЧ (0x1210+,
включая heartbeat) обновляется ТОЛЬКО по команде/пульсу VFD_FLAG.

## ADR-RC-005: Карта регистров — universal3, валидация DW-выравнивания

**Дата:** 2026-06-11 · **Статус:** принято

Источник карты — ТОЛЬКО `robot/universal3` (пара к текущему Lua): u2 расходится
(CFG=11 против 9, TLM=11, DCBUS_SCALE=10, REG_MODE, drawing-блок). DW-поля
обязаны лежать по чётным адресам (требование Lua-стороны) — проверяется при
сборке карты, а не в универсальном RegDW (это ограничение устройства, не Modbus).

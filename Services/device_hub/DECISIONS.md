# Services/device_hub — Архитектурные решения

## ADR-DH-001: Один мастер — процесс devices

**Дата:** 2026-06-11
**Статус:** accepted

Все соединения с устройствами принадлежат ОДНОМУ процессу `devices` (protected).
GUI и плагины ходят к нему по IPC. Честный анализ: IPC-хоп 0.2-2 мс против
Modbus-операции 2-50 мс — накладные расходы пренебрежимы; tight-loop'ы (feeder CVT)
живут внутри хоста.

**Отвергнуто:** владение соединением в pipeline-плагине (robot_io) — ручное
управление без рецепта невозможно; co-location трёх плагинов обязательна.

## ADR-DH-002: Bridge-транспорт шарит Lock носителя

**Дата:** 2026-06-11
**Статус:** accepted

Bridge-устройство (VFD через мост робота) использует RegisterTransport носителя
(RobotClient). RLock внутри ModbusDevice обеспечивает thread-safety: transaction
атомарна. Bridge-устройство НЕ имеет своего connect; is_connected делегируется
носителю; disconnect носителя -> зависимые в degraded.

## ADR-DH-003: YAML-протокол -> RegisterMap + meta

**Дата:** 2026-06-11
**Статус:** accepted (реализован в Фазе 0)

Файлы протоколов (`.yaml`) загружаются через `load_protocol` в `DeviceProtocol`
(RegisterMap для транспорта + RegisterMeta для GUI). Высокоуровневых write-команд
в файле НЕТ — командная семантика остаётся в клиентах.

## ADR-DH-004: Удаление носителя — блокировка, НЕ каскад

**Дата:** 2026-06-11
**Статус:** accepted

При попытке удалить устройство-носитель (robot), от которого зависят bridge-
устройства (vfd), операция БЛОКИРУЕТСЯ с RegistryIntegrityError. Оператор
должен сначала удалить зависимые устройства.

**Отвергнуто:** каскадное удаление зависимых — опасно для оператора, может
случайно удалить настроенные устройства.

**Отвергнуто:** каскадная деградация (оставить записи, отключить) — сложнее
реализовать, неочевидное поведение для пользователя.

# Assistant Vision — настольный голосовой ассистент (Фаза «Jarvis»)

Концепция, сценарии пользы, MCP-архитектура и советы по реализации.
Статус: идея одобрена (2026-07-20), предшествует фазе робота.

---

## Концепция

Вместо того чтобы начинать с робота — сначала **стационарный голосовой ассистент**
как отдельное устройство на рабочем месте. Камера + микрофон + динамики, без механики.

```
Уровень 1: Jetson Orin NX 16GB   — LLM 14B, диалог, RAG-память, tool-реестр, TTS-синтез
Уровень 2: Jetson Orin Nano 8GB  — аудио I/O, wake word, VAD, Whisper, малая LLM («подсознание»), depth/3D-жесты
Уровень 3: RPi 5 8GB + AI HAT+   — YOLO, лица, руки 2D (Hailo NPU), tracking, веб-дашборд
Механика (ESP32/Arduino)         — ОТЛОЖЕНА до фазы робота
```

**Почему это лучше исходного плана фаз:**

1. Исчезает миграция Whisper NX→Nano из Фазы 3 — Whisper *сразу* на Nano,
   GPU NX с первого дня целиком под LLM.
2. Весь интеллект (голос, зрение, память, планировщик, ROS2-мост) — это 80% софта
   будущего робота. Робот потом добавит только SLAM/Nav/моторы.
3. Механика — самая дорогая и рискованная часть (~60-80к руб). Откладывается,
   а полезный продукт появляется сейчас.

## Чем ассистент будет полезен — сценарии

### 1. Управление ПК (главный сценарий)

Голосовой интерфейс к рабочему компьютеру через MCP-серверы на ПК:

| Сценарий | Пример |
|----------|--------|
| Файлы и поиск | «Найди вчерашний скриншот с дефектом», «открой план по телеметрии» |
| Запуск задач | «Запусти обучение модели на буквах», «собери проект», «прогони тесты» |
| Контроль длительных задач | «Скажи, когда закончится обучение» → ассистент следит и сообщает голосом |
| Семантический поиск по коду/знаниям | qex уже есть — «где у нас обрабатывается hot-swap рецептов?» |
| Git-операции | «Что я коммитил вчера?», «есть ли незакоммиченные изменения?» |
| Браузер / поиск | «Найди даташит на XVF3800», «что нового по JetPack 7» |
| Диктовка и заметки | «Запиши идею: …» → заметка в базу знаний |
| Музыка, звонки, календарь | стандартные ассистентские функции поверх ПК |

### 2. Умный дом

Home Assistant имеет **официальную MCP-интеграцию** (MCP Server) — свет, розетки,
климат, сценарии становятся инструментами LLM без написания собственных драйверов.
Устройства покупаются любые, лишь бы поддерживались Home Assistant.

### 3. Рабочее место инженера (специфика владельца)

- Наблюдение за стендом Inspector_bottles: «какой FPS на конвейере?», «есть ли ошибки
  в логах за последний час?» — backend_ctl-подобный мост уже отработан в проекте.
- Мониторинг обучения ML-моделей: прогресс, метрики, «angle_mae перестал падать».
- GPU/температуры/дисковое место — проактивные предупреждения.

### 4. Зрение и присутствие

- Узнавание по лицу: «Привет, Сергей» + персонализация диалога из RAG-памяти.
- Жесты: 2D (RPi) + 3D-указание (depth на Nano) — «включи вон ту лампу» + жест.
- Присутствие/зоны: реагирует, когда человек подошёл; молчит, когда никого нет.
- «Кто приходил, пока меня не было?»

### 5. Зрительное управление ПК (жесты, взгляд)

Помимо голоса — управление компьютером жестами и взглядом:

| Сценарий | Механизм | Реалистичность |
|----------|----------|----------------|
| Мышь жестом | указательный палец → курсор, pinch → клик | Реально: hand-landmarks 30 fps на Hailo |
| Скролл / медиа жестом | свайпы ладонью, «щепотка» | Реально, проще мыши |
| Взгляд как контекст | на какой монитор/зону экрана смотрит | Реально (грубо, ±2-4°) |
| Взгляд как курсор | точное позиционирование глазами | Веб-камерой — НЕТ; нужен Tobii (~$250) |
| Мультимодальный дейксис | «открой ВОТ ЭТО» + взгляд/указание | Главная фишка — см. ниже |

Подробности архитектуры — раздел «Две плоскости управления» ниже.

### 6. База знаний голосом

Ассистент — голосовой интерфейс к личной базе знаний (KnowledgeOS vault + qex):
«что мы решали по телеметрии?», «найди статью про seqlock», «запиши в базу: …».
RAG-память ассистента и база знаний — два источника одного сборщика контекста.

### 7. Секретарь с памятью (RAG)

- «Запомни: прокладки на 3/4 дюйма лежат во втором ящике» → «где прокладки?» через месяц.
- Напоминания по контексту и времени: «когда я сяду за стол утром — напомни про отчёт».
- Ночная консолидация памяти (режим «сон») — уже спроектирована в SOFTWARE_ARCHITECTURE.md.

### 8. Проактивность (отличие Jarvis от колонки)

Отдельный контур («подсознание» на Nano) периодически оценивает контекст и решает,
есть ли повод заговорить: забытый включённый паяльник в кадре, завершившееся обучение,
человек ищет что-то по столу → «кружка слева от монитора». Обязателен rate-limit.

## MCP как нервная система ассистента

**Решение:** все внешние способности ассистента — через Model Context Protocol.
LLM-хост на Orin NX = MCP-клиент; каждая способность = MCP-сервер.

```
┌─── Orin NX: LLM-хост (MCP-клиент) ────────────────────────────┐
│  Qwen-2.5 14B + dialog manager + RAG + сборка контекста        │
└───┬───────────┬───────────────┬──────────────┬────────────────┘
    │ streamable HTTP / SSE по сети            │
    ▼           ▼               ▼              ▼
┌────────┐ ┌──────────┐ ┌──────────────┐ ┌──────────────┐
│ ПК     │ │ Home     │ │ Стенд        │ │ Робот        │
│ agent  │ │ Assistant│ │ Inspector    │ │ (БУДУЩЕЕ)    │
│ (MCP)  │ │ (MCP)    │ │ (MCP-мост)   │ │ motion tools │
└────────┘ └──────────┘ └──────────────┘ └──────────────┘
  файлы,     свет,        телеметрия,      «подъедь к…»
  shell,     климат,      логи,            (те же
  браузер,   сценарии     команды          контракты!)
  поиск
```

**Почему MCP — правильный выбор:**

1. **Стандарт, а не самодельный протокол** — готовые серверы (filesystem, shell,
   browser/Playwright, Home Assistant), SDK на Python, транспорт по сети (streamable HTTP).
2. **Расширяемость без изменения ядра** — новая способность = новый сервер в конфиге.
   Ядро (диалог, память, голос) не трогается.
3. **Робот = ещё один MCP-сервер.** Когда появится механика, «подъехать», «повернуть
   камеру» станут просто новыми инструментами — интеллект уже готов.
4. **Двухуровневый вызов инструментов:** малая LLM (Nano) может сама вызывать
   дешёвые/быстрые инструменты (свет, таймер, музыка) без пробуждения 14B —
   мгновенная реакция на простые команды.

**Внутренние потоки (аудио, кадры, состояние) — НЕ через MCP**, а через
multiprocess_framework (SHM/IPC) и ROS2 DDS между платами, как спроектировано.
MCP — для внешних интеграций и действий; realtime-восприятие — по шинам фреймворка.

## Две плоскости управления ПК (ключевое архитектурное решение)

Жест-мышь через LLM+MCP невозможна: цикл LLM — секунды, мыши нужны миллисекунды.
Поэтому ПК-агент имеет **два независимых входа**:

```
┌─── Плоскость ДЕЙСТВИЙ (секунды) ────────────────────────────────┐
│  голос → Whisper → LLM → MCP tool call → ПК-агент → результат    │
│  «запусти тесты», «найди файл», «включи свет»                    │
└──────────────────────────────────────────────────────────────────┘
┌─── Плоскость УПРАВЛЕНИЯ (миллисекунды, БЕЗ LLM) ────────────────┐
│  камера (RPi) → hand-landmarks (Hailo) → сглаживание (Nano)      │
│      → WebSocket/UDP → ПК-агент → SendInput (курсор, клик)       │
│  задержка цели: < 80 мс end-to-end                               │
└──────────────────────────────────────────────────────────────────┘
```

ПК-агент (Windows) — один процесс с двумя интерфейсами: MCP-сервер для LLM
и realtime-endpoint для потоков жестов/взгляда. LLM управляет *режимами*
плоскости управления («включи режим мыши»), но не участвует в её цикле.

### Жест-мышь — рецепт

- **Сглаживание — One Euro Filter** (стандарт для указательных интерфейсов:
  убирает дрожь без лага при быстром движении). Без него курсор трясётся.
- **Относительное управление** (как тачпад), не абсолютное: рука → дельта курсора.
  Абсолютный маппинг «палец = точка экрана» утомителен и неточен.
- **Clutch-жест обязателен** (проблема «руки Мидаса»): курсор двигается только
  в захвате (например, pinch удерживается) — иначе любое движение руки у стола
  дёргает мышь. Режим включается/выключается голосом или жестом.
- Клик — быстрый pinch; правый клик — pinch двумя пальцами; скролл — два пальца
  вертикально. Дребезг фильтровать гистерезисом по времени (>150 мс).
- Бюджет задержки: камера 33 мс + inference 10-20 мс + сеть 2-5 мс + фильтр
  ≈ 60-80 мс — заметно, но комфортно для управления (не для игр).

### Взгляд — честная оценка

- **Веб-камера/RGB даёт ±2-4° точности** (≈ ±2-4 см на мониторе с 60 см) —
  для курсора НЕ годится, ложные ожидания убьют фичу.
- Что взгляд даёт реально: **какой монитор/зона** в фокусе, смотрит ли человек
  на ассистента (детектор адресации — говорит мне или в телефон), автоскролл
  при чтении, «разбудить экран взглядом».
- Голова+глаза оценивать по RGB Nano/RPi (MediaPipe Iris, L2CS-Net); depth-камера
  добавляет устойчивую позу головы и работу в темноте.
- Если захочется настоящий gaze-курсор — только специализированное железо
  (Tobii Eye Tracker 5: USB, **только Windows**, macOS не поддерживается;
  Mac-совместимое — лишь линейка Tobii Pro за сотни тысяч).
- **Альтернатива без нового железа: голова-курсор по 335L.** Поза головы из
  depth+RGB оценивается точно и стабильно → «голова наводит грубо, pinch доводит
  точно». Проверенная accessibility-техника, ощущается почти как gaze-курсор.
- **Кроссплатформенность:** восприятие (взгляд/голова/жесты) живёт на платах
  ассистента, ПК-агент лишь принимает события курсора: Windows — SendInput,
  macOS — CGEvent. Один ассистент управляет обеими ОС; Tobii привязал бы
  фичу к одной Windows-машине.

### Мультимодальный дейксис — главная фишка связки

«Открой **вот это**» + указание пальцем / взгляд = разрешение «вот это» в объект:

```
голос (Nano, t₁) ──┐
жест 3D (Nano, t₂) ─┼→ FusionProcess: совместить по времени → «это» = окно/объект X
взгляд (RPi, t₃) ───┘        │
                             ▼
                LLM получает уже разрешённую ссылку
```

**Требование: синхронизация часов всех плат (chrony/PTP по Ethernet)** —
без общей шкалы времени события голоса и жеста не совместить. Заложить в подфазу (а).

### Безопасность (обязательно с первого дня)

| Мера | Зачем |
|------|-------|
| Allowlist инструментов на каждом MCP-сервере | ПК-shell без ограничений = дыра |
| Голосовое подтверждение деструктивных действий | «Удалить ветку — подтверди» |
| Идентификация по голосу/лицу для ПК-команд | чужой голос не управляет компьютером |
| Всё локально (LLM, STT, TTS, память) | приватность — конкурентное преимущество |
| Логи всех tool-вызовов | аудит «что ассистент сделал и почему» |

## Советы по реализации (зафиксировано из обсуждения 2026-07-20)

### Аудио-контур

- **reSpeaker XVF3800: динамики подключать ЧЕРЕЗ него — обязательно, не опционально.**
  Аппаратный AEC работает по опорному сигналу воспроизведения → возможен **barge-in**
  (перебить ассистента во время его речи). Динамик мимо массива = AEC не работает.
- Следствие: захват и воспроизведение на одной плате → `AudioIOProcess` на Nano;
  TTS-синтез на NX (или Nano), но playback — строго через XVF3800.
- **Проверить до покупки:** плата отдаёт и capture, и playback как USB-аудио
  (или имеет выход на динамик/I2S-усилитель).
- Wake word: openWakeWord (бесплатно, обучаемо на своё слово) на CPU Nano, перед VAD.
  Whisper не молотит постоянно.
- TTS для русского: **Silero TTS v4 ru** предпочтительнее Piper (качество на CPU).

### Latency — важнее выбора моделей

Streaming на каждом стыке: частичные результаты STT → потоковая генерация LLM →
TTS по предложениям, не дожидаясь конца ответа. Цель: **первый звук < 1.5 с**.
Закладывать в контракты процессов сразу: события-чанки, а не «запрос-ответ целиком».

### Малая LLM «подсознание» (Nano, ~3B Q4, ~2-2.5 GB)

Роли по ценности:
1. Быстрый контур (System 1): мгновенные реакции, пока 14B думает.
2. Роутер интентов: тривиальное → ответить/выполнить локально; сложное → к 14B.
3. Фоновый аннотатор: события восприятия → факты-кандидаты для RAG-памяти.
4. Детектор адресации: человек говорит ассистенту или по телефону?

**RAM-бюджет Nano:** Whisper-medium ~1 GB + 3B ~2.5 GB + depth ~1 GB + ОС 2 GB
≈ 6.5/8 GB — влезает, **но только пока нет SLAM**. При апгрейде до робота
подсознание не поместится рядом со SLAM+Nav (91% по плану) → либо ужать до 1.5B,
либо перенести на NX, либо принять как цену апгрейда. Решить при переходе.

### Зрение — разделение ролей (не делать жесты дважды)

- RPi/Hailo: 2D — лица, hand-landmarks, объекты, tracking.
- Nano + depth: 3D-фьюжн — **куда указывает рука**, расстояние, зоны присутствия, IR в темноте.
- Depth-камера (Orbbec 335L, ~30к руб) — самая дорогая покупка; можно отложить
  до подфазы (в), идея не пострадает. 335L (а не 335) — потому что потом уедет на робота.

### Контекст LLM из StateStore

RPi и Nano публикуют восприятие в дерево состояния; сборщик промпта на NX берёт
снапшот («в комнате Сергей, показывает на кружку, на столе ноутбук») + RAG-факты +
историю диалога. Готовая архитектура сенсорного фьюжна для LLM.

### Порядок ввода в строй — по одной плате

| Подфаза | Железо | Результат |
|---------|--------|-----------|
| (а) Голос | NX + Nano + XVF3800 + динамик | голосовой контур end-to-end, MCP: ПК + Home Assistant |
| (б) Зрение | + RPi 5 + камера | «что ты видишь?», лица, жесты 2D, веб-дашборд |
| (в) 3D | + Orbbec 335L | указание жестом, присутствие, проактивность |

Связь между платами — Ethernet-свитч (не Wi-Fi: DDS с видеотопиками задушит сеть).

### Риски

- **Nano 8GB — узкое место** и сейчас (6.5/8 GB), и в будущем (подсознание vs SLAM).
- XVF3800: проверить playback-путь до покупки.
- Доработки фреймворка те же, что в SOFTWARE_ARCHITECTURE.md (headless, ARM,
  ROS2 bridge, RAG-модуль) — план подтягивает их раньше, ничего не выбрасывается.

## Стол как интерактивная поверхность (335L смотрит вниз)

**Решение о роли камеры:** Orbbec 335L направляется НА СТОЛ (не на комнату) —
касания без глубины не сделать, а комнату закрывают RPi-камеры (лица, 2D-жесты)
+ mmWave-радар (присутствие). Если позже нужен 3D в комнате — второй Orbbec
(младший Gemini 335, ~$264, диапазон 0.26-3 м).

**Почему стол лучше воздуха:** mid-air жесты = «рука гориллы» (усталость за минуты);
на столе рука лежит, палец водит как по гигантскому тачпаду. Снимает главный
UX-риск жест-мыши.

### Что делать (по нарастающей)

| Идея | Вердикт | Механизм |
|------|---------|----------|
| Тачпад на столе | ДА — основной режим | плоскость стола (RANSAC) + палец ближе ~10 мм = касание, гистерезис на нажатие |
| Виртуальная клавиатура | НЕТ — честно | без тактильного отклика 10-20 слов/мин с ошибками (подтверждено Vision Pro); текст → диктовка Whisper |
| **Виртуальный Stream Deck** | ДА — вместо клавиатуры | зоны-кнопки на столе (печатный шаблон), тап = MCP-действие; крупные мишени прощают ±2 см |
| Сканер в базу знаний | ДА | «сфоткай документ» — камера уже смотрит на стол, выравнивание по плоскости |
| Осязаемые контроллеры | ДА, дёшево и эффектно | кубик/объекты с ArUco-метками: повернул = громкость, поставил на зону = сцена |
| Рукописный ввод → OCR | Опционально | для схем/формул; для текста диктовка быстрее |
| Проекция на стол | Потом | пико-проектор + depth = интерактивный проецируемый экран; начинать с бумажного шаблона |

### Ожидаемая точность и удобство (честные цифры)

| Параметр | Ожидание |
|----------|----------|
| Курсор | ±2-3 мм после One Euro ≈ тачпад ноутбука; мелкие цели — через «точный режим» (автозамедление) |
| Тапы по зонам 4-6 см | ~100% надёжность |
| Тап как клик по мелким целям | ~95%, двойной тап слабое место → клик на pinch |
| Задержка | 60-90 мс: навигация/кнопки — комфортно; рисование/игры — нет |
| Позиционирование | НЕ замена мыши, а второй канал: руки заняты/грязные, макро-зоны, управление на расстоянии |

### Проекция — камере НЕ вредит (снятое опасение)

Глубина 335L строится по ИК 850 нм; видимый свет проектора депт-канал не видит —
связка «проектор + depth» — проверенная классика (интерактивные столы на Kinect).
Реально страдает RGB-поток (картинка проецируется на руки → landmarks путаются).
Решения: (1) трекать руки по ИК/depth-стримам 335L вместо RGB — самое чистое;
(2) динамическая маска — проектор гасит UI под рукой (сегментация по глубине);
(3) светлое матовое покрытие. План: бумажный шаблон → проектор 300+ ANSI-люмен,
разовая калибровка проектор↔камера гомографией.

### Технические заметки

- Монтаж: ~60 см над столом (min-дистанция 335L = 0.17 м), FOV 90°×65° → зона ~1×0.7 м.
- Калибровка: плоскость стола на старте + гомография стол→экран один раз.
- Fingertip ID — hand-landmarks по RGB, контакт — по depth (фьюжн двух признаков).
- Матовый коврик на стол: тёмный глянец шумит в active-IR глубине.
- Active IR → работает в темноте.
- Конвейер = плагины фреймворка: depth-кадр → плоскость → палец → touch-события →
  ПК-агент (плоскость управления, без LLM).

### Оборудование для стола

| Что | ~Цена | Когда |

| Кронштейн/пантограф над столом | 1-2к ₽ | вместе с 335L |
| Матовый коврик | ~1к ₽ | вместе с 335L |

## Реализуемость: три уровня

### Уровень A — гарантированно получится (база продукта)

1. Голосовой диалог полностью локально: wake word → Whisper → LLM 14B → TTS, с barge-in.
2. Управление ПК голосом через MCP: файлы, запуск задач/тестов/обучения, git, поиск.
3. Контроль длительных задач: «скажи, когда закончится обучение» → следит и сообщает.
4. Умный дом через Home Assistant MCP (свет, розетки, климат, сценарии).
5. База знаний голосом (qex + vault) и RAG-память («запомни…» / «где лежит…»).
6. Зрение: «что ты видишь», приветствие по лицу, детекция объектов на столе.
7. Жесты 2D: медиа-контроль, свайпы, «тише/громче» ладонью.
8. Веб-дашборд с телефона (камера, статус, логи).

### Уровень B — получится с усилием (продвинутый продукт)

9. Жест-мышь (clutch + One Euro + относительный маппинг) — плоскость управления без LLM.
10. Подсознание 3B на Nano: роутер интентов, мгновенные реакции, фоновый аннотатор памяти.
11. Проактивность с rate-limit (заговорить по событию: обучение завершилось, человек ищет вещь).
12. Мультимодальный дейксис: «открой вот это» + указание/взгляд (нужны depth + синхронизация часов).
13. Идентификация по голосу (speaker verification) — ПК-команды только от владельца.
14. Взгляд как контекст: какой монитор в фокусе, детектор адресации, автоскролл.
15. Ночная консолидация памяти + рефлексия (режим «сон» из SOFTWARE_ARCHITECTURE.md).

### Уровень C — инновационные задачи (этого нет у Alexa/Google/Яндекс)

16. **Экзокортекс — журнал рабочего дня.** Подсознание непрерывно аннотирует
    происходящее (кто был, что делалось, что лежит где) → «что я делал в четверг?»,
    «когда я в последний раз калибровал камеру?». Возможно ТОЛЬКО потому,
    что всё локально — облачный ассистент такое не может по приватности.
17. **Инвентарь мастерской через зрение.** «Где мультиметр?» — ассистент видел его
    вчера на столе и помнит (детекции → RAG-память с координатами и временем).
18. **Голосовой SCADA для стенда.** Мост к Inspector_bottles (паттерн backend_ctl):
    «какой FPS на конвейере?», «есть ли брак за смену?», «останови линию» с подтверждением.
19. **Руки заняты — голос+кивок.** Пайка/сборка: «покажи следующий шаг», подтверждение
    кивком, таймер жестом — сценарий, где ассистент бьёт телефон и колонку.
20. **Обучаемые жесты on-the-fly:** «запомни этот жест = открыть терминал»
    (few-shot по landmarks, без переобучения моделей).
21. **Присутствие-охрана:** «кто подходил к столу, пока меня не было?» — лица + время,
    всё локально.

Правило порядка: A целиком → B по ценности (9-11 раньше) → C по вкусу.
C-задачи почти не требуют нового кода восприятия — они комбинируют A+B.

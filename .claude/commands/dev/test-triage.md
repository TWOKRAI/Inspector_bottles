**Opt-in bounded triage** полного тест-suite. Команда **диагностирует** падения и
**предлагает** фиксы — но **сама их не применяет**. Применение любого фикса требует
явного одобрения человека (`human-approves-any-fix`).

> **OPT-IN — не запускается автоматически.** Эта команда выполняется **только** когда
> пользователь явно вызвал `/dev:test-triage`. Она НЕ зарегистрирована ни в каком
> Claude Code hook, ни в cron/scheduled-агенте, ни в `settings.json`. «Ночной триаж» =
> человек запускает её перед уходом / в начале дня, а не harness по таймеру. Это
> сознательное ограничение (ROADMAP § E.4 «Ралф отклонён»): автономия bounded, человек
> на необратимом.

Аргумент: $ARGUMENTS — опц. подмножество (`tests/unit`, путь, `-k <expr>`). Пусто = весь suite.

## Жёсткие потолки (hard caps)

Один запуск `/dev:test-triage` ограничен — **что наступит раньше**:

- **≤ 3 итерации** диагноза (re-run suite + diagnose). Счётчик ведётся в prose-отчёте.
- **≤ 50k токенов** на весь run. Бюджет читается из той же телеметрии, что и `/dev:pipeline`
  (`data/pipeline-metrics.jsonl`, Stop-hook `token-budget-meter`). Превышение → **СТОП**,
  выведи частичный отчёт с пометкой `BUDGET-EXCEEDED`.

Потолки существуют, чтобы opt-in ночной прогон не превратился в неограниченный цикл.
**Это не autonomous runner** — `ai-judge` владеет stop-условием, команда не «крутится».

## Цикл (≤ 3 итерации)

**1. Run** — прогон suite с захватом вывода в файл:
`uv run pytest $ARGUMENTS -v 2>&1 | tee /tmp/triage_out.txt`
(нет `uv` → `pytest`; scoped, не bare из корня — см. `_stack.md`).
- Всё зелёное → **СТОП**, `VERDICT: GREEN`, отчёт «нечего триажить». Это успех, не no-op.

**2. callable gate (детерминированный)** — `python scripts/red_gate.py --report /tmp/triage_out.txt`.
Это тот же парсер, что в `/dev:pipeline` §3 (S3 RED-block), переиспользуется как сигнал
«падения настоящие, а не сломанный setup»:
- `FAILED` + `AssertionError`/`NotImplementedError` → настоящие падения, есть что диагностировать.
- `ImportError`/`SyntaxError`/collection-error → сломан **setup/окружение**, а не код под тестом →
  чини setup (env, импорты), это **не** баг-триаж. Нестандартный вывод парсер не классифицирует →
  эскалация к **ai-judge** (S3-семантика).

**3. ai-judge gate (stop-condition owner)** — запусти агента **ai-judge** (Opus, fresh context)
с машинным сигналом = `/tmp/triage_out.txt` текущей итерации + краткий diff к набору падений
прошлой итерации (имена тестов). Судья отвечает одним вердиктом:
- `VERDICT: PASS` → набор падений **изменился / есть новый actionable-сигнал** (например,
  применённый человеком фикс сдвинул картину, или flaky-тест надо подтвердить ещё одним прогоном) →
  разрешена следующая итерация.
- `VERDICT: BLOCK` → **нет прогресса** (тот же набор падений повторяется) либо сигнала недостаточно →
  **СТОП цикла**, переходи к отчёту. Это и есть защита от бесконечного триажа.

ai-judge — **bounded owner** stop-условия (см. `agents/ai-judge.md`): один сигнал → один вердикт,
он не правит код и не «гоняет» цикл сам.

**4. Diagnose (read-only)** — для каждого кластера падений запусти агента **debugger** (Sonnet)
в диагностическом режиме: воспроизвести → root cause → **предложить** фикс (набросок diff, 1-5 строк).
- debugger **не коммитит и не применяет** фикс в этом потоке — он выдаёт диагноз + предложение.
- Несколько независимых кластеров → можно параллельно (по одному debugger на кластер).

**5. Human gate (human-approves-any-fix)** — собери предложения в отчёт и **останови цикл для
решения человека**. Команда сама фикс **не применяет**. Человек выбирает и применяет одобренный
фикс отдельным шагом — через `/dev:debug` (диагноз с фиксом-в-scope) или `/dev:tdd`
(failing-тест → фикс), уже под управлением человека. После применённого фикса можно вызвать
`/dev:test-triage` снова — он начнёт с пустого счётчика итераций.

Итерация инкрементит счётчик. На 3-й итерации (или при `BUDGET-EXCEEDED`) — **СТОП** независимо
от вердикта судьи, с пометкой в отчёте.

## Отчёт

В конце выведи компактную сводку:
- `VERDICT`: `GREEN` / `TRIAGED` (есть предложения, ждут одобрения) / `BLOCKED` (ai-judge: нет прогресса) /
  `BUDGET-EXCEEDED`.
- Итераций потрачено: `N/3`. Токенов: `~K/50k`.
- Кластеры падений: тест(ы) → root cause → предлагаемый фикс (не применён).
- Следующий шаг для человека: какой `/dev:debug` / `/dev:tdd` запустить на одобренный фикс.

## Когда вызывать

- Накопились падения между сессиями — нужен системный разбор без немедленной починки.
- Подозрение на flaky — пара bounded-прогонов под надзором `ai-judge` отделит flaky от настоящих.
- Старт дня: быстрый triage «что сломалось ночью» с предложениями, но без авто-фикса.

## Когда НЕ вызывать

- Одно известное падение с понятной причиной → сразу `/dev:debug` (не нужен bounded-цикл).
- Реализация фичи / новый юнит → `/dev:tdd` или `/dev:implement` (это не триаж).
- Нужна полная фича-петля с ревью и ship → `/dev:pipeline`.
- Желание «пусть само чинит, пока не позеленеет» — **намеренно не поддерживается** (human-approves-fix).

Подмножество suite / фокус: $ARGUMENTS

Task 1.1 плана `plans/backend-ctl-proof-discipline.md` закрыт (`cdb62f24`): типизированные
обёртки `RouterStats`/`QueueDepths`/`WorkerStatus`/`MemoryStats` больше не подставляют `0`/
пустой словарь вместо отсутствующих данных. Поле без показания = `None`, его имя = в `missing`.

**Три состояния, которые нельзя схлопывать** (главная тонкость — их путают):

1. **Ключа не было** → `None` + имя в `missing`. Расхождение формы: сервер переименовал поле
   или ручка ответила не тем.
2. **Сервер прислал значение** (в т.ч. `0`, `{}`) → значение, `missing` пуст. Ноль и пустой
   словарь — валидные показания.
3. **Сервер прислал явный `null`** — только для `MemoryStats`: это ОТВЕТ «подсистема
   недоступна» по best-effort контракту `introspect.memory`. Секция `None`, но в `missing`
   НЕ попадает. Записать её в `missing` = сделать аномалией штатную норму.

**Why:** `int(stats.get("sent_ok", 0) or 0)` превращал «переименовали поле» и «ручка не
ответила» в «трафика не было» — ложь, неотличимую от факта, ради доверия к которой
инструмент и существует. Это часть цены ultra-ревью 4.5/10.

**How to apply:** добавляя поле в любую обёртку `protocol.py`, реши, к какому из трёх
состояний оно относится, и проведи приёмку ПАРОЙ (значение+`missing==[]` / переименованный
ключ→`None`+`missing`) — одиночный зелёный не отличает «прочиталось» от «подставилось».
`missing == []` на ЖИВОМ бэкенде — единственное доказательство, что имена сверены с сервером,
а не додуманы. Потребитель типизированных полей ровно один — `overview.py` (аномалия
`counter_missing`); пробы `backend_ctl/probes/*` идут мимо обёрток на сырых dict через
`unwrap(..., leaf=True)`, поэтому контракта не касаются. Связано: [[backend-ctl-signal-integrity]].

Стек хранения «на перспективу» — **embedded-first: SQLite через существующий `Services/sql` + плоские
файлы. Ноль новых зависимостей.** Отклонены: PostgreSQL (опцион по триггеру — диалектная граница уже
существует в `adapters/postgresql.py` + `core/ddl_builder.py`, её надо консервировать тестом, а не
строить), InfluxDB 3 / GreptimeDB (сервер + LSM ради ~2 строк/сек; write-amplification убивает SD-карту
Pi), Qdrant (RAM-конфликт с GPU-инференсом на Jetson), DuckDB (лишний слой).

**Why:** владелец предполагал серверный стек под многопроцессность + будущие вектора, с прицелом на SCADA
и развёртывание Windows + Jetson + Raspberry Pi. Два независимых раунда анализа по коду показали: узкое
место — не движок БД, а **картинки** (0.9-3.5 ГБ/сутки при 20 FPS против 30-40 МБ/сутки телеметрии, разница
2-3 порядка). Смена движка эту задачу не решает; решают файловая политика и ретенция.

**How to apply:** не предлагать серверные БД / Parquet / DuckDB / векторные БД без сработавшего триггера.
Триггеры (числами): →файл-БД-на-день при устойчиво >1-2 ГБ/сутки; →PostgreSQL при (а) второй машине,
пишущей по сети, (б) контрактном требовании SQL-доступа от SCADA/MES-интегратора, (в) внешней системе,
пишущей к нам; →`sqlite-vec` (не Qdrant) при первой реальной similarity-фиче ≤1 млн векторов; →`asyncua`
когда названа конкретная SCADA с OPC UA. Также вне игры: логи в БД, undo в БД (решено аудитом
`docs/audits/2026-06-18_command-undo-system.md` §4 — in-memory), рецепты в SQLite (остаются YAML ради
ruamel-комментариев и git-диффов). План декомпозиции — `plans/storage-stack-embedded-first.md`.
См. [[project-sql-insert-many-debt]].

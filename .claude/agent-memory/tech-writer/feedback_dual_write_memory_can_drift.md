В Inspector_bottles память ведётся в двух местах (правило dual-write из корневого `CLAUDE.md`): `docs/claude/memory/` (git, project+feedback) и `C:\Users\INNOTECH\.claude\projects\<hash>\memory\` (локально, + user/reference). Ожидание — что оба `MEMORY.md` растут синхронно.

На 2026-07-21 (заход Task 0.1 `backend-ctl-proof-discipline.md`) обнаружился разрыв: git-tracked `MEMORY.md` — 149 строк, локальный — только 111, при этом ХВОСТ (последние ~20 записей) у обоих совпадал дословно. То есть где-то в истории один из файлов пропустил порцию записей по центру, и с тех пор оба продолжали расти параллельно, но с разным смещением. Ни один из существующих файлов от этого не «сломан» — но слепое предположение «оба файла одной длины, вставлю по номеру строки» дало бы неверную вставку в локальный.

**Why:** dual-write полагается на дисциплину каждого агента в каждой сессии; если хотя бы раз одно из двух мест было пропущено, разрыв не самоисправляется и не бросается в глаза (оба файла продолжают выглядеть валидными по отдельности).

**How to apply:** перед добавлением новой строки в MEMORY.md (любое из двух мест) — читать фактический хвост ИМЕННО ТОГО файла, куда пишешь (`tail`/`Read` с большим offset), а не переносить номер строки или структуру из другого места. Вставлять новую запись как «последнюю строку перед концом файла» (через уникальный anchor — например, текст последней существующей записи), а не по вычисленному номеру строки. Не пытаться реконструировать/чинить исторический разрыв самостоятельно, если это не входит в задачу — это отдельная (и не факт что маленькая) задача.

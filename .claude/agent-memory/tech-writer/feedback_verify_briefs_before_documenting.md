Владелец (Two Krai, проект Inspector_bottles) при постановке задачи на документирование механизма (Task 0.2, `backend-ctl-proof-discipline.md`, 2026-07-21) явно написал: «Я перепроверил механизм по коду лично» — и всё равно попросил не доверять этому слепо, а перепроверить и сказать прямо при расхождении. Он сам поймал одну ошибку в это же утро (обвинил fence-фильтр вместо гейта топологии) только благодаря другому агенту.

При независимой трассировке (`router_manager.py`, `builtin_commands.py`, `process_manager_module/`) нашлась реальная неполнота брифа: утверждение «receive-мидлварь процесса-получателя судит трафик драйвера» верно только когда получатель — дочерний `ProcessModule`. Когда target — само `ProcessManager` (частый случай: `system_command`, `state.*`, большинство `introspect.*`), у хоста receive-мидлвари нет вовсе ни для кого (`BuiltinCommands._register_message_guards()` инстанциируется только в `process_module.py:654`, `process_manager_module/` не вызывает `add_receive_middleware` нигде) — это не «обход для драйвера», а отсутствие судьи в принципе. Бриф этого не различал.

**Why:** координатор проверяет по своей модели понимания кода в моменте; она может быть неполной даже при добросовестной проверке (особенно когда область — межпроцессная мидлварь с несколькими независимыми плоскостями). Документ, который агенты читают перед работой (AGENTS.md), кодифицирует ошибку в факт, если её не поймать здесь.

**How to apply:** для ADR/AGENTS.md/ARCHITECTURE.md по не-тривиальному механизму — всегда самостоятельно трассировать код у ключевых утверждений брифа (не только у тех, что выглядят подозрительно), особенно проверять "судит ли X" на РАЗНЫХ целях/сценариях, не только на том одном сценарии, что цитирует бриф. Расхождение — не повод молчать и не повод переписывать бриф в отчёте гладко: называть прямо, с file:line, в финальном докладе координатору.

# Семантический поиск (qex MCP) — эффективность

Цель — **быстрее и полнее** ответить на вопрос по коду, а не сделать фиксированное число вызовов. qex и Grep **дополняют** друг друга; порядок зависит от задачи.

## Что делает `search_code` (гибрид — используй осознанно)

Один вызов `mcp__qex__search_code` уже включает **оба** плеча индекса qex (после `index_codebase`):

| Плечо | Роль |
|--------|------|
| **Векторный поиск** | Запрос → эмбеддинг (Ollama) → поиск по смыслу в **Qdrant** (cosine). Хорош для перефразировок и «похожих по смыслу» фрагментов. |
| **BM25 (лексика)** | Полнотекстовый ранг по чанкам (**Tantivy**, локально). Хорош для **точных имён** символов, редких строк, ключевых слов. |
| **Итог** | **Гибридный ранжировщик** qex объединяет сигналы dense + sparse; отдельно «только вектор» или «только BM25» через MCP обычно не вызывается — **`search_code` = гибрид**. |

**Формулировки под гибрид:** смешивай **имена из кода** и **смысловое описание** в одном запросе (например: `"RouterManager send_message IPC targets"`). Чисто разговорный запрос тоже ок; для редких идентификаторов добавь их в строку явно.

Пайплайн: `docs/claude/qex/SETUP_GUIDE.md` (поток `search_code`).

## Предварительные условия

```bash
docker start qdrant   # при необходимости
ollama serve
```

### Индекс и актуальность (важно)

**Автоматически** перед каждым `search_code` ничего не обновляется: актуальность индекса зависит от того, вызовет ли агент проверку/переиндексацию и есть ли у qex фоновая синхронизация (не полагайся на неё при массовых правках).

**Перед серией семантических поисков** (и особенно во время **крупного рефакторинга**, много переименований/переносов файлов):

1. `mcp__qex__get_indexing_status()` — убедись, что индекс не пустой; если в статусе видно сильное отставание от репозитория или сомнения — переиндексируй.
2. Полное обновление: `mcp__qex__index_codebase(force=True)` — **пересборка всего индекса** (эмбеддинги через Ollama + запись в Qdrant + BM25). На большом репозитории это легко **10–30+ минут**; нагрузка на CPU/GPU и диск — нормально. Не жди завершения «между двумя промптами»: запускай **в конце дня**, на перерыве или в отдельной сессии/терминале, пока продолжаешь работать с **Grep / Read** (они всегда смотрят актуальные файлы).
3. Во время **активного** рефакторинга не гоняй `force=True` после каждой правки — слишком дорого. Схема: правишь много → **один** полный reindex, когда этап стабилизировался; до этого семантику используй как **грубый компас** или отложи.
4. Если индекс давно не трогали и правилось много файлов — один осознанный `force=True` лучше, чем доверять сильно устаревшим чанкам (в выдаче могут быть старые пути).

Slash-команда: `/qex-reindex` — тот же вызов `index_codebase(force=True)`.

## Когда начинать с `Grep`

- Известно **уникальное** имя класса, функции, константы, строки канала — и нужен **исчерпывающий** список текстовых вхождений.
- Один понятный файл + «найди в нём» → `Read` / `Grep` по пути.
- Чистый синтаксический/regex-поиск без смыслового «что ещё похоже».

После Grep: если объект **публичный API**, **межпроцессный контракт** или правка **широкая** — добавь **один** `search_code`, чтобы поймать косвенные места (строки в конфиге, динамический импорт, регистрация по имени, обёртки).

---

## Когда начинать с `search_code`

- Вопрос сформулирован **по смыслу**, имени символа нет или оно неуникально.
- Нужны **смежные** места: «что трогает канал / процесс / тип сообщения», «как устроена цепочка …».
- Перед **крупным** рефакторингом или сменой поведения — обзор темы одним-двумя запросами.
- После Grep результат **подозрительно узкий** (мало попаданий при «важном» символе) — проверь семантикой.

## Как формулировать запрос к qex

`mcp__qex__search_code(query="...")` — русский или английский. Это и есть использование **векторного + BM25 + гибридного** ранжирования (см. блок выше).

- С именем **и** намерением: `"кто вызывает или импортирует RouterManager"`, `"создание и запуск ProcessManagerProcess"`.
- По поведению: `"graceful shutdown процесса"`, `"маршрутизация сообщения по channel"`.
- Без имени: `"регистрация процесса в лаунчере"`.

**Лимит:** по умолчанию **один** запрос; второй — только если топ явно мимо темы (переформулируй: синонимы, английский, «caller», «instantiate»).

Открой **3–5** лучших совпадений `Read`; затем при необходимости **Grep** по конкретным символам из найденного.

---

## Сводное правило

| Ситуация | Сначала |
|----------|---------|
| Точное имя, нужен полный список вхождений | `Grep` → при риске для API/IPC — +1 qex |
| Размытый вопрос, обзор, влияние по смыслу | `search_code` → затем `Grep` для проверки символов |
| Уже прочитал файл, вопрос локальный | `Read` / `Grep`, qex не обязателен |

---

## Чего избегать

- Подряд несколько близких запросов qex без новой информации.
- Полагаться **только** на Grep при ответе «всё, что сломается от смены контракта сообщений / публичного API».
- Вызывать qex, если ответ уже однозначен после `Grep` + `Read` и риск побочных эффектов низок.

# `.claude/skills/` — Project Claude Skills

[Claude Skills](https://docs.claude.com/en/docs/agents-and-tools/skills) are small, focused capabilities Claude invokes on demand. They live in this directory and are loaded automatically when Claude Code starts.

## Included in this seed

Six framework-agnostic skills. Four curated from [mattpocock/skills](https://github.com/mattpocock/skills) (MIT); two written for this seed to close gaps before `/plan` and before `/ship`:

| Skill | What it does | When it fires |
|-------|--------------|---------------|
| **`caveman/`** | Ultra-compressed communication, ~75% fewer tokens, no loss of technical substance. Persists across the session until `stop caveman`. | User says "caveman mode", "be brief", "less tokens", or types `/caveman`. |
| **`grill-me/`** | Relentless interview about a plan/design — walks every branch of the decision tree, one question at a time, with a recommended answer for each. | User wants a plan stress-tested, says "grill me", or has a vague design that needs cornering. |
| **`zoom-out/`** | Forces the agent to step up an abstraction layer and produce a map of relevant modules and callers using the project's domain glossary. `disable-model-invocation: true` — user-invoked only via `/zoom-out`. | You're touching unfamiliar code and need the bigger picture before editing. |
| **`prototype/`** | Routes between two throwaway-prototype branches: terminal app for state/logic questions ([`prototype/LOGIC.md`](prototype/LOGIC.md)) or several toggleable UI variations ([`prototype/UI.md`](prototype/UI.md)). | User says "prototype this", "let me play with it", "try a few designs", or wants to sanity-check a state model before committing. |
| **`brainstorm/`** | Pre-`/plan` option generation: 2–4 distinct approaches with mechanism/cost/risk/reversibility, then a recommendation. Distinct from `grill-me` (attacks existing plan); brainstorm is for when no plan exists yet. | User says "what could we do about", "how should we approach", "give me options for", "I'm not sure how to", "brainstorm", "/brainstorm". |
| **`verify-done/`** | Pre-`/ship` verification gate. Forces "did the fix actually work" — reproduces original problem, runs entry point end-to-end, checks edge cases & side effects. Tests passing ≠ done. | Agent about to claim "done"/"ready"/"shipped"; before `/ship` / `/pipeline` finishes; user says "verify", "is it really done". |

**Activation order (typical):**

```
fuzzy idea  → brainstorm → /plan → grill-me → /implement → /test → verify-done → /ship
            (options)             (stress)                          (truth check)
```

These complement — they don't replace — the agent roster in `.claude/agents/` (Director/Manager/Developer/Tester/Reviewer/etc.). Skills are inline instructions; agents are isolated processes with their own context and model.

## graphify — not in this directory

[graphify](https://github.com/safishamsi/graphify) is a knowledge-graph skill that ships as a **CLI tool with native deps** (tree-sitter, networkx, scipy). It must be installed via `uv tool install ...` and writes its own `SKILL.md` into `~/.claude/skills/graphify/` system-wide. Seed-managed copy would be wrong because the skill file is regenerated by the CLI on each install/update.

See [`../mcp/graphify/SETUP_GUIDE.md`](../mcp/graphify/SETUP_GUIDE.md) for install. The `.claude/` integration is **user-global**, not project-local — graphify is registered once per machine and available in every project.

## Skill structure (for your own)

```
.claude/skills/<skill-name>/
└── SKILL.md
```

`SKILL.md` frontmatter:

```yaml
---
name: <skill-name>          # kebab-case, must match folder
description: >
  One-line description of when this skill activates.
  Claude reads this to decide whether to invoke — be specific
  about triggers, not just capability.
# Optional: prevent automatic LLM invocation (user-only via /skill-name)
# disable-model-invocation: true
---

# Full skill prompt and workflow goes here
```

## When to write a skill

- A workflow you find yourself triggering manually ≥3 times
- A pre-flight or post-flight checklist that needs to run consistently
- A domain-specific task (e.g. "lint our markdown wiki structure") that doesn't fit existing slash commands

## When NOT to write a skill

- One-shot task → just describe it in chat
- Already covered by an existing slash command (`/plan`, `/review`, etc.)
- Pure code change → use a slash command + Developer agent instead

## Skill vs slash command vs agent

| | Slash command | Skill | Agent |
|--|---------------|-------|-------|
| Invocation | Explicit `/command` | Auto-invoked when description matches user intent (unless `disable-model-invocation`) | Spawned via Task tool with explicit type |
| Location | `.claude/commands/` | `.claude/skills/` | `.claude/agents/` |
| Scope | One specific action | A behavioural pattern or workflow | A whole role with its own context window |
| Trigger | User types it | Claude decides from `description` | Parent agent / orchestrator |
| Model | Inherits | Inherits | Defined in agent frontmatter |

Pick the lightest tool that fits — most repeatable tasks belong in `commands/`, not `skills/`.

# aggregate_context — per-module knowledge aggregator

Generic фреймворк для per-module markdown-файлов (CONTEXT.md, DECISIONS.md).
Сканирует проект, собирает per-module знания в единый root-registry
с marker-based replacement.

**Зачем:** дать агенту (и человеку) одну точку входа в архитектурное знание
проекта. Когда агент берёт task в модуле X — он первым делом читает
`<module>/CONTEXT.md` (или `<module>/DECISIONS.md`) и знает Gotchas, design
decisions, glossary. Без этого агент идёт в код «слепым».

Pattern портирован из Inspector_bottles (`scripts/sync/`), обобщён.

## Конвенция

Папка с `CONTEXT.md` и/или `DECISIONS.md` = «модуль» (auto-discovery,
без ручного реестра). Имя модуля = имя папки.

### `CONTEXT.md` — лёгкий per-module документ

Все секции опциональные. Aggregator считает наличие/отсутствие.

```markdown
# <module> — Context

## Purpose         (1-2 фразы что и зачем)
## Key decisions   (ссылки на ADR / DECISIONS.md этого модуля)
## Gotchas         (footguns, неочевидные грабли)
## Glossary        (local terms)
## Open questions  (что не решено)
## Migration notes (важные миграции)
```

Шаблон: [`templates/CONTEXT.template.md`](../../templates/CONTEXT.template.md).

### `DECISIONS.md` — формальные ADR (для тяжёлых модулей)

Заголовки нумерованы: `## ADR-{CODE}-NNN: title`. Код модуля либо из
frontmatter (`module_code: BM`), либо auto-derived (uppercase initials).

```markdown
---
module_code: BM
---

# base_manager — Архитектурные решения

## ADR-BM-001: <title>
**Дата:** YYYY-MM-DD
**Статус:** Принято

### Контекст
...

### Решение
...

### Альтернативы
...

### Последствия
...
```

Шаблон: [`templates/DECISIONS.template.md`](../../templates/DECISIONS.template.md).

### `PROJECT_CONTEXT.md` — root registry (генерируется)

В корне `docs/PROJECT_CONTEXT.md` (создаётся из шаблона
`templates/PROJECT_CONTEXT.template.md`). Содержит три блока между маркерами:

```html
<!-- CONTEXT-INDEX:BEGIN — generated by aggregate_context — DO NOT EDIT -->
<!-- CONTEXT-INDEX:END -->

<!-- ADR-CODES:BEGIN — generated by aggregate_context — DO NOT EDIT -->
<!-- ADR-CODES:END -->

<!-- ADR-INDEX:BEGIN — generated by aggregate_context — DO NOT EDIT -->
<!-- ADR-INDEX:END -->
```

Текст вне маркеров — рукописный, sync не трогает.

## CLI

```bash
# Write-режим: обновить docs/PROJECT_CONTEXT.md
python -m scripts.aggregate_context

# CI-режим: показать diff и exit 1 при дрифте
python -m scripts.aggregate_context --check

# Один sync-модуль (render_context | render_adr)
python -m scripts.aggregate_context --only render_context

# Список активных модулей
python -m scripts.aggregate_context --list

# Альтернативный target
python -m scripts.aggregate_context --target docs/ARCHITECTURE.md
```

Slash-command обёртка: `/core:quality:sync-context`.

### Добавить новый sync-модуль

Реализуй `SyncModule` Protocol:

```python
from pathlib import Path

class RenderGotchas:
    name = "render_gotchas"
    description = "Сводка всех Gotchas из CONTEXT.md модулей"

    def __init__(self, root: Path, target_file: Path):
        self.root = root
        self.target_file = target_file

    def render(self) -> dict[Path, dict[str, str]]:
        # ... парсинг + рендеринг
        return {self.target_file: {"GOTCHAS-INDEX": content}}
```

Зарегистрируй в `__main__.py:_build_modules()`. Добавь маркер в
`PROJECT_CONTEXT.template.md`. Покрой тестами в `tests/test_render_gotchas.py`.

## Когда использовать в проекте

- **≥3 нетривиальных модулей** — без сводного registry агент тратит токены
  на чтение исходников чтобы понять архитектуру
- **Долгоживущий проект** — design decisions накапливаются, забываются,
  переоткрываются. ADR per-module — реестр.
- **Несколько контрибьюторов / агентов** — общая ground truth для
  «почему здесь так»

## Когда НЕ использовать

- < 5 файлов целиком — overhead не окупается
- Прототип / spike — нечего фиксировать
- Один автор, одна сессия работы — память агента справится

## Связь с другими частями seed

- `commands/dev/adr.md` — slash-команда для создания **глобальных** ADR
  (`docs/decisions/NNNN-*.md`). Per-module ADR создаются через
  `templates/DECISIONS.template.md` руками или агентом.
- `skills/module-contract` — при создании нового публичного модуля
  рекомендует создать CONTEXT.md если есть Gotchas или Key decisions.
- `commands/quality/sync-context.md` — slash-обёртка над CLI этого пакета.

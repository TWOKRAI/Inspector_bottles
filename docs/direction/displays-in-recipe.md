# Дисплеи как часть рецепта

Рецепт становится самодостаточным: определения дисплеев хранятся внутри рецепта,
а не в глобальном файле. Переключение рецепта автоматически меняет набор дисплеев.

---

## 1. Назначение

Сейчас все дисплеи описаны в глобальном `displays.yaml` — они общие для всех рецептов.
Это делает рецепт неполным: взяв только YAML-файл рецепта, невозможно запустить систему
без внешнего файла дисплеев.

После этой фичи рецепт содержит всё необходимое:
- **определения дисплеев** (размер, формат, параметры отображения) — секция `displays`
- **привязки выходов нод к дисплеям** (node_id → display_id) — секция `blueprint.displays`

Глобальный `displays.yaml` упраздняется.

---

## 2. Модель данных рецепта

### Структура YAML на диске

```yaml
name: color_inspect
version: 3
description: "Инспекция по цвету"

# --- НОВАЯ секция: определения дисплеев рецепта ---
displays:
  - id: main
    name: Основной дисплей
    width: 1280
    height: 720
    format: BGR           # BGR | RGB | GRAY | RGBA
    fps_limit: 30.0       # 0.0 = без ограничений
    ring_buffer_blocks: 3 # количество SHM-блоков
    # параметры отображения (render-слой, не влияют на SHM):
    position:             # позиция окна на экране, пиксели
      x: 100
      y: 50
    fit: contain          # contain | cover | stretch | none
    scale: 100            # масштаб в %; 100 = 1:1, хранится как int, минимум 10
    rotate: 0             # 0 | 90 | 180 | 270
    flip: none            # none | horizontal | vertical | both
    crop:                 # прямоугольник обрезки; null = без обрезки
      x: 0
      y: 0
      w: 1280
      h: 720

blueprint:
  name: color_inspect
  processes:
    - ...
  wires:
    - ...
  # --- СУЩЕСТВУЮЩАЯ секция: привязки нод → дисплеям ---
  displays:
    - node_id: painter.contour_draw.frame
      display_id: main
      # display_name убран — имя резолвится по display_id из секции displays
```

### Две секции — разные роли

| Секция | Содержимое | Кто правит |
|--------|-----------|------------|
| `displays` (top-level) | Определения: id, name, размер, формат, параметры preview | Вкладка «Дисплеи» |
| `blueprint.displays` | Привязки: node_id → display_id | Редактор графа (Pipeline) |

### Поля определения дисплея

| Поле | Тип | По умолчанию | Описание |
|------|-----|--------------|----------|
| `id` | string | — | Уникальный ID (ключ ссылки), адресация по нему |
| `name` | string | — | Отображаемое имя; если пустое — показывается `id` |
| `width` | int, px | 1280 | Ширина SHM-сегмента |
| `height` | int, px | 720 | Высота SHM-сегмента |
| `format` | enum | BGR | Формат пикселей: BGR, RGB, GRAY, RGBA |
| `fps_limit` | float | 30.0 | Лимит FPS (0.0 = без лимита) |
| `ring_buffer_blocks` | int | 3 | Количество блоков ring-buffer |
| `position.x` | int, px | 0 | Позиция окна по горизонтали, персистируется в рецепт |
| `position.y` | int, px | 0 | Позиция окна по вертикали, персистируется в рецепт |
| `fit` | enum | contain | Режим вписывания кадра |
| `scale` | int, % | 100 | Масштаб preview в процентах; минимум 10, максимум 1000, шаг 10 |
| `rotate` | enum | 0 | Поворот preview: 0/90/180/270 |
| `flip` | enum | none | Отражение: none/horizontal/vertical/both |
| `crop` | объект | null | Обрезка до вывода — пиксели SHM-кадра {x, y, w, h} |

> `name` живёт только в определении (`displays[].name`). Адресация везде — по `id`.
> `display_name` в `blueprint.displays` (DisplayInstance) **удалён** — дублировал имя из определения.

## 3. Изменения domain-модели

### Entity DisplayDefinition

Новая Pydantic-сущность `DisplayDefinition` (frozen, наследует `SchemaBase`):
- Поля: `id`, `name`, `width`, `height`, `format`, `fps_limit`, `ring_buffer_blocks` + render-параметры (`position{x,y}`, `fit`, `scale`, `rotate`, `flip`, `crop{x,y,w,h}`).
- Отличается от `DisplayInstance` (привязка `node_id → display_id`): `DisplayDefinition` описывает сам дисплей, `DisplayInstance` — связь ноды с дисплеем.
- Конфигурация: `extra="forbid"`, все поля со значениями по умолчанию кроме `id`.

### Поле в Recipe

- В `Recipe` добавляется поле `displays: tuple[DisplayDefinition, ...]` (по умолчанию пустой кортеж).
- Конфигурация `Recipe` сейчас имеет `extra="forbid"` — добавление поля без правок domain-кода приведёт к ошибке парсинга. Поле обязано быть объявлено в модели явно.
- `Recipe.from_dict` должен парсить top-level ключ `displays` и передавать его в конструктор `DisplayDefinition`.
- `Recipe.to_dict` должен сериализовать `displays` обратно в список dict (через `to_dict` каждого `DisplayDefinition`).

### RecipeStoreFromManager._denormalize

- Метод `_denormalize` собирает структуру рецепта перед сохранением. Новое поле `displays` обязано войти в его whitelist — иначе определения дисплеев будут потеряны при сохранении через RecipeStore.

### Инвариант уникальности id

- `Recipe.displays[].id` должны быть уникальны в пределах рецепта.
- Валидация выполняется на domain-уровне (`@model_validator` или `__post_init__`), не только при парсинге YAML.
- Загрузка рецепта с дублирующимся `id` → ошибка с указанием конфликтующего значения.

### Dict-at-Boundary контракт (boot и apply_topology)

- `unwrap_recipe` (launch.py) сейчас тянет только `blueprint` и `display_bindings`. Top-level секция `displays` будет потеряна при boot-сборке без явного проброса.
- Требование: при boot и при `apply_topology` — `displays` (список dict определений) передаётся в backend вместе с blueprint. Формат на границе — `list[dict]`, не Pydantic-объекты.
- Backend использует переданные определения для наполнения `DisplayRegistry`.

### Затронутые тесты

При реализации domain-изменений потребуется обновить:
- `test_entities_roundtrip` — добавить round-trip с полем `displays`.
- `test_recipe_store` — проверить, что `_denormalize` сохраняет определения.
- `test_demo_recipe` — убедиться, что демо-рецепты проходят новую схему.

## 4. Жизненный цикл DisplayRegistry при переключении рецепта

### Ownership: кто и когда наполняет реестр

- **Backend** наполняет `DisplayRegistry` в рамках `apply_topology` (транзакция switch), не в `set_active`.
- Привязка re-populate к `set_active` — нежелательна: `set_active` только фиксирует выбор, не запускает процессы.
- **GUI** не наполняет реестр самостоятельно. GUI подписывается на событие `RecipeActivated` и перечитывает состояние реестра для обновления вкладки «Дисплеи».

### Метод reload(entries) в DisplayRegistry

`DisplayRegistry` должен иметь метод `reload(entries: list[dict])`, выполняющий следующий порядок:

1. Отправить сигнал закрытия открытым окнам превью дисплеев, которых нет в `entries`.
2. Остановить SHM-каналы старых записей (cleanup SHM-сегментов).
3. Очистить внутренний реестр (`clear`).
4. Зарегистрировать новые записи из `entries`.
5. Выделить SHM-сегменты для новых записей.

Порядок относительно процессов:
- `reload` вызывается **после** `stop` старых процессов и **до** `start` новых.
- SHM для нового рецепта выделяется внутри `reload` — до запуска воркеров-потребителей.

### Персистентность: только рецепт

- Определения дисплеев хранятся в рецепте (`recipe_store`), НЕ в `displays.yaml`.
- `displays.yaml` упраздняется как источник истины и удаляется после миграции.
- При сохранении формы дисплея GUI вызывает `recipe_store.save(recipe)` — отдельного файла дисплеев нет.

## 5. Зависимости и порядок реализации

### Hard prerequisite

**`recipe-orchestrator-unify` Phase 2 (`PM.apply_topology`)** должен быть DONE до начала реализации этой фичи.

Re-populate реестра встраивается в транзакцию `apply_topology`. Без готовой Phase 2 — нет точки для корректного вызова `DisplayRegistry.reload`.

### Soft prerequisite

**Phase 3 (`recipe-orchestrator-unify` — RecipeManager как владелец активного рецепта, развязка «активация ≠ применение»)** не является жёстким блокером, но без неё потребуется переделка при дальнейшем развитии.

Рекомендация: дождаться Phase 2, желательно Phase 3, и только затем начинать реализацию фичи.

### Явный запрет: раньше Phase 2 не начинать

Реализация фичи (domain-модель, `reload`, GUI-вкладка, миграция) **начинается после того, как Phase 2 unify помечена как DONE**.

## 6. Сценарии пользователя

### 6.1 Создать новый дисплей

1. Открыть вкладку **«Дисплеи»**.
2. Заполнить форму: ID, название, ширину, высоту, формат, FPS limit, ring buffer.
3. Нажать **«Создать»**.
4. Дисплей появляется в списке карточек и записывается в секцию `displays` активного рецепта.

> Дисплей создаётся независимо от привязки — он может существовать без назначенного выхода ноды.

### 6.2 Задать параметры отображения (fit, scale, rotate, flip, crop)

1. Выбрать карточку дисплея.
2. В форме появляется раздел **«Параметры отображения»** (ниже основных полей).
3. Изменить нужные значения.
4. Нажать **«Сохранить»** или **«Создать»** — изменения записываются в рецепт.
5. Открыть **превью** кнопкой «Открыть превью» — результат виден немедленно.

### 6.3 Увидеть результат трансформации в preview

- Изменение `fit` / `scale` / `rotate` / `flip` / `crop` мгновенно применяется в окне превью.
- SHM-кадр не меняется — рендер-слой применяет трансформации только к отображению.
- Если превью уже открыто, оно обновляется без перезапуска.

### 6.4 Привязать выход ноды к дисплею

1. Открыть вкладку **«Pipeline»** (редактор графа).
2. Перетащить или выбрать дисплей из палитры правой панели; узел подписан именем дисплея (fallback `id`).
3. Соединить выход нужной ноды (например `painter.contour_draw.frame`) с блоком дисплея.
4. Сохранить граф — в секцию `blueprint.displays` записывается `{node_id, display_id}`.

> Привязать можно только дисплей, определённый в секции `displays` текущего рецепта.

### 6.5 Переключить рецепт — набор дисплеев сменился

1. Открыть вкладку **«Рецепты»** и активировать другой рецепт.
2. Backend выполняет `apply_topology`; внутри — `DisplayRegistry.reload` из `displays` нового рецепта.
3. Вкладка «Дисплеи» обновляется по событию `RecipeActivated` и показывает только дисплеи нового рецепта.
4. Открытые окна превью обрабатываются по правилу:
   - Окна дисплеев, которых **нет в новом рецепте** — закрываются автоматически.
   - Окна дисплеев с **совпадающим `id`** — переподключаются к новым кадрам (окно не закрывается, источник сменяется).

### 6.6 Дисплей без привязки

Дисплей в `displays` рецепта может не иметь ни одной записи в `blueprint.displays`.
Он отображается в списке вкладки «Дисплеи», его можно открыть в превью (канал будет пустым),
но при запуске системы он не получает кадры.

## 7. Render-pipeline (конвейер отображения)

### Компонент

Preview открывается как **отдельное окно** (`frontend/widgets/displays/`, функция `open_for_display`).
Это НЕ `ImagePanelWidget` и НЕ `DisplaySlot` (эти компоненты предназначены для встроенных in-app панелей и поддерживают только `KeepAspectRatio`).

### Конвейер трансформаций

```
SHM-кадр (raw numpy array)
  → numpy slice / crop    (обрезка по {x, y, w, h} в пикселях SHM-кадра)
  → cv2.resize            (scale в %)
  → cv2.rotate / flip     (поворот и отражение через OpenCV)
  → QImage                (конвертация BGR→RGB при необходимости)
  → QPixmap               (для Qt-рендеринга)
  → fit в QLabel          (вписывание в контейнер окна по выбранному режиму fit)
  → вывод на экран
```

Трансформации выполняются через **OpenCV/numpy на сыром кадре**, не Qt-transforms на QPixmap. При 22 FPS и разрешении 1280×720 суммарные затраты на трансформации составляют порядка 1–2 мс — реалистично без пропуска кадров.

### Режимы fit

| Режим | Поведение |
|-------|-----------|
| `contain` | Кадр вписывается целиком, пропорции сохранены; боковые поля заполняются фоном (по умолчанию чёрный) |
| `cover` | Кадр заполняет всё окно, пропорции сохранены; края обрезаются |
| `stretch` | Кадр растягивается на весь контейнер без сохранения пропорций |
| `none` | Кадр отображается в оригинальном размере без масштабирования; если не влезает — появляются полосы прокрутки или обрезается |

### scale

Значение в процентах (int). Диапазон: **10–1000%**, шаг: **10%**, по умолчанию: **100%** (= 1:1).
Минимум 10 — значение 0% бессмысленно (кадр не виден). Применяется поверх `fit`.
В YAML хранится как целое число (например `scale: 100`).

### rotate

Кадр поворачивается на 0 / 90 / 180 / 270 градусов по часовой стрелке через `cv2.rotate`.
Поворот влияет на компоновку: при 90/270 ширина и высота кадра меняются местами.

### flip

Применяется после `rotate` через `cv2.flip`:
- `none` — без отражения
- `horizontal` — зеркало по вертикальной оси
- `vertical` — зеркало по горизонтальной оси
- `both` — оба отражения

### crop

Обрезка задаётся в **пикселях SHM-кадра** (абсолютные координаты). Формат `{x, y, w, h}`:

| Поле | Описание |
|------|----------|
| `x` | Левый край вырезаемой области, пиксели |
| `y` | Верхний край, пиксели |
| `w` | Ширина области, пиксели |
| `h` | Высота области, пиксели |

`null` (или отсутствие поля) = обрезка отключена.

### Порядок применения трансформаций

```
SHM-кадр (raw)
  → crop      (обрезка, пиксели SHM-кадра)
  → scale     (масштабирование в %)
  → rotate    (поворот 0/90/180/270°)
  → flip      (отражение)
  → fit       (вписывание в контейнер окна preview)
  → вывод на экран
```

## 8. Вкладка «Дисплеи» (recipe-scoped)

### Фазы UI

Вкладка реализуется в двух фазах:

**Phase 1 (MVP):**
- Список карточек (паттерн `ProcessCard` / `SectionedForm`) + форма с render-параметрами.
- Persist определений в рецепт.
- Миграция старых YAML (перенос из `displays.yaml`, удаление `display_name`).
- Preview-трансформации через OpenCV.

**Phase 2 (после MVP):**
- Поле расстановки (canvas) с position/наложением.
- Drag-and-drop карточек по canvas + pin позиции.
- Тумблер «Без наложения» (режим плитки).

> Текущая базовая структура вкладки — `BaseListNavTab` (список + форма). Canvas-расстановка несовместима с этим компонентом и реализуется в Phase 2 как отдельный компонент замены.

### Структура (Phase 1)

```
[ Кнопки действий ] | [ Область карточек дисплеев ] | [ Форма ]
     160 px                   230 px                    остаток
```

### Область карточек (левая панель)

- Каждый дисплей отображается **карточкой** с рамкой и заголовком-вывеской, где написано имя дисплея (fallback: `id`).
- Стиль карточки — переиспользовать паттерн **ProcessCard** (`frontend/widgets/tabs/processes/widgets/process_card.py`: QFrame StyledPanel + header QLabel с objectName, индикатор, кнопки) и/или **SectionedForm** (`frontend/widgets/primitives/sectioned_form.py`: секции QGroupBox). Новый компонент рамки не изобретать.
- Список показывает дисплеи **только активного рецепта**; при смене рецепта перестраивается автоматически.
- Если рецепт не активирован — список пуст.

### Поле расстановки и опция наложения (Phase 2)

- **Тумблер «Без наложения»** (режим плитки): при включении дисплеи авто-примыкают друг к другу без перекрытий. При выключении — наложение разрешено, дисплеи могут перекрываться.
- Позиции дисплеев (`position.x`, `position.y`) персистируются в рецепт.
- Ручное перетаскивание карточек по полю расстановки + возможность «зафиксировать» (pin) позицию.

### Имя дисплея — везде как подпись

Имя дисплея показывается:
- на карточке вкладки «Дисплеи» (заголовок),
- в узлах/привязках вкладки «Pipeline»,
- **как надпись на самом окне preview** (оверлей/заголовок).

Если `name` пустое — везде подставляется `id`.

### Форма (правая панель)

**Базовые параметры:**

| Поле | Элемент | Диапазон |
|------|---------|----------|
| ID | QLineEdit (read-only при выборе) | slug, уникален в рецепте |
| Название | QLineEdit | — |
| Ширина | QSpinBox | 1..7680 px |
| Высота | QSpinBox | 1..7680 px |
| Формат | QComboBox | BGR / RGB / GRAY / RGBA |
| FPS limit | QDoubleSpinBox | 0.0..240.0 (0 = без лимита) |
| Ring buffer | QSpinBox | 1..32 |

**Параметры отображения (render-слой):**

| Поле | Элемент | Диапазон / значения |
|------|---------|---------------------|
| Позиция X | QSpinBox | 0..7680 |
| Позиция Y | QSpinBox | 0..7680 |
| fit | QComboBox | contain / cover / stretch / none |
| scale | QSpinBox | 10..1000 %, шаг 10, default 100 |
| rotate | QComboBox | 0° / 90° / 180° / 270° |
| flip | QComboBox | none / horizontal / vertical / both |
| crop X | QSpinBox | 0..width |
| crop Y | QSpinBox | 0..height |
| crop W | QSpinBox | 1..width |
| crop H | QSpinBox | 1..height |

Галочка «Обрезка включена» включает/выключает секцию crop.

### Кнопки действий

| Кнопка | Активна | Действие |
|--------|---------|----------|
| Создать | всегда (с правом `tabs.displays.edit`) | Создать дисплей из формы |
| Удалить | при выборе | Удалить из рецепта |
| Дублировать | при выборе | Копировать с суффиксом `_copy` |
| Открыть превью | при выборе | Открыть окно превью SHM-канала |

## 9. Инварианты и ограничения

1. **Рецепт самодостаточен:** секция `displays` — единственный источник определений дисплеев. Глобального `displays.yaml` нет.

2. **Валидация ссылок:** каждый `display_id` в `blueprint.displays` обязан присутствовать в `recipe.displays`. Загрузка рецепта с нарушением → ошибка с указанием недопустимого `display_id`.

3. **`width` / `height` задают SHM-сегмент:** эти поля определяют физический размер кадра в разделяемой памяти. Параметры render-слоя (`fit`, `scale`, `rotate`, `flip`, `crop`) не влияют на размер SHM.

4. **Render-слой работает только в preview:** трансформации применяются в окне превью GUI. Кадр, который процессы-потребители читают из SHM, не трансформируется.

5. **Дисплей без привязки допустим:** создать дисплей можно до того, как в графе есть нода, которую нужно к нему привязать. Такой дисплей регистрируется в реестре, но не получает кадры.

6. **`position` персистируется:** поле `position` задаёт начальное положение окна превью и сохраняется в рецепт. Ручное перемещение окна может обновлять значение (задел для Phase 2).

7. **Уникальность `id` в рамках рецепта:** два дисплея с одинаковым `id` в одном рецепте — ошибка domain-валидации (не только YAML-парсинга).

8. **Без жёсткого лимита на число дисплеев:** лимит не enforce'ится ошибкой. Практический ориентир — до ~21 дисплея; при большем числе производительность не гарантирована.

9. **`display_name` в `DisplayInstance` удалён:** адресация исключительно по `display_id`; имя резолвится из `displays[].name` по `id`. Удаление — отдельный шаг (Phase 0/pre-migration), см. раздел 10.

10. **Имя в YAML — только в `displays[].name`:** поле `name` не дублируется ни в каком другом месте рецепта.

11. **scale ≥ 10%:** значение 0% бессмысленно (кадр не виден). Минимум зафиксирован на domain-уровне и в UI (QSpinBox minimum = 10).

12. **Dict-at-Boundary:** при boot и `apply_topology` определения дисплеев передаются в backend как `list[dict]`. Pydantic-объекты не пересекают границу процесса.

## 10. Миграция старых рецептов

### Phase 0: удаление display_name (pre-migration, отдельный шаг)

**Почему отдельный шаг:** поле `display_name` присутствует в живом коде:
- `display.py`, `color_inspect.yaml`, `region_pipeline.yaml` (поле в YAML),
- Pipeline editor (`display_node_item.py`, `io.py`) — читает и показывает имя.

`DisplayInstance` имеет `extra="forbid"` — при удалении поля без миграции существующие YAML **немедленно сломаются** при загрузке.

Шаги Phase 0:
1. Pipeline editor переводится на резолвинг имени через `recipe.displays[].name` по `display_id` (fallback: `id`).
2. Из `DisplayInstance` удаляется поле `display_name`.
3. YAML мигратор (`update_yaml_preserving`) убирает `display_name` из всех `blueprint.displays` записей в рецептах.
4. Phase 0 коммитится и проходит тесты до начала Phase 1.

### Phase 1: перенос определений из displays.yaml в рецепты

Разовый мигратор:
1. Читает глобальный `displays.yaml`.
2. Для каждого рецепта: копирует определения дисплеев, на которые есть ссылки в `blueprint.displays`, в новую секцию `displays` рецепта.
3. Добавляет render-параметры со значениями по умолчанию (`fit: contain`, `scale: 100` и т.д.).
4. Перезаписывает файлы рецептов через `update_yaml_preserving` (ruamel round-trip, merge).
5. Глобальный `displays.yaml` переименовывается в `displays.yaml.bak` и помечается устаревшим.

> **Обязательно:** мигратор использует `update_yaml_preserving` (ruamel round-trip), НЕ `yaml.safe_dump`. Использование `yaml.safe_dump` приведёт к полной перезаписи файла с потерей комментариев и форматирования (был прецедент в ветке `recipe-v3-engine-decouple`).

### Пример: `color_inspect.yaml` после миграции

```yaml
name: color_inspect
version: 3

displays:
  - id: main
    name: Основной дисплей
    width: 1280
    height: 720
    format: BGR
    fps_limit: 30.0
    ring_buffer_blocks: 3
    position: {x: 0, y: 0}
    fit: contain
    scale: 100
    rotate: 0
    flip: none

blueprint:
  ...
  displays:
    - node_id: painter.contour_draw.frame
      display_id: main
      # display_name убран
```

## 11. Решения по параметрам (зафиксировано 2026-06-07)

Все вопросы закрыты. Решения обязательны к реализации.

| # | Параметр / тема | Решение |
|---|----------------|---------|
| 1 | Единицы `crop` | Абсолютные **пиксели SHM-кадра**. Формат `{x, y, w, h}`. Нормализация 0..1 не применяется. |
| 2 | `scale` — диапазон и формат | **10–1000 %**, шаг 10, default 100 (= 1:1). Минимум 10 (0% бессмысленно). В YAML — **int**. В UI — QSpinBox. |
| 3 | Порядок трансформаций | `crop → scale → rotate → flip → fit`. Трансформации через OpenCV/numpy, не Qt-transforms. |
| 4 | Превью при переключении рецепта | Вариант **А**: окна дисплеев без совпадающего `id` закрываются; совпадающие `id` переподключаются без закрытия. |
| 5 | Лимит числа дисплеев | **Без жёсткого лимита**. Мягкий ориентир ~21 дисплей — не enforce'ить ошибкой. |
| 6 | `position` и расстановка | Координаты персистируются в рецепт. Canvas + drag + тумблер «без наложения» — Phase 2. |
| 7 | `name` и `display_name` | `name` живёт **только** в `displays[].name`. `display_name` из `DisplayInstance` удаляется в Phase 0 (отдельный шаг с мигратором). |
| 8 | Ownership реестра | Backend наполняет `DisplayRegistry` в `apply_topology`, не в `set_active`. GUI подписан на `RecipeActivated`. |
| 9 | Prerequisite | Hard: `recipe-orchestrator-unify` Phase 2 DONE. Soft: Phase 3 (желательно). |
| 10 | Render-компонент | Отдельное окно `open_for_display`, не `ImagePanelWidget`/`DisplaySlot`. |
| 11 | Мигратор YAML | `update_yaml_preserving` (ruamel round-trip), НЕ `yaml.safe_dump`. |

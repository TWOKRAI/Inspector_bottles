# Dev Company — Агентная система разработки

## Концепция

Dev Company — система агентов Claude Code, организованная как компания разработки.
Каждый агент имеет свою роль, модель (Opus/Sonnet/Haiku) и зону ответственности.
Director (главный разговор с Opus) управляет всей командой.

```
┌─────────────────────────────────────────────────┐
│  DIRECTOR (Opus) — CEO                          │
│  = главный разговор Claude, НЕ агент            │
│  Управляется через .claude/CLAUDE.md            │
│  Анализ задачи → выбор стратегии → оркестрация  │
└────────┬───────────────┬────────────────────────-┘
         │ Agent tool    │ Agent tool
    ┌────▼────┐     ┌────▼─────┐
    │ MANAGER │     │ TEAMLEAD │
    │ (Opus)  │     │ (Opus)   │
    │ ТЗ, план│     │ Senior+  │
    └────┬────┘     └────┬─────┘
         │               │
    ┌────▼─────┐    ┌────▼──────┐    ┌───────────┐
    │DEVELOPER │    │  TESTER   │    │DOCS WRITER│
    │ (Sonnet) │    │ (Sonnet)  │    │  (Haiku)  │
    │ Код      │    │ Тесты     │    │ Докумен-  │
    └────┬─────┘    └─────┬─────┘    │ тация     │
         │                │          └─────┬─────┘
         │                │                │
    ┌────▼────────────────▼────────────────▼──┐
    │  REVIEWER (Opus)                        │
    │  Код-ревью: архитектура, безопасность,  │
    │  IPC, PyQt, соответствие плану          │
    └──────────────────┬──────────────────────┘
                       │ результат
    ┌──────────────────▼──────────────────────┐
    │  DIRECTOR — финальная оценка и решение  │
    └─────────────────────────────────────────┘
```

> **Director — это не файл в `agents/`.** Это сам Claude в главном разговоре (всегда Opus).
> Его инструкции — `.claude/CLAUDE.md`. Все остальные роли — агенты, которых Director
> вызывает через Agent tool. Агенты не могут вызывать друг друга.

## Роли

### Director (main conversation, Opus)

**Аналогия:** CEO / Генеральный директор.

**Обязанности:**
- Принимает задачу от пользователя
- Оценивает сложность (правило порога)
- Выбирает стратегию: сам или делегирует
- Управляет агентами: кому, что, в каком порядке
- Финальное решение по коду

**Особенность:** Director — это **не файл в `agents/`**, а сам Claude (Opus) в главном разговоре. Его конфигурация — `.claude/CLAUDE.md`. Он единственный, кто может вызывать агентов через Agent tool. Агенты не видят друг друга и не могут вызывать друг друга — вся координация через Director.

---

### Manager (Opus)

**Аналогия:** Начальник отдела / Project Manager.

**Вызов:** `/plan <задача>`

**Обязанности:**
- Получает этап/фичу от Director
- Изучает код и архитектуру
- Декомпозирует на атомарные подзадачи (Task X.Y)
- Каждой задаче назначает уровень сложности и исполнителя
- Определяет порядок и зависимости
- Сохраняет план в `plans/`

**НЕ делает:** не пишет код, не запускает тесты, не делает git.

**Формат выхода:** файл плана в `plans/<имя>.md` с Task X.Y.

---

### TeamLead (Opus)

**Аналогия:** Техлид / Ведущий разработчик.

**Вызов:** через Director (Agent tool, subagent_type: teamlead)

**Обязанности:**
- Реализует задачи уровня Senior+ (слишком сложные для Sonnet)
- Архитектурные изменения, сложный рефакторинг
- Экспресс-ревью работы Developer'а
- Принимает технические решения в рамках scope
- Записывает архитектурные решения в DECISIONS.md

**Когда вызывать вместо Developer:**
- Задача затрагивает архитектуру
- Нужен рефакторинг с изменением контрактов
- Developer дважды не справился с задачей

---

### Developer (Sonnet)

**Аналогия:** Разработчик / Middle+ инженер.

**Вызов:** `/implement <Task X.Y или описание>`

**Обязанности:**
- Реализует код строго по ТЗ
- Читает все указанные файлы перед работой
- Smoke-test после изменений
- Коммитит с осмысленным сообщением

**НЕ делает:** не выходит за scope, не рефакторит «заодно», не добавляет фичи.

**При блокере:** СТОП, сообщает конкретно что неясно.

---

### Tester (Sonnet)

**Аналогия:** QA-инженер.

**Вызов:** `/test <что тестировать>`

**Обязанности:**
- Пишет pytest-тесты по acceptance criteria
- Запускает и проверяет результаты
- Покрывает edge cases
- Сообщает о багах, но не чинит

**НЕ делает:** не меняет логику приложения.

---

### Docs Writer (Haiku)

**Аналогия:** Технический писатель.

**Вызов:** `/docs <что документировать>`

**Обязанности:**
- Docstrings для публичных функций
- README.md, STATUS.md модулей
- DECISIONS.md (ADR)
- Следует языковым правилам проекта

**НЕ делает:** не меняет логику кода, не рефакторит имена.

---

### Reviewer (Opus)

**Аналогия:** Senior-ревьюер / Архитектор-инспектор.

**Вызов:** `/review <что ревьюить>`

**Обязанности:**
- Проверяет соответствие плану и acceptance criteria
- Базовый чеклист (качество, тесты, побочные эффекты)
- Доменные специализации:
  - **Архитектура** — Dict at Boundary, interfaces.py, DECISIONS.md
  - **IPC-роутинг** — targets vs channel не перепутаны
  - **Безопасность** — pickle, injection, секреты
  - **PyQt** — thread-safety, утечки QObject, блокировка GUI

**Выход:** `APPROVED` или `CHANGES REQUESTED` с конкретными правками (файл:строка:проблема:решение).

## Workflow

### Правило порога

| Размер задачи | Что делать |
|---|---|
| 1-2 файла, <50 строк | Director сам |
| 3-5 файлов | `/implement` → `/review` |
| 6+ файлов, архитектура | `/plan` → `/implement` → `/test` → `/review` → `/ship` |

### Полный цикл — автоматический (`/pipeline`)

Одна команда запускает весь конвейер:

```
/pipeline "Добавить модуль X"
```

Director автоматически:
1. Запускает Manager → план с Task 1.1–1.5
2. Показывает план пользователю, ждёт подтверждения
3. Для каждой задачи: Developer (или TeamLead для Senior+) → код → коммит
4. Tester → тесты по acceptance criteria
5. Reviewer → ревью → апрув или правки (макс 2 итерации)
6. Ship → validate + тесты + линтер → предложение push

Если на любом этапе 2 неудачи подряд — СТОП, эскалация к пользователю.

### Полный цикл — ручной (пошагово)

```
1. /plan "Добавить модуль X"
   → Manager создаёт план в plans/add-module-x.md
2. Director проверяет план
3. /implement "Task 1.1 — создать структуру"
   → Developer создаёт файлы, коммитит
4. /implement "Task 1.2 — обработка сообщений"
   → Developer реализует логику
5. /test "Тесты для модуля X"
   → Tester пишет и запускает тесты
6. /review
   → Reviewer → APPROVED или CHANGES REQUESTED
7. /docs "Документация для модуля X"
   → Docs Writer пишет README, docstrings
8. /ship
   → validate + тесты + линтер → push
```

### Worktree isolation

Для рискованных изменений Director запускает агента в изолированном worktree:

```
Agent(subagent_type: "developer", isolation: "worktree", prompt: "...")
```

Агент работает в отдельной копии репо. Если результат плохой — worktree удаляется без последствий для основной ветки. Полезно при рефакторинге, экспериментах, массовых изменениях.

### Средний цикл (3-5 файлов)

```
1. Director: оценивает — 4 файла, средний цикл
2. /implement "Задача: ..."
   → Developer реализует
3. /review
   → Reviewer проверяет → APPROVED
4. Done
```

### Малый цикл (1-2 файла)

Director делает всё сам. Агенты не нужны.

## Уровни сложности и режимы

Manager назначает уровень каждой подзадаче. **Правило: всегда на один выше минимально необходимого** — каждый агент должен "прыгнуть чуть выше головы", чтобы результат был на отлично.

| Уровень | Модель | Агент | Режим | Глубина анализа |
|---------|--------|-------|-------|-----------------|
| Senior+ | Opus | TeamLead | `deep` | Все edge cases, архитектурные последствия, совместимость, DECISIONS.md |
| Senior | Opus | TeamLead | `thorough` | Зависимости, побочные эффекты, smoke-test + импорты |
| Middle+ | Sonnet | Developer | `careful` | Контекст перед изменением, Grep по символам, smoke-test |
| Middle | Sonnet | Developer | `standard` | По ТЗ, smoke-test, acceptance criteria |
| Junior | Haiku | Docs Writer | `quick` | По шагам, не усложнять |

### Как режим работает технически

Claude Code **не имеет** параметров thinking mode или effort level per-agent. Режим реализован через **инструкции в промпте** — Director включает строку `Режим: <mode>` в начало промпта при вызове агента.

Каждый агент в своём `.md` файле имеет таблицу режимов, которая определяет:
- Насколько глубоко анализировать контекст перед работой
- Сколько проверок делать (Grep по зависимостям, smoke-test, импорты)
- Покрывать ли edge cases или только happy path

Пример вызова Director'ом:
```
Agent(subagent_type: "developer", model: "sonnet", 
      prompt: "Режим: careful. Task 2.1 — добавить обработку frame_data...")
```

## Экономика

### Стоимость моделей (относительная)

| Модель | Стоимость | Используют |
|--------|-----------|------------|
| Opus | 15x | Director, Manager, TeamLead, Reviewer |
| Sonnet | 3x | Developer, Tester |
| Haiku | 1x | Docs Writer |

### Где экономия

- ~70% работы (реализация + тесты) делает Sonnet/Haiku вместо Opus
- Каждый агент получает чистый контекст 200K — нет проблемы переполнения
- Специализированные промпты дают лучший результат, меньше итераций

### Где расход

- Каждый агент читает CLAUDE.md заново (~2K токенов)
- Передача контекста между агентами (план, ТЗ) — ~1-2K на агент
- Manager + Reviewer = 2 вызова Opus на крупную задачу

## Расширение штата

### /hire — создание нового агента

```
/hire api-specialist — Специалист по REST API, Sonnet, full tools
```

Команда создаёт нового агента по шаблону `_template.md`:
1. Задаёт имя, роль, модель, инструменты
2. Создаёт `agents/<имя>.md`
3. Опционально создаёт `commands/<имя>.md`

### Шаблон агента

Файл `agents/_template.md` содержит структуру с плейсхолдерами:
- `{{NAME}}`, `{{DESCRIPTION}}`, `{{MODEL}}`, `{{TOOLS}}`
- Секции: Роль, Правила, Что делать, Чего НЕ делать, Формат вывода

### Примеры агентов для найма

| Имя | Роль | Модель |
|-----|------|--------|
| `api-specialist` | REST/GraphQL API дизайн и реализация | Sonnet |
| `db-engineer` | Миграции, запросы, оптимизация БД | Sonnet |
| `devops` | Docker, CI/CD, деплой | Sonnet |
| `performance` | Профилирование, оптимизация | Opus |

## Переносимость

### Что универсально (копируется)

```
.claude/
├── CLAUDE.md              ← Workflow (не трогать)
├── settings.json          ← Разрешения (адаптировать стек)
├── agents/                ← Все агенты команды
├── commands/plan|implement|review|test|docs|ship|hire.md
└── hooks/                 ← Хуки безопасности и форматирования
```

### Что проектно (создаётся заново)

```
CLAUDE.md (корень)         ← Проектный контекст
.claude/commands/          ← Проектные команды (validate, fw-test, ...)
.claude/skills/            ← Проектные скиллы
.claude/mcp.json           ← MCP-серверы проекта
```

### Шаги переноса

1. Скопировать `.claude/` в новый проект
2. Создать корневой `CLAUDE.md` — архитектура, стек, правила проекта
3. Удалить/адаптировать проектные commands/ и skills/
4. Обновить `mcp.json` и `settings.json` под стек
5. Готово — `/plan`, `/implement`, `/review`, `/ship`, `/hire` работают

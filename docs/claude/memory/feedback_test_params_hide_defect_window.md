Ревью 2026-07-10 нашло 2 HIGH в constructor-master, оба «под зелёными тестами»:
- ложное `recovered` (process_monitor): тест-хелпер использовал `backoff_sec=0.0` — окно между планированием и диспатчем рестарта закрыто, дефект невидим (прод-дефолт 2.0);
- петля drain↔tap (observability): wiring-тесты писали INFO (ниже ERROR-порога tap'ов) и `adapter=None` — оба условия дубля обойдены.

**Why:** параметр теста, «упрощающий» сценарий (нулевые таймауты, severity ниже порога, None-зависимость), может ровно закрывать окно, в котором живёт дефект. Зелёный сьют ≠ доказательство.

**How to apply:** при написании тестов на механизмы с окнами/порогами (backoff, throttle, min_level, буферы) — хотя бы один тест с ПРОД-значениями параметров и с реальными зависимостями (не stub), особенно на пути ошибок. При ревью — искать, чем параметры тестов отличаются от прод-дефолтов. Связано: [[hardware-recipes-no-headless-boot]] (live-проверки), уроки Ф3.7/Ф4.2 «истину вскрыл живой прогон».

Стойло фонового агента («no progress for 600s») ≠ смерть: после watchdog-нотификации оригинальный поток может ОЖИТЬ и продолжить писать. SendMessage-реанимация в этот момент создаёт второй инстанс — оба пишут одну задачу в одном дереве (случилось на Ф2.2 constructor-master, 2026-07-07).

**Why:** watchdog сообщает об отсутствии прогресса, а не о завершении; runtime может восстановить stream. Один task-id может нотифицировать несколько раз.

**How to apply:**
1. Перед реанимацией — вахта 1-2 мин на mtime файлов зоны агента (файлы меняются → оригинал жив, НЕ реанимировать, ждать).
2. В ТЗ агентам всегда: «коммить каждый шаг» — при коллизии выживает закоммиченное, а второй инстанс сможет строить поверх.
3. Инцидент-паттерн разрешения: владелец зоны — тот, у кого незакоммиченный in-progress; второй останавливается и докладывает (так и произошло — агент сам обнаружил коллизию по чужим mtime).

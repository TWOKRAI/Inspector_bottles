Task 1.1 плана `plans/backend-ctl-proof-discipline.md` закрыт (`cdb62f24`): типизированные
обёртки `RouterStats` / `QueueDepths` / `WorkerStatus` / `MemoryStats` больше не подставляют
`0` / пустой словарь вместо отсутствующих данных. Поле без показания = `None`, его имя лежит
в новом поле `missing`; `missing == []` читается как «форма ответа полная».

**Три состояния, которые нельзя схлопывать** — главная тонкость контракта:

| Что произошло | Значение поля | В `missing`? |
|---|---|---|
| ключа не было в ответе (переименовали / ручка ответила не тем) | `None` | **да** |
| сервер прислал значение, в т.ч. `0` или `{}` | значение | нет |
| сервер прислал явный `null` (только `MemoryStats`) | `None` | **нет** |

Третья строка — про best-effort контракт `introspect.memory`: явный `null` секции означает
«подсистема недоступна», это ОТВЕТ сервера. Записать его в `missing` = сделать аномалией
штатную норму.

**Why:** `int(stats.get("sent_ok", 0) or 0)` превращал «сервер переименовал поле» и «ручка не
ответила» в «трафика не было» — ложь, неотличимую от факта, ради доверия к которой инструмент
и существует. Продолжение болезни из [[project-backend-ctl-signal-integrity]] (ложный ноль),
часть цены ultra-ревью 4.5/10.

**How to apply:** добавляя поле в любую обёртку `backend_ctl/protocol.py`, сначала реши, к
какому из трёх состояний оно относится, и проводи приёмку **парой** (значение + `missing == []`
/ переименованный ключ → `None` + `missing`) — одиночный зелёный не отличает «прочиталось» от
«подставилось дефолтом». `missing == []` на ЖИВОМ бэкенде — единственное доказательство, что
имена сверены с сервером, а не додуманы по исходникам. Потребитель типизированных полей ровно
один — `backend_ctl/overview.py` (аномалия `counter_missing`); пробы `backend_ctl/probes/*`
идут мимо обёрток на сырых dict через `unwrap(..., leaf=True)` и контракта не касаются.

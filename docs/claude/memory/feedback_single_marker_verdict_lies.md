Инструмент, выносящий вердикт «произошло/не произошло» по одному наблюдаемому маркеру,
ошибается в обе стороны. Пример (Ф4 truth-holes, `process_restart_verified`, 2026-07-23):
вердикт «рестарт был» считался по смене `pid` →

- **ложное «да»**: новый инстанс поднялся и сразу умер (битый конфиг) — pid сменился,
  `restarted=True`, хотя процесса нет;
- **ложное «нет»**: ОС переиспользовала pid (Windows это делает охотно) — pid совпал,
  `restarted=False`, хотя `instance_restarts` вырос.

Починка: `restarted = (pid ИЛИ instance_restarts изменились) И alive`, а полуудача
(замена была, инстанс мёртв) — отдельное поле `replaced`, не спрятанное ни в успехе,
ни в «ничего не было».

**Why:** оба маркера по отдельности — следствия события, а не само событие; совпадение
двух независимых следствий плюс признак жизни отсекает и подмену, и совпадение.

**How to apply:** прежде чем писать `verdict = (X изменился)`, спросить: (1) может ли X
измениться без события? (2) может ли событие произойти без изменения X? (3) отличает ли
вердикт «сделано» от «сделано и работает»? Если хоть один ответ «да» — нужен второй
маркер и отдельное имя для полуудачи.

Тест, легализующий такой дефект, распознаётся по несовпадению имени и assert'а:
`test_dead_new_instance_is_not_counted_as_restarted` проверял `alive is False`, а не сам
вердикт — то есть не доказывал того, что обещал. См. [[feedback_plausible_is_not_verified]],
[[feedback_check_red_on_main_first]].

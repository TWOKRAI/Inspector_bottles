Владелец (2026-07-22): «главное чтобы эти флаги не были костылями».

**Why:** dark-launch-флаг легитимен только с полным жизненным циклом
`OFF → замер → default-ON → **удалить флаг и OFF-ветку**`. Если остановиться на флипе
дефолта, OFF-ветка живёт в коде вечно, тест-матрица удваивается, и реестр (уже 18 флагов)
становится памятником нерешённым спорам. «Переключить дефолт» ≠ «убрать флаг».

**How to apply:**
- В плане с флагами заводить ОТДЕЛЬНУЮ фазу снятия лесов с acceptance `grep FW_X` пуст —
  не ограничиваться пунктом «флип дефолтов». Пример: Фаза 6 в `plans/truth-holes-closure.md`.
- План не считать закрытым, пока флаги живы.
- Если у двух веток есть РЕАЛЬНАЯ долгосрочная причина сосуществовать — это сигнал, что нужен не
  флаг, а структурное различие. Пример: `FW_STATE_COALESCE` добавляет до 120 мс задержки —
  для уровней (FPS/глубины) невидимо, для фронтов (ошибка/смерть/переход) вредно. Правильный
  конец — не флаг, а выбор по природе сигнала (levels коалесцируются, edges идут мимо буфера).
- Тот же водораздел levels-vs-edges нужен [[project-telemetry-pull-on-demand]].

(прецедент: 9 флагов флипнуты по одному с замером).

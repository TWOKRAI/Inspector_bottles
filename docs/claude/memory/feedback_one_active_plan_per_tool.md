Для одного инструмента/модуля ведётся ОДИН активный план. Новая задача по нему вносится в этот план (новая фаза/секция), а не в отдельный файл рядом. Если параллельная работа всё же требует нескольких файлов (мини-планы под конкретную фичу) — родитель обязан быть прочитан ПЕРЕД стартом, а не просто существовать как ссылка.

**Why:** раздельность треков backend_ctl (семь файлов: hardening, debug-console, d1/d2/d4-мини-планы, framework-module, плюс сам framework-module ссылался на debug-console как на предшественника) дала два конкретных дефекта:
1. **Воскрешение отменённого решения.** GATE G3 (2026-07-13) явно пометил Task 1.8 (record/replay) «пропустить, вернуться при реальной нужде». Через 6 дней план `backend-ctl-d4-flight-recorder` (в другом треке, не знавшем о GATE G3) построил ровно этот record/replay — 6 MCP-инструментов + session-режим. Не обязательно ошибка по существу (нужда могла настать), но отмена не была пересмотрена осознанно — это совпадение процесса, не решение (см. [[project_backend_ctl_gaps_2026_07]]).
2. **Простой санкционированного рефакторинга.** Владелец одобрил сплит `driver.py`/`mcp_tools.py` (Task 2.1/2.2: вынос `registers.py`/`dispatch.py`) решением 2026-07-17. Решение осталось записанным в `backend-ctl-framework-module.md`, но файл продолжал расти новыми фичами (Phases D/E/F из `debug-console.md`) вместо исполнения сплита — к моменту архивации план вырос до 505 строк, а сплит так и не был сделан.

**How to apply:** при заведении новой задачи по backend_ctl проверить `plans/backend-ctl-*.md` — должен остаться РОВНО один файл (`backend-ctl-proof-discipline.md` на 2026-07-21, поглотил все предыдущие шесть — архив в `plans/_archive/`). Любой план, живущий дольше одного дня, обязан иметь строку в `plans/QUEUE.md` (тот же урок с другой стороны — см. правило в шапке QUEUE.md, добавленное 2026-07-20). Дублируется на класс «параллельные треки не знают друг о друге» — см. также [[feedback_plan_queue_must_list_every_plan]] (пока не заведена отдельно).

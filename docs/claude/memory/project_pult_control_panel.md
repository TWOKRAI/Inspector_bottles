Ветка `feat/pult-control-panel`. План `plans/pult-control-panel.md`.

**Сервис «Пульт» (`Services/control_panel/` + GUI-вкладка Services «Пульт»):** рецептный — контролы хранятся в config-поле `controls` ноды `control_panel` (source-плагин, пул портов out_1..out_8). Phase 1-3 DONE + qt-mcp smoke. Контролы: button/toggle/slider/number/text эмитят значение на свой порт (pipeline-сигнал), привязываются к потребителю в редакторе Pipeline. add/remove персистятся в рецепт через `SetPluginConfig` (editor-топология → save). Демо `recipes/pult_demo.yaml`.

**Грабли (smoke поймал):** секция Services ОБЯЗАНА реализовать `action_buttons()` — иначе весь Services-таб падает «Ошибка загрузки» (unit-тест спеки не ловит; есть регресс-тест на поверхность SectionProtocol). qt-mcp дерево-навигация ненадёжна по координатам — кликать прямой ref видимого элемента после End-скролла.

**Кнопка «Рисовать» в phone_sketch (DONE, verified live):** пульт-сигнал out_1 → провод → `robot_draw`. robot_draw получил generic pipeline-триггер: register `trigger_source` (ключ сигнала = имя ИСХОДНОГО порта, провода НЕ переименовывают ключ!) + входной порт `trigger`; сигнал взводит `_armed` (как cmd_send), путь уходит на ближайшем кадре с draw_points. Сигнал и draw_points приходят в РАЗНЫЕ process()-вызовы (chain_targets маршрутизирует, не wires). Универсально: любой контрол вяжется к роботу конфигом, без правок кода.

**Дашборд (Phase 5) — DONE + live verified (2026-06-16):** вынос ВЫБРАННЫХ параметров/команд ДРУГИХ нод в пульт. `ControlSpec.source ∈ {local,param,monitor,action}` (+target_process/index/field/command, +value_arg/command_args для action). Реализация в `frontend/.../control_panel/`: `catalog.py` (NodeCatalog: ноды из топологии, поля через `extract_fields`+PluginRegistry, команды через `PluginRegistry.get().commands`; билдеры make_param/action_spec; `field_value` тянет текущее значение из config ноды — НЕ min) + `picker.py` (NodePickerDialog: процесс→плагин→param/action→поле/команда; тип контрола следует типу поля). Роутинг presenter по source: local→set_control, **param→`SetPluginConfig`** (live field-write + персист, app.py listener), **action→`bridge.on_action_command`**, monitor→read-only (deferred). 5.4 monitor отложен. 65 тестов.

**ГРАБЛИ Phase 5 (важно):** топологию в GUI брать ТОЛЬКО из `services.topology.load().to_dict()` (канон, как Pipeline/Processes) — у конкретного `TopologyBridge` НЕТ свойства `topology` (`getattr`→None→пустой пикер). Тем же фиксом починен legacy `_resolve_node` (персист контролов читал мёртвый bridge.topology). action со значением: фреймворк хранит команды как `{имя:метод}` БЕЗ схемы аргументов → GUI не авто-открывает аргументы, пользователь задаёт value_arg+command_args (param/поля — авто из register-схемы).

**Узел `robot_scale` (DONE, вписан в phone_sketch live verified):** strokes_to_points → identity (ПИКСЕЛИ), `robot_scale` первым в процессе `points`, вписывает кадр src_width×src_height(px) в прямоугольник листа по углам ЛВ(x0,y0)/ПН(x1,y1) **в мм** (Y-вверх: y0>y1). phone_sketch: углы 0/144/192/0, src 640×480. Углы (x0..y1) — register-поля live-tunable, вынесены в pult как param-контролы (правятся вживую). `points_render` рисует лист: прямоугольник + 4 угла + подписи мм-координат (register `show_sheet`, из `draw_bounds`).

**Робот-параметры в пульте (action, DONE):** скорость+перо в pult phone_sketch. Скорость → `robot_draw_set_speed{device_id,pct}`. Перо вниз/вверх (опускание к плоскости / подъём при перемещении) → `robot_draw_set_pen{device_id,down|up}` — сделан **частичный апдейт** в `RobotDriver._op_draw_set_pen` (down ИЛИ up, второе сохраняется) → два независимых контрола на одну команду. Все три — `number` (по подтверждению ввода, не флудит робот как слайдер). pen/speed — НЕ register-поля (живут в RobotDriver процесса `devices`), потому action, не param.

**Модель «A4 один раз → рисунок внутри» + crop (DONE 2026-06-16, live verified):** курс владельца — углы x0/y0/x1/y1 robot_scale = прямоугольник листа A4 на столе (задаётся ОДИН раз), а рисунок ДВИГАЕТСЯ/МАСШТАБИРУЕТСЯ внутри листа. В robot_scale добавлены `draw_scale`(доля листа)+`offset_x`/`offset_y`(мм): `x_mm=x0+offset_x+px*(x1-x0)/w*scale`. `draw_bounds` остаётся = A4 (не рисунка) → на дисплее точек видно, как рисунок ездит внутри листа. Новый узел `Plugins/processing/crop/` (crop_x/y/w/h px, +out_width/height; вырезает регион и РЕСАЙЗИТ обратно к размеру исходного кадра — размерности тракта стабильны, strokes/robot_scale src не плывут; всё 0 = passthrough) вписан первым в процесс `lines` phone_sketch (wire seg.image_adjust.frame→lines.crop.frame). Все поля (crop x/y/w/h, draw_scale, offset_x/y) — param-контролы в пульте. 73 теста.

**Поворот листа 90° + ровный показ + пропорции + перо Z (DONE 2026-06-16, владелец verified):** реальная рабочая зона A4 в коорд. робота — ВЛ(325,−223), НП(544,−17), лист повёрнут 90° на столе. Добавлен `swap_axes` в robot_scale (робот-X из ВЕРТИКАЛИ кадра py, робот-Y из ГОРИЗОНТАЛИ px) — координаты для робота пересчитываются под повёрнутый лист, а `points_render` тоже `swap_axes` → на экране портрет показывается РОВНО (как исходник), углы подписаны робот-координатами (ВЛ верх-лево). `keep_aspect` (robot_scale, default True) — единый масштаб по меньшей стороне + центрирование, без горизонтального сжатия (важно: зона почти квадрат 219×206, кадр 4:3). Урок: ориентацию дисплея определять по УПОРЯДОЧЕННЫМ углам draw_bounds [x0,y0(ВЛ),x1,y1(НП)] + swap_axes, НЕ по min/max+Y-инверсии (та путала верх/низ при Y-вниз зоне). Перо Z робота: низ к плоскости −90, подъём −80 (`robot_draw_set_pen` партиал-апдейт). flip_y в points_render — отдельный визуальный тумблер (вверх ногами), сейчас false.

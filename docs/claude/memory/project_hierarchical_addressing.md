Требование владельца (2026-05-31, обсуждение целевой коммуникационной архитектуры «RouterManager как единый хаб + каналы»): **адрес получателя в `Message` (билете) обязан быть иерархическим**, не только имя процесса.

Аналогия владельца: почтовый адрес — Страна → Город → Улица → Дом → Квартира → Человек. Так же и здесь: **процесс → воркер → (возможно глубже)**.

Правила иерархии:
- Можно указать ТОЛЬКО процесс (доставка на уровень процесса — как сейчас).
- Нельзя указать воркер БЕЗ процесса (нарушает порядок). Сначала процесс, потом воркер, потом глубже.
- Отсутствие нижних уровней — нормально (prefix-маршрутизация: доставляем на самый глубокий заданный уровень).
- Порядок/вложенность обязательны — нельзя «перепрыгнуть» уровень.

**Why:** адресация должна масштабироваться до суб-процессной гранулярности (воркер и глубже) без передизайна шины. RouterManager читает иерархический адрес билета и маршрутизирует на нужный уровень. Это прямо связано с долгом #2 `assigned_worker` (Фаза 2) — `assigned_worker` это и есть уровень «воркер» в адресе.

**How to apply:** при проектировании конверта `Message` и маршрутизации `RouterManager` в коммуникационном рефакторе моделировать `target` как УПОРЯДОЧЕННЫЙ иерархический адрес (например list `[process, worker, ...]` или dotted `process.worker.…`) с prefix-семантикой доставки. Ось «адрес» (куда, иерархия) ортогональна оси «канал/kind» (тип груза: command/data/frame/event/state). Обе живут в билете. Не путать с Router-каналом (см. `ROUTING_GLOSSARY.md`). Целевая архитектура и каналы — в [`multiprocess_framework/docs/COMMUNICATION_MAP.md`](../../../multiprocess_framework/docs/COMMUNICATION_MAP.md).

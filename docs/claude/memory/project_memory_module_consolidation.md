Требование владельца (2026-07-14, во время G.5 В3): **память должна быть ОДНИМ модулем**, не размазанной по framework без необходимости — «без костылей, как полагается». У модуля должен быть **фасад, интерфейс, взаимозаменяемость** (в пределе — подмена на Rust-библиотеку вроде iceoryx2 и т.п.).

**Why:** система инспекции — сложные pipeline-цепочки + межпроцессное общение через роутер/сообщения; SHM-транспорт кадров — критичный и универсальный кусок конструктора. Если память размазана (часть в `shared_resources_module.MemoryManager`, часть в `router_module.FrameShmMiddleware`: free-list/refcount/seqlock/handle-кэш/loan-протокол), то смена реализации (на боевую Rust-библиотеку) или ревизия становятся дорогими и хрупкими.

**How to apply:**
- На ревью Fable G.5 (после 8-углового) — явно попросить Fable оценить: (1) архитектуру и **эффективность** новых правок (система и до этого работала — оправданы ли изменения, не усложнили ли зря); (2) **не размазана ли память** — стоит ли frame-pool/loan/seqlock-логику из `FrameShmMiddleware` (router_module) свести в модуль памяти (`shared_resources_module`) за фасадом+интерфейсом, чтобы транспорт кадров был взаимозаменяем (DDS/iceoryx2-совместимый контракт уже заявлен в G.4).
- Кандидат-следствие: выделить/консолидировать «модуль памяти» с Protocol-интерфейсом (module-contract), FrameShm как одна из реализаций за фасадом. Возможно отдельная H-задача/рефактор после G.5, не ломая dark-launch флаги.
- Не рефакторить это в разгар G.5 (сначала доделать фазу + ревью); зафиксировать как архитектурную цель.

**Последовательность закрытия G.5 (владелец, 2026-07-14):** доделать фазу (d-2 release + e reclaim) → **инкрементальный qex-reindex** (`/mcp-qex:qex-reindex`, чтобы семантический поиск видел новый код) → **8-угловое ревью** → **ревью Fable** (с двумя директивами выше: архитектура+эффективность, память-один-модуль) → merge.

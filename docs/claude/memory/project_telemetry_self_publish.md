Телеметрия процессов (FPS/latency) переведена на **self-publish** (2026-06-04, ветка feat/comm-system-target-architecture, план plans/telemetry-self-publish-redesign.md).

**Архитектура:** каждый процесс САМ публикует свои метрики в дерево StateStore через `ProcessHeartbeat._publish_metrics_to_tree` → `state_proxy.set(...)` — тот же проверенный путь, что и статус (зелёный индикатор). НЕ через старую центральную heartbeat-агрегацию `ProcessMonitor._publish_process_aggregate` (она оказалась хрупкой/мёртвой для метрик — Task 2 плана = её удалить).
- Агрегат процесса: `processes.{p}.state.fps` = max(effective_hz), `state.latency_ms` = max(cycle_duration_ms) = время самого медленного воркера (узкое горло).
- Per-worker: `processes.{p}.workers.{w}.status/effective_hz/cycle_duration_ms` (коммит 61b02761).

**Корневой баг (почему FPS не работал у агентов раз за разом):** `CycleMetricsRecorder.effective_hz` считался как `1/cycle_duration`, а длительность мерилась `time.monotonic()`. На Windows у monotonic гранулярность ~15 мс → sub-миллисекундная работа consumer-воркеров (DataReceiver/PipelineExecutor) округлялась в 0 → `effective_hz=0.0` хотя cycles росли ~21/с. Камера работала только из-за throttle-sleep ~47 мс. **Фикс (b6ce2bb8):** effective_hz = частота ЗАВЕРШЕНИЯ циклов (интервал между `record()` через `time.perf_counter()`); consumer-раннеры тоже перешли на perf_counter.

**GUI per-worker (7e59e259):** `GuiStateBindings.bind_fanout(pattern, callback, owner)` — переиспользуемый fan-out (callback на каждую matching дельту + replay из кэша) для динамического обнаружения ключей. `SingleProcessPanel` обнаруживает рантайм-воркеров из телеметрии и подмешивает в `WorkerTable` read-only строками — потому что `presenter.get_workers()` возвращает только конфиг-топологию (WorkerSpec) + синтетический message_processor, БЕЗ рантайм-воркеров пайплайна (data_receiver/pipeline_executor/source_producer_*).

**Frame-trace конверт (2026-06-04, план plans/frame-trace-envelope.md):** пер-сегментная трассировка кадра. `item["trace"]` — список спанов (`transport from→to` / `process node:plugin`), едет in-band в метаданных кадра через всю цепочку. Гейт env `INSPECTOR_FRAME_TRACE=1` (по умолчанию off → no-op). Транспорт меряют раннеры (`SourceProducer`/`DataReceiver`/`PipelineExecutor`, `node_name` из GenericProcess), обработку — декоратор `frame_trace.traced`, авто-навешен на `process`/`produce` ВСЕХ плагинов через `ProcessModulePlugin.__init_subclass__` (node из `plugin._trace_node`, ставит `PluginOrchestrator.boot`). GUI (`app.py _on_frame_received`) дописывает финальный transport + дампит таймлайн в лог. Файл `multiprocess_framework/modules/process_module/generic/frame_trace.py`. Коммиты 48caea37/20b151e3. Также: сквозная задержка `capture_ts` + GUI-метки «FPS цепочки»/«Задержка цепочки» (08160b3d), system.health avg_fps/active из дерева (067af248, GUI подписан на `system.**`). Вывод по замеру: время уходит в IPC-передачу, не в CV.

**Task 3 GUI-таблица + терминология (2026-06-04, коммит 95a122cf):** таблица «Участок · Тип · Среднее, мс» со строкой «Итого» в `AllProcessesPanel` («Все процессы»), скрыта без `INSPECTOR_FRAME_TRACE=1`. Поток: `MainWindow.record_trace_spans` копит пер-сегмент в `_on_frame_received`, `reset_trace_segments` усредняет за секунду, `app.py._update_fps` публикует `system.trace_segments` (список) через `bridge.dispatch`, панель ловит через `bind_fanout`. Заодно **терминология телеметрии**: «FPS» корректен только для кадров → процесс-метрики переименованы «FPS»→«Циклов/с» (частота), «Latency»→«Время цикла» (мс, узкое место), health «Средний FPS»→«Средняя частота»; «FPS цепочки»/«Задержка цепочки» сохранены (реальные кадры). `effective_hz` переписан на **оконное среднее за 1 с** (deque завершений в `cycle_metrics`) вместо мгновенного 1/interval — стабильное «циклов/с» (запрос владельца). Осталось: БД-сток истории (async+sample, плагин io/database), fan-in trace (region_split/stitcher). **Orphan-shutdown FIXED — финал через OS-примитив (de2ee9c1):** закрытие окна/краш оставляли воркеров-сиротами; psutil-обход ненадёжен (рвётся PPID-цепочка после kill PM, гонки рестарта/переиспользования PID — наблюдался 1 выживший). Правильное решение без костылей — **`ProcessTreeGuard`** (`launcher/process_tree_guard.py`): **Windows Job Object** с `KILL_ON_JOB_CLOSE` (создать ДО спавна PM, `AssignProcessToJobObject(PM)` — дети наследуют job; `TerminateJobObject` в stop + авто-kill при закрытии хэндла launcher), **POSIX** — оркестратор `setsid()` (new_session в run_process_function) + `killpg` по его группе; psutil-fallback если примитив недоступен. ВАЖНО: ctypes-сигнатуры kernel32 задавать явно (иначе 64-бит HANDLE усекается до c_int → access violation). Покрывает и `subprocess.Popen` плагинов (hikvision SDK). Live Windows: (1) закрытие окна → TerminateJobObject → все 8 мертвы; (2) краш launcher одиночным kill → KILL_ON_JOB_CLOSE → дерево умирает само (psutil так не мог). Архитектура дерева: `launcher → PM(оркестратор) → воркеры` (камера/плагины — под PM; лишний процесс в дереве = venv-shim от uv, не наш код). Промежуточный фикс-снимок был в 1812b204, заменён guard'ом.

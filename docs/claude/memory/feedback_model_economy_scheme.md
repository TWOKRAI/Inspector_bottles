Решение владельца 2026-07-14 («эффективно и не супер дорого», после session-limit на Fable-оркестрации G.3).

**Схема:**
1. **Главный чат (оркестратор): Opus 4.8**, обычное окно (НЕ `[1m]`), без fast mode, дефолтный уровень размышления. 80% ходов главного чата — маршрутизация/уведомления фоновых агентов; Fable-цена там не окупается.
2. **Fable — точечно, на вердикты:** свод полного ревью рисковой задачи и merge-gate. Вызов: субагент `model: "fable"` с готовым свод-файлом на входе (один вызов) ИЛИ короткий отдельный Fable-чат. НЕ держать Fable главным на всю сессию.
3. **Финдеры ревью: Sonnet 5, НЕ Opus.** Доказано ревью G.3: 8 Sonnet-углов нашли 8 HIGH (2 воспроизвели скриптами) на работе Opus-исполнителя. Opus-финдеры ≈ ×2.5 цены без доказанного прироста. Углов по риску: 8 — только рисковые вскрытия (G.4-класс), 2–3 — механика, группой — S-задачи.
4. **Исполнители: Sonnet 5 дефолт; Opus 4.8 только верхний край** (конкурентность/протоколы: G.4, G.8). Урок G.6/G.3: промахи исполнителей — охват брифа, не качество строк; лечится брифом, не тиром.
5. **Верификация находок:** финдеры сами воспроизводят (скрипты) — отдельные агенты-верификаторы на каждую находку не нужны; отсев делает свод.

**Why:** за G.3 сожжён session-limit до полудня: Fable-главный на ~15 ходов уведомлений + 8 финдеров + Opus-исполнитель ×2 захода. Самая дорогая строка — Fable-токены главного цикла на рутинных ходах.

**How to apply:** новый чат стартовать на Opus 4.8; при свод/merge — спавнить Fable-вердикт-агента; тяжёлые фан-ауты (8 углов) запускать сразу после сброса лимита. Точки, где НЕ экономить: merge-вердикт (Fable), исполнитель G.4 (Opus), репродьюсеры до правок. Разовая экономия контекста всех чатов: прополоть MEMORY.md (~−4k/ход), дедуп CLAUDE.md (−2k), выключить неиспользуемые плагины sentry/mcp-github/sequential-thinking (−2–3k) — см. аудит /context-budget 2026-07-14.

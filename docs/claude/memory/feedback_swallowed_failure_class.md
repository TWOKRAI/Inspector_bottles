За один день 2026-07-21 нашлось **четыре** экземпляра одного класса дефекта: сбой
происходит, его следствие видно (счётчик, распухший файл, отклонённая запись), а
ПРИЧИНА проглочена — либо `except ...: pass`, либо запись на уровне `debug`.

Экземпляры:

1. `TopologyGateMiddleware` не писал `context["rejection_reason"]` → ответ приходил
   безымянным `"middleware"`. Стоило полного расследования агентом Opus (~140k токенов),
   чтобы узнать имя виновника. Починено `f58eff45`.
2. `_SafeRotatingFileHandler.doRollover`: `except PermissionError: pass` → ротация не
   работает, лимит 10 МБ не соблюдается, файл вырос до 645 МБ, следа нет
   (`logger_module/channels/log_channel.py:51-56`).
3. `RouterManager` на неудачной отправке: `_inc_stat("errors")` + `_log_debug(...)`
   (`router_module/core/router_manager.py:365`) → счётчик показывает 23% потерь,
   причина невидима на дефолтном уровне.
4. `config_reload` отвечает `{'success': True, 'applied': {'log_level': 'DEBUG'}}`,
   а эффекта нет — ложный успех вместо честного отказа.

**Why:** следствие без причины хуже, чем отсутствие следствия. Оно выглядит как факт,
провоцирует построить неверную гипотезу и потратить расследование. Я сам на этом
попался: увидев отказ `reason: "middleware"`, публично назвал виновником fence-фильтр —
неверно, опровергнуто другим агентом. Ответ не давал ничего, кроме общего слова, и я
достроил недостающее догадкой.

**How to apply:**
- При правке любого `except`, гасящего ошибку: спроси, как о ней узнает тот, кто будет
  разбираться через месяц. Нет ответа — добавь счётчик и троттлённое предупреждение.
  «Fail-open, чтобы не уронить» и «молча» — разные вещи; первое обязательно, второе нет.
- Отказ обязан называть себя. Конвенция в проекте уже есть — `context["rejection_reason"]`
  (`ThrottleMiddleware` → `"throttled"`, `ValidationMiddleware` → `"validation"`);
  новая мидлварь обязана ей следовать.
- Причина отказа на уровне `debug` = причины нет. Отказ — событие как минимум `warning`.
- Grep-подсказки для поиска новых экземпляров: `except.*:\s*pass`, `_log_debug` рядом с
  `_inc_stat("errors")`, `success.*True` рядом с отсутствующей верификацией применения.
- Это же требование записано как правило: [[bctl-adr-007]] — сигнал без провенанса не
  считается подключённым.

# Аудит жизненного цикла рецептов: сохранение, переключение, остановка/запуск, кадры

**Дата:** 2026-07-12 · **Метод:** 4 code-разведки (switch-оркестрация, SHM-кадры, полнота сохранения, pipeline-граф vs рецепт) + живой эксперимент через backend-ctl/qt-mcp (boot → switch → GUI-save → обратный switch → shutdown) на main 119dc279.
**Повод:** владелец: «что-либо не сохранялось в рецепт; при переключении рецептов процессы не останавливались, конфликт — особенно по кадрам; такая же история с pipeline-графами». Мандат: без компромиссов и костылей — лучшая архитектура; все сценарии покрыть тестами.

## TL;DR

Исторический «жёсткий» баг (незавершённая остановка → cleanup по «сигнал подан», а не «процесс мёртв») **был реально починен 2026-07-04** (`fcf6dacd`, план topology-switch-hardening — его текст дословно описывает симптом владельца). Но вокруг осталось **семь классов дефектов**, из которых три воспроизводимы прямо сейчас, а кадровый конфликт по-прежнему возможен четырьмя путями. Плюс главная архитектурная причина: **у рецепта нет единого владельца жизненного цикла** — 4 несинхронизируемые копии топологии и 4 независимых писателя одного YAML.

## Подтверждено живым экспериментом (backend-ctl/qt-mcp)

| # | Наблюдение | Статус |
|---|---|---|
| Ж-1 | Остановка pipeline-процессов при switch **работает** (2 переключения: старые убиты на уровне ОС, постоянные PM/devices/gui живут) | ✅ симптом «не останавливаются» на симуляторе не воспроизводится |
| Ж-2 | **State-дерево врёт** (LP-4): убитые процессы висят `running/health ok`, uptime растёт; у пережившего `camera_0` — конфиг СТАРОГО рецепта; у новых процессов config в state отсутствует; `pid: null` у всех | 🔴 воспроизводится каждый switch |
| Ж-3 | В момент switch тихий `WARNING: Command 'state.merge' failed: Поле 'data' обязательно и должно быть dict` — шаг обновления state падает на валидации | 🔴 кандидат в корень Ж-2 |
| Ж-4 | Shutdown системы **дважды за сессию** оставил живого ребёнка (добит вручную) | 🔴 класс «процесс пережил остановку» жив на пути shutdown |
| Ж-5 | Readiness нового процесса через liveness-fallback («event не получен за 0.5s») + шум «Failed to set priority» ×N на каждый switch | 🟡 |

Дыры инструмента = дыры state (Ж-2: без PID и честного статуса сопоставить state↔ОС невозможно) — чинятся в RS-2, после чего backend-ctl становится полноценным отладчиком и бэка, и фронта.

## Механизмы (из code-разведок, файлы:строки — в отчётах агентов, ключевое здесь)

### A. «Не сохранялось в рецепт»

- **A-1 🔴 Кнопка «Сохранить» на вкладке Recipes падает ВСЕГДА** — тихий no-op: `recipes/presenter.py:547` — `load()` возвращает `Topology`-entity (без `.get()`; `or {}` не срабатывает), строки вне try/except → AttributeError из Qt-слота. Тест проходил, потому что фейк возвращал dict вместо entity (контракт фейка ≠ Protocol). **Верифицировано Fable.**
- **A-2** Pipeline-Save затирает `blueprint.name/description` → `default`/`''` (= LP-1, `graph_to_blueprint` с дефолтами).
- **A-3** Два Save-пути расходятся: pipeline пишет name/description/metadata; recipes-путь — только processes/wires/displays (стёр бы позиции, если бы не падал).
- **A-4** Поля, никогда не редактируемые через рецепт: `workers/queues/log_dir` нет в blueprint-схеме; `queue_size`/circuit-breaker-поля не входят в whitelist `_pick` extras. `Process.workers` доезжает до диска, но не до рантайма («мёртвая» конфигурация).
- **A-5** `active_services` пишется один раз `[]` при создании — GUI-редактирования нет, заморожен навсегда.
- **A-6** 4 независимых write-пути одного YAML (2×Save, devices-upsert, Displays-CRUD) с read-modify-write полного файла → гонки перезаписи чужих секций. Только layout-autosave пишет точечно.
- **A-7** Displays-tab — третий, более строгий read-путь (`extra=forbid`-entities): ValidationError на легаси-рецепте не ловится в `load()` → падение вкладки.

### B. «Процессы не останавливались / конфликт по кадрам»

- **B-1** Планировщик — честный full-replace (никакого reuse по имени), остановка graceful→terminate→kill **с подтверждением смерти**, фазы строго последовательны. Это хорошо.
- **B-2 🔴 `protected`-ловушка**: `protected` читается из ТЕКУЩЕГО конфига работающего процесса; protected-имя исключается и из остановки, И из нового blueprint → процесс живёт со старым конфигом вечно, switch «успешен».
- **B-3** `unstoppable` (пережил kill): apply падает → rollback, зависший исключается из cleanup/пересоздания «насовсем» — честно, но тихо для GUI.
- **B-4 🔴 Обрыв 5-фазного конвейера**: любой сбой команды прерывает список — cleanup-хвост ДРУГИХ старых имён не исполняется → их state/PSR-записи остаются (кандидат-механизм LP-4/Ж-2).
- **B-5 🔴 `ProcessStatusMonitor` держит алиас старого list-объекта** (`remove_process` ребиндит список) → перечисляет все процессы с момента старта PM, включая снесённые.
- **B-6 🔴 Data-plane (кадры) не фенсится вообще** — сознательное решение (ADR-PMM-010/014, перенос = Ф7 G.4): кадры пережившего процесса доезжают без фильтрации.
- **B-7 🔴 Имя SHM-сегмента — буквальный литерал `output_frames_N`** без владельца/поколения; при switch новый процесс создаёт сегмент С ТЕМ ЖЕ именем. Очереди получателей не дренируются → старое сообщение читает НОВЫЙ кадр по совпавшему имени (механизм «перепутанных кадров»). SHM-handles protected-процессов (GUI) замораживаются на spawn навсегда (рабочий путь — fallback по имени из сообщения). Мультикамера: два ЖИВЫХ источника с одинаковым ключом коллидируют даже без switch.
- **B-8** Seqlock отсутствует (torn-frame — задокументированный долг Ф7 G.3), ring buffer де-факто одно-слотовый/blind round-robin (Ф7 G.4).
- **B-9** HP-5 (unlink SHM × in-flight кадр при switch) — официально «митигировано, но не сверено» (gate-вопрос Ф7).

### C. Pipeline-графы

- **C-1 🔴 4 копии топологии** (диск / TopologyRepositoryStore / PipelineModel / живой бэкенд) без dirty-флага и транзакций; каждый переход — отдельное явное действие.
- **C-2 🔴 Переключение рецепта молча выбрасывает несохранённые правки графа** И одновременно заменяет живую систему — без подтверждения.
- **C-3** Undo структурных правок меняет редактор, но НЕ живую систему (только config-diff полей переигрывается) → гарантированное расхождение после Apply+Ctrl+Z.
- **C-4** Save в рецепт — без валидации; «Загрузить из файла» обходит домен-валидацию целиком (можно записать в рецепт цикл/дубли имён).
- **C-5** Перезапуск GUI теряет применённую-но-несохранённую топологию (бэкенд продолжает исполнять то, чего GUI больше не видит).

## Целевая архитектура (вектор «как правильно»)

1. **Один владелец жизненного цикла рецепта** (RecipeSession): SSOT-топология + dirty-флаг + версия; ВСЕ читатели/писатели (оба Save, devices, displays, activate) — через него; секционные записи вместо read-modify-write всего файла.
2. **Switch как транзакция с контрактом фаз**: prepare → stop+confirm → cleanup+verify → provision → start → ready → commit (state-cleanup + конфиги + epoch) — с компенсацией при сбое НА ЛЮБОЙ фазе (доисполнение cleanup-хвоста, никогда не «оборвали и ушли»), результат каждой фазы — в state, ошибки громкие (не WARNING).
3. **Кадровый тракт с поколениями** — Ф7 G.3/G.4 (имя сегмента с owner+incarnation, seqlock, боевое кольцо, fencing data-plane, дренаж очередей получателей на switch, refresh handles protected-процессов) — строго одним вскрытием, RS-волна его НЕ трогает.
4. **Контракт тестовых дублёров**: фейки обязаны реализовывать Protocol (runtime-контракт-тест) — класс «фейк разошёлся с Protocol» (A-1) закрывается системно.

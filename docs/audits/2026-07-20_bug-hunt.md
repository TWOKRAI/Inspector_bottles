Ревью в этом проекте были **диффовыми**: «корректен ли этот дифф?». Ни один сегодняшний дефект в ревьюируемые диффы не попадал — `stop_worker` лежит нетронутым, вероятно, с начала.

Глубже: behavior-preserving рефакторинг **обязан** сохранять баги — это определение. А характеризационные тесты («бит-в-бит прежнее поведение», паритет G.2, гейт «флаг off = прежнее») технически **закрепляют** латентный дефект как эталон. Сегодня это видели буквально: снапшот `build` честно хранил `priority: null` как ожидаемое значение.

Единственный раз, когда инструмент навели на подсистему целиком, а не на дифф, — ultra-ревью `backend_ctl` — он выдал 23 находки и 4.5/10. Аппарат работает; его не наводили на ядро.

## 3. Runbook: soak на webcam_sketch

```bash
# из корня репозитория
python -m backend_ctl.probes.g7_soak_probe \
    --recipe webcam_sketch.yaml --duration 7200 --interval 300 \
    --out logs/g7_soak_webcam.jsonl
```

Проба `backend_ctl/probes/g7_soak_probe.py` обобщена под произвольную топологию: процессы берутся из живой системы через `system_overview`, воркеры сводятся по `effective_hz` без зашитых имён. 18 тестов на чистые функции.

**Грабли, пойманные заранее:**

- **Поднимется окно GUI.** Процесс `gui` объявлен инлайном прямо в рецепте (не приходит из presentation-overlay), headless-флаг его НЕ перебивает. Это документированный edge case в шапке `frontend/presentation.yaml`
- **Камера должна быть свободна** — иначе `capture` не поднимется
- **Порт 8766 занят** всё время прогона
- **stdout буферизуется** при редиректе — следить по JSONL (он пишется с flush на каждый сэмпл), а не по консоли
- **Форматы JSONL разные** у синтетики (старый: `source`/`consumer`) и webcam (новый: per-process). Оба самоописательные
- **Флаги** включаются пробой автоматически: 9 из лесенки + `FW_PERF_PROBES`. `FW_GC_SCHEDULED` намеренно OFF — measurement-gated, на шаге 9 условие активации не выполнилось

**Что считать успехом:** leak-счётчики (`torn_reads`/`stale_drops`/`loan_exhausted`/`pickle_fallbacks`) = 0; `frame_handle_cache_size` не растёт; RSS без тренда вверх; pid процессов не меняются; FPS ровный.

**Сравнение с baseline:** только same-tier. Синтетика сравнивается с синтетикой лесенки. Для webcam baseline all-off НЕ снят — поэтому webcam-прогон доказывает **отсутствие утечек и деградации во времени**, но не «p99 не хуже, чем было».

## 5. Находки

Severity: **CRITICAL** — неверные данные/физические последствия · **HIGH** — потеря ресурса или невидимая деградация · **MED** — достижимо при определённых условиях.

### ✅ Подтверждено лично (сверено с текущим кодом)

#### C-1. CRITICAL — робот едет по сфабрикованной координате поворота
`Services/device_hub/drivers/robot_driver.py:321-328`

```python
try:
    rz_now = self._client.read_position().rz_deg
except Exception:
    rz_now = 0.0
    self._record_err()
place = (px, py, pz, rz_now + dovorot)   # задание НЕ отменяется
```

Комментарий двумя строками выше декларирует замысел: «читаем **РЕАЛЬНЫЙ** R робота (телеметрия)… кладём под R = реальный + доворот». Фолбэк нарушает собственное намерение кода. `0.0` неотличим от честно прочитанного нуля; `_record_err()` фиксирует «была ошибка связи», но не «поехали по выдумке».

**Последствие:** деталь укладывается с неверным поворотом. Для тракта «буква + угол», где весь смысл в угле, — прямо в цель.
**Живая проверка:** оборвать телеметрию робота при активном задании с `place is not None`, сверить фактический R укладки с ожидаемым.

#### C-2. CRITICAL — health-снапшот теряется безвозвратно
`multiprocess_framework/modules/process_module/health/state.py:325-329` и `:459-472`

`take_dirty()` сбрасывает `_dirty = False` **до** возврата снапшота. Далее каждый `proxy.set` обёрнут в `except: pass`. Публикация провалилась → снапшот съеден, флаг снят, ретрая нет; следующий такт вернёт `None` («изменений нет»).

**Последствие:** процесс ушёл в `degraded`/`failed`, дерево состояния навсегда показывает последнее удачное `ok` — до следующего *изменения* health. GUI, `backend_ctl` и watchdog смотрят на зелёное. Ровно тот класс, против которого затевалась «громкость» Ф7.
**Живая проверка:** заставить `proxy.set` бросать, перевести процесс в degraded, читать `state_get` по health-пути.

#### H-1. HIGH — `stop_worker` лжёт об успехе → треды-сироты
`worker_module/lifecycle/worker_lifecycle.py:137-141`, `worker_module/core/worker_manager.py:119-120`

**Найдено двумя независимыми охотниками** (гонки и утечки) — сильнейший сигнал документа.

```python
if worker_info["thread"].is_alive():
    worker_info["thread"].join(timeout=timeout)
...
return True          # ← безусловно, результат join НЕ проверяется
```
а `remove_worker` результат `stop_worker` **вообще игнорирует** и делает `unregister` безусловно.

**Последствие без всякой гонки:** воркер завис в `cv2.read()` (5-30 с) → `join` истёк → запись снята из реестра → тред жив и невидим для `stop_all_workers` и GUI. Следующий `create_worker` с тем же именем стартует **второй** тред на том же плагине. Для source-воркера — два писателя одного SHM-кольца; `LoanLedger._bind_writer` (`loan_ledger.py:93-99`) поймает это как `RuntimeError`, но постфактум.
**Связь:** вероятный кусок известного «graceful-stop debt» (5 с ханг при switch/shutdown). Гипотеза: система не «не может остановить», а **считает, что остановила**, и идёт дальше.

**Дополнительно (гонка, тот же узел):** `worker_registry.get()` отдаёт живой dict (докстринг сам ссылается на «внешнюю блокировку», которой нет); `stop_worker` читает `worker_info["thread"]` дважды раздельно (`:137` `.is_alive()`, `:138` `.join()`), а `_restart_worker_internal` присваивает новый тред на `:235` и стартует его на `:243` — окно в 8 строк, где там лежит несстартованный `Thread`. Interleaving даёт `RuntimeError: cannot join thread before it is started`.
**Живая проверка:** снять воркер, зависший в `read()`, сравнить `introspect_status.workers` с реальным числом тредов процесса.

#### H-2. HIGH — плагин, упавший в `configure()`, не попадает в shutdown
`process_module/generic/plugin_orchestrator.py:163-165`, `:194`

`plugin._do_configure(ctx)` стоит строкой **раньше** `self._plugins.append(plugin)`. Исключение → плагина нет в списке → `shutdown()`, итерирующий `zip(self._plugins, self._contexts)`, его не увидит никогда.

**Последствие:** всё, что плагин захватил до броска, висит до конца жизни OS-процесса без владельца. Достижимо: `Services/ml_inference/plugin/plugin.py:126` создаёт `onnxruntime.InferenceSession` (CUDA) внутри `configure()` — VRAM остаётся занятой. Для source-плагинов — заблокированная камера.
**Фикс:** переставить `append` ДО `_do_configure` (обратное направление корректно: `_do_shutdown` короткозамыкается только на `STOPPED`).

#### H-3. HIGH — «удалить процесс» в GUI расходится с рантаймом и рапортует успех
`multiprocess_prototype/frontend/widgets/tabs/processes/presenter.py:437-443`

```python
self._services.topology.save(new_topo)                 # персист УЖЕ выполнен
try:
    self._topology_bridge.hot_remove_process(name)     # cmd "process.hot_remove"
except Exception:   # nosec B110
    pass                                               # провал проглочен
return True                                            # вызывающему — успех
```

**У команды `process.hot_remove` нет приёмника.** Проверено лично: все вхождения в репозитории — отправляющая сторона (`system_commands.py:97` строит, `topology_bridge.py:428` шлёт, `presenter.py:441` вызывает); `register_command("process.hot_remove", ...)` — ноль совпадений. В `_register_builtin_commands` (`process_manager_process.py:296-326`) есть `process.create/start/stop/pause/resume/restart/status/list`, но нет `hot_*`; своя таблица `TopologyManager._execute_command` их тоже не содержит. Долг частично задокументирован: `multiprocess_framework/docs/COMMUNICATION_MAP.md:222` (про парный `hot_add`).

**Последствие:** топология сохранена без процесса, а процесс продолжает работать. Расхождение конфига и рантайма, ни одного сигнала пользователю. После рестарта процесс исчезает — выглядит как «сработало, просто нужен был перезапуск», поэтому баг не заводят.

Парный `process.hot_add` (`system_commands.py:77`) тоже без приёмника, но его отправители (`topology_bridge.py:401`, `:567`) в проде недостижимы — только self-calls и тесты. Формальный мисматч, практического отказа нет.

**Живая проверка:** `introspect_handlers` на PM — убедиться, что `process.hot_remove` отсутствует; затем удалить процесс из GUI и посмотреть `introspect_status`, жив ли он.

#### M-1. MED — `bootstrap.py:36` не защищён от явного null
`multiprocess_prototype/backend/state/bootstrap.py:36` — `.get("priority", "normal")` срабатывает только на ОТСУТСТВУЮЩЕМ ключе; GUI-редактор пишет `priority:` пустым скаляром → `None` попадает в дерево состояния.

### ❌ Проверено и ОТВЕРГНУТО (не тратить время повторно)

Раздел существует намеренно: в аудитах обычно пишут только найденное, из-за чего одни и те же тупики перепроверяют по кругу.

**О-1. `_FRAME_ID_MODULO = 121` — НЕ баг.** В планах висело как «гигиена-остаток, мооты TTL-выселением, но лучше выровнять» с хеджем. Доведено до чисел:

```
оборот seq_id при 21 fps = 121/21 ≈ 5.8 с
TTL набора джойна        = 2 × timeout_sec = 0.16 с
запас ≈ 36×   (при 60 fps ≈ 12×)
```
Коллизия требует, чтобы набор дожил до возврата того же `seq_id` — TTL должен вырасти в 36 раз. **Хедж можно снять:** это выравнивание ради единообразия, не дефект.

**О-2. Класс «`.get(key, default)` против явного null» — в основном безвреден.** Первичный grep дал ложные срабатывания (паттерн `^\s+key:\s*$` не отличает пустой скаляр от вложенного блока — так в список попали `transport`/`params`/`plugins`/`config`). Парсер показал: явный null имеют ровно 5 ключей — `target_process` ×49, `category` ×49, `description` ×46, `priority` ×11, `crop` ×10. Потребители почти везде используют проверку на истинность или `or ""`, где `None` ведёт себя как `""`. Осталось настоящего: только M-1 выше + лгущие аннотации типов (`target_process: str`, содержащий `None`).

**О-3. Нарушений Dict-at-Boundary в проде не найдено.** Прогон по всему репо на Pydantic/непиклимые объекты в payload: совпадения только в тестах. Путь до `queue.put` типизирован `Dict[str, Any]` на каждом звене; кадры ходят через SHM по имени. Историческая дыра `No handler for key 'register_update'` закрыта fallback'ом на `register_class` (`plugins/base.py:311-313`).

### 🔍 Не находки, но структурные наблюдения

- **У правила №1 нет рантайм-стража.** `mp.Queue.put` пиклит в фоновом feeder-потоке → непиклимый payload НЕ бросит на месте вызова: `put` вернётся успешно, `send_to_process` отдаст `True`, объект умрёт с трейсбеком в stderr чужого потока. Дисциплина держится на конвенции и ревью. Предложение: assert-guard на `send_to_process`, отвергающий не-JSON-совместимый payload, под dev-флагом. *(Механизм — стандартная семантика CPython; на конкретном коде стоит подтвердить вживую.)*
- **Каналы-сироты.** `FieldRouting(channel=...)` объявлен в проде 8 раз без зарегистрированных каналов. Отказа нет: `channel` не читается на живом пути (`command_catalog.py:122-129` берёт только `process_targets`), а код, который бы его читал, помечен мёртвым. LOW сейчас, **HIGH при реактивации** `routing_map` — там при заданном `channel` ранний `return []` отменяет fallback по `targets`.
- **Две системные дыры логирования:** `Services/device_hub/` и `config_module/core/config.py` — по нулю вхождений logger/logging. При этом `registers_module` имеет ADR-RM-004 «Логирование вместо silent except». Стандарт в проекте сформулирован — эти два пакета из него выпали.

### Новые живые находки

**LIVE-1. HIGH — канал `system_events` мёртв, ошибки PM растут бесконечно.**
`EventManager.emit_event` (`events/core/manager.py:131-138`) шлёт каждое событие в канал `system_events` (`targets: ["ProcessManager"]`) — **потребителя не существует** (grep по PM и прототипу: 0; live: `listening: false`, очередь стоит на 94). Очередь набивается на старте; далее каждый emit бьётся о never-drop-политику → put падает → `errors` PM растут **~13/с** (baseline) и **~57/с** при активной GUI-подписке (замерено: 1886→6985 за ~90 с, дельта `queue_system_evict_blocked` совпадает до штук). Следствия: (а) счётчик `errors` PM перманентно замусорен — настоящую ошибку в нём не увидеть; (б) реальный GUI генерирует этот поток всю жизнь сессии. Живой родственник «каналов-сирот» из §5. Throttled-ERROR из queue-manager при этом **никуда не пишется** (его `_log_error` не подключён к логгеру — ни в файле, ни в observability-плоскости).

**LIVE-2. CRITICAL (для G.7 Ф3) — вытеснение из data-очереди теряет loan → перманентная смерть кольца.**
Репро: webcam_sketch + 9 флагов лесенки (проба G.7). На старте `lines` грузит TEED ~10 с → data-очередь переполнена → `remove_old_if_full` (drop_oldest, `queues/core/manager.py:336-341`) выбрасывает сообщения **вместе с loan-тикетами**. Release шлёт только обработавший кадр исполнитель (`pipeline_executor.py:287-321`; drop исполнителем тоже релизит — `test_g5c_executor_drop`), за вытесненное из очереди не релизит никто; `reclaim_reader` (`frame_shm_middleware.py:523`) покрывает ТОЛЬКО мёртвого читателя. Итог: ≥4 вытеснений → free-list `seg/output_frames` исчерпан **навсегда**, «кадры не сохраняются», skipped растёт со скоростью FPS (наблюдалось 1→1200 за минуту). Без флагов (В1 round-robin) симптома нет — потому вчерашняя синтетика жила 2 ч. **Это открытая fault-инъекция G.7 «2.4 slow-consumer», пойманная не инъекцией, а первым же живым запуском.** Направления фикса (дизайн-решение Ф7, не хотфикс): release при вытеснении (транспорту нужен доступ к тикетам конверта) либо owner-side TTL-реклейм застрявших слотов под защитой В1 re-check («занижение безопасно» — §8.2 G.5). **Webcam-soak Ф3 заблокирован до фикса.**

**LIVE-3. HIGH — main-процесс не выходит после system.shutdown.**
Дети терминированы (после 5с-таймаутов), порты свободны, а главный процесс жив 3+ минуты: **1 тред, CPU 0** — вечный `join()`/`wait()` на хвосте shutdown. Прямое объяснение вчерашних «сирот от харнесса». Диагноз до строки не доведён (py-spy в venv нет).

**Мелочь:** `updated_at: 0.0` в health-снапшоте (и через `health.status`, и в state-дереве) — таймстемп не проставляется никогда.

### Оценка backend_ctl как инструмента (владелец просил)

Работает и окупается: `capabilities` закрыл H-3 одним вызовом; `system_overview.anomalies` сам вывел на LIVE-1; `introspect_memory` дал живое свидетельство A-1; `watch_like_gui` = приёмный профиль GUI без GUI (и его включение стало нагрузочным тестом, вскрывшим ×4 ускорение LIVE-1). Задокументированные дыры инструмента:

1. `events` не ограничивает пачку по байтам — 40 событий = 155 КБ (replay-снапшот подписки 102 КБ одной дельтой); E.3-усечение на events не распространяется
2. `system_command`: описание инструмента даёт пример `{'command': ...}`, сервер требует `'cmd'` — пример из собственной доки не проходит
3. Плоскости `logs`/`errors` слепы ко всему, что мимо ObservableMixin/LoggerManager: loguru-логи Services (MODBUS-ретраи каждую секунду) и неподключённые логгеры framework-модулей (throttled-ERROR queue-manager) в них не попадают
4. На старте пробного харнесса PM отвечает «No handler» на `introspect.*`/`router.relay` — гонка: опрос раньше регистрации хендлеров (шумит WARNING'ами в лог)
5. gui-процесс без командной поверхности («no channel resolved») — известно; для внутренностей GUI связка с qt-mcp обязательна

## Открыто — по приоритету

### 1. Потеря 22-23% never-drop трафика ProcessManager (в работе)

**Механизм доказан** арифметическим тождеством, проверен независимо дважды:
```
attempted − middleware_dropped − sent_via_targets − sent_via_channel == errors   (сошлось точно 2/2)
errors +336 ↔ queue_system_evict_blocked +338                                    (корреляция 1:1)
```
100% ошибок PM — ветка «no channel resolved» (`router_manager.py:349`), но текст врёт: каналы резолвятся, а `_deliver_by_targets` упирается в **полную never-drop (system) очередь**. 99% отправок идут через `send_async`, результат никто не читает → сообщение исчезает без следа. Нарушен инвариант, дословно записанный в `qos.py`.

**Не регрессия ветки** — файлы идентичны `main`.

**Получатель УСТАНОВЛЕН — `gui`** (коммит `f90f7513`). Живой прогон с правкой наблюдаемости называет сразу: очередь `system` процесса `gui`, **10057 потерь → 103 строки** лога. То есть ProcessManager заливает GUI потоком `state.changed`, тот не успевает разгребать, never-drop очередь глубиной 100 переполняется.

Сходится с известной болью: память `project_webcam_sketch_freeze` (блокирующий IPC-шторм в главном потоке GUI) и ADR-136 (`project_gui_telemetry_read_model`: запись-всегда / чтение-локально). Вероятно один и тот же корень — **проверить эту связь первым делом**, возможно правка ADR-136 не покрыла путь `state.changed`.

Почему логи молчали (диагноз агента): штатная плоскость `self._log_*` у `QueueRegistry` молчит **по построению** — ни один продовый вызов `SharedResourcesManager(...)` не передаёт logger (`spawner.py:69`, `bundle_builder.py:53`, `process_runner.py:148`), плюс `ObservableMixin.__getstate__` выкидывает `_registry` при pickle. Починено через модульный stdlib-логгер с троттлингом 5 с.

**ОТКРЫТО и важно — противоречие в глубине очереди.** `system_overview` показывает у `gui` `queues: {"system": 0}`, расследование сделало 184 замера за 25 с — максимум 2 из 100. При этом очередь отвергает ~17 сообщений/с. Глубина читается почти нулевой, пока идёт отказ. Либо очередь заполняется и опустошается между замерами (всплески), либо `introspect_queues` меряет не тот объект. **Потенциально четвёртая слепая зона инструмента** — разобрать до того, как чинить саму потерю, иначе будем чинить по ложному показанию.

`RouterStats` открыл все счётчики отправки — коммит `b74f00f6`.

Решение владельца: **сначала наблюдаемость, потом чинить потерю**. Сам фикс (`dropped_reliable`, backpressure) не начат сознательно.

Отдельно, архитектурное и НЕ решённое: высокочастотный `state.changed` едет с `queue_type: "system"` по надёжной очереди глубиной 100, хотя по QoS у `state` профиль `best_effort`/`drop_oldest`. Класс груза противоречит очереди — похоже на корень перегрузки. Это разговор с владельцем, не задача.

Дом находки — [`plans/transport-single-policy.md`](../../plans/transport-single-policy.md), не план backend_ctl.

## Грабли захода — не наступать снова

- **Два бэкенда на одном порту.** Параллельные live-suite на 8765 дают каскад ложных `BackendUnavailable`: получил 8 красных вместо 3 и чуть не отчитался этим baseline'ом. Live-прогоны — **строго одиночные**.
- **Мой монитор соврал «finished».** `pgrep -f "pytest backend_ctl"` не сматчил процесс, цикл вышел мгновенно. Не доверять самодельным детекторам завершения — брать факт из вывода.
- **MCP-сервер держит код на момент своего старта.** `telemetry_snapshot` через MCP не показывал поля, добавленные коммитом того же дня; прямой `BackendDriver` показывал. Версия сборки агенту не сообщается. Проверять свежие правки — перезапуском сервера или напрямую.
- **`logs/prototype_2/` — НЕ мёртвый каталог**, это штатный `log_dir` текущего прототипа (`system.yaml:10`). Я дважды назвал его мусором и был неправ оба раза. Проверять перед `rm`.
- **`config.reload` для подъёма уровня логов не работает** (см. открытый пункт 2) — поднимать уровень правкой конфига файлом.

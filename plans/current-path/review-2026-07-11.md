## 1. Сводные оценки

| Домен | Оценка | Одной строкой |
|---|---|---|
| **Ядро-конструктор / composition root** | **4/10** | Субстрат зрелый, но «рыбы» нет: новое приложение = форк ~29 файлов `backend/` + 2 подкласса-оркестратора |
| **Контрактная плоскость IPC** | **5/10** | Строгость декларативная, не enforced: warn-only + `extra="allow"`, двойственность args/data, 5 осей адресации |

| **Модель плагинов/сервисов** | **6/10** | Контракт-ядро сильное (8/10), но манифеста нет (2/10), таксономия дрейфует, ctx падает поздно и немо |
| **GUI как фреймворк** | **6/10** | «Переиспользуемо форком, не зависимостью»: каркас вкладок в прототипе, идентичность «Inspector» зашита |
| **Supervision** | **6.5/10** | Крепкий, но ПЛОСКИЙ: нет дерева/стратегий/depends_on/экспоненты backoff/эскалации |
| **Наблюдаемость** | **7/10** | ~70% пути (store+вкладки+tail+debug-plane); нет trace-id, alerting, агрегации метрик |
| **Модульное ядро как библиотека** | **7/10** | 0 циклов, чистые слои, фасады; но не отделима: общий pyproject, тяжёлые deps обязательны, `INSPECTOR_*` в ядре |

**Интегрально: ~5.5–6/10.** Инженерная база выше среднего (слоение, живучесть, наблюдаемость, тесты ~6.9k), но сама ЦЕЛЬ — «легко собрать любое приложение» — ещё не материализована: не хватает «рыбы» (app_module + minimal_app), отделимости библиотеки и enforce-контрактов.

Метрики среза: sentrux quality 7078 (bottleneck modularity 5668), DSM 0 циклов / propagation cost 24 / чистое слоение; файловое тест-покрытие 750/2913 (~26%, дыры в основном GUI).

## 3. Главные выводы

1. **Узкое место одно — «рыбы» нет.** Все агенты сходятся: субстрат (IPC, supervision, наблюдаемость, плагины) зрелый, но собрать второе приложение без форка прототипа невозможно. Задачи 5.11–5.13 — сердце всей цели, их статус в плане недооценён.
2. **Вторая ось отсутствует в планах целиком: отделимость библиотеки** (packaging/extras/de-brand/env/публичный API) — ни одна фаза Ф0–Ф8 её не несёт. Без неё «конструктор» остаётся внутренним.
3. **Контракты пока диагностика, не гарантия.** Целевая модель — три ортогональные оси (Адрес / Kind+QoS / Схема) с единой декларацией; конкретные шаги уже в G.2/G.4/4.3/4.9, не хватает задачи strict-валидации.
4. **Достроить манифестную плоскость симметрично:** app.yaml (5.11) + plugin-манифест (4.4, расширить) + service-маркер (5.11) — один формат `version + extras`.
5. **Быстрые дешёвые победы, не дожидаясь фаз:** README (UTF-8, quickstart), CI-шаблон, категории-Enum, `build_form_for_schema`, AppIdentity, экспоненциальный backoff.

Новые задачи NEW-1…NEW-10 сформулированы в [`plan.md`](plan.md) §4.

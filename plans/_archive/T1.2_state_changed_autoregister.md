# План: T1.2 — Авто-регистрация state.changed handler в ProcessModule

**Дата:** 2026-05-08
**Статус:** DRAFT
**Ветка:** main
**Scope:** только `multiprocess_prototype/` (плюс минимальная правка типизации в `multiprocess_framework/`)

---

## Обзор

Фреймворк уже поддерживает авто-регистрацию: `ProcessModule._init_state_proxy()` вызывается
в `initialize()` и регистрирует `state.changed` handler, если `state_proxy` передан в конструктор.
Проблема — все 6 конкретных процессов прототипа и 3 плагина создают `StateProxy` внутри
`_init_application_threads()` / `_init_custom_managers()` (то есть ПОСЛЕ `super().__init__()`),
а не передают в `super().__init__(state_proxy=...)`. Из-за этого авто-регистрация не срабатывает,
а явный вызов `router_manager.register_message_handler("state.changed", ...)` дублируется.

Цель задачи:
1. Перенести создание `StateProxy`/`GuiStateProxy` в каждом процессе ДО `super().__init__()` и
   передать его через `state_proxy=` — тогда фреймворк сам зарегистрирует handler.
2. Удалить ставшие лишними явные вызовы `register_message_handler("state.changed", ...)`.
3. Заменить аннотацию `state_proxy: Any | None` → `IStateProxy | None` в ProcessModule.
4. Закрыть ADR-SS-006 (обновить статус с TODO на «принято, реализовано»).
5. Плагины (`ProcessModulePlugin`) — **не трогать**: они не наследуют ProcessModule,
   механизм авто-регистрации к ним не применим.

---

## Архитектурный контекст

**Ключевой инвариант:** `_init_state_proxy()` вызывается из `initialize()` → `ProcessLifecycle.initialize()`.
К этому моменту `router_manager` уже создан (`_init_managers()` идёт раньше). Значит если
`state_proxy` задан в `__init__`, авто-регистрация гарантированно сработает.

**Текущий порядок в `_init_application_threads` (проблема):**
```
super().__init__()        # state_proxy=None → авто-регистрация пропускается
...
StateProxy(...)           # создаётся позже
router.register_handler() # ручная регистрация — дублирует фреймворк
```

**Целевой порядок (после миграции):**
```
proxy = StateProxy(...)   # создаётся ДО super().__init__()
super().__init__(..., state_proxy=proxy)   # фреймворк регистрирует handler
# router.register_handler() — строка удалена
```

**Проблема с DatabaseProcess:** создаёт StateProxy в `_init_custom_managers()`, который вызывается
уже внутри `initialize()`. Нужно вынести создание proxy в `__init__` (как в RobotProcess).

## Риски и ограничения

- **DatabaseProcess:** StateProxy создаётся в `_init_custom_managers()` — нетипичная точка.
  Решение: создать lazy-прокси в `__init__` через вынесение конфигурации proxy.
- **GuiProcess использует GuiStateProxy (не StateProxy)** — нужно учесть импорт `GuiStateProxy`.
- **Двойная регистрация (баг):** если забыть убрать явный вызов — handler будет вызван дважды.
  Acceptance criterion должен это явно проверить.
- **Плагины оставить как есть** — `ProcessModulePlugin` не передаёт proxy в ProcessModule,
  авто-механизм к ним не применим.

### Задача 2.1 — Миграция RobotProcess

**Уровень:** Middle (Sonnet, normal thinking)
**Исполнитель:** developer
**Цель:** Перенести создание `StateProxy` в `__init__` RobotProcess и удалить ручную регистрацию handler

**Контекст:**
`RobotProcess.__init__` уже переопределён (строки 24-27). StateProxy сейчас создаётся в
`_init_application_threads()` (строки 43-52). Нужно перенести создание в `__init__` до
`super().__init__()` и передать как `state_proxy=...`.

**Важно:** `StateProxy` требует `router=self.router_manager` — но `router_manager` ещё не создан
в `__init__`. Решение: StateProxy принимает `router` как опциональный параметр и router можно
передать позже через атрибут, **или** создать proxy с `router=None` и присвоить router в
`_init_application_threads()` уже только через `self._state_proxy.router = ...`.

Проверить сигнатуру `StateProxy.__init__` — если `router` обязательный, то нужна другая стратегия:
передавать router в ProcessModule и переопределить момент создания proxy.

**Альтернативная стратегия (если router обязательный):** переопределить хук `_init_state_proxy()`
в RobotProcess чтобы создать proxy там и дёрнуть `super()._init_state_proxy()`. В этом случае
стандартный механизм `state_proxy=...` в `__init__` не используется, но ручная регистрация
handler всё равно удаляется — она делегируется переопределённому хуку.

**Третья стратегия (рекомендуемая, если router не нужен при создании):** StateProxy создаётся в
`_init_application_threads()` как `self._state_proxy = StateProxy(...)`, затем присваивается
`self.state_proxy = self._state_proxy` ДО строки с `register_message_handler`. Потом
`_init_state_proxy()` вызывается ещё раз явно: `self._init_state_proxy()`.
**Это ломает контракт** — не делать так.

**Правильная стратегия (финальная):** изучить, есть ли в StateProxy возможность создания
без router (lazy init). Если нет — добавить хук `_post_init_state_proxy()` в ProcessModule или
переопределить `_init_state_proxy()` в процессе.

> **Примечание разработчику:** перед реализацией прочитать сигнатуру
> `StateProxy.__init__` в `multiprocess_framework/modules/state_store_module/core/state_proxy.py`.
> Это ключевой факт для выбора стратегии.

**Файлы:**
- `multiprocess_prototype/backend/processes/robot/process.py`

**Шаги:**
1. Прочитать `StateProxy.__init__` — выяснить, обязателен ли `router`.
2. **Если `router` необязателен (или принимает `None`):**
   - В `RobotProcess.__init__` создать proxy ДО `super().__init__()`: `proxy = StateProxy("robot", router=None, server_target="ProcessManager")`
   - Передать в `super().__init__(*args, **kwargs, state_proxy=proxy)`
   - Сохранить как `self._state_proxy = proxy`
   - В `_init_application_threads()` присвоить router: `self._state_proxy._router = self.router_manager` (или через публичный метод если есть)
   - Удалить строку `self.router_manager.register_message_handler("state.changed", ...)`
3. **Если `router` обязателен:** переопределить `_init_state_proxy()`:
   ```python
   def _init_state_proxy(self) -> None:
       # proxy создаётся здесь, после router_manager готов
       pass  # пустой хук — регистрацию делаем в _init_application_threads вручную
   ```
   В `_init_application_threads()` оставить создание proxy, но удалить явный вызов
   `register_message_handler` — вместо этого вызвать `super()._init_state_proxy()`... нет,
   это рекурсия.
   **Правильно:** вызвать `self.router_manager.register_message_handler` убрать, а proxy
   назначить на `self.state_proxy` и вызвать метод `_init_state_proxy()` сразу после создания:
   ```python
   self._state_proxy = StateProxy(...)
   self.state_proxy = self._state_proxy   # перехватит стандартный хук
   # register_message_handler — УДАЛИТЬ
   self._init_state_proxy()               # явный вызов хука
   ```
4. Убедиться что `self.state_proxy is self._state_proxy`.

**Критерии приёмки:**
- [ ] Строки с `register_message_handler("state.changed", ...)` в robot/process.py удалены
- [ ] `grep -n "register_message_handler.*state.changed" multiprocess_prototype/backend/processes/robot/process.py` — пусто
- [ ] Файл импортируется без ошибок: `python -c "from multiprocess_prototype.backend.processes.robot.process import RobotProcess"`

**Вне scope:** не менять логику команд, адаптер, сервис.

### Задача 2.2 — Миграция CameraProcess

**Уровень:** Middle (Sonnet, normal thinking)
**Исполнитель:** developer
**Цель:** Удалить ручную регистрацию `state.changed` handler в CameraProcess, передать proxy через авто-механизм

**Контекст:**
StateProxy создаётся в `_init_application_threads()` строки 63-82 с `camera_id`-параметризацией.
`camera_id` берётся из конфига `app_cfg = self.get_config("config") or {}` — это доступно только
после `initialize()`. Значит ранний перенос в `__init__` невозможен без рефакторинга.

Применить стратегию «назначить после создания»: после создания `self._state_proxy = StateProxy(...)`
сразу установить `self.state_proxy = self._state_proxy`, удалить явный `register_message_handler`,
затем явно вызвать `self._init_state_proxy()`.

**Файлы:**
- `multiprocess_prototype/backend/processes/camera/process.py`

**Шаги:**
1. В `_init_application_threads()` после строки `self._state_proxy = StateProxy(...)`:
   - Добавить: `self.state_proxy = self._state_proxy`
   - Удалить: `self.router_manager.register_message_handler("state.changed", self._state_proxy.on_state_changed)`
   - Добавить: `self._init_state_proxy()`
2. Убедиться, что `self._init_state_proxy()` вызывается только **один раз** — `initialize()` тоже
   вызывает его, но к тому моменту `state_proxy` был `None`. После нашей правки `state_proxy`
   уже назначен, и повторный вызов из `initialize()` будет cold — нужно проверить, идемпотентен ли
   `_init_state_proxy()` (регистрирует ли handler дважды при двойном вызове).
   - Если не идемпотентен: добавить флаг `_state_proxy_registered: bool = False` в ProcessModule.
   - Если идемпотентен (RouterManager игнорирует дублирующую регистрацию) — оставить как есть.
3. Проверить `grep -n "register_message_handler.*state.changed"` — убедиться что строки нет.

**Критерии приёмки:**
- [ ] Строки с `register_message_handler("state.changed", ...)` в camera/process.py нет
- [ ] Файл импортируется без ошибок
- [ ] `self.state_proxy` назначается до вызова `_init_state_proxy()`

**Вне scope:** не трогать SHM middleware, ring buffer, воркер, команды.

**Edge cases:** двойная регистрация handler — выяснить и либо предотвратить, либо задокументировать что RouterManager идемпотентен.

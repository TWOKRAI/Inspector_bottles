## 1. Цель

Заменить ручное поддержание сводных разделов в документации **расширяемым sync-каркасом**:

- Single source of truth — там, где факт **возникает** (модульный `DECISIONS.md`, тело корневого `DECISIONS.md`).
- Сводки и оглавления **генерируются** sync-модулями между маркерами.
- Один CLI: `python -m scripts.sync [--check] [--only NAME] [--list]`.
- `scripts/validate.py` зовёт `--check` → дрифт физически невозможен.
- Архитектура расширяется на любые будущие синхронизации (например, `MODULES_STATUS.md`) без переписывания.

## 2. Что делает sync framework

| Что синхронизирует | sync-модуль | Источник истины | Целевой раздел |
|--------------------|-------------|-----------------|----------------|
| Таблица «Модульные решения» в корневом `DECISIONS.md` | `adr_modules` | заголовки `## ADR-{CODE}-NNN: Title` в `modules/<X>/DECISIONS.md` | между `<!-- ADR-INDEX:BEGIN/END -->` |
| Таблица «Коды модулей» в `docs/ADR_REGISTRY.md` | `adr_modules` | те же | между `<!-- ADR-CODES:BEGIN/END -->` |
| Оглавление глобальных ADR в корневом `DECISIONS.md` | `adr_toc` | заголовки `## ADR-NNN: Title` ниже `<!-- ADR-TOC:END -->` | между `<!-- ADR-TOC:BEGIN/END -->` |
| Раздел «Устарело» в корневом `DECISIONS.md` | `adr_obsolete` | ADR с `Статус: устарело` в теле документа | между `<!-- ADR-OBSOLETE:BEGIN/END -->` |

`adr_lint` (проверка кросс-ссылок и оборванных `ADR-XXX` после миграции) — **не входит** в текущий PR. Запланирован отдельной задачей T2.5.

## 3. Архитектура

### 3.2. `registry.py` — общий механизм

Минимальный API:
```python
@dataclass
class SyncModule(Protocol):
    name: str                                  # 'adr_modules', 'adr_toc', ...
    description: str                           # для --list
    def render(self) -> dict[Path, dict[str, str]]:
        """Вернуть {file_path: {marker_key: rendered_content}}."""

def apply_sync(modules: list[SyncModule], *, check: bool) -> int:
    """write или check; возвращает exit code."""

def replace_between_markers(text: str, key: str, content: str) -> str:
    """Заменить содержимое между <!-- KEY:BEGIN ... --> и <!-- KEY:END -->.

    Если маркеров нет — поднимает MarkerNotFound с понятным сообщением
    (надо один раз обернуть существующий блок маркерами вручную).
    """
```

Маркеры:
```markdown
<!-- ADR-INDEX:BEGIN — generated by scripts/sync/adr_modules.py — DO NOT EDIT -->
... сгенерированная таблица ...
<!-- ADR-INDEX:END -->
```

### 3.3. CLI (`__main__.py`)

```bash
python -m scripts.sync                   # write всё
python -m scripts.sync --check           # exit 0 если sync, exit 1 + diff если drift
python -m scripts.sync --only adr_toc    # выборочно
python -m scripts.sync --list            # перечислить sync-модули и описания
```

Регистрация sync-модулей — явный список в `__main__.py`:
```python
from . import adr_modules, adr_toc, adr_obsolete
SYNC_MODULES = [adr_modules.module(), adr_toc.module(), adr_obsolete.module()]
```
Добавить новый sync — одна строка импорта + одна строка в списке.

### 3.4. Алгоритмы sync-модулей

**`adr_modules`:**
- Сканирует `multiprocess_framework/modules/*/DECISIONS.md`.
- Парсит `^## ADR-([A-Z]+)-(\d+)(?: \(was ADR-[\w]+\))?: (.+)$`.
- Валидирует:
  - один `{CODE}` на модуль (иначе fail);
  - один модуль на `{CODE}` глобально (защита от ADR-CM-коллизии);
  - модуль присутствует в `_adr_layers.MODULE_LAYERS` (иначе fail с подсказкой «добавь строку в `_adr_layers.py`»).
- Рендерит:
  - **корневой `DECISIONS.md` § «Модульные решения»**: таблица `Модуль | Файл | Слой | Статус`. `Статус = "ADR-{CODE}-{first}…{last} ({first_title}, ..., {last_title})"` (если ADR один — без "..."; если два — `({title1}, {title2})`).
  - **`docs/ADR_REGISTRY.md` § «Коды модулей»**: таблица `Код | Модуль | Файл решений`.
- Порядок строк — из `_adr_layers.MODULE_LAYERS`.

**`adr_toc`:**
- Читает корневой `DECISIONS.md`.
- Берёт всё **ниже** маркера `<!-- ADR-TOC:END -->` до раздела «Устарело» / «Модульные решения».
- Извлекает `^## ADR-(\d+): (.+)$` (только глобальные ADR с числовым `NNN`, без префиксов).
- Сортирует по числу.
- Помечает `(устарело)` те, у кого в теле `^- Статус: устарело$` или `Суперсед:`.
- Рендерит bullet-список:
  ```markdown
  - [ADR-001](#adr-001-observablemixin-остаётся): ObservableMixin остаётся
  - [ADR-099](#adr-099-...): ... *(устарело, см. ADR-100)*
  ```

**`adr_obsolete`:**
- Тот же скан тела `DECISIONS.md` после `<!-- ADR-TOC:END -->`.
- Фильтрует ADR с `Статус: устарело`.
- Рендерит **только заголовки + ссылки**, без полного тела:
  ```markdown
  - **ADR-099**: Пакет `multiprocess_prototype.registers_2` — слой над `schema_v3` без второго RegistersManager *(Суперсед: **ADR-100**, 2026-03-27)*
  ```
- **Разовая чистка перед первым `--write`:** найти ADR, тело которых сейчас дублируется в текущем разделе «Устарело» (на сегодня — только `ADR-099`). Удалить дубль из «Устарело» руками; полное тело остаётся в основной части документа. Оборачиваем секцию маркерами `<!-- ADR-OBSOLETE:BEGIN/END -->`.

### 3.5. ADR об архитектуре sync

Создаётся **`ADR-119: Sync framework для генерируемых разделов документации`** в корневом `multiprocess_framework/DECISIONS.md`:
- Контекст: ручные сводки рассинхронизуются (4 модуля выпали из реестра, ADR-CM коллизия, дубль ADR-099).
- Решение: каркас `scripts/sync/` с маркерами и плагинными sync-модулями. CLI `python -m scripts.sync`. Подключён к `validate.py`.
- Последствия: новый тип синхронизации = новый файл в `sync/`, не правка монолита. Агенты следуют простому правилу: «после правок документации — `python -m scripts.sync`».
- Альтернативы (отклонены): pre-commit hook **только** (теряем явный CLI), один монолитный скрипт (не масштабируется).

## 7. Риски и тонкости

1. **Текущая ручная таблица «Модульные решения» содержит flavor-текст** в столбце «Статус». Новый формат `(first_title, …, last_title)` будет иначе. Косметическая разница ожидаема и принимается reviewer-ом — генератор даёт стабильность взамен.
2. **`config_module`, `shared_resources_module`** имеют **смешанные** ссылки (глобальные + локальные ADR). Сводка генерится только по локальным `ADR-{CODE}-NNN`. Глобальные ссылки в теле не теряются (живут в основном теле документа). Текущая ручная пометка `"ADR-023 (global) + ADR-143…146"` упростится до `"ADR-CFG-001…004"`.
3. **`adr_obsolete` чистка дублей.** На сегодня в репо один такой ADR (`ADR-099` дублируется в основном теле + в «Устарело»). Чистка тривиальна. Но в принципе скрипт должен **детектировать дубли** при первом запуске и фейлиться с подсказкой («тело `ADR-NNN` найдено дважды — оставь его в основной части документа, я сгенерирую раздел `Устарело`»).
4. **`scripts/validate.py`** сейчас использует `sys.path.insert` — добавление вызова `from scripts.sync import run_check` потребует чтобы `scripts/sync/` был импортируемым пакетом. Делаем это аккуратно: либо вызов через `subprocess.run([sys.executable, "-m", "scripts.sync", "--check"])` (чище, без path-хакинга), либо добавляем `scripts/__init__.py`.
5. **Если ADR-119 будет первым ADR который добавляется автогенератором**, он попадёт в `--write` сразу как часть TOC. Reviewer должен это видеть и не считать «лишним diff».
6. **Атомарность PR.** Маркеры + первичная генерация + правки `validate.py` + `ADR-119` коммитятся одним PR. Иначе промежуточное состояние ломает `/validate`.

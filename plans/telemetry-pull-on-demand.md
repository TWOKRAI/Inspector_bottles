# telemetry-pull-on-demand — телеметрия по опросу, а не по push

> **Статус: DRAFT** (2026-07-22) — направление сформулировано владельцем, ждёт go-ahead.
> Слаг: `telemetry-pull-on-demand`. Ветки нет.
> Возникло из live-приёмки Фазы 1 `truth-holes-closure`
> ([аудит](../docs/audits/2026-07-22_phase1-flip-webcam-sketch.md)).

## Формулировка владельца

> «Любая телеметрия должна работать при опросе (зайти в окно наблюдения или сделать
> команду опроса). А до этого — просто записывать у себя в памяти, в самом процессе.»

## Зачем (что осталось нерешённым после Фазы 1 и ADR-136)

Фаза 1 `truth-holes-closure` и ADR-136 решают **разные половины**, но обе — на уровне
«трафик уже создан»:

| Слой | Что решено | Чем |
|---|---|---|
| Чтение в GUI | дёшево, без блокирующего IPC | ADR-136 read-model (`gui-telemetry-read-model`) |
| Транспорт | поток состояния не душит команды | Ф1: коалесцирование + очередь класса `state` |
| **Генерация** | **не решено** | ← этот план |

Live-факт (baseline `webcam_sketch`, флаги OFF): **1728 попыток отправки за ~30 с,
1461 отказ, 597+ безвозвратных потерь** — и это на штатной нагрузке, без искусственного
флуда. Флип Ф1 сделал доставку 100 %, но **сообщения продолжают генерироваться**.
Дешевле всего то сообщение, которого нет.

Главный приз pull-модели: **закрытое окно = ноль трафика**. Push-модель этого даром не
умеет — ей нужен явный вентиль на каждую метрику (ADR-PM-018 такой вентиль дал, но им
надо управлять). Pull получает это как свойство.

## Ключевое проектное решение: делить по природе сигнала, не по слову «телеметрия»

Наивный «pull для всего» ломается: события между опросами исчезают бесследно.

| Класс | Примеры | Модель | Почему |
|---|---|---|---|
| **Уровни (levels)** | FPS, latency, глубина очереди, uptime, счётчики, температура | **pull** | пропуск промежуточного значения безвреден — важно текущее |
| **Фронты (edges)** | ошибка, процесс умер, устройство отвалилось, задание завершено, значимый переход состояния | **push** | пропуск = потеря события навсегда; переход A→B→A между опросами исчезнет |

Правило: **показывает → опрос; запускает действие или сообщает о необратимом → push.**
Согласуется с ADR-PM-018 (ошибки прибиты always-on) — тот вентиль остаётся верным.

## Обязательные условия, иначе шторм вернётся с другой стороны

Опрос выигрывает **только при всех трёх** разом:

1. **Частота опроса < частоты изменений.** Обычно так (меняется 30/с, на экране нужно
   1/с). Если опрашивать чаще изменений — стало хуже: запрос+ответ = **два** сообщения
   против одного push.
2. **Пакетный опрос.** Один запрос на процесс → много метрик. Иначе 7 процессов × 20
   метрик = 140 круговых рейсов вместо 7.
3. **Опрашивается только видимое.** Закрытая вкладка/окно не опрашивает вовсе.

## Что нужно в процессе

Локальное кольцо: последнее значение каждой метрики + короткая история (для графиков).
Память копеечная, IPC нулевой. По запросу отдаётся снимок (при необходимости
прореженный до частоты отображения).

## Открытые вопросы (до go-ahead)

- Ревизия классификации: какие из текущих `processes.**` — уровни, какие фронты.
  Нужен инвентарь по факту, а не по названию.
- Судьба дерева состояния: остаётся для **настоящего** реактивного состояния
  (конфиг-эффекты, переключения топологии), из него уходят только уровни.
- Кто владеет опросом: GUI-таймер видимой вкладки? `TelemetryViewModel` (ADR-136 уже
  единый владелец «снимок→виджет» — вероятно, он же).
- Дедлайн/актуальность: помечать снимок временем сбора, чтобы GUI отличал «свежо» от
  «процесс завис».
- Совместимость с `backend_ctl`: у инструмента опрос уже естественен
  (`telemetry_snapshot`/`telemetry_history`) — свести к одному механизму, не плодить второй.

## Соотношение с соседями

- **Не отменяет Фазу 1** `truth-holes-closure`: коалесцирование и очередь `state`
  остаются страховкой транспорта для того, что всё-таки push (фронты, реактивное
  состояние). Ф1 — пол, этот план — снижение потолка.
- **Не отменяет ADR-136**: read-model остаётся владельцем «снимок→виджет»; меняется
  источник снимка (не поток дельт, а ответ на опрос).
- **Не воскрешает** отклонённый snapshot-канал из `telemetry-delivery-simplification`
  (SUPERSEDED): там был третий **push**-путь бэкенд→GUI; здесь — request/response
  поверх существующего командного пути, без нового data-plane.

# Ф7 hot-path — план исполнения (брифы для Opus/Sonnet)

> Создан 2026-07-13 на закрытии GATE G3. Статусы задач — ТОЛЬКО в [plan.md](plan.md) (этот файл — брифы и порядок).
> Роли: **исполнители — Sonnet 5 / Opus 4.8** (model override per задача), **Fable — ревью каждой задачи + merge + вердикт G.7**.
> Sentrux-baseline фазы снят 2026-07-13: quality 7088 (скан 3956 файлов / 588 708 строк).

## 1. Честная оценка: что было → что будет

### Что есть сейчас (замерено/из кода, не предположение)

| Ось | Факт |
|---|---|
| Границы процесса на кадр | **3–9** против нормы 1–2 (статика по рецептам: modbus_demo 9, region_pipeline 8, letter_angle 7, hikvision 5). Каждая граница = Pydantic→to_dict→pickle→mp.Queue→роутер: десятки–сотни мкс **+ джиттер**. Механизм снижения уже влит (C6(d)/(e): цепочка внутри воркера) — Ф7 замеряет и дожимает |
| SHM-налог на кадр (full-HD ~6 МБ) | **2 полные копии** (запись ~0.5–1 мс + чтение `.copy()` ~0.5–1 мс × каждого читателя) **+ open/mmap/close на каждый кадр** (основной cross-process путь) + resource_tracker. Итого ~1.5–4 мс на кадр только на транспорт |
| Корректность | Ring-of-3 **без синхронизации**: перезапись слота под читателем возможна по построению → torn frame (принятый долг). Осиротевшие SHM-сегменты после kill -9 висят |
| Тихие деградации | Сбой SHM-write → кадр молча едет pickle-через-Queue (латентность ×3, метрик ноль); дроп кадров тихий; system-сообщения делят политику переполнения с data |
| p99-хвосты | 7 TRACE в on_receive на горячем пути; тысячи мелких аллокаций на кадр (dict, Pydantic Message) → GC-паузы → выбросы p99 |

### Что будет после Ф7

| Ось | Результат | Чем подтверждается |
|---|---|---|
| Границы/кадр | ≤ 1–2, виден runtime-счётчик | G.6 счётчик + G.1/G.7 baseline |
| SHM-налог | кэш handles (0 syscalls/кадр) + zero-copy чтение (0 копий у читателя); в пределе — 0 копий/0 syscalls (спутник «запись у источника» — по возможности SDK) | G.3/G.5, замер до/после |
| Корректность | torn frame **невозможен по построению** (seqlock + пул с владением по цепочке); осиротевших сегментов после рестарта 0; stale-процесс не пишет (fence/incarnation) | репродьюсеры G.3 (torn + HP-5) |
| Громкость | pickle-fallback, дроп, исчерпание пула — всё кричит (счётчики → heartbeat → state → вкладка Pipeline); system никогда не дропается | acceptance G.3(d)/G.4 |
| p99 | TRACE=0, GC по расписанию + `gc.freeze()`, per-frame путь без Pydantic | G.1/G.9, soak G.7 без GC-выбросов |
| Мульти-камера | 2–3+ камер, у каждой своя цепочка: независимые пулы, глубина по длине СВОЕЙ цепочки, деградация одной не трогает другие | G.4, frame-pool принцип 6 |

### Насколько лучше — честно, без маркетинга

- **Латентность/джиттер:** выигрыш ~несколько мс на кадр (копии+syscalls) и заметное сужение p99 (GC, TRACE, тихий fallback). При бюджете 33 мс (30 FPS) это не «в разы быстрее в среднем» — это **вычищенные хвосты и предсказуемость**.
- **FPS:** вырастет только если лимитером был тракт. Честно: на sketch-рецептах лимитером может быть инференс (TEED/YOLO) или сама камера — тогда FPS не изменится, но тракт перестанет быть узким местом и появится запас на 2–3+ камер. Поэтому вердикт — только по замеру G.1→G.7 (лесенка: синтетика → вебкамера → Hikvision, сравнение same-tier).
- **Безопасность/корректность — крупнейший выигрыш фазы:** класс багов «порванный/устаревший кадр, тихая деградация» закрывается по построению, а не тестами. Это дороже нескольких мс.
- **Главный риск (не замалчиваем):** протокол владения слотами — падение читателя с невозвращённым слотом = утечка → исчерпание пула → дропы. Нужен reclaim по incarnation/супервизору + soak. Это самая сложная часть, поэтому G.3/G.4 — Opus.
- **Страховка:** каждый под-шаг за feature-flag, откат = флаг off (бит-в-бит прежнее поведение по G.2-паритету).

## 2. Порядок и раскладка по моделям

Строгая последовательность (зависимости, «одним вскрытием», один агент единовременно, ветка `feat/constructor-f7`, worktree):

| # | Задача | Модель | Почему такая модель |
|---|---|---|---|
| 1 | G.6 trace-id + счётчик границ | **Sonnet 5** | S-усилие, семантика полей, не hot-path-риск |
| 2 | G.1 TRACE=0 + perf-пробы + baseline (tier: синтетика) | **Sonnet 5** | механика + замер по готовому чек-листу |
| 3 | G.2 характеризационные тесты + kind-каналы за флагом + единый конверт команд | **Opus 4.8** | контрактно-чувствительно: паритет бит-в-бит, снятие shape-sniffing `state.merge`, слом двух билдеров конверта |
| 4 | G.3 FrameShm: одна стратегия + seqlock + header под пул + cleanup + громкий fallback + HP-5-репродьюсер | **Opus 4.8** | конкурентность (seqlock), формат header'а на годы вперёд |
| 5 | G.4 QoS-профили + пулы/кольца per-camera + владение по цепочке + drop-на-источнике | **Opus 4.8** | протокол владения/reclaim — главный риск фазы |
| 6 | G.5 снятие двойной конверсии + zero-copy чтение | **Sonnet 5** | механика, безопасность уже дана seqlock'ом G.3; строго ПОСЛЕ G.3 |
| 7 | G.9 GC-дисциплина + per-frame без Pydantic (msgspec/dict) | **Opus 4.8** | перф-чувствительно, стык с TECH_STACK §4; строго ПОСЛЕ G.5 |
| 8 | G.8 drain→detach→stop воркера | **Opus 4.8** | лайфсайкл-конкурентность (полукадры) |
| 9 | G.7 приёмка: flip флага, soak, метрики | **Sonnet 5** (замер) + **Fable** (вердикт) | исполнение по чек-листу; gate-решение — не исполнителю |

После КАЖДОЙ задачи: ревью → merge в main → следующая. Без формального ревью merge запрещён (правило проекта).

**Экономия ревью (решение владельца 2026-07-13, после G.6):** трёхуровневая схема вместо полного протокола на каждую задачу —
- **Полное 8-угловое** (финдеры+верификаторы на Sonnet, свод — Fable): только рисковые вскрытия G.2/G.3/G.4;
- **Лёгкое** (reviewer-агент Opus, 2–3 угла: line-by-line + cross-file + фасады): S-задачи и механика — G.1, G.5, G.7-замер, пост-ревью-фиксы;
- **Группой**: смежные S-задачи на одной ветке (напр. G.6-фиксы + G.1) — одно ревью на пару перед одним merge; правило «формальное ревью до merge» сохраняется, merge просто реже.

**Модельная политика (уточнено 2026-07-13):** Sonnet 5 — «near-Opus» на кодинге/агентной работе при цене в ~2.5 раза ниже → законный дефолт исполнителя, включая средние задачи; Opus 4.8 — только верхний край (G.3 seqlock/конкурентность, G.4 протокол владения/reclaim — «одно вскрытие», длинный автономный заход). G.2/G.9/G.8 — кандидаты на Sonnet 5 с явным охватом в брифе; если ревью покажет паритет находок с Opus — экономим и там. Урок G.6: промахи исполнителя были по ОХВАТУ (буквальное исполнение брифа), не по качеству строк — лечится брифом, а не тиром модели.

## 3. Брифы задач (вставлять агенту как ТЗ вместе со ссылками)

Общее для всех брифов — «шапка агента»:
- Читать: `plan.md` §Ф7 (строка своей задачи — суть+acceptance = единственный скоуп), [frame-pool-idea.md](frame-pool-idea.md), [observability-messaging-vision.md](observability-messaging-vision.md) §5 (для G.2–G.4), `docs/audits/2026-07-12_recipe-lifecycle-audit.md` (B-6..B-9, для G.3/G.4).
- Ветка `feat/constructor-f7` от актуального main (сверить SHA базы! — грабли worktree-stale-base), isolation: worktree.
- Каждый под-шаг с рантайм-эффектом — за feature-flag (конфиг + env), дефолт = старое поведение.
- Коммиты: Conventional Commits + `Why:`/`Layer:` + `Refs: plans/2026-07-06_constructor-master/plan.md`; коммитить каждый законченный шаг.
- Тесты: `python scripts/run_framework_tests.py` + прицельный pytest своего модуля; прод-значения параметров хотя бы в одном тесте (грабли «тест-параметры прячут окно дефекта»); BackendHarness-smoke при рантайм-эффекте.
- Правила: Dict at Boundary; per-frame путь БЕЗ Pydantic-валидаций; НЕ трогать скоуп соседних G-задач; qex-first при поиске использований.
- **Наблюдаемость строго через фасады (требование владельца, 2026-07-13):** логи → LoggerManager, ошибки → ErrorManager (фактический маршрут — ctx.log_error), статистика → StatsManager/штатный stats-механизм модуля. Никаких самодельных счётчиков с колбэками/lock'ами на горячем пути — образец: `FrameShmMiddleware.frame_boundary_crossings` (plain int) + суммирование в `RouterManager.get_stats()`.
- **Конверт кадра (решение владельца, 2026-07-13):** системные поля кадра (`trace_id`, `capture_ts`, `frame_hops`) переносит ДВИЖОК (`PluginRunner._carry_system_fields`), плагин может переопределить, но не потерять. Новые системные поля добавлять в `_CARRIED_SYSTEM_FIELDS`, не в carry-списки плагинов. Формализация конверта как типизированной структуры — стык G.9 (msgspec).
- **Брифы под Sonnet 5 — с явным охватом:** модель исполняет буквально и не обобщает («добавь поле» ≠ «проверь все места, где item пересобирается») — границы и «проверь везде, где…» проговаривать явно (урок ревью G.6).

### G.6 (Sonnet) — trace-id + счётчик границ

Скоуп: семантические поля `trace_id` (hex, OTel-совместимый формат W3C trace context, БЕЗ OTel SDK) в frame_trace и LogRecord; runtime-счётчик пересечений границы процесса на кадр (инкремент в момент IPC-отправки кадра, едет в метаданных кадра, виден в `introspect.router_stats`/stats).
Вход: `router_module` middleware (`frame_shm_middleware.py`), логгер (`channel_routing_module`/LoggerManager), место рождения кадра (source-плагины) — trace_id назначается у источника, дальше только пробрасывается.
Acceptance: по trace_id в логах восстанавливается путь кадра (лог↔кадр коррелируются); счётчик границ на кадр доступен runtime (для baseline G.1); оверхед — только поля, никаких новых аллокаций-объектов на кадр сверх dict-полей.
НЕ делать: не трогать каналы/конверт (G.2), не трогать SHM-запись (G.3).

### G.1 (Sonnet) — TRACE=0 + perf-пробы + baseline tier-1

Скоуп: снять 7 TRACE из on_receive; perf-пробы latency за флагом (HP-1); повторный baseline в `baseline.md`: FPS, p50/p99, **границ/кадр** (счётчик G.6), аллокации — **tier «синтетика»** (синтетический источник кадров нужного размера/частоты; вебкамера и Hikvision — дозамер позже, tier помечать).
Acceptance: grep TRACE hot-path = 0; числа в baseline.md с пометкой tier.

### G.2 (Opus) — характеризационные тесты + kind-каналы + единый конверт

Скоуп — по строке G.2 plan.md (там всё перечислено): паритет-тесты доставки на дефолте `"queue"` (GATE перед любой правкой!); флаг `use_kind_channels` + проводка `resolve_channel_kind` в `_resolve_channels` (router_manager.py:951, ~15 строк); один билдер конверта команд вместо двух, `sql_manager._normalize_command` удалён; `state.merge` — явный маркер конверта вместо shape-sniffing.
Acceptance: флаг off = бит-в-бит прежнее; тесты доставки зелёные до/после; fence-инвариант 4.2 не сломан.

### G.3 (Opus) — FrameShm: seqlock + header под пул + громкий fallback

Скоуп — строка G.3 plan.md (a-d) + внесённое на G3: header слота сразу под пул (`state`/`generation`/опц. `refcount` — ОДИН формат); кэш SHM-handles читателя (инвалидация по имени/incarnation); SHM-имя owner+incarnation (B-6); torn-frame-репродьюсер (до — воспроизводится, после — 0); HP-5-репродьюсер (`replace_blueprint`/unlink × in-flight кадр); startup-cleanup осиротевших сегментов; громкий pickle-fallback (счётчик в state/heartbeat).
Дизайн-рамка: [frame-pool-idea.md](frame-pool-idea.md) — контракт слота держать совместимым с iceoryx2 loan/publish.
Acceptance — по plan.md G.3 (расширенный).

### G.4 (Opus) — QoS + пулы per-camera + владение по цепочке

Скоуп — строка G.4 plan.md: QoS-профили kind (system=reliable/never-drop, data=keep_last+drop-oldest со счётчиками); пул = кольцо + free-list + владение по цепочке (release последним читателем, fan-out — refcount); drop-на-источнике при исчерпании (громкий); **пулы per-camera, камер 2–3+, цепочки у камер разные** → глубина пула = длина СВОЕЙ цепочки × 2, изоляция цепочек; **reclaim слотов при смерти читателя** (incarnation/супервизор — падение процесса не должно утекать слоты; это главный риск — покрыть fault-injection-тестом); B-7 дренаж data-очередей получателей на switch; B-8 refresh handles protected.
Acceptance — по plan.md G.4 (расширенный) + тест: kill -9 читателя посреди чтения → пул восстанавливается, дропов после reclaim нет.

### G.5 (Sonnet) — снятие двойной конверсии

Скоуп: `return_messages=False` на data-plane; `restore_frame(copy=False)` (view + seqlock-проверка поколения после чтения → drop + метрика при гонке). Строго после G.3.
Acceptance: паритет G.2 зелёный; p99 ≤ baseline; при принудительной перезаписи под читателем — drop+счётчик, не torn frame.

### G.9 (Opus) — GC-дисциплина + аллокации

Скоуп — строка G.9 plan.md: `gc.freeze()` после старта воркера + сборка по расписанию; профиль аллокаций/кадр (tracemalloc на soak); per-frame путь без Pydantic → msgspec.Struct/голые dict (бенч msgspec-vs-pickle из TECH_STACK Волна 1 — здесь); Pydantic остаётся на конфигах/границах. Строго после G.5.
Acceptance: p99 без GC-выбросов (замер до/после freeze); 0 Pydantic-валидаций на per-frame пути (grep/трейс); аллокации/кадр в baseline.md.

### G.8 (Opus) — drain→detach→stop

Скоуп: управляемая остановка воркера: drain входа → detach от источника → stop; без полукадров при detach; стык со стоп-механикой C6(e) (сентинелы+cancel+wait=True) — переиспользовать, не дублировать.
Acceptance: нет полукадров при detach (тест); switch/стоп рецепта не оставляет занятых слотов пула.

### G.7 (Sonnet-замер + Fable-вердикт) — приёмка

Скоуп: flip `use_kind_channels` on → soak оба рецепта (tier, доступный на момент прогона); FPS ≥ baseline, p99 ≤ baseline (same-tier!); socket-канал backend_ctl жив (регресс); drop-счётчики видимы; аудит opt-out `manages_own_reply`; p99 без GC-выбросов.
Вердикт gate — Fable, по числам.

## 4. Правила фазы (напоминание себе-оркестратору)

- Один агент единовременно, «одним вскрытием» — никаких параллельных вскрытий hot-path.
- Fable-ревью после каждой задачи (формальное, в транскрипте) → merge → следующая.
- `session_end` sentrux в конце фазы (baseline 7088 снят на старте).
- Статусы задач отмечать ТОЛЬКО в plan.md; QUEUE.md — при закрытии блока.
- При любом конфликте скоупа — стоп и вопрос владельцу, не расширять вскрытие.

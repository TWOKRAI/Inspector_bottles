# Ф4.2 — Реестр контрактов сообщений + fencing-token

**Статус:** дизайн (2026-07-08, открытие Ф4)
**Ветка:** `feat/constructor-f4`
**Задачи плана:** 4.2 (реестр контрактов + warn-middleware) + требование владельца по fencing-token (message-fence по incarnation/epoch)
**Связь:** [[project-topology-fencing-token]], ADR-PMM-010 (routing-epoch, Ф3.1)

## Проблема (два требования, одна точка встройки)

1. **Контракты сообщений (4.2 плана).** Сейчас `command`/`data_type` → обработчик по
   конвенции, без реестра схем. Опечатка в имени команды или лишнее/недостающее поле
   payload молча проходит. Нужен реестр `command|data_type → схема` + middleware,
   который на receive control-plane **предупреждает** о несоответствии (strict — опц.).

2. **Fencing-token (требование владельца 2026-07-08).** После `topology switch` старый
   процесс НЕ должен вкинуть stale-сообщение в новую топологию. Сейчас (ADR-PMM-010)
   `incarnation`/`epoch` существуют, но:
   - используются лишь для CLEANUP очередей (выживший сбрасывает стейл-очередь);
   - guard `epoch <= last_seen → ignore` применяется ТОЛЬКО к самому `routing.refresh`;
   - остаётся задокументированное ОКНО ГОНКИ: сообщение, отправленное до обработки
     refresh, может проскочить.
   Жёсткая гарантия = штамповать конверт + отбрасывать stale на receive.

**Ключевое наблюдение:** оба требования — это **receive-middleware поверх конверта**.
Реестр контрактов и fencing живут в одном pipeline (`add_receive_middleware` →
`_recv_mw.apply` в `router_manager.receive():688`; middleware, вернувший `None`,
дропает сообщение и растит счётчик `middleware_dropped`). Делаем их одним заходом.

## Что уже есть в коде (грунтовка codegraph)

- **Конверт** `Message` (`message_module/core/message.py`): поля `id/type/sender/targets/`
  `timestamp/priority/channel/metadata/data`. `to_dict()` (ADR-008 Dict at Boundary)
  триммит поля по типу через `MESSAGE_TYPE_EXCLUDE_FIELDS`. Полей `incarnation`/`epoch`
  в конверте НЕТ.
- **Создание** — `MessageAdapter.create()` (`message_adapter.py:65`) → `Message.create()`;
  единая фабрика, к которой привязан каждый sender (`self._sender`). Удобная точка
  штампа fence.
- **Истина epoch/incarnation** — PM: `_incarnations: dict[name→int]`, `_routing_epoch: int`
  (`process_manager_process.py`), `_bump_incarnation`/`_bump_routing_epoch`, зеркало в
  PSR-metadata (`routing_epoch`/`routing_incarnation`). Дети узнают epoch через
  `routing.refresh` (полный снимок incarnations + epoch).
- **Приём** — `receive()`: `_recv_mw.apply(msg_dict)` ПЕРВЫМ шагом (до resolve pending,
  до worker-route, до kind-router). Значит fence-drop отсекает stale ДО любой обработки.

## Дизайн

### A. Реестр контрактов (`message_module`, ядро framework)

- `MessageContractRegistry`: `register(key, schema, *, plane="control")`, `get(key)`,
  где `key` = имя команды или `data_type`. Схема — Pydantic-класс (или облегчённый
  дескриптор полей). Наполнение — декларативно при регистрации обработчиков/плагинов.
- **warn-middleware на receive** (control-plane): по `command|type` находит контракт,
  валидирует payload; несоответствие → `WARNING` с diff полей (лишние/недостающие),
  сообщение НЕ дропается (диагностика, не барьер). Флаг `strict` (env
  `FW_CONTRACTS_STRICT`, дефолт warn) → в strict дропает + счётчик.
- **Инвариант для Ф7:** после G data-plane валидируется ТОЛЬКО payload-валидатором 4.3,
  НЕ этим реестром (зафиксировать текстом в разделе G плана). Здесь — только control-plane.
- **Контактная книжка v1 (из 1.9):** `introspect.capabilities` отдаёт `params_schema`
  из реестра. См. `capability-manifest-idea.md`.

### B. Fencing-token (envelope fence + drop-middleware)

- **Конверт** — добавить опциональные поля fence: `sender_incarnation: int|None`,
  `epoch: int|None` (nested `_fence` во избежание коллизий/утечки в payload-схемы,
  либо top-level — решить в реализации; nested предпочтителен для чистоты `to_dict`).
- **Штамп (sender-side, control-plane).** Процесс знает свою `incarnation` и последний
  виденный `epoch` (из `routing.refresh`). `MessageAdapter`/`RouterManager` штампует
  исходящие control-plane сообщения. **Data-plane (hot-path кадров) — НЕ штампуем**
  (сознательное решение ADR-PMM-010); перенос на data-plane — Ф7 G.4 под флагом.
- **Drop-middleware (receiver-side).** `add_receive_middleware(fence_filter)`: если у
  сообщения есть `epoch` и он `< self._known_epoch` → **drop** (`return None`) + счётчик
  `fence_dropped` + throttled WARNING «отброшено stale от <sender> epoch=X < Y». Сообщения
  без fence (легаси/data) проходят прозрачно (обратная совместимость).
- **Источник `_known_epoch` у получателя.** Тот же `routing.refresh`, что уже обновляет
  incarnations — расширить обработчик, чтобы он клал текущий epoch в поле, читаемое
  fence-middleware. PM — источник истины; дети — зеркало.

### C. Композиция с ADR-PMM-010

Fence — это **ужесточение** epoch-guard'а с одного сообщения (`routing.refresh`) до
всех control-plane. Cleanup-очередей (drop_process_queues) остаётся как есть — fence
закрывает окно гонки «отправлено до refresh», cleanup закрывает «очередь мертва».
Два слоя, не дубль.

## Порядок реализации

1. `MessageContractRegistry` + contract-тесты (пустой реестр = ноль оверхеда).
2. warn-middleware (control-plane) + `FW_CONTRACTS_STRICT`; live: опечатка команды → WARNING с diff.
3. Fence-поля в конверте + штамп в `MessageAdapter` (control-plane) за флагом `FW_FENCE`.
4. Fence drop-middleware + счётчик `fence_dropped` + проводка `_known_epoch` из `routing.refresh`.
5. **Live RED→GREEN (ОБЯЗАТЕЛЬНО e2e, НЕ unit):** через `backend-ctl` driver /
   BackendHarness (файлом, свой порт ≥8770, свой `INSPECTOR_PID_FILE`) — switch →
   старый процесс шлёт по стейл-epoch → сообщение **не доходит до handler'а**
   (`fence_dropped` растёт), новая топология чиста. Как fault_injection_live в Ф3.7.
6. `introspect.capabilities` v1 отдаёт `params_schema`; перегенерация `CAPABILITIES.yaml`/`.md`.

> **⚠️ Урок Ф3.7 — не повторять.** Авто-рестарт (Ф3.1) прошёл unit-тест «IPC отправлен»,
> но живьём НЕ работал (после P4.4.1 нужен был `type=command`, а слался `system`). Мораль:
> **юнит «сообщение отброшено» ≠ доказательство fencing.** Для fencing-token (как и для
> любого supervisor/routing-пути) доказательство — только **живой e2e с реальным switch**:
> поднять бэкенд с `BACKEND_CTL=1`, через driver проверить, что stale от старого процесса
> физически не долетает до нового handler'а. Unit-тесты (шаги 1-4) — необходимы, но
> недостаточны; зелёный unit без шага 5 = НЕ закрытый fencing.

## Acceptance

- Опечатка в команде → WARNING с diff полей (warn-режим); в strict — drop + счётчик.
- После switch stale-сообщение от старого процесса **физически не доходит** до handler'а — доказано **live e2e через backend-ctl/BackendHarness** (не unit), `fence_dropped++`.
- Сообщения без fence и data-кадры проходят без изменений (регресс паритет).
- `introspect.capabilities` отдаёт схемы; CI-gate `test_dump_matches_committed` зелёный.
- Откат: `FW_FENCE=0` (fence off) / `FW_CONTRACTS_STRICT` unset (warn-only). Hot-path не затронут.

## Риски

- **Штамп epoch у детей** требует, чтобы ребёнок гарантированно получил `routing.refresh`
  до первой отправки после switch — иначе он штампует старым epoch и сам себя отбросит
  на приёмнике. Митигация: пока epoch неизвестен — не штамповать (сообщение проходит как
  легаси), fence активен только когда обе стороны знают epoch. Проверить на live switch.
- Nested `_fence` не должен попадать в payload-схемы контрактов (исключать в валидаторе B/A).
- Пересечение с 4.3 (payload по Port) и 4.9 (StateStore ревизии — тот же монотонный счётчик):
  не плодить три разных «номера поколения» — свериться при реализации 4.9.

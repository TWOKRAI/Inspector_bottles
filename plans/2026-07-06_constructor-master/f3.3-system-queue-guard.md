# Ф3.3 Guard system-очереди — план реализации

**Ветка:** worktree от `feat/constructor-f3`. **Мастер-план:** `plans/2026-07-06_constructor-master/plan.md:138`.
**Acceptance:** переполнение data-очередей не выбивает `process.stop`. Усилие S.

## Контекст (проверено разведкой)

- `remove_old_if_full` (`shared_resources_module/queues/core/manager.py:266-271`) при
  `queue.full()` молча делает `get_nowait()` — вытесняет самый старый. **Принимает только
  `queue: Queue`, НЕ знает `queue_type`** → system-очередь вытесняется как data.
- Единственный call-site: `QueueRegistry.send_to_queue` (`manager.py:144-165`), строка `:156`
  ДО каждого put. `send_to_queue` ЗНАЕТ `queue_type` (аргумент `:147`), но не прокидывает.
- `process.stop`/`heartbeat` → system-очередь (`router_manager.py:275-278`: `type in
  command/system` → "system"). System-очередь конечна: `maxsize:100`
  (`process_manager_process.py:166`) → может переполниться и выкинуть `process.stop`. Это баг acceptance.
- Счётчика дропов в QueueRegistry нет; есть `self._stats` (`:55`) + `_merge_stats` (`:277-291`);
  QueueRegistry наследует ObservableMixin → доступен `_log_error`. Образцы drop-счётчиков:
  `router_module/core/_sender.py:75,142-144`.
- Канонические строки типов: `process_data.py:32-36` (`QUEUE_SYSTEM="system"` и т.д.).
- Cross-ref: Ф7 G.4 (`plan.md:213`) ЯВНО помечен `(поглощает 3.3)` — полную QoS-модель
  (drop_policy/history/deadline/UI) НЕ трогать. Ф3.3 = только предохранитель + счётчик.

## Дизайн

Прокинуть `queue_type` в guard; для `"system"` — **не вытеснять**, а ERROR-лог (throttled) +
инкремент счётчика; put при полной system-очереди пусть падает штатно (put_nowait → Full →
существующая обработка ошибки в send_to_queue), НЕ молчаливая потеря system-команды.

## Шаги

1. `shared_resources_module/queues/core/manager.py`:
   - `remove_old_if_full(self, queue, queue_type=None)` (обратно совместимо — дефолт None =
     старое поведение вытеснения; сторонние вызовы не ломаются). Для `queue_type == "system"`:
     НЕ вытеснять; если `queue.full()` → `_log_error` (throttle: раз на окно, не на каждый put —
     hot-path) + `self._stats["system_evict_blocked"] += 1`; вернуть без вытеснения.
   - `send_to_queue` (`:156`): передать `queue_type` в вызов.
   - Новый ключ `"system_evict_blocked"` в `self._stats` инициализировать (`:55`).
2. **Тест** (`shared_resources_module/tests/test_queue_registry.py`, паттерн maxsize из
   `test_routing_epoch_psr.py`): system-очередь `maxsize=2`, заполнить, послать `process.stop` →
   старый элемент НЕ вытеснен, счётчик `system_evict_blocked` вырос; data-очередь `maxsize=2`
   переполнить → вытеснение РАБОТАЕТ как прежде (регресс не сломан). Проверить `get_stats`
   отдаёт счётчик.
3. **Доки**: статус Ф3.3 НЕ править в plan.md (параллельный агент Ф3.5 правит plan.md —
   конфликт; статус проставит владелец/оркестратор при merge). Короткий комментарий в коде
   со ссылкой на Ф7 G.4 (полная модель).

## Риски и откат

- Обратная совместимость: `queue_type=None` → старое поведение; существующие тесты
  (`test_queue_registry.py`, broadcast) зелёные без правок.
- Не трогать data-путь — вытеснение data остаётся (это by design до G.4).
- System-очередь переполнилась и put падает: это ВИДИМАЯ ошибка (лог+счётчик) вместо тихой
  потери — цель задачи. Реальное переполнение system при maxsize=100 маловероятно; если станет
  проблемой — QoS-модель G.4.
- Откат: revert одного коммита (правка локальна в manager.py + тест).

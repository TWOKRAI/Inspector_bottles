# Ф3.6 + Ф3.7 + Ф3.8 — Supervisor v2 policy + fault-injection + GATE G1

**Ветка:** `feat/constructor-f3`. **Мастер-план:** `plans/2026-07-06_constructor-master/plan.md:141-143`.
**Решение владельца G1 (2026-07-08):** ДА, включить RestartPolicy per-process для source/hub.
Цепочка последовательная (3.7 тестирует 3.6, 3.8 включает) — один исполнитель.

## Контекст (проверено разведкой)

**Ф3.6 уже ~70% готова** в `process_manager_module/monitor/process_monitor.py`:
- `_try_auto_restart` (`:597-659`): give-up при `count >= max_retries` → snap+PSR `status="failed"`;
  иначе backoff-план в `_pending_restarts`, инкремент `_restart_counts[name]` (`:654`, `:96`).
- `_dispatch_due_restarts` (`:661-699`): IPC `process.restart` в PM (не прямой вызов — гонки нет).
- `reset_restart_count` (`:701-707`) существует, но **нигде авто не вызывается**.
- `RestartPolicy` (`core/restart_policy.py:13-28`): `enabled=False`(!), `max_retries=3`,
  `backoff_sec=2.0`, `restart_on_crash/unresponsive`. **Политика ГЛОБАЛЬНАЯ** (одна на монитор,
  `process_manager_process.py:109-122`).

**Недостающее (это и есть Ф3.6):**
1. **Окно стабильности N/T отсутствует** — `_restart_counts` пожизненный, без временных меток.
2. **Per-process политика отсутствует** — тянуть из `_process_configs[name]`, не глобально.
3. **give-up в `state.status`, НЕ в `health.status`** — acceptance требует health. Монитор пишет
   `processes.<name>.state.status` (`:791`), а `health.status` (`process_module/health/schema.py:37-42`,
   `FAILED="failed"`) не трогает.
4. **reset_restart_count не проведён** — стабильная работа не сбрасывает счётчик.

**Ф3.7 — kill живьём:** PID ребёнка НИГДЕ не публикуется (`introspect.status`
`builtin_commands.py:422-437` без pid). `process.stop` — graceful (exitcode 0 → "stopped" →
авто-рестарт НЕ триггерится). Нужен настоящий SIGKILL (crash, exitcode≠0 → монитор ловит по
`is_alive()` за ≤ poll 0.5с). Реестр PM даёт PID: `core/process_registry.py` `get_process_by_name(name).pid`.
Существующие live-тесты (`test_routing_epoch_live.py:152`) рестартят ВРУЧНУЮ, смерти→авто-рестарта нет.

**Пороги:** poll 0.5с, heartbeat_timeout 15с, backoff 2с. SIGKILL-путь ≈ 0.5+2+0.5(ready)+refresh.

## Шаги

### Ф3.6 — policy (framework)
1. **RestartPolicy расширить** (`core/restart_policy.py`): добавить `window_sec: float = 60.0`
   (окно стабильности; 0 → пожизненный счётчик как раньше). Оставить `enabled=False` дефолт
   (включение — Ф3.8 в рецептах, не глобально).
2. **Окно N/T** (`process_monitor.py`): `_restart_counts: dict[str,int]` → `_restart_history:
   dict[str, list[float]]` (timestamps). В `_try_auto_restart`: отфильтровать метки старше
   `now - window_sec`, `count = len(history_in_window)`; give-up при `count >= max_retries`
   В ОКНЕ. `time.monotonic()` (НЕ Date.now). Сохранить публичное поведение `reset_restart_count`
   (очистка истории имени).
3. **health=failed при give-up**: в give-up-ветке (`:623-644`) дополнительно публиковать
   `processes.<name>.health.status="failed"` + `health.degraded_reason="supervisor give-up: N
   рестартов за Tс"` + `health.updated_at` через тот же `_publish_state`. State.status оставить
   (обратная совместимость), health — добавить.
4. **Per-process policy**: в `_try_auto_restart` резолвить политику по имени из `_process_configs`
   (PM хранит), фолбэк на глобальную `self.restart_policy`. Прокинуть доступ к per-process конфигу
   в монитор (ссылка/колбэк, как `_active_wires` в Ф3.5).
5. **reset-проводка**: в `_check_heartbeats` — если процесс держит `running` дольше `window_sec`
   (стабилен) → `reset_restart_count(name)`. Метка «running since» per-process.
6. **Blueprint per-process поле** (`process_module/generic/blueprint.py`): `ProcessConfig`
   (`:58-167`) + `restart_policy: dict = {}`; проброс в `as_generic_config` (`:124-158`) рядом с
   `protected`/`priority`. Так рецепт задаёт политику на процесс.
7. **Unit-тесты** (`tests/test_process_monitor.py:164-224` расширить): N рестартов в окне → give-up
   + `health.status="failed"` в опубликованном state; вне окна (метки протухли) → счётчик сброшен,
   рестарт продолжается; стабильный running > window → reset; per-process политика перекрывает
   глобальную; protected не рестартится; `enabled=False` → нет авто-рестарта.

### Ф3.7 — fault-injection (harness + live)
8. **Kill-механизм**: `BackendHarness.kill_child(name)` (`backend_ctl/harness.py`) — получить PID
   через driver (новая тест-совместимая точка: либо добавить `pid` в `introspect.status`
   `builtin_commands.py:422-437` — полезно и для debug-plane, либо PM-команда `debug.pid(name)`).
   Рекомендация: добавить `pid` в `introspect.status` (аддитивно, честная наблюдаемость) → harness
   читает pid → `os.kill(pid, SIGKILL)`. Guard: имя не найдено → явная ошибка.
9. **Fault-injection фикстура + e2e** (`backend_ctl/tests/test_fault_injection_live.py`, порт 8783
   — свериться с занятыми 8770-8782): рецепт/конфиг с `RestartPolicy(enabled=True, max_retries=3,
   window_sec=60)` на source; boot → baseline поток кадров (probe/fps-дельта preprocessor) →
   `harness.kill_child("<source>")` (SIGKILL) → дождаться авто-рестарта (монитор ≤0.5с + backoff 2с
   + ready) → **данные снова идут** (fps/health-дельта возобновилась) + wire re-issue (Ф3.5) +
   routing.refresh (Ф3.1) отработали. Дедлайн events ~20с, sleep 3с после kill (паттерн
   routing-epoch `test_routing_epoch_live.py:127-172`).
10. **Give-up e2e (опц., если бюджет)**: `max_retries=1, window_sec` большой → два kill подряд →
    `health.status="failed"` виден через `state_subscribe`/probe.

### Ф3.8 — GATE G1 (prototype, решение владельца ДА)
11. **Включить RestartPolicy в обоих рецептах** per-process для source/hub:
    - `multiprocess_prototype/recipes/phone_sketch.yaml` — source-процессы (камера) + hub:
      `restart_policy: {enabled: true, max_retries: 3, window_sec: 60}`.
    - `multiprocess_prototype/recipes/hikvision_letter_robot.yaml` — hikvision hub + source.
    - НЕ включать для protected (gui/PM — они и так skip) и для чисто-вычислительных воркеров
      (по усмотрению; source/hub — приоритет по решению владельца).
12. **qt-smoke обоих рецептов** (или BackendHarness-smoke, если qt недоступен headless): оба
    бутятся, процессы running, авто-рестарт активен (не срабатывает при здоровье). Откат = убрать
    `restart_policy` из рецепта (или `enabled: false`).
13. **Доки**: статусы 3.6/3.7/3.8 [x] в plan.md; **GATE G1 — отметить решение владельца** в секции

## Порядок и риски

Порядок строгий: 6.1-6.7 (policy) → 6.8-6.10 (fault-тест доказывает механизм) → 6.11-6.13 (включение
в рецептах на доказанном коде). Коммит после каждого шага.

- **Риск flap-петли**: без окна N/T give-up был пожизненный — теперь окно защищает от вечных
  рестартов, но и от преждевременной сдачи (метки протухают). Проверить оба края unit-тестом.
- **Риск Ф3.8**: включение авто-рестарта в проде может маскировать баги (процесс тихо рестартует
  вместо падения). Митигация: health=failed при give-up виден в дереве + fault-тест; окно N/T
  ловит flap. Откат — флаг в рецепте.
- **SIGKILL на macOS**: убедиться, что монитор ловит crash (exitcode≠0) за poll, а не ждёт
  heartbeat_timeout 15с (иначе тест медленный/флак).
- **enabled=False дефолт** сохранить — прод без явного рецепт-флага не меняет поведение (безопасно).

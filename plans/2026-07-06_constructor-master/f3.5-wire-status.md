# Ф3.5 Wire-статусы first-class — план реализации

**Ветка:** `feat/constructor-f3`. **Мастер-план:** `plans/2026-07-06_constructor-master/plan.md:140`.
**Acceptance:** kill+restart камеры → кадры снова идут; `broken_wires ≠ 0` в момент разрыва. Усилие M.

## Контекст (проверено разведкой)

**Два независимых SHM-механизма кадров — НЕ путать:**
- **A. generic data-path** (живой поток камеры): `FrameShmMiddleware` в конструкторе процесса
  (`process_module/generic/generic_process.py:70-82`), receiver берёт `shm_name`/`owner` per-frame
  из msg (`data_receiver.py:219-233`). **Самовосстанавливается** при рестарте (mw пересоздаётся,
  имя приходит в каждом сообщении).
- **B. wire.* абстракция** (PM-управляемые «провода», GUI `connect_wire`): статический
  `FrameShmMiddleware` через `wire.configure` (`process_module/commands/builtin_commands.py:882-931`),
  трекается в `self._wire_middlewares` (инициализируется ПУСТЫМ в конструкторе, `:29`).
  **НЕ восстанавливается** — это и есть баг.

**Где теряется связь** (`process_manager_module/process/process_manager_process.py`):
- `restart_process` (`:1302-1352`): новый инстанс на `:1330` с пустым `_wire_middlewares`;
  `wire.configure` НЕ переигрывается до `return`; PM-трекер `_active_wires[wire_key]["status"]`
  остаётся `"active"` (`:577`) — провод висит на мёртвом инстансе.
- `apply_topology` (switch, `:1429-1557`): план топологии wire-команд не содержит → та же болезнь.
- SHM-имя партнёра зависит от инкарнации owner'а и меняется при рестарте
  (`router_module/middleware/frame_shm_middleware.py:63-80`) → статический wire протухает вдвойне.

**broken_wires hardcode 0**: `process_manager_module/monitor/process_monitor.py:298`
(`_publish_state("system.health.broken_wires", 0)`, TODO рядом `:273-276`). Источник истины —
`ProcessManager._active_wires` (`:61`, статусы `pending`/`active`), но монитор к нему не обращается.

**Паттерн публикации**: `ProcessMonitor._publish_health` (`:263-298`) → `_publish_state` (`:231-244`)
в StateStore, вызов в цикле `:351`. Сюда же встроить `_publish_wires`.

**Пересечение с сделанным**: в `restart_process` уже есть Ф3.1 (`reuse_queues`, `_bump_routing_epoch`
+ `_broadcast_routing_refresh("process.restart")` на `:1349-1350`) и Ф3.2 (`_wait_processes_ready`
`:1343-1346`). Wire re-issue встроить ПОСЛЕ `_wait_processes_ready` (инстанс готов принимать команды)
и рядом с refresh — ортогонально epoch/ready, инварианты не трогать.

## Дизайн

1. **Wire re-issue при restart**: после `_wait_processes_ready` в `restart_process` — перебрать
   `_active_wires`, где `source_process == name or target_process == name`; для каждого переиграть
   `wire.configure` в перезапущенный инстанс (роль sender/receiver как в `_cmd_wire_setup:554-573`),
   при необходимости переаллоцировать SHM (owner мог смениться — свериться с incarnation Ф3.1).
   Тот же перебор — в success-ветке `apply_topology` для затронутых switch-ем процессов.
2. **broken_wires реальный + system.wires.***: дать ProcessMonitor доступ к `pm._active_wires`
   (ссылка при создании монитора); `_publish_wires` публикует `system.wires.<wire_key>.status`
   (active/broken/pending) и агрегат `system.health.broken_wires` = число проводов, чей source
   или target не `is_alive` / статус broken. Заменить hardcode `:298`.
3. **Статус wire honest**: при рестарте помечать задетые `_active_wires[...]["status"]="broken"`
   ДО re-issue, после успешного re-issue → `"active"`. Тогда `broken_wires ≠ 0` в момент разрыва
   (acceptance) и снова 0 после восстановления.

## Шаги (пути от `multiprocess_framework/modules/`)

1. **PM re-issue helper**: `process_manager_process.py` — `_reissue_wires_for(process_name)`
   (перебор `_active_wires`, отправка `wire.configure`; broken→active вокруг). Вызов в
   `restart_process` после `_wait_processes_ready` и в `apply_topology` success-ветке.
   Guard'ы: `_active_wires` пуст (нет проводов) → no-op; memory_manager/communication mock → no-op.
2. **Broken-marking**: помечать задетые провода `"broken"` в начале restart (перед create) и в switch.
3. **Monitor broken_wires + system.wires**: `process_monitor.py` — ссылка на `_active_wires`
   (через `self.process` или явный параметр конструктора `:104-122` в PM); `_publish_wires`;
   заменить hardcode `:298` на реальный счёт. Троттлинг публикаций — как у `_publish_health`.
4. **Тесты** (`process_manager_module/tests/`):
   - unit re-issue: mock `_active_wires` с проводом source=restarted → после `restart_process`
     ушёл `wire.configure` в новый инстанс; провод снова `active`.
   - unit broken_wires: провод с мёртвым target → `_publish_wires` даёт `broken_wires>=1`;
     все живы → 0. **Обновить существующие ожидания 0**: `test_process_monitor.py:515,593`
     (TestSystemHealth), `Plugins/io/telemetry_sink/tests/test_telemetry_sink_plugin.py:239,248`,
     `frontend/widgets/tabs/processes/tests/test_health_summary.py:48-52` — там, где 0 был
     hardcode, теперь 0 при всех живых (проверить, что тесты стартуют со здоровой топологией →
     остаются зелёными; где ожидание завязано на «всегда 0» — переформулировать на «0 при живых»).
   - **live** (`backend_ctl/tests/`, порт уникальный — 8770-8781 заняты, бери 8782+): boot →
     наблюдать поток кадров preprocessor (state-дельты / probe) → `process.restart` источника →
     **кадры снова идут** после рестарта; в момент разрыва `system.health.broken_wires ≠ 0`
     (subscribe перед рестартом). Если провод B в headless-рецепте не заведён — задокументировать
     и проверять на generic-пути A (кадры восстанавливаются) + broken_wires на synthetic wire.
5. **Доки**: статус Ф3.5 [x] в plan.md; ADR-дополнение `process_manager_module/DECISIONS.md`
   (wire re-issue при рестарте, honest broken_wires; граница A vs B) + `scripts.sync`; validate.py.

## Риски и откат

- **A vs B в acceptance**: если живой рецепт использует generic-путь A (не wire.* B), «кадры снова
  идут» верифицируется на A (он и так самовосстанавливается через routing-epoch Ф3.1) — тогда
  вклад Ф3.5 в этой части = honest broken_wires + re-issue для B-проводов (GUI). Явно зафиксировать
  в ADR, какой путь тестируется live.
- Изменение broken_wires с hardcode 0 на реальный счёт ломает тесты-ожидания 0 → аккуратно
  переформулировать (0 при здоровье, не «константа»).
- Откат: feature-флаг `wire_reissue_enabled` (дефолт True) в конфиге PM; при False — старое
  поведение (провода не переигрываются, broken_wires снова 0). Re-issue ортогонален epoch/ready.

# Ф3.1 routing-epoch — план реализации

**Ветка:** `feat/constructor-f3` (уже создана). **Мастер-план:** `plans/2026-07-06_constructor-master/plan.md:136`.
**Acceptance:** peer→peer send после switch доставляется (сейчас молча теряется). Усилие M.

## Контекст (проверено по коду)

Каждый дочерний процесс получает при спавне **замороженный снимок** routing_map
(`process_manager_module/core/process_registry.py:84-87` → bundle → `runner/bundle_builder.py:107-116`
наполняет ЕГО локальный `ProcessStateRegistry`). После `apply_topology`/`restart_process` PM создаёт
НОВЫЕ Queue-объекты, а соседи остаются со стейл-ссылками. Fallback'и router'а (канал Ф1.1b, hub-relay Ф1.7,
`router_module/core/router_manager.py:339-362`) срабатывают **только если очереди НЕТ** — а у стейл-соседа
она есть (мёртвая): `put_nowait` в осиротевшую очередь возвращает успех → **тихая потеря**
(задокументировано в docstring `_cmd_process_relay`, `process_manager_process.py:832-850`).

**Жёсткое ограничение:** `multiprocessing.Queue` нельзя запиклить вне spawn
(«Queue objects should only be shared through inheritance») → `routing.refresh` НЕ может нести Queue-объекты, только данные.

**Data-plane идёт через эти же очереди** (chain_targets → `send_to_queue(target, "data")`) → нельзя
после restart навсегда сажать соседей на hub-relay (все кадры через system-очередь PM).

## Дизайн: гибрид A (switch) + B (restart)

- **B — restart:** очереди **переиспользуются** (identity сохраняется) — стейл-ссылки соседей остаются
  валидными, hot-path не деградирует. Новый ребёнок наследует те же Queue через bundle. Мусор прошлой
  жизни дренируется перед стартом (`get_nowait` до `Empty`, НЕ `clear_queue` — тот спит ~0.2с).
- **A — switch:** `routing.refresh` несёт `epoch` + авторитетную карту `{имя: incarnation}`. Выжившие
  дети сбрасывают очереди имён с изменившейся/исчезнувшей incarnation → последующий send падает в
  существующий `_relay_via_hub` (Ф1.7) → PM со свежим PSR доставляет. Refresh — **декларативная сверка
  полного снимка** (не дельта): повторная/потерянная рассылка самовосстанавливается; guard
  `epoch <= last_seen → ignore`.
- C (Manager-proxy очереди) отклонён: сокет-раундтрип на каждый put — недопустимо на пути кадров.

## Контракт `routing.refresh` (Dict at Boundary)

```python
{"type": "command", "command": "routing.refresh", "sender": "ProcessManager", "queue_type": "system",
 "data": {"epoch": 7, "hub": "ProcessManager", "reason": "topology.apply|process.restart|rollback",
          "processes": {"camera_0": {"incarnation": 0}, "preprocessor": {"incarnation": 2}}, "ts": ...}}
```

Handler у ребёнка (все ветки идемпотентны): `epoch <= last_seen` → ignored; имя отсутствует в снимке →
сбросить его очереди; incarnation ≠ локальной → сбросить очереди + запомнить; свою запись и `hub` не трогать;
в конце сохранить epoch в metadata своей PSR-записи + счётчик `routing_refresh_applied`.

Bundle дополняется аддитивным ключом `routing_meta = {"epoch": N, "incarnations": {...}}` — новые дети
рождаются с актуальными значениями.

## Риски и откат

- **Окно гонки**: сообщения, уже лежащие в мёртвых очередях / отправленные до обработки refresh, теряются (окно ≈ switch + один цикл message-loop). Sender-side epoch-check на каждом send — сознательно вне scope (hot-path); зафиксировать в ADR.
- Потерянный refresh лечится следующим (полный снимок). Переполнение system-очереди — тема Ф3.3.
- SHM/wire-каналы управляются отдельно (`wire.configure`, Ф3.5) — вне scope, отметить в ADR.
- **Откат**: env `FW_ROUTING_REFRESH=0` / конфиг `routing_refresh_enabled: false` (bump+broadcast off, handler no-op); `restart_reuse_queues: false` — старое поведение рестарта. Оба — чистый возврат к main.

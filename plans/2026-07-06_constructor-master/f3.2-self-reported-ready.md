# Ф3.2 Self-reported ready — план реализации

**Ветка:** `feat/constructor-f3`. **Мастер-план:** `plans/2026-07-06_constructor-master/plan.md:137`.
**Acceptance:** switch/boot закрывается по ready, не по таймеру. Усилие M.

## Контекст (проверено разведкой)

- Ребёнок НЕ сообщает PM о завершении init. Статус READY пишется только в СОБСТВЕННЫЙ SRM
  (`process_module/core/process_module.py:160-161`); PM узнаёт «running» по `is_alive()` монитора
  с задержкой poll 0.5с (`process_monitor.py:368-407`), первый heartbeat — через ~5с.
- Switch-барьер `_wait_started_ready` (`process_manager_process.py:1552-1611`) — death-watch:
  здоровый switch ВСЕГДА ждёт весь settle-window `start_ready_timeout_s` (дефолт 0.5с);
  `True` = «не умер», НЕ готовность.
- Boot-барьера НЕТ: PM ставит `_system_ready_event` сразу после своего initialize
  (`process_manager_process.py:211-213`), дети могут ещё выполнять init
  (докстринг `system_launcher.py:196-202` прямо выносит это «в будущую задачу» = Ф3.2).
- **ЖЁСТКОЕ ОГРАНИЧЕНИЕ (дедлок):** heartbeat/любое IPC-сообщение и `topology.apply`
  обрабатываются ОДНИМ `message_processor`-потоком PM — блокирующе ждать IPC-`ready` в барьере
  НЕЛЬЗЯ (комментарий `process_manager_process.py:1561-1572`).

## Дизайн: per-process `mp.Event` ready (вне message-loop)

`ready` — НЕ IPC-сообщение, а `multiprocessing.Event`, который PM создаёт при спавне и передаёт
ребёнку через bundle `custom["ready_event"]` (inheritance при spawn — легально, как stop_event;
см. комментарий об mp.Event в `process_registry.py:91-96`). Runner ставит event сразу после
успешного `initialize()` — ДО `_run_lifecycle`. Барьер PM ждёт event'ы + death-watch —
никакого message-loop, дедлок исключён. Settle-window остаётся фолбэком.

Семантика барьера (сохранить контракт `ready: dict[name→bool]`):
- event set → `True` **немедленно** (здоровый switch больше НЕ ждёт весь window — acceptance);
- окно истекло, процесс жив, event нет → `True` + WARNING «ready не получен — фолбэк по liveness»
  (прежнее поведение; MockProcess в тестах event не ставит — существующие тесты зелёные);
- умер в окне → `False` (как сейчас);
- `start_ready_timeout_s == 0` → барьер выключен, `{}` (как сейчас).

## Шаги (пути от `multiprocess_framework/modules/`)

1. **Bundle/Registry**: `process_manager_module/core/process_registry.py` — в `_create_process`
   создать `mp.Event()`, положить в `custom["ready_event"]` bundle, сохранить у себя
   `self._ready_events[name]`; публичный `get_ready_event(name)`; очистка в
   `remove_process`/unregister. `runner/process_runner.py` — после успешного
   `process_instance.initialize()` (стр. ~177-189): `bundle.get("custom", {}).get("ready_event")`
   → `.set()` (guard None — обратная совместимость со старыми bundle в тестах).
   **Не забыть**: `ready_event` в EXCLUDE-ключи broadcast custom (`monitor/process_monitor.py:24`)
   — event не пиклится через очереди.
2. **Барьер switch**: `_wait_started_ready` (`process_manager_process.py:1552-1611`) — цикл
   poll 0.05с: death-watch (как сейчас) + проверка `registry.get_ready_event(name).is_set()`
   → ранний выход `True` для этого имени. Фолбэк-семантика из дизайна. Guard'ы: registry/event
   могут отсутствовать (mock) → чистый фолбэк на прежнее поведение.
3. **Барьер boot**: в `initialize()` PM (`process_manager_process.py:~179-213`) — после
   `_create_processes_from_config` и старта монитора, ПЕРЕД `_system_ready_event.set()`:
   тот же helper ожидания ready всех стартованных детей с конфигом `boot_ready_timeout_s`
   (дефолт 5.0с; 0 → пропустить). По таймауту — `_system_ready_event.set()` ВСЁ РАВНО
   + WARNING со списком не-ready (boot не блокировать навсегда). Это initialize-поток PM,
   не message_processor — блокировка допустима.
4. **restart_process** (`:1100-1120`): пересоздать ready_event для нового инстанса
   (старый event set — обнулить заменой объекта, не `.clear()` — у старого ребёнка ссылка);
   после `start` — тот же барьер на один процесс.
5. **Наблюдаемость**: в ответ `apply_topology` поле `ready` уже есть; добавить в лог барьера
   «ready via event / via liveness-fallback / dead» per-process; счётчик fallback'ов в
   introspect PM необязателен (S-scope).
6. **Тесты** (`process_manager_module/tests/`):
   - runner: успешный init → event set; провал init → event НЕ set (`test_process_runner.py` дополнить);
   - барьер: event set → ранний выход (время << window); event нет + жив → True (фолбэк);
     умер → False; timeout 0 → {} — существующие `TestReadinessBarrier` зелёные БЕЗ правок ожиданий;
   - boot: дети ready → `_system_ready_event` после event'ов; таймаут → set + WARNING;
   - restart: новый event, барьер отрабатывает;
   - live-smoke (`backend_ctl/tests/`, паттерн test_health_live, ПОРТ уникальный — проверь занятые
     grep'ом по backend_ctl/tests, 8778/8779 заняты Ф3.1): boot harness → `topology.apply` →
     `response["ready"]` все True; время switch заметно меньше прежнего «всегда полный settle»
     (не жёсткий assert по секундам — флак; достаточно факта ready=True).
7. **Доки**: статус Ф3.2 [x] в plan.md; ADR-дополнение в `process_manager_module/DECISIONS.md`
   (почему event, а не IPC — дедлок message_processor) + `python -m scripts.sync`; validate.py.

## Риски и откат

- Старые bundle без `ready_event` (тесты, сторонние спавны) → guard None, фолбэк = прежнее поведение.
- `boot_ready_timeout_s` дефолт 5с: медленный ребёнок (ML-веса) может не успеть → boot НЕ ломается
  (set + WARNING), но launcher получит ready позже прежнего. Откат: `boot_ready_timeout_s: 0`.
- Откат целиком: `start_ready_timeout_s: 0` (switch-барьер выкл) + `boot_ready_timeout_s: 0`
  (boot как в main); event-механика пассивна без потребителей.

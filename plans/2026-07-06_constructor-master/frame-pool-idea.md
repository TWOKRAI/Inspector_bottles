# Frame-pool с владением по цепочке (идея владельца, 2026-07-13)

**Статус:** идея-вход для Ф7 G.3/G.4 (hot-path вскрывается ОДНИМ заходом — отдельной задачи до Ф7 нет).
**Контекст:** кадр проходит цепочку процессов (источник → разрез → рисование → GUI), каждый хоп — через
`FrameShmMiddleware` (`router_module/middleware/frame_shm_middleware.py`). Сегодня SHM-слоты — ring из 3
на writer'а, round-robin **без синхронизации**: writer не знает, дочитали ли хвостовые звенья слот,
перезапись возможна по построению (torn frame — принятый долг до G.3(b) seqlock).

## Идея (владелец, 2026-07-13)

Много ячеек — **на каждый кадр каждого пишущего звена**: слот пишется один раз, читается по цепочке и
**занят, пока кадр не завершил свой участок цепочки**; после — возвращается в оборот (release/«кеш»).
Эффект: перезапись невозможна по построению, а конвейерность становится безопасной и глубокой — новый
кадр обрабатывается в первых звеньях, пока предыдущий ещё в последних; локальное замедление одного звена
не рушит и не рвёт кадры остальных.

## Аудит текущих накладных (2026-07-13, код `frame_shm_middleware.py`)

На кадр full-HD BGR (~6 МБ) сейчас:

1. **Копия №1 при записи** — `write_images` копирует ndarray в SHM-буфер (~0.5–1 мс).
2. **Копия №2 при чтении** — `_read_shm_from_actual_name`: `frombuffer(...).copy()` (`:88`) — memcpy на
   каждого читателя.
3. **open/mmap/close на каждый кадр** — fallback-путь каждый раз открывает `SharedMemory(name=...)` и
   закрывает (`:75`, `:93`); судя по [TRACE]-логам, cross-process это ОСНОВНОЙ путь
   (`MemoryManager.read_images` работает только при handles внутри процесса). Плюс resource_tracker
   на каждый open.
4. **Гонка ring-буфера** — `.copy()` не атомарен, writer может перезаписать слот под читателем уже сейчас.

Теоретический предел: ноль копий и ноль syscalls на кадр — producer пишет прямо в слот, consumer читает
view, по очереди едут только координаты. Пункты ниже к нему ведут.

## Дизайн-принципы (зафиксировано при обсуждении)

1. **«Ячейка на кадр» ≠ «SHM-сегмент на кадр».** shm_open/mmap/unlink на кадр съест выигрыш
   (+resource_tracker). Форма: **один предвыделенный сегмент на N слотов** + free-list внутри.
   «Создать ячейку» = взять слот из free-list, «удалить» = вернуть; «кеш» = переиспользование слота
   и есть, отдельного механизма не нужно.
2. **Кто освобождает.** Линейная цепочка — владение передаётся с сообщением, слот освобождает
   **последнее звено** после чтения; refcount не нужен. Fan-out (GUI + запись на диск читают один кадр)
   — atomic refcount в header слота.
3. **Backpressure — явная политика.** Пул конечен; при исчерпании — **drop на источнике** (кадр с камеры
   пропускается, счётчик dropped++ в stats), НЕ блокировка capture и НЕ очередь: старый кадр с конвейера
   бесполезен. Стыкуется с инвариантом G.4 «дроп только громкий» и перф-ревью п.3 «громкий slow-path».
4. **Звенья порождают НОВЫЕ кадры** (разрез — другой размер, рисование — другой контент): слоты нужны
   «кадр × звено-производитель» — у каждого пишущего звена свой набор слотов (или свой пул). Входной
   слот освобождается, когда звено дочитало вход, а не когда весь конвейер закончил.
5. **Глубина пула** = число звеньев × 2 (запас на джиттер). Больше — уже latency, не throughput.
6. **Конвейер, не эстафета (уточнение владельца, 2026-07-13).** Новый кадр запускается в первое
   звено, как только ОНО освободилось — не дожидаясь, пока предыдущий кадр пройдёт всю цепочку;
   в полёте одновременно ~столько кадров, сколько звеньев. Ровно это пул и делает безопасным
   (слот нового кадра не может затереть слот старого — владение). Честная граница механизма:
   глубина пула/буфер сглаживает **джиттер** (временные замедления звена), но НЕ лечит устойчивое
   отставание — если цепочка стабильно медленнее камеры, буфер любой глубины заполнится, и лишние
   кадры честно дропаются на источнике (принцип 3: старый кадр конвейеру бесполезен). Устойчивое
   отставание лечится параллелизмом пула воркеров C6(e) — несколько кадров считаются одновременно
   на разных тредах (пока хватает ядер); тогда на стоке (GUI/статистика/вердикт) нужна дисциплина
   порядка по frame_id — кадры могут финишировать не по порядку.
7. **Мульти-камера с разными цепочками (владелец, 2026-07-13):** камер может быть 2–3 и больше, и у
   каждой камеры может быть СВОЯ цепочка (разные звенья, разная длина). Следствия: пулы строго
   per-camera/per-источник (общих слотов нет — уже принцип 4 + кольца G.4); глубина каждого пула
   считается по длине цепочки ИМЕННО этой камеры (п.5 применяется per-chain, не глобально);
   маршрутизация по stream-id, ключи без коллизий (B-9); изоляция — замедление/дроп цепочки камеры K
   не влияет на кадры камеры M (у каждой свой free-list и свой drop-счётчик).

## Ускорения-спутники (тот же заход Ф7, по соотношению выигрыш/усилие)

1. **Кэш SHM-handles в читателе** (дёшево, первым): dict `shm_actual_name → SharedMemory`, инвалидация
   при смене имени в сообщении (grow-only реаллокация — имя меняется редко). Убирает п.3 аудита.
2. **Zero-copy чтение**: view вместо `.copy()` + seqlock-проверка поколения после чтения (G.3(b) даёт
   header; G.5 `restore_frame(copy=False)` — это оно). Убирает копию №2 у каждого читателя.
3. **Запись сразу в SHM у источника**: camera/декодер пишет в ndarray, backed by `shm.buf` (если SDK
   позволяет буфер назначения). Убирает копию №1. Сложнее — трогает capture-код.

## Стыковка с планом (не дублировать, а расширить)

| Куда | Что даёт пул |
|------|--------------|
| G.3(b) seqlock | тот же header слота расширяется полями `state` (free/writing/ready/reading), `generation`, опц. `refcount` — один формат header'а, не два |
| G.3(c) декларативные регионы | предвыделенный пул под max-размер кадра = та же статика; снимает use-after-unlink |
| G.4 RingBuffer per-camera | кольцо = частный случай пула (без release-протокола); пул = кольцо + free-list + владение по цепочке; «старый слот НЕ перезаписывается» получается по построению, а не по глубине |
| G.4 QoS/дроп | политика «drop на источнике + счётчик» — тот же громкий drop-counter → heartbeat → state |
| G.5 copy-elision | reader-side zero-copy (п.2 спутников) — уже в плане, строго после seqlock |
| RS→Ф7 (B-6..B-9) | SHM-имя owner+incarnation и refresh handles на switch — header/handle-кэш пула должен это учесть |
| iceoryx2 (TECH_STACK_2026 «👀 наблюдать») | loan/publish = ровно эта семантика; пул руками сейчас → миграция на iceoryx2 потом = замена транспорта под тем же контрактом, не переделка пайплайна |

## Аналоги в индустрии (валидация идеи; реплика владельца 2026-07-13: «такой механизм должен быть у мощных библиотек»)

Верно — это стандартный паттерн зрелых zero-copy/vision-систем, мы не изобретаем, а повторяем проверенную семантику:

| Библиотека | Механизм | Совпадение с идеей |
|---|---|---|
| **iceoryx / iceoryx2** | `loan()` слота из пула → publish → release последним подписчиком | 1:1 — уже помечен в плане как будущий транспорт (TECH_STACK_2026 §7) |
| **GStreamer** | `GstBufferPool` — предвыделенный пул, буфер возвращается в пул по unref (refcount) | пул + refcount + backpressure «нет свободного буфера» |
| **Fast DDS / CycloneDDS** | zero-copy loans (data-sharing delivery), reader держит loan до `return_loan` | владение по цепочке, drop-политика QoS |
| **GenICam GenTL** (по нему работает Hikvision MVS SDK) | announce/queue буферов: потребитель отдаёт пул буферов SDK, камера DMA-пишет прямо в них, после обработки буфер re-queue | ровно «спутник №3» — запись у источника без копии: MVS позволяет announce НАШИХ буферов (SHM-slot) |
| GPU swapchain (Vulkan/DXGI) | acquire → render → present → fence освобождает image | «слот занят, пока кадр не завершил участок» |

Вывод для триажа: семантика подтверждена индустрией; контракт слота (state/generation/refcount + free-list) держать совместимым с loan/publish iceoryx2 — миграция потом будет заменой транспорта, не переделкой. GenTL-announce — конкретный путь исполнения спутника №3 для Hikvision-источника.

## Триаж

- До Ф7 — ничего не трогать (консолидация «одним вскрытием», решение владельца 2026-07-09).
- На GATE G3 — внести в acceptance G.3/G.4: header слота проектировать сразу под пул
  (state/generation/refcount), free-list + владение по цепочке + drop-на-источнике — в G.4 вместе
  с per-camera кольцами; кэш handles и zero-copy — G.3/G.5 как уже запланировано.

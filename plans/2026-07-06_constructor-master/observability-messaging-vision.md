> **СОГЛАСОВАНО 2026-07-09** — разложено по фазам мастер-плана ([plan.md](plan.md)): non-hot-path наблюдаемость → **Ф5** (5.14/5.16/5.17/5.19), hot-path кадры + доставка → **Ф7** (G.2–G.4, одним вскрытием). 4 инварианта §3 внесены в «Сквозные правила исполнения». Честно: что подтвердилось, что сейчас, куда идём, чего **не** делать, и что решено.

> Правило чтения: целевую берём **не единым пакетом**. Диагноз подтверждён по исходникам; амбициозную «всё-в-одном» версию критика отклонила — вместо неё минимальный корректный срез сейчас + отложенное build-on-demand.

## 1. Твоя идея — что подтвердилось (и одна поправка на каждое)

| Твоя мысль | Вердикт | Поправка-предохранитель |
|---|---|---|
| Модуль = прибор с 5 гнёздами (вход / выход / лог / ошибки / статистика), удобный фасад с ООП | ✅ верная рамка | Метафора — **документационная**, НЕ единый Python-интерфейс `IPort`. Иначе получим третий транспортный слой поверх двух существующих. |
| Единый конверт-Message ВЕЗДЕ + доставка на выбор (напрямую / через роутер) | ✅ верно | Конверт = единая **схема полей**, а не обязательный Pydantic-объект. На потоке кадров Message **не конструируется** — там плоский dict. |
| Крупные данные (изображение) — в письме только ссылка на SHM | ✅ уже работает | Есть эталонный Claim-Check (`FrameShmMiddleware`). Довести до канона, но не строить заново. |
| Можно и ссылки на БД | ⚠️ пока пусто | DB-by-reference сегодня **нет вообще** (не задел). Строить только когда появится реальный потребитель (иначе YAGNI). |
| Фронт — не один кабель, а несколько каналов | ✅ верно | Каркас уже есть (модель подписок в `backend_ctl/driver.py`), поднять её в дашборд. |

## 2. Где мы сейчас (честно, с файлами)

По **каждой** оси сосуществуют два поколения, и мост между ними не достроен:

- **Наблюдаемость.** Боевое — push: `ObservableMixin` → `_call_manager` → `Logger/Stats/ErrorManager` → сразу в sink. Новое — pull: `ObservabilityHub` + 3 `BoundedChannel` (Ф5.15). **Hub production-мёртв**: ни один composition root его не создаёт и не зовёт `drain_*`; эмиссия уходит в буфер и не доходит до sink.
- **Сообщения.** Номинально единый `Message` — плоский god-object ~30 полей, `extra=allow`. Но **два конверта команд** (`MessageAdapter`→`args` vs `command_envelopes`→`data`), из-за чего `sql_manager._normalize_command` склеивает `{**cmd,**args,**data}`. Доставка тоже двойная (channel-routing vs targets-addressing) с магической строкой `'queue'`.
- **Крупные данные.** Claim-Check работает (пиксели в SHM, в письме координаты). Но: **нет seqlock → torn-frame** (тихая порча = ложный брак/пропуск, severity high); `RingBufferWriter/Reader` — мёртвый код; осиротевшие SHM-сегменты после `kill -9`; GUI читает per-frame `open/close` (mmap-syscall на 25-30 FPS).
- **БД by-reference.** Нет. БД — write-only сток by-value (`str(data)` в TEXT). Обратного пути (row-id → resolve) не существует.
- **Фронт.** Уже многоканален (данные + состояние), но выделенных вкладок Логи/Ошибки/Метрики нет; FPS инъектируется **поддельными** state-дельтами; готовая подписочная модель живёт только в driver'е.

**Итог:** ~50-70% готовности, застряли на середине миграции. Примитивы построены, боевой путь на старом поколении, мосты отсутствуют.

## 3. Целевая архитектура (с инвариантами-предохранителями)

Единый **эмиттер** (hub) на входе + сменные **sink** (менеджеры) на выходе; между ними **drain-петля**, решающая direct/routed. Один конверт-схема. Тяжёлое — по ссылке. Фронт — несколько каналов с подпиской.

```
producer (класс/подмодуль)
   │ удобные методы (log/record/track_error/emit_data) — знает ТОЛЬКО фасад
   ▼
ФАСАД-ПРИБОР (ObservabilityHub, 3 гнезда сейчас → +data/control)
   │ единый конверт-схема {kind,module,ts,payload|payload_ref}
   ▼
DRAIN-ПЕТЛЯ (composition root, по heartbeat)
   ├─ direct  → локальный Logger/Stats/ErrorManager        (в процессе, без Message-объекта)
   └─ routed  → RouterManager → QueueChannel/SocketChannel  (в оркестратор / на фронт)
                     │
   Оркестратор: «глобальные» менеджеры = его СОБСТВЕННЫЕ менеджеры (новый класс не нужен)
                     │
   Фронт: несколько именованных каналов в state_store (log/error/stats/data/state) + подписка+фильтр
```

**Четыре инварианта-предохранителя** (без них метафора прячет баги — нашла критика):

1. **Hot-path.** На data/frame-канале носитель остаётся **плоским dict**; `Message(**…)` per-frame **не конструируется** (иначе per-frame валидация 30 полей + двойное поле времени `hub.ts` vs `Message.timestamp` → просадка FPS).
2. **Overflow по природе гнезда.** Наблюдаемость — `drop_oldest` + счётчик потерь (терять можно, молчать нельзя). Data/command — **backpressure/явная ошибка отправителю**, НЕ тихий drop (потерянный кадр/команда — это баг корректности, не «допустимая потеря»).
3. **Crash-логи.** `error/critical` — **синхронный fast-path** прямо в sink (или дренаж в `atexit`/signal-handler), а не только буфер+heartbeat: иначе последний лог перед падением теряется ровно тогда, когда нужнее всего. И на время миграции модуль **либо** синхронный путь, **либо** hub — не оба сразу на один менеджер.
4. **SHM lifetime.** Fence по epoch/incarnation **детектирует** stale, но кадр всё равно теряется при grow-realloc с немедленным `unlink`. Нужен **refcount / окно удержания поколения** (не `unlink` сразу, пока in-flight письма несут старое имя). Fence необходим, но недостаточен.

**Единый конверт** = общая схема (владелец — `message_module.Message` или `SchemaBase`), один билдер вместо двух. Тяжёлые тела — типизированный `payload_ref {backend: shm|db|file, key, meta, epoch}`, обобщающий и `FrameRef`, и `DataReference`.

## 4. Чего НЕ делать (костыли/баги — избегать)

- ❌ Строить `IPort`-надмножество над `IChannel` + сливать два буферных контракта (`IBufferStrategy` push-flush vs `BoundedChannel` pull-drain) — это **третий** механизм. Метафора 5 гнёзд — дока, не единый интерфейс.
- ❌ Гнать Message-объект на каждый кадр.
- ❌ `drop_oldest` на гнёздах data/command.
- ❌ Перестраивать доставку (`_deliver_by_targets`/`send_to_queue`/резолв `channel`/`targets`) **без характеризационных тестов** — сначала тесты как GATE, потом трогать.
- ❌ Строить DB-by-reference спекулятивно (нет потребителя). Сначала нормализовать `str(data)` → адресуемый PK.
- ❌ Держать мёртвый `RingBufferWriter/Reader` рядом с наивным путём — включить или удалить.
- ❌ Тащить наследование `ErrorManager→LoggerManager` и протечку в приватные поля CRM (`_channel_registry/_buffer`) в новую модель — развести на композицию **до**.

## 5. Scope: что сейчас, что потом

### Сейчас — минимальный корректный боевой срез (Ф5.16 + hardening)
1. **Drain-consumer + подключение hub — ОДНОВРЕМЕННО** (иначе hub мёртв). Владелец петли — `ProcessModule`. *(Ревью 2026-07-09: гнёзда ProcessModule уже non-None — stats создаётся всегда, error по дефолту; реальная предварительная чистка — канонизация имени слота `errors`→`error` + CRM-развязка, см. §6.1.)* Дренаж требует **адаптера dict-запись→метод sink** (у `LoggerManager/ErrorManager/StatsManager` нет `dispatch/track/record` — есть `log`/`log_exception`/`record_metric`).
2. **Один билдер конверта**, убрать `_normalize_command` и второй конверт команд.
3. **seqlock/version в заголовок буфера** (`format/buffer.py`) — torn-frame это тихая порча для инспекции.
4. **startup-cleanup на рантайм-slot `output_frames`** (или статическое объявление региона) — убрать осиротевшие сегменты.
5. **Синхронный fast-path для error/critical** (инвариант 3).
6. **`ErrorManager` → брат `LoggerManager`, не наследник** (РЕШЕНО): общий предок всех трёх — `ChannelRoutingManager`; общий слой записи лога — только для Logger+Error (Stats его НЕ наследует); убрать протечку в приватные поля CRM.
7. **RingBuffer — РЕШЕНО: Вариант A (дожать)** — см. «Кадровый транспорт — доработка» ниже (мульти-камера, буфер ≥ неск. кадров, дроп только громкий).
8. **Характеризационные тесты доставки** — жёсткий GATE перед правкой резолва.
9. **Фронт-подписки из driver'а → в дашборд**, раздельные каналы, убрать поддельные FPS-дельты.

### GUI: вкладки Логи / Ошибки / Статистика (РЕШЕНО — переиспользовать существующее)

Требование владельца: 3 вкладки (Статистика / Ошибки / Логи), у каждой **целая история**; **один** переиспользуемый виджет-список на все три (очистка / копирование / обновление).

- `history/` MVP (`settings/history/`) — таблица + **Сохранить / Очистить / автоскролл** за `HistoryView`-Protocol. *(Ревью 2026-07-09: это домен **undo/redo** через `services.commands`, НЕ канал наблюдаемости — из слияния исключён, основа = `AuditLogPanel`.)*

**План** (ревью 2026-07-09 разнёс на две задачи):

Создать стор (аналог `SqliteAuditStorage`) + дренаж-в-стор (целая история пагинацией) и hub→GUI-форвардинг живого хвоста через `process._bridge` (переиспользуется ПАТТЕРН подписок `driver.py`, сам driver — control-plane `BACKEND_CTL=1`). Пре-условие 5.20 — дренированный hub (5.16).

### Кадровый транспорт — доработка (РЕШЕНО: Вариант A, мульти-камера)

Требование владельца: кадры **нельзя терять** молча; несколько потоков **одновременно** (напр. **3 камеры**), у каждого свои регионы в памяти.

- **Вариант A принят:** дожать `RingBuffer`/`index_usage` — боевой **многослотовый** кольцевой буфер (сейчас `index_usage` не инкрементируется → слот всегда 0, де-факто одно-слотовый).
- **Кольцо на КАЖДУЮ камеру отдельно:** N независимых потоков → N независимых регионов SHM (`frame_cam0/1/2…`), у каждого своё кольцо слотов; middleware маршрутизирует по stream-id в нужный регион. Общего слота между камерами нет.
- **Глубина кольца ≥ несколько кадров** — буфер поглощает отставание обработки: не успела обработать — кадр ждёт в следующем слоте, старый **не перезаписывается**.
- **Дроп только громкий, не тихий:** живую камеру тормозить нельзя (backpressure на неё невозможен). Буфер поглощает всплески; при устойчивой перегрузке (обработчик стабильно медленнее камеры) — **дроп с счётчиком + алярм** (теряем бутылки → надо знать и чинить пропускную способность). Цель — ноль дропа; механизм — глубокий буфер + громкая детекция.
- **seqlock на каждый слот каждого кольца** (см. §5.3) — читатель не поймает рваный кадр.
- **Именование регионов per-camera + fencing:** имя = stream-id + incarnation/epoch — устаревшее имя после realloc/hot-swap не даёт тихий `frame=None`.
- **Все кадровые регионы объявить заранее** (декларативно в рецепте/blueprint, под max-размер кадра): GUI открывает handle каждого региона **один раз** (чинит open/close-тормоз, §5.7), а startup-cleanup знает про них (чинит осиротевшие сегменты после kill, §5.4). Плюс: регион под max-размер → нет grow-realloc → инвариант-4 (use-after-unlink) в основном отпадает сам.

### Потом / build-on-demand (в open-decisions, не в целевой сейчас)
- **DB-by-reference** (ReferenceMiddleware + SQL-резолвер + обратный row-id-путь) — **РЕШЕНО отложить** (владелец: привёл как пример, не срочно; сделаем позже).
- **`IPort`-надмножество** — остаётся метафорой-докой.
- **refcount-удержание поколений SHM** (инвариант 4) — в основном снимается статическими регионами под max-размер (см. выше); отдельно — только при hot-swap с ростом кадра.

## Суть проблемы (подтверждено чтением кода)

Одна и та же `multiprocessing.Queue` доступна через две двери с разной политикой переполнения:

- **Дверь А** (`_deliver_by_targets` → `QueueRegistry.send_to_queue` → `remove_old_if_full`): QoS-политика (drop_oldest/never-drop), счётчики `data_evicted`/`system_evict_blocked`, хук `on_evict` (release-on-evict, ADR-RTR-010).
- **Дверь Б** (`_do_send` → `channels[N].send` → `QueueChannel.send`, `queue_channel.py:52-67`): голый `put(block=True, timeout=1.0)` — стойл потока-отправителя до 1 с, затем `{"status":"error"}` (фактически drop_newest), без QoS, без атрибутируемых счётчиков, без `on_evict`. Кадры уходят **синхронным** `router.send` с потока-исполнителя (`send_to_process` → `router_manager.send`), поэтому стойл бьёт по продьюсеру кадров напрямую.

Политика переполнения зависит от способа резолва адреса — произвольная связь двух независимых осей. Фикс LIVE-2 (`8b8a3c54`) на боевой раскладке kind-каналов не исполняется: `on_evict` навешивается только в `_deliver_by_targets`.

Дополнительно подтверждён (статикой) латентный баг `release_evicted`: `loan_ledger.py:191-208` декрементит refcount по **любому** имени reader'а — множества loan-aware потребителей в леджере нет. `_on_frame_evicted` (`router_manager.py:484-533`) шлёт release с `reader = владелец очереди`, а конверт кадра несёт SHM-координаты и для copy-out получателей (GUI): `frame_shm_middleware.py:462-463` проставляет `owner`/`shm_owner` во **все** исходящие кадровые конверты. Вытеснение из очереди GUI сняло бы чужую долю → досрочное освобождение слота. Сегодня ветка мертва для кадров, но **унификация политики её оживит** — фикс обязан идти раньше унификации.

## Порядок и обоснование

1. **Фаза 0 — доказательная база.** Оба направления (транспорт и инструмент) упираются в одно: мы не знаем фактическое распределение трафика по дверям и не знаем, чем был снят блокер webcam-soak. Резать транспорт вслепую нельзя, а закрытие этих вопросов дёшево (счётчики + один короткий прогон). Сюда же — минимальный срез доказательности backend_ctl (строгие имена счётчиков), потому что именно им будет проверяться результат Фазы 1.
2. **Фаза 1 — одна политика.** Транспорт кадров — это data-path продукта: невидимая перегрузка + защита «чужим лимитом» + мёртвый путь release-on-evict. Fix-forward, слоёв становится меньше (одна реализация политики вместо двух).
3. **Фаза 2 — продукт.** Вывести потерю кадров (~11% на seg) в телеметрию и GUI — прямая продуктовая ценность, зависит от Фазы 1 только счётчиками.
4. **Фаза 3 — backend_ctl.** Только «типизированный ноль» для вердиктных счётчиков. Полная программа провенанса — отдельным планом после закрытия транспорта; новых инструментов не добавлять (урок ultra-ревью 4.5/10).

LIVE-3 (ханг main) в план не входит: не воспроизводится, ловится py-spy'ем оппортунистически. Хвосты §8.6 (EventManager unbounded, `updated_at: 0.0`, красный `test_history_graph` и пр.) — backlog вне плана.

### Task 1.2 — Единая политика: дескриптор канала + умная дверь
**Level:** Senior+ (Opus)
**Assignee:** teamlead
**Goal:** решение «что делать с полной очередью» существует в репозитории ровно один раз; канальная дверь применяет ту же QoS-политику, счётчики и `on_evict`, что и targets-дверь. Разрез ответственности: **знание — у канала (статичная декларация при конструировании), поведение — в двери роутера (динамика: политика, evict-хук, гейт loan, счётчики)**.
**Files:** `router_module/interfaces.py`, `router_module/channels/base_channel.py`, `router_module/channels/queue_channel.py`, `router_module/core/router_manager.py`, `process_module/communication/process_communication.py`, `config_module/feature_flags.py`, тесты router + queues + process

**Дизайн (зафиксирован пересмотром 2026-07-21, проверен по коду):**
- `ProcessQueueDescriptor` — frozen dataclass в `router_module/interfaces.py`: `process: str`, `queue_type: str`, `to_dict()`. Присутствие дескриптора = «канал — фасад над процессной очередью из реестра». Поле `descriptor: Optional[ProcessQueueDescriptor] = None` на `MessageChannel` (ctor-параметр, по образцу log-инъекции), сериализуется в `get_info()["descriptor"]` → автоматически доезжает до `introspect_router_stats` через существующий `get_stats()["channels"]` (`router_manager.py:1222`).
- Дескриптор проставляет **единственное место, где истина известна** — точка конструирования (`register_router_channels`: свои очереди и cross-process соседи). `_local` (ThreadQueue), Socket, Modbus, тестовые каналы — без дескриптора → прежний прямой `send`, ни бита изменений.
- Умная дверь: `_do_send` после резолва для каждого канала (single И broadcast-ветки) читает `ch.descriptor` — plain-атрибут, ~наносекунды. Дескриптор есть и флаг ON → `qr.send_to_queue(d.process, d.queue_type, msg, on_evict=self._on_frame_evicted при self._frame_loan_active)` — бит-в-бит тот же гейт, что в `_deliver_by_targets:479-482`. `False` → `{"status": "error", ...}`. Дескриптора нет → `ch.send(msg)` как сегодня.
- Опрос состояния канала **сообщением на send-пути не выполняется никогда** (21 кадр/с × N получателей): дескриптор — иммутабельная декларация, читаемая локально. Опрос сообщением — только диагностика, отложен (см. «НЕ делаем»).
- Флаг `FW_CHANNEL_QUEUE_POLICY` (реестр `feature_flags.py`, default OFF): OFF = прежний `put(block=True, timeout=1.0)` бит-в-бит. Флип — в Task 1.3 живым прогоном (same-session control, урок флип-лесенки Ф7).

**Steps:**
1. Контракт: дескриптор в `interfaces.py` + поле на `MessageChannel` + surface в `get_info()`; контракт-тест.
2. Проводка дескриптора в `register_router_channels` (оба места создания mp-каналов: свои + cross-process; `_local` — без дескриптора).
3. Дверь в `_do_send` + счётчик `sent_via_channel_policy` (сколько канальных отправок делегировано в политику; отличим от прямого `ch.send`) — это и есть типизированный ноль для Фазы 3.
4. Убрать зависимость корректности от соотношения «кольцо < очередь»: политика работает при любом соотношении (тест с data-очередью `maxsize=2`).
5. Характеризационные тесты доставки обновить осознанно; semantic diff перечислить в PR-описании.

**Semantic diff (осознанные изменения при флаге ON):**
- Команды через канальную дверь на полной system-очереди: раньше стойло до 1 с + error; теперь немедленная видимая ошибка + `system_evict_blocked` (паритет с targets-дверью — политика одна). Масштаб командного трафика через эту дверь известен из Task 0.2 до флипа.

**Out of scope:** изменение адресации/резолва; тюнинг глубин; богатое самоописание канала и опрос сообщением; политика других поверхностей переполнения (буфер AsyncSender, кольцо SHM).
**Проверка:** статика + тесты; живое доказательство — Task 1.3.

**Rejected (перенести в ADR Task 1.4 с обоснованиями):**
- **A — канал делегирует в `send_to_queue` сам.** Каналу нужны ссылка на `queue_registry`, router-овский `_on_frame_evicted` и **живой** гейт `_frame_loan_active` (пересчитывается при `register/unregister_frame_middleware`, `router_manager.py:165-170` — снапшотом при конструировании не передаётся) → связка канал→роутер вплоть до цикла; два поведения в одном классе. Знание у канала уместно, поведение — нет.
- **C — роутер определяет сорт канала (isinstance / разбор имени).** Доказано кодом нерабоче: `QueueChannel` — один класс и над `mp.Queue`, и над `ThreadQueue` `_local` (`queue_channel.py:35`, `process_communication.py:105-106`) — isinstance не различает; разбор `{proc}_{qtype}` — stringly-typed, имена процессов содержат `_`.
- **E — политика как свойство очереди (PolicedQueue-обёртка).** Очереди пересекают границу процессов через PSR/pickle → обёртке нужна pickle-безопасность и пере-навеска хуков в каждом процессе; счётчики всё равно per-process; воспроизводит уже существующую per-process точку политики (`send_to_queue`) плюс добавляет слой — против правила «меньше слоёв».
- **Binding-map в роутере** (`register_channel` строит побочную карту name→(proc, qtype)). Дублирует знание вторым состоянием, требует синхронизации на `unregister_channel`; прямое чтение атрибута канала проще и не рассинхронизируется.
- **D в максималистской форме — опрос описания сообщением на send-пути.** Message-roundtrip на каждый кадр недопустим; оставлена локально читаемая декларация, опрос сообщением — диагностика, отложен до реального потребителя.
- Из прежней редакции: **выключение kind-каналов** (лечит один путь из трёх, теряет O(1)-резолв; годится лишь как аварийный рычаг), **«каналы без транспорта»** (ломает `IMessageChannel` — Socket/Modbus транспорт по существу; добавило бы слой диспетчеризации вместо удаления).

**Связь с P0.2:** контракт `AddressAwareQueueChannel` (`routing/address_aware_channel.py`) декларирует тот же целевой инвариант — «канал — тупая труба, транспорт за каналом = queue_registry». Схема дескриптора достигает инварианта без нового подкласса канала; при возврате к P3 transport-router-hub контракт P0.2 сверить с этим решением.

# План: одна политика переполнения для транспорта кадров

**Slug:** `transport-single-policy`
**Ветка:** продолжение `fix/bug-hunt-live-findings` (при желании — новая `fix/transport-single-policy` от неё)
**Основание:** `docs/audits/2026-07-21_frame-transport-doors.md`, `docs/audits/2026-07-20_bug-hunt.md` §9-§11
**Автор плана:** teamlead (архитектурный пересмотр 2026-07-21)

## Суть проблемы (подтверждено чтением кода)

Одна и та же `multiprocessing.Queue` доступна через две двери с разной политикой переполнения:

- **Дверь А** (`_deliver_by_targets` → `QueueRegistry.send_to_queue` → `remove_old_if_full`): QoS-политика (drop_oldest/never-drop), счётчики `data_evicted`/`system_evict_blocked`, хук `on_evict` (release-on-evict, ADR-RTR-010).
- **Дверь Б** (`_do_send` → `channels[N].send` → `QueueChannel.send`, `queue_channel.py:52-67`): голый `put(block=True, timeout=1.0)` — стойл потока-отправителя до 1 с, затем `{"status":"error"}` (фактически drop_newest), без QoS, без атрибутируемых счётчиков, без `on_evict`. Кадры уходят **синхронным** `router.send` с потока-исполнителя (`send_to_process` → `router_manager.send`), поэтому стойл бьёт по продьюсеру кадров напрямую.

Политика переполнения зависит от способа резолва адреса — произвольная связь двух независимых осей. Фикс LIVE-2 (`8b8a3c54`) на боевой раскладке kind-каналов не исполняется: `on_evict` навешивается только в `_deliver_by_targets`.

Дополнительно подтверждён (статикой) латентный баг `release_evicted`: `loan_ledger.py:191-208` декрементит refcount по **любому** имени reader'а — множества loan-aware потребителей в леджере нет. `_on_frame_evicted` (`router_manager.py:484-533`) шлёт release с `reader = владелец очереди`, а конверт кадра несёт SHM-координаты и для copy-out получателей (GUI): `frame_shm_middleware.py:462-463` проставляет `owner`/`shm_owner` во **все** исходящие кадровые конверты. Вытеснение из очереди GUI сняло бы чужую долю → досрочное освобождение слота. Сегодня ветка мертва для кадров, но **унификация политики её оживит** — фикс обязан идти раньше унификации.

## Порядок и обоснование

1. **Фаза 0 — доказательная база.** Оба направления (транспорт и инструмент) упираются в одно: мы не знаем фактическое распределение трафика по дверям и не знаем, чем был снят блокер webcam-soak. Резать транспорт вслепую нельзя, а закрытие этих вопросов дёшево (счётчики + один короткий прогон). Сюда же — минимальный срез доказательности backend_ctl (строгие имена счётчиков), потому что именно им будет проверяться результат Фазы 1.
2. **Фаза 1 — одна политика.** Транспорт кадров — это data-path продукта: невидимая перегрузка + защита «чужим лимитом» + мёртвый путь release-on-evict. Fix-forward, слоёв становится меньше (одна реализация политики вместо двух).
3. **Фаза 2 — продукт.** Вывести потерю кадров (~11% на seg) в телеметрию и GUI — прямая продуктовая ценность, зависит от Фазы 1 только счётчиками.
4. **Фаза 3 — backend_ctl.** Только «типизированный ноль» для вердиктных счётчиков. Полная программа провенанса — отдельным планом после закрытия транспорта; новых инструментов не добавлять (урок ultra-ревью 4.5/10).

LIVE-3 (ханг main) в план не входит: не воспроизводится, ловится py-spy'ем оппортунистически. Хвосты §8.6 (EventManager unbounded, `updated_at: 0.0`, красный `test_history_graph` и пр.) — backlog вне плана.

## Фаза 1 — Одна политика, две адресации

**Гейт выхода:** вытеснение кадра доказуемо исполняется на живой раскладке (счётчик `frame_loans_released_on_evict` > 0 в probe-рецепте), кольцо переживает перегрузку, стойл писателя на полной data-очереди устранён.

### Task 1.1 — Guard чужого reader'а в release-on-evict (ДО унификации)
**Level:** Senior+ (Opus)
**Assignee:** teamlead
**Goal:** release вытесненного кадра декрементит refcount только если reader входит в loan-aware множество владельца; copy-out получатель (GUI) не снимает чужую долю.
**Files:** `multiprocess_framework/modules/shared_resources_module/memory/pool/loan_ledger.py`, `.../frame_shm_middleware.py` (проводка имён потребителей), `router_module/core/router_manager.py` (`_on_frame_evicted`), тесты
**Steps:**
1. Выбрать точку истины: (а) owner-side — при `commit` фиксировать множество ожидаемых reader'ов (требует проводки имён, сейчас есть только число `num_consumers`), или (б) метка в конверте per-target «loan-aware», проверяемая в `_on_frame_evicted`. Решение оформить коротким ADR (модульный, shared_resources).
2. Реализовать + юнит-тест: вытеснение конверта из очереди не-loan-aware процесса НЕ меняет refcount.
3. Обновить докстринг `release_evicted` (сейчас декларирует dedup как достаточную защиту — это неверно для чужих имён).
**Acceptance criteria:**
- [ ] тест: `release_evicted(reader="gui")` при `num_consumers=2` (lines, points) → freed == 0
- [ ] штатные release/reclaim не регрессируют (полный suite shared_resources)
**Out of scope:** сама унификация политики.
**Проверка:** статика + юнит-тесты (живого прогона не требует — ветка станет достижимой только после 1.2).

### Task 1.2 — Единая реализация политики переполнения
**Level:** Senior+ (Opus)
**Assignee:** teamlead
**Goal:** решение «что делать с полной очередью» существует в репозитории ровно один раз; дверь Б применяет ту же QoS-политику, счётчики и `on_evict`, что и дверь А.
**Files:** `shared_resources_module/queues/core/manager.py`, `router_module/channels/queue_channel.py`, `router_module/core/router_manager.py`, `process_module/communication/process_communication.py`, тесты всех троих
**Steps:**
1. Дизайн-выбор (зафиксировать в ADR): предпочтительно — mp-backed `QueueChannel` для процессных очередей делегирует в `QueueRegistry.send_to_queue(process, qtype, …)` (kind известен из имени канала `{proc}_{qtype}`), т.е. в репо остаётся **одна** точка put-с-политикой. Альтернатива (если делегация ломает слои) — извлечь policy-applier в отдельную функцию/объект и звать из обоих мест с общими счётчиками. Каналы, не обёрнутые над процессными очередями (`_local` ThreadQueue, Socket), поведения не меняют.
2. Проводка `on_evict`: `_do_send` передаёт `self._on_frame_evicted` на data-kind канальный путь при `_frame_loan_active` (та же гейт-логика, что в `_deliver_by_targets`).
3. Убрать зависимость корректности от соотношения «кольцо < очередь»: политика работает при любом соотношении.
4. Отвергнутые альтернативы фиксировать в ADR: выключение kind-каналов (лечит только один путь из трёх, теряет O(1)-резолв; годится лишь как аварийный рычаг отката), «каналы без транспорта» (ломает IMessageChannel — Socket/local каналы транспорт по существу; добавило бы слой диспетчеризации вместо удаления).
**Acceptance criteria:**
- [ ] юнит: полная data-очередь на канальном пути → drop_oldest + `data_evicted` + вызов `on_evict`; system → never-drop + `system_evict_blocked`; стойла put(timeout=1.0) нет
- [ ] юнит из Task 1.1 остаётся зелёным при живом `on_evict` на канальном пути
- [ ] `test_live2_evict_release.py` проходит на ОБОИХ путях адресации
- [ ] полный suite router + queues + process (`python scripts/run_framework_tests.py`) зелёный
**Out of scope:** изменение адресации/резолва; тюнинг глубин очередей.
**Проверка:** статика + тесты; живое доказательство — Task 1.3.

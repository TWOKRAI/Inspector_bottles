### Задача 1.1 — ADR-RM-006: «Регистры vs State Store — когда что применять»

**Уровень:** Middle+ (Sonnet, extended thinking)
**Исполнитель:** developer
**Цель:** Написать ADR-RM-006 в `registers_module/DECISIONS.md`, который чётко разграничивает применение `registers_module` и `state_store_module` через decision matrix.

**Контекст:** Оба модуля дают pub/sub — разработчик может запутаться, что выбрать.
`registers_module` — типизированные Pydantic-инстансы, per-field подписки, FieldRouting fan-out, snapshot при изменении, предназначен для **именованных конфигов** (устройства, UI-настройки).
`state_store_module` — произвольное dict-дерево, glob-pattern подписки, delta-only доставка, middleware pipeline (throttle, validation), предназначен для **real-time состояния** (метрики, телеметрия, динамические топологии).
ADR фиксирует эту границу официально.

**Файлы:**
- `multiprocess_framework/modules/registers_module/DECISIONS.md` — добавить секцию ADR-RM-006 в конец файла

**Шаги:**

1. Открыть `multiprocess_framework/modules/registers_module/DECISIONS.md`, найти последнюю запись (ADR-RM-005).
2. Добавить секцию `## ADR-RM-006: Регистры vs State Store — когда что применять` сразу после ADR-RM-005.
3. Структура ADR-RM-006:
   - **Дата:** 2026-05-08
   - **Статус:** принято
   - **Контекст:** оба модуля предоставляют pub/sub; без документированной границы разработчик выбирает инструмент интуитивно.
   - **Решение:** секция с decision matrix (таблица — см. ниже) + явное правило выбора.
   - **Decision matrix** — таблица из 3 колонок: «Критерий», «Регистры (`registers_module`)», «State Store (`state_store_module`)»; строки:
     - Структура данных
     - Типизация
     - Паттерн подписки
     - Доставка изменений
     - Middleware / валидация
     - IPC fan-out
     - Типичные применения
     - Типичные антипаттерны (когда НЕ использовать)
   - **Явное правило:** «Если поле имеет схему, имя и FieldRouting — регистр. Если структура динамическая, иерархическая или realtime-метрика — state store.»
   - **Граничные случаи:**
     - UI-форма настроек камеры → регистр (типизировано, FieldRouting нужен)
     - Текущий FPS процесса → state store (runtime-метрика, меняется 30/с)
     - Список активных процессов (динамический) → state store (glob-подписка)
     - Таблица рецептов (фиксированная схема) → регистр
   - **Почему не объединять:** у каждого модуля своя модель доставки; объединение создаст «швейцарский нож» без гарантий производительности (delta-IPC несовместимо с FieldRouting fan-out).
   - **Последствия:** новый разработчик обращается к этому ADR, не изучая оба модуля.
   - **Связанные решения:** ADR-RM-001, ADR-SS-001, ADR-SS-013 (PersistenceManager), CONSTRUCTOR_BLUEPRINT §4 паттерн 8.

**Критерии приёмки:**
- [ ] Секция `## ADR-RM-006` присутствует в `registers_module/DECISIONS.md` после ADR-RM-005
- [ ] Таблица decision matrix содержит не менее 7 строк
- [ ] Упомянуты все 4 граничных случая (UI-форма, FPS, список процессов, рецепты)
- [ ] Явное правило выбора сформулировано одним предложением
- [ ] Секция «Почему не объединять» присутствует
- [ ] Ссылки на ADR-RM-001, ADR-SS-001 указаны в «Связанные решения»

**Вне рамок:** не трогать код модулей, не добавлять новых тестов, не редактировать другие ADR.

**Граничные случаи:** Не нумеровать ADR как RM-002 (занят) — только RM-006.

**Зависимости:** нет

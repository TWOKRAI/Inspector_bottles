> ⛔ **ОТЛОЖЕНО / ВЛИТО В P2 (решение владельца 2026-06-01).** Отдельной фазой не делаем.
> Причина: P1.5b не разблокирует отладку прототипа сейчас (логи уже пишутся в файлы, внешней
> двери в систему нет — это P2). Push-стрим окупается только ПОСЛЕ P2 и там получается **бесплатно**:

> Реестр подписок в `ChannelRoutingManager` + `route_stream` (этот документ) — **отвергнуто** как
> переусложнение. Документ сохранён ради разведки кода (§3 — иерархия классов, `MessageType`,
> `error`=LOG с level≥ERROR) и как обоснование решения.

## 1. Цель

Подписчик (driver — позже под MCP; на P1.5b — тестовый таргет) получает **push-поток**
`log` / `error` / `event` от выбранного процесса **через штатный `RouterManager`** — а не
поллингом `introspect.*`. Доставка тем же путём, что у обычных сообщений
(`Message` → router → очередь получателя). Закрывает «послал команду → вижу, что произошло».

---

## 2. Принцип

**«Подписчик = ещё один `targets`».** Сообщения логов уже едут через router. Подписка не
плодит транспорт — лишь **добавляет адрес получателя** в fan-out существующего/нового `Message`.
Согласовано с `transport-router-hub` (один транспорт) и P2 (driver = «GUI по сокету»).

Поскольку kinds = все три, механизм делается **единым (generic)**: один реестр + один путь
роутинга `route_stream(kind, …)`, у которого `log`/`error`/`event` — лишь три вызывающих.

## 4. Реестр подписок (Вариант A — единый в базе)

### 4.1. Где живёт

Единый `StreamSubscriptionRegistry` в **`ChannelRoutingManager` (CRM)** — подтверждённый общий
предок Logger/Error/Stats (см. §3.2). Один реестр → команда `stream.subscribe` пишет один раз →
fan-out единообразен для `log`/`error`/`event`.

- ➕ DRY, одна точка правды, дешёвая `introspect.subscriptions` (`feedback_constructor_modularity`).
- ⚠️ База читается из горячего пути → строгая потокобезопасность (§7.7).
- Реестр **per-process** (подписка адресная: «поток ЭТОГО процесса»). `stream.subscribe`
  маршрутизируется в нужный процесс через router и обрабатывается его builtin-handler'ом.

### 4.2. Структура

```
subscription = {
  subscriber: <target-name>,                  # куда push'ить (имя очереди/канала получателя)
  kinds:      set[{"log","error","event"}],
  min_level:  "DEBUG|INFO|WARNING|ERROR|...",  # фильтр для log/error (event — без уровня)
}
registry: dict[subscriber -> subscription]
```

Горячий путь читает **снимок** таргетов по kind (§7.7), не держит лок на доставку.

## 5. Единый механизм роутинга `route_stream` (DRY для трёх kinds)

В общей базе — один метод, через который идут все три потока:

```python
def route_stream(self, kind, message_type, *, level=None, payload):   # база
    if not (self.is_enabled("router_routing") and self._router_manager):
        return
    targets = self._stream_targets(kind, level)        # снимок подписчиков + базовый таргет
    if not targets:                                    # fast-path: никого → выходим
        return
    # собрать Message один раз, targets — все получатели; send (try/except pass)
```

- **log:** `_route_via_router` логгера переписывается на `route_stream("log", MessageType.LOG,
  level=record.level, payload=record)` — базовый таргет `"logger"` сохраняется как и был.
  `ErrorManager(LoggerManager)` идёт **этим же путём** (ошибка = LOG с `level≥ERROR`).
- **error:** НЕ отдельный вызов — это `log`-поток, отфильтрованный по `level ≥ ERROR`. Подписчик с
  `kinds={"error"}` получает LOG-сообщения только при `level in {error, critical}`. Реализуется
  внутри `_stream_targets` (kind `error` ⟺ kind `log` + жёсткий нижний порог ERROR).
- **event:** см. §6 — эмиттеры зовут `route_stream("event", MessageType.EVENT, payload=evt)`
  (через `MessageAdapter.event()`).

`_stream_targets(kind, level)` для входящего LOG возвращает подписчиков, у кого `"log" in kinds`
(с `level ≥ min_level`) **или** `"error" in kinds` (с `level ≥ ERROR`); для EVENT — у кого
`"event" in kinds`. Плюс опц. постоянный базовый таргет (`"logger"` для логов). Пусто →
`route_stream` выходит сразу (fast-path).

## 6. EVENT — определение источника (решение: включаем) `[verify@code]`

`event` — **lifecycle/доменные события процесса**, не логи и не ошибки. Источник — НЕ новый
подсистемный механизм, а тонкий эмиттер поверх того же `route_stream`:

- **Базовый набор (P1.5b):** lifecycle процесса/воркера — `worker.created/started/stopped`,
  `process.state_changed`, `command.applied` (напр. `register_update` применён). Эти точки уже
  существуют как команды/переходы — добавляем в них вызов `route_stream("event", …)` при наличии
  подписчиков (fast-path no-op без них).
- **Опционально (отдельная задача, НЕ P1.5b):** трансляция изменений `state_store_module`
  (может быть шумно — отдельно с фильтром по glob-подписке).
- **`MessageType.EVENT` уже есть** + билдер `MessageAdapter.event(event_type, event_data, …)` —
  заводить тип не нужно. `[verify@code]`: эмитит ли кто-то EVENT через router сейчас, или
  lifecycle-эмиттеры добавляем в P1.5b.

> Зачем сразу: владелец выбрал все три. Чтобы EVENT не раздул scope, на P1.5b он ограничен
> lifecycle-точками (их немного и они дёшевы), без StateStore-трансляции.

## 7. Анти-рекурсия, анти-спам, перфоманс (КРИТИЧНО — горячий путь)

1. **Анти-рекурсия log→route→log.** В `route_stream` и пути доставки **запрещено** вызывать
   `self.log(...)`/любой эмит того же kind. Существующий `try/except: pass` логгера сохранить.
   Тест: доставка N логов не порождает рост числа логов.
2. **Запрет самоподписки.** `subscriber` не может быть `"logger"`/собственным таргетом менеджера —
   валидация в `stream.subscribe`.
3. **Fast-path при пустом реестре** (инвариант для прода без driver): `_stream_targets` отдаёт
   пусто → `route_stream` выходит до сборки `Message`. Накладные = одна проверка «снимок пуст».
   У логгера сегодня `_route_via_router` всегда шлёт на `"logger"`; новый путь сохраняет это
   поведение, подписчики — это **доп.** таргеты.
4. **Backpressure = drop, не block.** Очередь подписчика переполнена → **дроп** сообщения
   подписчику (прод важнее полноты стрима у агента); считать `dropped` (в `introspect.subscriptions`).
   Никаких блокировок в `log()`/`route_stream`.
5. **`min_level`** применять рано (на продьюсере, до сборки `Message`); `should_log` логгера уже
   фильтрует грубо — per-subscriber фильтр поверх.
6. **Один build — много targets.** `Message` собирается один раз; fan-out — забота router'а.
7. **Потокобезопасность реестра.** Запись (команда из приёмного потока) и чтение (горячий путь из
   рабочих потоков) — короткий лок только на снятие снимка set'а таргетов; **лок не держится** на
   время сборки/`send`.
8. **Контракт P0.5:** стрим — **fire-and-forget** (push без ответа), `request()` НЕ используется →
   дедлок-контракт не задет.

## 9. Жизненный цикл

- Подписки процесса **очищаются при его остановке**.
- P1.5b — явный `stream.unsubscribe` + clear-on-stop. Авто-отписка по разрыву сокета — **P2**.

## 10. Тест-план (через router-путь, не handler напрямую)

- `subscribe` → лог процесса доходит до таргета через `router.receive` (fan-out, +базовый `"logger"`).
- `unsubscribe` → доставка прекращается.
- kinds: подписка только на `error` не приносит `log`; `event` — отдельно (lifecycle-точка).
- `min_level`: `WARNING`-подписка не получает `INFO`.
- **анти-рекурсия:** доставка K логов не порождает новых логов (счётчик стабилен).
- **fast-path:** без подписчиков поведение прежнее (логгер шлёт только на `"logger"`; error/event — no-op).
- **самоподписка** отклоняется.
- **backpressure:** переполненная очередь подписчика → drop + счётчик, хот-путь не блокируется.
- три kinds через **один** `route_stream` (DRY проверяется: один путь, три вызывающих).
- `introspect.subscriptions` отражает реестр + `delivered`/`dropped`.
- reachability: команды видны в `message_dispatcher` после регистрации.

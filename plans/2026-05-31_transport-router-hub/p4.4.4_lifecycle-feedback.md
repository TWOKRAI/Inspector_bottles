## Зачем (выгода, подтверждена с владельцем 2026-06-06)

Сегодня команда идёт через `request()` → поток **блокируется до единственного финального ответа или таймаута**. Промежуточного состояния нет. Боль проявляется на **долгих** командах (быстрые — worker CRUD, field-write, state.get — выгоды не дают):

| Боль | Где в проекте |
|------|---------------|
| Нет «принято» — UI кажется зависшим | Активация рецепта / `replace_blueprint` (спавн N процессов) |
| Таймаут под долгую операцию задаётся вслепую | Старт источников, загрузка ML-модели, рестарт пайплайна |
| Нет стадий/прогресса | «остановлено k/M, запущено j/N процессов» |
| Падение непрозрачно — только финальная ошибка | На какой стадии запуска рецепта упало |

**Реальный потребитель в этом плане:** `ProcessManagerProcess.replace_blueprint` ([process_manager_process.py:667](../../../multiprocess_framework/modules/process_manager_module/process/process_manager_process.py#L667)) — у него естественные стадии: остановка каждого `to_replace` (цикл :737), старт каждого нового (цикл :758). Это и есть «запуск рецепта», который владелец хочет видеть в GUI.

## Recon (2026-06-06, read-only, на актуальном `main` после P4.4.1/1b/6)

> Investigation-first (принцип родителя). Recon от 2026-06-05 (p4.4_command-bus §8) актуализирован: влились P4.4.1/1b/6, имена/строки сдвинулись.

### R1. ⚠ ГЛАВНАЯ НАХОДКА — `ctx` нельзя передать «вторым аргументом»
`Dispatcher.dispatch()` зовёт обработчик **строго одним позиционным аргументом** ([dispatcher.py:402-403](../../../multiprocess_framework/modules/dispatch_module/core/dispatcher.py#L402-L403)):
```python
handler_data = message if handler_info.expects_full_message else message.get(data_field, {})
result = handler_info.handler(handler_data)
```
Recon P4.4.0 предполагал «ctx — опциональный 2-й аргумент» — это **невозможно** без правки горячего диспетчера (отвергнуто: ядро не трогаем).
**РЕШЕНИЕ:** bus кладёт `ctx` в билет под transport-internal ключом **`message["_ctx"]`** (по аналогии с уже существующими `_address`, `_receive_info`, `_source_channel`); lifecycle-команда регистрируется с **`expects_full_message=True`** и читает `ctx = message.get("_ctx")`. Это явный контракт opt-in.
**Инвариант безопасности:** `_ctx` — локальный объект, **никогда не пересекает границу процесса** (не сериализуется, билет с `_ctx` не пере-отправляется). Гарантируется тем, что инъекция — на приёмной стороне в `_dispatch_command`, после десериализации, и `processed` дальше не уходит в `send`.

### R2. Поток `receive()` (router_manager.py) — точки вставки подтверждены
`receive()` ([:539](../../../multiprocess_framework/modules/router_module/core/router_manager.py#L539)): poll → `_recv_mw` → pending-resolve (`type=="response"`, [:590](../../../multiprocess_framework/modules/router_module/core/router_manager.py#L590)) → `_route_to_worker` ([:599](../../../multiprocess_framework/modules/router_module/core/router_manager.py#L599)) → `_route_by_kind` ([:605](../../../multiprocess_framework/modules/router_module/core/router_manager.py#L605)).
- **Место для progress-subscriber check:** МЕЖДУ pending-resolve (:590) и `_route_to_worker` (:599). Если `type in {"event","progress"}` И `cid in _progress_subscribers` → callback + `continue` (consumed, без диспатча и warning-шума).
- `type="event"+correlation_id` проходит pending-resolve мимо (резолв только `type=="response"`, R2 подтверждён [:590](../../../multiprocess_framework/modules/router_module/core/router_manager.py#L590)) — как и предсказывал recon §8.

### R3. `_dispatch_command` — единая точка для авто-lifecycle
`_dispatch_command` ([:463](../../../multiprocess_framework/modules/router_module/core/router_manager.py#L463)) уже знает `cid` (через `_extract_correlation_id`) и `info = cm.get_command_info(cmd_name)` (metadata). Сюда вешаются: создание `ctx`, авто-`accepted` (вход), авто-`completed`/`failed` (выход/исключение).
**Коллизия с `manages_own_reply`:** `process.command`/`blueprint.replace` отвечают САМИ (PM строит свой `process.command.response`, [:856](../../../multiprocess_framework/modules/process_manager_module/process/process_manager_process.py#L856)) → помечены `metadata.manages_own_reply=True`, авто-reply пропускается ([:502](../../../multiprocess_framework/modules/router_module/core/router_manager.py#L502)). Lifecycle должен это уважать: **финальный `response` остаётся за командой** (или generic reply), `ctx` шлёт только lifecycle-**события** (accepted/progress) + терминальный **event**-маркер (completed/failed) для подписчиков — НЕ второй response.

### R4. Реестр pending — готовый образец
`_pending_requests` + `_PendingRequest` + `_pending_lock` ([:120](../../../multiprocess_framework/modules/router_module/core/router_manager.py#L120)) — образец для `_progress_subscribers` (но: не блокирует, many-events, cleanup на финале/таймауте). `request()` ([:353](../../../multiprocess_framework/modules/router_module/core/router_manager.py#L353)) расширяется опц. параметром `on_progress`.

### R5. CommandManager — место для opt-in флага
`register_command` ([command_manager.py:152](../../../multiprocess_framework/modules/command_module/core/command_manager.py#L152)) принимает `metadata` → флаг `lifecycle` кладём в metadata (или отдельным kwarg → metadata). `get_command_info` ([:286](../../../multiprocess_framework/modules/command_module/core/command_manager.py#L286)) уже отдаёт metadata в `_dispatch_command`. **Реальный потребитель:** `_cmd_blueprint_replace(self, data=None, **kwargs)` ([:826](../../../multiprocess_framework/modules/process_manager_module/process/process_manager_process.py#L826)) — `expects_full_message=False`. Для lifecycle придётся перевести на `expects_full_message=True` + читать `data`/`_ctx` из message (мелкий рефактор сигнатуры).

### R6. Envelope-билдеры
[command_envelopes.py](../../../multiprocess_framework/modules/message_module/builders/command_envelopes.py) — добавить `build_progress_event(...)`: `{type:"event", command:"<cmd>.progress", data:{stage, progress, payload}, request_id:cid, seq, sender, targets:[initiator]}`.

### R7. Доставка event'а инициатору — ОТКРЫТЫЙ вопрос (проверить в P4.4.4.2)
`_select_queue_type` ([:243](../../../multiprocess_framework/modules/router_module/core/router_manager.py#L243)): `type in (command,system)` → "system", иначе ("event"/"data") → **"data"-очередь**. Значит progress-event с `type="event"` поедет инициатору в **data-очередь**. **Проверить:** опрашивает ли процесс-инициатор (GUI-host / driver) свою data-очередь на приёме (нужно, чтобы progress дошёл до `_progress_subscribers`). Если нет — слать progress с `queue_type="system"` (как ответы), чтобы попал в system-приёмник инициатора. **Это блокер P4.4.4.2 — решить до реализации подписки.**

### R8. GUI-мост recipe-launch — переиспользуем (pipeline-live-control)
Активация рецепта в GUI → `replace_blueprint` через IPC-мост уже работает (memory `project_pipeline_live_control_stage1`: «IPC-мост GUI→PM DONE»). P4.4.4.4 подписывается на progress этого же запроса (`request(on_progress=...)`), не изобретая транспорт. **Точный call-site инициации** (`launch_active_recipe`/recipes-presenter) уточнить в P4.4.4.4 (investigation-first перед GUI-правкой).

## Целевая архитектура

```
ИНИЦИАТОР (GUI/driver)                          ИСПОЛНИТЕЛЬ (PM/процесс)
 router.request(msg, on_progress=cb)             receive() → _dispatch_command(processed):
   cid = uuid                                       if lifecycle && cid:
   _progress_subscribers[cid] = cb                     ctx = ExecutionContext(router, cid, initiator)
   send(msg, request_id=cid)                           ctx.report("accepted")            # event seq=0
   block on pending.event.wait ───────┐                processed["_ctx"] = ctx
                                       │             result = cm.handle_command(processed)
 receive() (отдельный поток):         │                # handler внутри: ctx.report("running", progress=k/N)
   event+cid in subscribers → cb(msg) │             on OK:    ctx.finish("completed")   # event terminal
   (progress bar обновляется)         │             on EXC:   ctx.finish("failed", err)
                                       │             reply_to_request(...) / own response → type="response"
   response+cid → resolve pending ◀────┘                (закрывает request())
   request() возвращает → finally: unsubscribe(cid)
```

### Три сущности (минимум, наследует «тупые реестры» родителя)

| Сущность | Где | Ответственность |
|---|---|---|
| `ExecutionContext`/`ProgressReporter` | `command_module` (новый файл) | `report(stage, data=None, progress=None)` + `finish(stage, ...)`; держит `seq`-счётчик (монотонный per-cid), строит progress-event через `build_progress_event`, шлёт через `router.send` |
| `_progress_subscribers` + `subscribe_progress`/`unsubscribe_progress` | `router_module` (RouterManager) | реестр `cid→callback`; consume event/progress в `receive()`; cleanup на финале/таймауте; не блокирует |
| авто-lifecycle | `router_module` (`_dispatch_command`) | создать ctx (если opt-in+cid), авто-`accepted`/terminal, инъекция `_ctx`, уважать `manages_own_reply` |

### Семантика стадий (зафиксировать в ADR-COMM-006)
- `accepted` — авто на входе (event, seq=0).
- `running`/`progress` — вручную из хендлера через `ctx.report(...)` (event, seq++, опц. `progress: float 0..1`).
- `completed`/`failed` — авто на выходе/исключении: терминальный **event** (для чистых подписчиков) + **финальный `response`** закрывает `request()` (НЕ дублировать response, см. R3).
- **Порядок:** в envelope `seq` (монотонный per-cid); потребитель упорядочивает, поздний `accepted` после терминала отбрасывает.
- **Backward:** команда без `lifecycle=True` ИЛИ билет без cid → ноль изменений (только финальный response, как сегодня).

## Зачем (проблема)

Сегодня приём команды проходит **два одинаковых диспетчера** `dispatch_module.Dispatcher`, сцепленных цепочкой, оба резолвят по `command`:

```
receive() (router_manager.py:467)
  → message_dispatcher.dispatch(key="command")     ← резолв #1: все N ключей → одна generic-H
  → _make_command_handler closure (process_lifecycle.py:134)
      → cm.handle_command(msg)                      ← timing/metrics/log (command-only)
          → cm.dispatcher.dispatch(key="command")  ← резолв #2 (настоящий)
      → router.reply_to_request(msg, result)        ← reply закопан в lifecycle-closure
```

Три структурных дефекта:
1. **Двойной резолв** по одному `command` (две инстанции одного `Dispatcher`: `RouterManager.message_dispatcher` router_manager.py:94 и `CommandManager.dispatcher` command_manager.py:88).
2. **`message_dispatcher` знает каждый ключ команды** (`register_commands_with_router` process_lifecycle.py:104 регистрирует команды по одной) — протечка ответственности CommandManager наверх.
3. **Наблюдаемость и reply — несквозные**: timing/metrics впаяны в `CommandManager.handle_command` (command-only), `reply_to_request` — в closure (только команды, прошедшие через lifecycle-регистрацию).

## Целевая архитектура (first-principles, «как полагается»)

Один **входной конвейер** на процесс; сквозное — в стадиях, реестры — чистые таблицы. Три непересекающиеся сущности:

| Сущность | Ответственность | НЕ его дело |
|---|---|---|
| **kind-router** (в `receive()`) | роутинг по **виду** `type`: `COMMAND`→command bus; `STATE`/`EVENT`→свои хендлеры; `RESPONSE`→correlation | не резолвит имена команд |
| **CommandRegistry** (бывш. `CommandManager`) | чистая библиотека `{имя → handler + metadata/tags/options}` + тонкий `resolve+run` | транспорт, correlation, наблюдаемость |
| **inbound-middleware** (конвейер вокруг исполнения) | observability (timing/metrics/log) + lifecycle-feedback + reply — **единообразно для всех видов** | бизнес-логика команд |

Резолв по `command` — **ровно один раз** (в CommandRegistry). `message_dispatcher` маршрутизирует по `type` (зеркало kind→channel на отправке из P0.2/P3, только на приёме). Паттерн — **Command Bus + middleware-pipeline** + message-router по виду.

```
InboundPipeline.process(msg):           # расширение receive()
    msg = _recv_mw.apply(msg)            # СУЩЕСТВУЕТ (pre-dispatch transform)
    if correlation.resolve_if_response(msg): return     # P0.5, type=="response"
    if route_to_worker(msg): return                     # P2.2 control-plane (ПОРЯДОК: до kind-router!)
    kind = msg["type"]
    with observability(msg):             # шов вокруг исполнения (context-manager/inline, НЕ новый pipeline)
        result = kind_router[kind](msg)  # COMMAND → registry[msg["command"]](msg["data"], ctx?)
    if wants_reply(msg): reply(msg, result)             # стадия, единообразно
```

> **УТОЧНЕНИЕ ПОСЛЕ РЕВЬЮ (2026-06-05, NEEDS REWORK → исправлено):**
>
> **(1) Резолв двухуровневый, не «двойной».** Сегодняшний дефект — `message_dispatcher` И `CommandManager.dispatcher` резолвят по **одному** ключу `command` (дубль). Цель — **разные** ключи: `message_dispatcher` по `type` (kind), CommandRegistry по `command`. Это корректные два уровня (вид→имя), резолв по `command` один.
>
> **(2) ⚠ Блокер ревью: есть command-хендлеры МИМО CommandManager.** Напрямую в `message_dispatcher` (по ключу-команде) зарегистрированы как минимум `process.command` (`process_manager_process.py:173`), `register_update` (`plugin_orchestrator.py:259`), `state.changed` (`process_module.py:276`). Наивный kind-router «`type=="command"` → CM» **украдёт** их сообщения (в CM их нет → потеря). Отсюда — **развилка дизайна, решается в P4.4.0:**
>   - **Вариант A (kind-router + fallback):** `type=="command"` → если `command` есть в CM → CM; иначе fallback в `message_dispatcher` по ключу. CM остаётся исполнителем-под-шиной. Минус: два резолвера сосуществуют.
>   - **Вариант B (один диспетчер):** CommandManager **регистрирует свои хендлеры прямо в `message_dispatcher`** (через `register_message_handler`), а свой второй `Dispatcher` снимает с живого пути. Тогда CM-команды, `process.command`, `register_update` сосуществуют как ключи в ОДНОМ диспетчере — коллизии нет by-design, дубль исчезает, CM = чистый реестр (ближе к замыслу «библиотека»). Наблюдаемость/reply — шов в `receive()` поверх единого диспетчера.
>
>   **РЕШЕНИЕ ВЛАДЕЛЬЦА (2026-06-05, изначально): вариант B** — «меньше слоёв». ⚠ **ПЕРЕСМОТРЕНО ПОСЛЕ RECON (2026-06-05) → выбран B2 (kind-router)**, не B1. Recon показал: (1) B1 «один диспетчер» делает CommandManager полым фасадом и оставляет `message_dispatcher` свалкой (имя врёт) — это противоречит разделу «Целевая архитектура» этого плана; (2) все команды несут `type=command` → kind-router реализуем, обходные хендлеры (process.command/register_update) сворачиваются в обычные команды CM, коллизии нет. B2: `message_dispatcher`→`kind_router` (резолв по `type`), `CommandManager` владеет таблицей команд (резолв по `command` — 1 раз, в CM). «Меньше слоёв» = меньше полых/дублирующих компонентов, а не -1 dict-lookup. **Детали и таблица сравнения — во врезке «ПЕРЕСМОТР РАЗВИЛКИ» (секция recon).**

## Lifecycle-feedback (новое требование владельца, опционально)

Команда может **опционально** репортить ход выполнения обратно инициатору: `accepted` (принята) → `running`/`progress` (выполняется, опц. с долей) → `completed`|`failed`. Дизайн:

- **Opt-in при регистрации:** `register_command(..., lifecycle=True)` (дефолт `False` → нынешнее поведение, ноль регрессии).
- **Транспорт через correlation_id (P0.5):** промежуточные репорты идут как `type="event"` (или `type="progress"`) с `correlation_id` входящего билета — **НЕ `type="response"`**, чтобы не резолвить pending-future инициатора раньше времени (receive() router_manager.py:518 резолвит только `type=="response"`). **Финальный** ответ = `type="response"` (completed/failed) — он и закрывает `request()`.
- **ExecutionContext / ProgressReporter:** если команда opt-in И билет несёт correlation_id — bus передаёт хендлеру `ctx` с `ctx.report(stage, data=None, progress=None)`. `accepted` шлётся автоматически на входе, `completed`/`failed` — автоматически на выходе/исключении; `running`/`progress` — вручную из хендлера. Команды без opt-in `ctx` игнорируют (сигнатура совместима: `ctx` — опциональный 2-й аргумент или через `expects_full_message`).
- **Backward:** инициатор без подписки на progress получает только финальный `response` — как сегодня.

> **⚠ Блокер ревью — механизм подписки на progress (детализировать в P4.4.0).** Сегодня инициатор ждёт ответ через `request()` (блокирует поток на `pending.event.wait`), а `message_dispatcher` резолвит по **ключу** (command/type), НЕ по correlation_id. Значит progress-события (`type="event"`+correlation_id) сейчас **некуда деться** — у `message_dispatcher` нет handler'а для них → warning-шум на каждый репорт. Нужен **отдельный реестр progress-callbacks по correlation_id** (аналог `_pending_requests`, но: не блокирует, принимает МНОГО событий, чистится по `completed`/`failed`/timeout). API: `router.subscribe_progress(correlation_id, callback)` / авто-снятие на финале. Это самостоятельная подсистема, а не «минимальный API».
>
> **Гарантия порядка.** Между процессами — очереди, порядок `accepted`/`progress`/`completed` не гарантирован latency. В envelope progress класть `seq` (монотонный счётчик) + `stage`; потребитель упорядочивает по `seq`, поздний `accepted` после `completed` отбрасывает. Иначе bar «прыгает».
>
> **Ошибки.** Исключение в хендлере → `failed`-event (lifecycle) + error-`response` (reply). Порядок: сначала reply (закрывает `request()`), потом/вместе lifecycle `failed`. Зафиксировать в ADR.

> **Design decision (зафиксировать в ADR):** интерим-репорты ≠ финальный ответ по `type`. Это сохраняет семантику P0.5 «request() резолвится первым `response`».
>
> **Опция разбиения:** lifecycle-feedback (P4.4.4) — самая объёмная и наименее зрелая часть. Допустимо **вынести P4.4.4 в отдельный подплан** после P4.4.0, чтобы не блокировать основной рефактор P4.4.1–P4.4.3/5 (он самоценен и без lifecycle). Решение — по итогам recon.

## Принципы (наследуются от родителя)

1. **No big-bang / strangler** — прототип запускаем после каждой задачи; старый путь живёт адаптером до финала. qt-smoke (FPS ~21) после каждого шага.
2. **No crutches** — каждый шаг доводится до конца либо не начинается.
3. **Fewer links** — целевой путь короче (1 резолв вместо 2), не длиннее.
4. **Один вход / один диспетчер / тупые реестры** — CommandRegistry без диспетчеризации; observability/reply — конвейер, не менеджер.
5. **Reuse-first** — `MiddlewarePipeline` (`_recv_mw`), `Dispatcher`, `reply_to_request`/`correlation_id` (P0.5), `StatsManager` (`_record_metric`/`_record_timing`) уже есть; достраиваем шов «вокруг исполнения», не вводим параллельных сущностей.
6. **Investigation-first** — P4.4.0 обязателен первым.
7. **Сохранить request/response** — `reply_to_request` и резолв pending по correlation_id не ломать (вкл. state.*).

### ⚠ ПЕРЕСМОТР РАЗВИЛКИ (2026-06-05, после recon): выбран B2 (kind-router), не B1

Ранний ярлык «вариант B = один диспетчер» по факту **противоречил** разделу «Целевая архитектура (first-principles)» этого же плана (там нарисован kind-router). Recon-факт «все команды несут `type=command`» снял причину, по которой B уходил от kind-router (страх коллизии обходных хендлеров). **Решение владельца 2026-06-05: B2.**

| | B1 (один диспетчер) | **B2 (kind-router + CM-owns-commands) ✅** |
|---|---|---|
| Резолвов на приёме | 1 | 2 (kind по `type` ~5 ключей + имя в CM) |
| «Команды только в CommandManager» | ложь (ключи в message_dispatcher) | **правда** |
| Полый/лишний слой | да (CommandManager-фасад) | **нет — у каждого 1 ответственность** |
| 3 обходных хендлера | остаются прямыми | **растворяются** (process.command/register_update → команды CM; state.changed → event-sink) |
| `auto_register_ipc=False` хак | остаётся | **уходит** (одна таблица команд = в CM) |
| Имя `message_dispatcher` | врёт (свалка) | **→ `kind_router`, честное** |
| Миграция | меньше | больше (свернуть обходные, переименовать) |

«Меньше слоёв» в правильном прочтении = меньше дублирующих/полых компонентов с одной ответственностью у каждого, а не «-1 dict-lookup» (2-й lookup по таблице ~5 типов — наносекунды, к тому же это control-plane, кадры идут мимо через SHM). По этому критерию B2 строго чище.

### Черновик ADR-COMM-005 (финал — в P4.4.6)

> **ADR-COMM-005: Command Bus — kind-router по `type` + CommandManager владеет командами + inbound-seam**
> **Контекст:** приём команды сегодня проходит два инстанса `Dispatcher` (message_dispatcher + cm.dispatcher), оба резолвят по `command` (двойной резолв); `message_dispatcher` — свалка (команды-копии + state + 3 прямых хендлера); наблюдаемость/reply несквозные (timing только в handle_command + дубль в cm.dispatcher; reply в closure).
> **Решение (B2):** (1) `message_dispatcher` → **`kind_router`**: маршрутизация по `type` (`command`→`CommandManager.handle_command`; `event`→event/progress-sink; `response`→correlation [уже спец-кейс receive():518]; `data`→data-plane). (2) `CommandManager` **владеет** таблицей `имя→handler` (его `dispatcher` — единственный command-резолвер). Резолв по `command` — **ровно один**, в CM. (3) 3 обходных хендлера сворачиваются: `process.command`/`register_update` → обычные команды CM; `state.changed` (`type=event`) → event-sink. Хак `auto_register_ipc=False` удаляется (двойной регистрации больше нет). (4) Наблюдаемость (timing/metrics/log) и `reply_to_request` — **единый seam в `receive()`** вокруг kind-route-исполнения (context-manager/inline, НЕ новый pipeline), единообразно. (5) Опциональный lifecycle-feedback по `correlation_id`: интерим = `type="event"`+`seq`, финал = `type="response"`; реестр progress-callbacks по correlation_id (не блокирует, cleanup на финале).
> **Инварианты:** `_route_to_worker` (P2.2) — ПЕРЕД kind-router; «финал=response, прогресс=event»; request/response + state.* паритет.
> **Альтернативы:** B1 (один диспетчер, CM-фасад) — отвергнут: команды физически не в CM, полый слой, имя врёт. Вариант A (kind-router + fallback в message_dispatcher для обходных) — отвергнут: fallback не нужен, обходные просто становятся командами.
> **Последствия:** -1 резолв; `message_dispatcher` ужимается до ~5 kind-маршрутов; observability на 3 бывших обходных; уходит auto_register_ipc-хак; больше миграции (свернуть обходные).

# MODE=3 RETURN — возврат буквы на ленту (правка прошивки)

Парный контракт к Python-стороне (`Services/robot_comm/core/registers.py`,
`client.do_return`, `RobotDriver._execute_return`, sim `RobotSimCore._handle_return`).
**Адреса/шкалы обязаны 1:1 совпадать** — иначе координаты «уедут». Этот документ —
ТЗ для добавления режима в `cvt_universal_full.lua` (владелец флешит робот).

## Зачем

После укладки слова оператор жмёт кнопку на телефоне → робот возвращает ВСЕ выложенные
буквы обратно на конвейер по одной. ПК шлёт координату СЛОТА (откуда забрать), робот сам
исполняет линейную траекторию и хват. Существующие режимы (CVT/DRAW/MANUAL) так не умеют:
CVT — забор с ДВИЖУЩЕЙСЯ ленты по энкодеру; DRAW — перо без хвата; MANUAL — без Z и хвата.

## Карта регистров (зона 0x1350)

| Рег | Адрес | Доступ | Смысл |
|-----|-------|--------|-------|
| `REG_MODE` | 0x1109 | W | **= 3** для RETURN (0=CVT, 1=DRAW, 2=MANUAL) |
| `REG_RET_FLAG` | 0x1350 | RW | ПК пишет 1 = задание готово; робот сбрасывает в 0 при приёме |
| `REG_RET_X` | 0x1351 | RW | X слота (забор), 0.1 мм, s16 |
| `REG_RET_Y` | 0x1352 | RW | Y слота (забор), 0.1 мм, s16 |
| `REG_RET_Z` | 0x1353 | RW | Z слота (высота захвата), 0.1 мм, s16 |
| `REG_RET_BUSY` | 0x1354 | R | 1 = робот выполняет возврат, 0 = свободен |

Масштаб XY_SCALE=10 (как у job/place). Зона 0x1350..0x1354 — в свободной дыре 0x130C..0x13FF
(между CONFIG 0x130B и DRAW 0x1400), НИЖЕ буфера точек рисования 0x1420..0x154B.
⚠️ Раньше блок стоял на 0x1510 — это ВНУТРИ буфера рисования: рисунок ≥80 точек затирал
RET-регистры. Перенесён, чтобы режимы не конфликтовали по адресам.

## Handshake (как у рисования)

ПК (`do_return`) пишет `ret_x/ret_y/ret_z`, затем `ret_flag=1` (МАРКЕР — последним) и ждёт:
1. `ret_flag → 0` — приём задания;
2. `ret_busy → 1` — старт исполнения;
3. `ret_busy → 0` — возврат завершён.

Робот не получит следующую букву, пока `ret_busy` не упадёт → координаты не перезапишутся
на лету. Между буквами ПК шлёт задания по одному (очередь в драйвере).

## Последовательность движения (Motion, mode=3)

Смещения — **константы Lua** (по ТЗ владельца): `RET_LIFT_MM = 20`, `RET_PUSH_MM = 100`.
От координаты слота (sx, sy, sz):

```
при REG_RET_FLAG == 1:
  REG_RET_FLAG = 0            -- приём
  REG_RET_BUSY = 1; REG_FREE = 0
  sx = REG_RET_X/10; sy = REG_RET_Y/10; sz = REG_RET_Z/10
  MovL → (sx, sy, sz + RET_LIFT_MM)   -- подвод над слотом (перо/схват вверх)
  MovL → (sx, sy, sz)                 -- опуститься к диску
  DO(DO_GRIP, 1); DELAY(GRIP_S)       -- ЗАХВАТ (пневмо)
  MovL → (sx, sy, sz + RET_LIFT_MM)            -- подъём +20
  MovL → (sx + RET_PUSH_MM, sy, sz + RET_LIFT_MM)  -- сдвиг +100 по X (над лентой)
  MovL → (sx + RET_PUSH_MM, sy, sz)            -- опускание −20 к ленте
  DO(DO_GRIP, 0); DELAY(GRIP_S)       -- ОТПУСТИТЬ → диск падает на ленту
  MovP("GL_HOME")                     -- домой
  REG_RET_BUSY = 0; REG_FREE = 1      -- готово
```

Знак `RET_PUSH_MM` (направление к ленте) и оси — подобрать на железе (как у CVT). Линейные
ходы — `MovL` (только линейные, как просил владелец).

## Захват (DO)

`DO(DO_GRIP, 1/0)` — пневмо. В CVT-укладке (`run_job`) DO сейчас **закомментирован** —
для реальной укладки И возврата его нужно раскоммитить/завести с правильным номером DO.
Со стороны ПК хват НЕ управляется (по решению владельца): вся логика хвата — в Lua.

## Эталон поведения

Логику handshake точно воспроизводит `Services/robot_comm/server/sim_core.py`
(`_handle_return`) — на нём проходят e2e-тесты `do_return` без железа. Lua должен дать
тот же handshake (flag→0, busy 1→0).

## Проверка после флеша

1. Симулятор не нужен — на железе: `robot_set_mode {mode: return}` свободному роботу.
2. `robot_return_job {device_id: robot_main, x_mm, y_mm, z_mm}` (через device_hub) →
   робот должен забрать из (x,y,z) и сбросить на ленту по траектории выше.
3. В рецепте `hikvision_letter_robot`: выложить слово → кнопка телефона (signal_1) →
   все буквы вернулись → ввод нового слова.

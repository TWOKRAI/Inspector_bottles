# Git для одного разработчика: методичка

Краткое руководство по работе с Git в соло: что за что отвечает и какие команды вводить. Команды ниже даны для **PowerShell** или **Git Bash** в Windows; пути к проекту подставьте свои.

---

## 1. Зачем это нужно

- **История** — можно вернуться к старой версии файла или всего проекта.
- **Ветки** — изолировать задачу, не ломая рабочую линию `master`.
- **Резервная копия** — `push` на GitHub/GitLab хранит код на сервере.

---

## 2. Термины (простыми словами)

| Термин | Что это |
|--------|--------|
| **Репозиторий (repo)** | Папка проекта + скрытая папка `.git` с историей. |
| **Коммит (commit)** | Снимок проекта в момент времени + сообщение «что сделано». |
| **Ветка (branch)** | Именованная линия коммитов. Обычно **`master`** — стабильная основа. |
| **Удалённый репозиторий (remote)** | Копия на сервере, чаще всего `origin` = GitHub. |
| **Индекс (staging)** | «Корзина» перед коммитом: что попадёт в следующий коммит. |
| **Рабочая копия** | Текущие файлы на диске (могут отличаться от последнего коммита). |

---

## 3. Рекомендуемый процесс для одного человека

1. Держите в **`master`** только то, что **собирается и не ломает** основной сценарий.
2. Любая задача (фича, рефакторинг, крупный фикс) — **отдельная ветка** от актуального `master`.
3. Закончили — **слияние в `master`**, **push** на сервер, при желании удалите ветку.

Pull Request на GitHub **не обязателен** в соло; можно мержить локально.

## 4. Команды по порядку изучения

Ниже — команда, затем **зачем** и **что делает**.

### 4.1 Настройка (один раз на компьютер)

```bash
git config --global user.name "Ваше Имя"
```

- **Зачем:** имя в истории коммитов.
- **Делает:** сохраняет настройку для всех репозиториев на машине.

```bash
git config --global user.email "you@example.com"
```

- **Зачем:** тот же email, что на GitHub (для привязки коммитов к аккаунту).
- **Делает:** записывает email в конфиг Git.

```bash
git config --global init.defaultBranch master
```

- **Зачем:** новые репозитории по умолчанию создадут ветку `master` (если хотите именно её).
- **Делает:** меняет значение по умолчанию (у многих стоит `main`).

---

### 4.2 Клонирование (уже есть репозиторий на GitHub)

```bash
cd C:\Users\ВАШ_ПОЛЬЗОВАТЕЛЬ\Desktop
git clone https://github.com/ОРГ/РЕПО.git
cd РЕПО
```

- **Зачем:** скачать проект первый раз.
- **Делает:** создаёт папку `РЕПО`, копирует историю и ветки по умолчанию с сервера.

---

### 4.3 Где я и что изменилось

```bash
cd C:\Путь\к\Inspector_bottles
```

- **Зачем:** перейти в корень репозитория перед командами `git`.
- **Делает:** меняет текущую папку в терминале.

```bash
git status
```

- **Зачем:** понять, на какой ветке вы, что изменено, что уже в индексе.
- **Делает:** показывает краткий отчёт (modified / untracked / staged).

```bash
git branch
```

- **Зачем:** список локальных веток; звёздочка `*` — текущая.
- **Делает:** печатает имена веток.

```bash
git log --oneline -15
```

- **Зачем:** последние 15 коммитов одной строкой на коммит.
- **Делает:** показывает хеш и сообщение коммита.

```bash
git diff
```

- **Зачем:** посмотреть **нестажеенные** изменения в файлах.
- **Делает:** выводит отличия текущих файлов от последнего коммита.

```bash
git diff --staged
```

- **Зачем:** что уже добавлено в индекс и попадёт в коммит.
- **Делает:** diff между индексом и последним коммитом.

---

### 4.4 Сохранить работу в коммит

```bash
git add путь/к/файлу.py
```

- **Зачем:** включить конкретный файл в следующий коммит.
- **Делает:** кладёт версию файла в **индекс (staging)**.

```bash
git add .
```

- **Зачем:** добавить все изменения в текущей папке и ниже (осторожно: лишнее тоже попадёт).
- **Делает:** индексирует все отслеживаемые изменения и новые файлы в дереве.

```bash
git commit -m "Кратко: что сделано"
```

- **Зачем:** зафиксировать снимок с понятным сообщением.
- **Делает:** создаёт коммит из индекса; без `git add` ранее нечего коммитить.

Сообщение пишите **повелительным наклонением** или фактом: `fix: путь к modules в validate`, `docs: добавить GIT_WORKFLOW`.

---

### 4.5 Ветка под задачу

```bash
git checkout master
```

- **Зачем:** перейти на основную ветку перед обновлением или новой веткой.
- **Делает:** переключает рабочую копию на указанную ветку.

```bash
git pull origin master
```

- **Зачем:** подтянуть с сервера последние коммиты в `master`.
- **Делает:** скачивает и **сливает** изменения с `origin/master` в текущую ветку (если вы на `master`).

```bash
git checkout -b feature/kratkoe-opisanie
```

- **Зачем:** начать задачу в изолированной ветке.
- **Делает:** создаёт ветку `feature/kratkoe-opisanie` **от текущего коммита** и сразу переключается на неё.

Альтернатива (новый синтаксис Git 2.23+):

```bash
git switch master
git switch -c feature/kratkoe-opisanie
```

- **`git switch`** — только смена ветки (без путаницы с `checkout`).

---

### 4.6 Влить задачу в master

Когда закончили в `feature/...`:

```bash
git checkout master
git pull origin master
git merge feature/kratkoe-opisanie
```

- **Зачем:** перенести все коммиты фичи в `master`.
- **Делает:** `merge` создаёт коммит слияния (или fast-forward, если можно без лишнего коммита).

```bash
git branch -d feature/kratkoe-opisanie
```

- **Зачем:** удалить локальную ветку после успешного merge.
- **Делает:** удаляет ветку, если она уже влита.

```bash
git push origin master
```

- **Зачем:** отправить обновлённый `master` на GitHub.
- **Делает:** публикует локальные коммиты, которых ещё нет на `origin`.

Первый push новой ветки на сервер:

```bash
git push -u origin feature/kratkoe-opisanie
```

- **Зачем:** сохранить ветку на GitHub и запомнить связь.
- **Делает:** `-u` (upstream) — дальше достаточно `git push` / `git pull` в этой ветке.

---

### 4.7 Отмена и откат (осторожно)

```bash
git restore путь/к/файлу
```

- **Зачем:** выкинуть **нестажеенные** правки в файле и вернуть как в последнем коммите.
- **Делает:** перезаписывает файл из HEAD.

```bash
git restore --staged путь/к/файлу
```

- **Зачем:** убрать файл из индекса, но не трогать содержимое на диске.
- **Делает:** «unstage».

Старые имена тех же действий:

```bash
git checkout -- путь/к/файлу
git reset HEAD путь/к/файлу
```

---

### 4.8 Временно отложить изменения (stash)

```bash
git stash push -m "wip: перед переключением ветки"
```

- **Зачем:** быстро переключить ветку, не коммитя недоделанное.
- **Делает:** сохраняет изменения в стек и убирает их из рабочей копии.

```bash
git stash list
```

- **Зачем:** посмотреть отложенные пачки.

```bash
git stash pop
```

- **Зачем:** вернуть последнюю пачку и применить к файлам.

---

### 4.9 Связь с удалённым репозиторием

```bash
git remote -v
```

- **Зачем:** увидеть URL `fetch`/`push` (обычно `origin`).

```bash
git fetch origin
```

- **Зачем:** скачать новые ветки и коммиты с сервера **без слияния** в вашу ветку.
- **Делает:** обновляет `origin/*` локально.

## 5. Мини-памятка: типичный день

```text
git checkout master
git pull origin master
git checkout -b feature/moja-zadacha
# … правки …
git add .
git commit -m "feat: описание"
git checkout master
git merge feature/moja-zadacha
git push origin master
git branch -d feature/moja-zadacha
```

Из корня `Inspector_prototype` (или как у вас в README):

## 7. Частые ошибки новичка

| Ситуация | Что сделать |
|----------|-------------|
| Закоммитил не то сообщение (последний коммит ещё не push) | `git commit --amend -m "новое сообщение"` |
| Забыл файл в коммите | `git add файл` → `git commit --amend --no-edit` (если не пушили) |
| Конфликт при `merge` | Открыть файлы с маркерами `<<<<<<<`, выбрать код, затем `git add` → `git commit` |
| Нужно отменить последний коммит, сохранив файлы | `git reset --soft HEAD~1` |

## 9. Краткий глоссарий команд

| Команда | Назначение |
|---------|------------|
| `git clone URL` | Склонировать репозиторий |
| `git status` | Статус файлов и ветки |
| `git add` | Добавить в индекс |
| `git commit` | Создать коммит |
| `git log` | История коммитов |
| `git diff` | Отличия от последнего коммита |
| `git branch` | Список / создание веток |
| `git checkout` / `git switch` | Сменить ветку |
| `git merge` | Влить ветку в текущую |
| `git pull` | Скачать и слить с сервера |
| `git push` | Отправить на сервер |
| `git fetch` | Скачать без слияния |
| `git stash` | Временно спрятать изменения |
| `git restore` | Откатить файл / снять со staging |

---

*Файл можно дополнять под ваши сценарии (например, только `main` вместо `master` — замените команды соответственно).*
